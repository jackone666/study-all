# RAG 检索增强生成全解

---

## 一、RAG 核心概念与架构

### 1.1 RAG 是什么

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与文本生成相结合的技术范式。其核心思想是：在 LLM 生成答案之前，先从外部知识库中检索相关文档片段，将其作为上下文注入 Prompt，从而让模型基于真实资料作答，而非仅凭自身参数记忆。

> **一句话定位**：RAG 是大模型的外部知识插件，将事实与推理解耦。
>
> **本质认知**：用外部非参数化知识对冲大模型不稳定的参数化记忆，从根源剥离记忆负担。

### 1.2 RAG 解决什么问题

| 问题 | 说明 |
|------|------|
| **知识滞后** | LLM 训练数据有截止日期，无法获知最新信息 |
| **幻觉** | 模型靠参数记忆回答问题，事实准确性无保障 |
| **隐私合规** | 企业私有数据不能用于模型训练，但可用作检索语料 |
| **可溯源性** | RAG 可标注引用来源，每句话有据可查 |

### 1.3 RAG 四大核心组件

| 组件 | 职责 | 技术选型 |
|------|------|----------|
| 索引器（Indexer） | 将原始文档转化为可检索的语义向量表示 | 嵌入模型（BGE、Sentence-BERT）、分块策略 |
| 检索器（Retriever） | 从知识库中检索与查询最相关的文档片段 | 向量数据库（Chroma、Milvus、FAISS）、BM25 |
| 生成器（Generator） | 结合检索结果生成最终自然语言回答 | GPT-4、Claude、提示工程模板 |
| 后处理器（Post-processor） | 优化生成结果的可信度与可解释性 | 来源标注、事实核查、答案排序 |

### 1.4 四阶段工作逻辑与基础流程

**四阶段工作逻辑**

1. **查询输入阶段**：接收用户原始提问，进行查询改写（Query Rewrite）或指代还原。
2. **检索阶段**：将改写后的查询向量化，在向量库中进行相似度匹配（余弦相似度等），同时可结合 BM25 关键词检索进行多路召回。
3. **生成阶段**：将检索到的文档片段与原始查询拼接为上下文，输入大模型生成回答。
4. **后处理阶段**：来源标注（注明引用文档与章节）、事实核查（交叉验证检索结果与历史回答记录）、置信度过滤（低于阈值时提示"根据现有资料无法确认"）。

**RAG 基础流程：Indexing → Retrieval → Generation**

**离线阶段——数据入库（Indexing）**

1. **文档解析**：处理 PDF/Word/HTML 等格式，尤其处理双栏排版、复杂表格、扫描件
2. **语义切片（Chunking）**：按语义边界切分，保持每块信息完整
3. **向量化入库**：通过 Embedding 模型将文本转为向量，存入向量数据库

**在线阶段——检索生成（Retrieval + Generation）**

```
用户提问 → Query 改写 → 混合检索 → Rerank 重排序 → 上下文组装 → LLM 生成 → 输出（附引用）
```

**代码示例：LangChain 实现 RAG 基础流程**

```python
from langchain.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA

# 1. 加载文档
loader = TextLoader("data.txt")
docs = loader.load()

# 2. 语义切分（500 tokens + 100 overlap）
text_splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=100)
chunks = text_splitter.split_documents(docs)

# 3. 向量化入库
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(chunks, embeddings)

# 4. 构建检索问答链
qa = RetrievalQA.from_chain_type(llm=ChatOpenAI(), retriever=vectorstore.as_retriever())
answer = qa.invoke("你的问题")
```

### 1.5 RAG 三大核心组件（面试关键词版）

| 组件 | 角色 | 职责 | 面试关键词 |
|------|------|------|-----------|
| **Embedding 模型** | 系统的眼睛 | 将文本映射到向量空间，决定召回质量 | Embedding 决定 RAG 下限 |
| **向量数据库** | 系统的大脑书柜 | 存储向量 + 快速近似最近邻搜索（ANN） | HNSW 算法、元数据过滤 |
| **大语言模型（LLM）** | 系统的大脑 | 理解上下文、推理生成答案 | 大模型决定生成上限 |

> **工业界共识**：「找得准」往往比「答得美」难得多。Embedding 拉垮 → 召回失败 → 即使最强模型也无用。

### 1.6 向量数据库在 RAG 中的角色

向量数据库是将非结构化数据转化为高维语义向量的工业级流水线，绝非简单的"建库建表"。

**构建五步流程**

| 步骤 | 操作 | 关键要点 |
|------|------|----------|
| 数据接入与解析 | PDF/Word/图片等多格式文档的解析 | 版面分析、智能OCR、表格转Markdown、多模态大模型生成图片描述 |
| 语义分块（Chunking） | 将文档切分为语义完整的片段 | 语义切分替代固定字数切分；父子文档分块策略（子块检索、父块召回）；10%-20%重叠度；全局上下文注入 |
| 向量化（Embedding） | 将文本片段转化为稠密向量 | 通用嵌入模型 vs 领域微调（对比学习）；稠密向量+稀疏向量混合表示 |
| 索引构建 | 存储向量并建立高效索引 | HNSW算法（高效社交网络式搜索）；量化压缩（32位浮点压至8位，内存节省数倍）；MD5哈希校验实现Upset覆盖更新 |
| 混合检索架构 | 多路召回 + 重排序 | 前置过滤（权限/时间）；向量检索（语义）+ BM25（关键词）双路并进；交叉编码器重排序模型精选Top5 |

**工程核心原则**：不谋全局者不足谋一域。数据接入的解析质量、分块的语义完整性、索引的工程效率，每一环都决定最终检索精度。

## 二、文档处理与切分策略

### 2.1 Chunking 切分平衡术

切分大小面临两难困境：

| 切分策略 | 问题 | 后果 |
|---------|------|------|
| **切太小（如 50 tokens）** | 语义破碎 | 主语在上一切片，谓语在下一切片，模型搜不到完整信息 |
| **切太大（如 2000+ tokens）** | 大海捞针 | 答案淹没在大量废话中，向量特征被稀释，且浪费 token 成本 |
| **最佳实践** | 500-1000 tokens + 10%-20% 重叠 | 平衡检索精度和上下文完整性 |

> **面试追问**：如何根据文档类型调整 Chunk Size？底层原则是什么？
>
> **答**：Chunk Size（分块大小）指将文档切分为检索单元时的单块长度（以 token 或字符计）。调整策略的核心原则是**保持语义完整性**，即每个 chunk 必须是一个自包含的逻辑单元，避免在句子中间或主题边界处切断。具体建议如下：
> - **代码文档**：chunk size 可大至 1000-1500 tokens，目的是保留完整的函数定义、类或模块——如果函数体被切到两个 chunk 中，检索时只搜到一半代码，模型无法理解上下文。
> - **新闻文章**：建议中等大小（500-800 tokens），因为新闻段落结构清晰，每个段落通常表达一个完整观点，切分过大会混入无关信息。
> - **FAQ / 问答对**：可以很小（200-300 tokens），因为每条问答本身就是独立单元，切大反而引入噪声干扰匹配。
> - **技术手册 / 产品文档**：中等至大（600-1000 tokens），按章节标题切分，确保概念定义、参数说明和代码示例在同一个 chunk 中。
> - **法律合同**：中等（500-800 tokens），按条款切分，因为每个条款是独立的权利义务单元，跨条款的 chunk 会导致模型混淆法律关系。
>
> 此外还需注意：chunk 重叠（overlap）通常设为 10%-20%，保护切分边界不丢失边缘信息；检索性能与 chunk 大小成反比——chunk 越小检索灵敏度越高，但上下文越破碎，需通过父子文档策略（小块检索、大块生成）平衡。

### 2.2 语义切分 vs 固定大小切分

| 切分方式 | 做法 | 优点 | 缺点 |
|---------|------|------|------|
| **固定字数切分** | 按固定字数一刀切（如每 500 字） | 实现简单 | 句子被切断，语义丢失严重 |
| **语义切分** | 根据标题、段落边界，甚至用模型判断断点 | 语义完整，保留自然段落 | 实现复杂度较高 |

> **原则**：永远不要按字符数硬切，必须保证每块语义完整（不切半句话）。

**代码示例：RecursiveCharacterTextSplitter 不同策略对比**

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 策略一：小切块 + 大重叠（适合 FAQ/问答对，检索灵敏）
small_chunk = RecursiveCharacterTextSplitter(chunk_size=200, chunk_overlap=50)

# 策略二：中切块 + 中重叠（通用推荐，平衡精度和上下文）
medium_chunk = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=100)

# 策略三：大切块 + 小重叠（适合代码文档/论文全文，保持完整）
large_chunk = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=100)

# 中文文档专用：定制分隔符优先级（段落 > 句子 > 标点 > 字符）
chinese_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100,
    separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""]
)

chunks = chinese_splitter.split_text("你的中文文档内容")
print(f"切分后共 {len(chunks)} 个片段，长度分别为 {[len(c) for c in chunks]}")
```

### 2.3 父子文档策略（Parent-Child Indexing）

这是解决「检索精度」与「上下文完整」矛盾的终极方案：

| 角色 | 大小 | 用途 |
|------|------|------|
| **子 Chunk（Child）** | 128-256 tokens | 用于检索（灵敏度高，精准匹配） |
| **父 Chunk（Parent）** | 512-1024 tokens | 给 LLM 生成（上下文完整，不断章取义） |

- **检索时**：用小 chunk 做索引，稍微沾边就能搜到
- **生成时**：召出子 chunk 后，取出其所属的父 chunk 或周围段落一并交给 LLM
- **效果**：既瞄得准，又给得全

### 2.4 重叠窗口设计（Overlap）

- **问题**：文档中写"WIFI 密码是 123456"，如果一刀切在冒号处，上一块是"WIFI 密码是"，下一块是"123456"，用户搜"WIFI 密码"永远搜不到 123456
- **方案**：设置 **10%-20% 的重叠窗口**，保护边缘信息不被切断

### 2.5 PDF 解析与中间存储格式

| 解析方案 | 方法 | 成本 | 效果 |
|---------|------|------|------|
| **傻瓜方案（PyPDF2/PyPDF）** | 按行读取，不管排版 | 免费 | 差（双栏文档直接变成天书） |
| **经济型方案** | LayoutLM 等视觉模型先画框再提取 | 中等 | 好 |
| **土豪型方案** | PDF 转图片，扔给多模态模型（如 GPT-4o）识别 | 高 | 最好 |

**中间存储格式必须用 Markdown**：
- 大模型训练时看了大量 GitHub 代码和文档，天生理解 Markdown
- `#` 表示标题，`|` 表示表格，保留 Markdown 就是保留文档骨架
- 错误做法：存成纯文本 TXT 或 JSON

> **一句话速记**：切分保语义（500-1000t + 20% overlap），检索用小块，生成用大块，解析认准 Markdown。

## 三、Query 改写与优化

### 3.1 不做 Query 改写的三大坑

| 痛点 | 示例 | 问题本质 |
|------|------|---------|
| **表达不规范** | 用户说「那啥报错咋整」，文档写的是「Error500 解决方案」 | 口语 vs 书面语脱节 |
| **语义不匹配** | 用户说「头疼发烧流鼻涕」，文档存的是「感冒的病理特征」 | 现象描述 vs 原理描述，字面无重合 |
| **意图太复杂** | 「A 和 B 有什么区别」需要多步推理的问题 | 单次检索搞不定 |

### 3.2 为什么要改写 Query（补充视角）

| 维度 | 问题描述 | 改写目标 |
|------|----------|----------|
| **语言风格断层** | 用户口语化提问 vs 知识库文档书面语，向量空间中距离遥远 | 将口头表达转换为文档书写体，弥合语义空间距离 |
| **上下文信息缺失** | 用户提问残缺模糊（如"应该怎么处理"），缺失隐含背景 | 挖掘隐含背景信息，转化为完整具体的检索命令 |
| **多轮对话连贯性** | 代词/模糊指代（"那个""它"）导致检索断裂 | 指代还原，将所有代词替换为明确实体名称 |

### 3.3 五大核心改写策略

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| **Query Rewrite** | 大模型将口语翻译为带技术关键词的标准查询 | 口语化严重的用户输入 |
| **Multi-Query** | 将一个问题扩展为 3-5 个不同角度变体，并行搜索 | 问题太宽泛，怕漏搜 |
| **Decomposition** | 将复杂问题拆解为多个子任务分别检索后再汇总 | 对比类、多步推理问题 |
| **HyDE** | 先让大模型生成一个假设性答案，用假答案去检索 | 问题和文档字面完全不搭界 |
| **Step-Back** | 后退一步先问高层级原理，拿回大背景知识再推理 | 用户问得太细，库里没有 |

### 3.4 HyDE（假设文档嵌入）深度解析

- **原理**：不直接用问题检索，先让 LLM 生成一段假设性答案（编造内容）
- **为什么有效**：虽然生成内容可能是幻觉（事实错误），但用词风格与真实文档相似，将"问答匹配"转化为"文档匹配"。核心逻辑：**用相似的东西搜相似的东西**
- **核心价值**：补全了专业术语的语义关联，匹配度大幅提升
- **适用场景**：长尾冷门问题、用户问题太短特征不足

**代码示例：HyDE 假设文档嵌入（LangChain）**

```python
from langchain.chains import HypotheticalDocumentEmbedder, LLMChain
from langchain.llms import OpenAI
from langchain.embeddings import OpenAIEmbeddings
from langchain.prompts import PromptTemplate

# HyDE 核心：先用 LLM 生成假设性答案，再用假答案去检索
# 将"问答匹配"转化为"文档-文档匹配"，弥合口语与书面语差距
hyde = HypotheticalDocumentEmbedder.from_llm(
    llm=OpenAI(),
    base_embeddings=OpenAIEmbeddings(),
    prompt_key="web_search"  # 内置"web_search"提示模板
)

# 使用 HyDE 向量检索
from langchain.vectorstores import Chroma
vectorstore = Chroma(embedding_function=hyde)
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
docs = retriever.get_relevant_documents("头疼发烧流鼻涕怎么办")
# HyDE 先编造"感冒症状及处理"的假答案 → 用假答案搜到"感冒病理特征"文档
```

> **⚠️ 面试追问**：Multi-Query 和 Decomposition 的区别是什么？各自适用于什么场景？
>
> **答**：Multi-Query（多查询扩展）和 Decomposition（查询分解）是两种截然不同的 Query 改写策略。
>
> **Multi-Query（横向扩展）**：将同一个用户问题从不同角度扩展为 3-5 个变体，并行执行检索，所有结果取并集。例如用户问"iPhone15 怎么样"，Multi-Query 可扩展为："iPhone15 性能评测"、"iPhone15 电池续航"、"iPhone15 摄像头效果"——三个方向同时搜索，谁搜到算谁。它的核心价值是**覆盖率**，适合用户提问太宽泛、无法确定具体检索角度，或问题涉及多个子主题的场景。
>
> **Decomposition（纵向拆分）**：将复杂问题拆解为多个有依赖关系的子任务，按顺序串联执行，前一步的输出作为后一步的输入。例如用户问"对比 A 公司和 B 公司 2023 年的营收增长率"，Decomposition 会先拆为："子任务 1：查 A 公司 2023 年营收" → "子任务 2：查 B 公司 2023 年营收" → "子任务 3：计算增长率并对比"——每个子任务依赖前一个结果，最终需要逻辑组合。它的核心价值是**推理深度**，适合对比类、多跳推理或需要逐步分析的问题。
>
> **类比理解**：
> - Multi-Query 像派出多个侦察兵分头探索不同区域，各自带回情报并汇总
> - Decomposition 像组装一台机器——先做零件 A，再做零件 B，最后组装到一起
>
> **工程选择建议**：如果问题本身简单但表述模糊（如"讲讲那个东西"），用 Multi-Query 做语义扩展即可；如果问题涉及多步推理或因果链条（如"为什么 A 导致 B，而 B 又影响了 C"），必须用 Decomposition 逐步拆解。

### 3.5 多轮对话 Query 改写

- **问题**：多轮对话中用户使用代词（「他」「那个」「这个」），机器不知道指代什么
- **方法**：将对话历史中的关键信息脱水提取，把模糊代词翻译成完整的检索需求
- **示例**：用户先问"iPhone15 电池多大"，再问"他和 14 比谁更强" → 改写为"iPhone15 和 iPhone14 相比谁的性能更强"

### 3.6 意图识别 + 路由分发

不是所有问题都需要 Query 改写：

| 问题类型 | 处理方式 |
|---------|---------|
| 简单问题（打招呼、问候） | 直接跳过改写 |
| 一般问题 | 标准 Query Rewrite |
| 复杂问题（多步推理、对比） | Multi-Query + Decomposition |
| 模糊问题（术语不匹配） | HyDE + Rewrite |

### 3.7 工程实战三大难点

| 难点 | 问题描述 | 解决方案 |
|------|---------|---------|
| **延迟** | 每次改写需额外调用大模型，用户多等数秒 | 使用微调后的小模型专项负责改写；改写任务与初筛检索并行执行（流水线并行） |
| **查询漂移** | 模型改写偏离原意（「苹果怎么吃」→「苹果手机售后」） | 引入语义相似度阈值检查（如0.75），改写结果与原提问相似度低于阈值时弃用改写版本，宁信原话 |
| **成本** | 每条请求都改写，Token 费用高昂 | 加意图路由：简单问题直接跳过改写，仅对复杂模糊问题开启高级改写，成本可降60%以上 |

> **一句话速记**：检索不准不一定是库不好，很可能是大模型根本没听懂用户在问什么——改写五策记心间：Rewrite、Multi-Query、Decomposition、HyDE、Step-Back。

## 四、Embedding 优化

### 4.1 垂直领域术语壁垒（两大痛点）

| 痛点 | 说明 | 示例 |
|------|------|------|
| **术语壁垒** | 通用模型不知道专业术语映射 | 不知道"心梗" = "AMI" |
| **分布差异** | 高频共现术语的向量被拉近 | 法律文中"原告"和"被告"常成对出现，向量很近，但搜"原告责任"却命中"被告责任" |

### 4.2 三种 Embedding 优化策略对比

| 方案 | 数据要求 | 成本 | 效果 | 适用阶段 |
|------|---------|------|------|---------|
| **HyDE（假设性文档嵌入）** | 无需数据 | 低 | 立竿见影 | 冷启动 / 起步阶段 |
| **微调 Embedding 模型** | 万级高质量业务问答对 | 高 | 效果最好，形成护城河 | 有数据有预算 |
| **混合检索 + Rerank** | 无需训练数据 | 中 | 业界黄金标准 | 生产落地首选 |

### 4.3 微调 Embedding：Triplet Loss 原理

三个核心要素构成三元组：

```
Anchor（用户 query）
    ↓ Pull（拉近）
Positive（正确答案）
    ↓ Push（推远）
Hard Negative（长相相似但语义错误的文档）
```

- **Hard Negative** 正是通用模型最容易翻车的地方
- 通过 Triplet Loss 训练，让 Embedding 模型学会区分细微语义差异

### 4.4 跨模型向量比较与空间对齐

- 不同 Embedding 模型产出的向量**不可直接比较**（向量空间不同）
- 切换 Embedding 模型后必须**重新索引**已有文档
- 如果必须混用：使用跨模型对齐技术（如无监督对齐或有监督映射）

### 4.5 Embedding 决定 RAG 的下限

> **工业级共识**：Embedding 模型是 RAG 系统的根基。70% 的精力应投入到优化 Embedding、索引和重排序链路——解决最关键的「找得准」问题。大模型已足够聪明，只要把正确干净的信息摆好，它大多能答对。

| 角色 | 类比 | 关键影响 |
|------|------|---------|
| Embedding 模型 | 系统的眼睛 | 决定召回质量 = RAG 的根基 |
| 大模型 | 系统的大脑 | 决定最终答案的专业性和条理性 |

> **一句话速记**：垂直领域 Embedding 有三招——HyDE 冷启动、微调建护城河、混合检索 + Rerank 黄金组合打天下。

## 五、检索增强

### 5.1 向量检索 vs BM25 关键词检索

| 检索方式 | 优势 | 劣势 | 适用场景 |
|---------|------|------|---------|
| **向量检索** | 理解语义，能找近义词，容忍拼写错误 | 对数字/专有名词/产品型号不敏感 | 语义搜索、意图匹配 |
| **关键词检索（BM25）** | 精确匹配，绝不含糊 | 不懂语义，无法理解近义词 | 搜合同号、错误代码、法律条款 |

**向量数据库的致命弱点**：
- **数字盲**：向量空间中"错误代码 503"和"错误代码 404"的语义太像，向量距离几乎贴在一起
- **精确度差**：对专有名词、数字、编号等精确匹配场景表现糟糕

> **⚠️ 面试追问**：为什么有了 Embedding 检索还需要 BM25？两者各有什么不可替代的优势？
>
> **答**：Embedding 检索（稠密向量检索）和 BM25（关键词检索）是两种互补的检索范式，各自有对方无法替代的能力盲区。
>
> **BM25（Best Matching 25）** 是一种基于词频-逆文档频率（TF-IDF）改进的关键词检索算法。它统计查询词在文档中出现的频率（TF）和该词在整个文档集合中的稀有程度（IDF），计算文档与查询的相关性得分。其核心优势是**精确匹配**：如果文档中出现了查询中的精确词汇，BM25 可以零误差地将其召回。
>
> **Embedding 检索** 将文本映射到高维语义向量空间，可以理解"近义词"和"上下文含义"，但对精确的数字、型号、代号不敏感。
>
> **BM25 不可替代的场景**：
> 1. **数字和代码**：搜"错误码 503"或"合同编号 X2035"，BM25 能精确命中，而向量检索因为"503"和"404"在语义空间距离太近，可能混淆。
> 2. **专有名词**：搜"GPT-4 Turbo"时，BM25 精确匹配大小写和横线，向量检索可能因为"GPT4"和"GPT-4"的向量相似度高而召出不相关结果。
> 3. **倒排索引速度**：BM25 基于倒排索引，毫秒级响应，而向量检索的 ANN（近似最近邻搜索）在大规模数据集上延迟更高。
>
> **Embedding 不可替代的场景**：
> 1. **同义词替换**：搜"头疼发烧"可以匹配到"感冒症状"，BM25 做不到。
> 2. **多语言对齐**：中文问句可以匹配英文文档，BM25 无法跨语言。
>
> **业界黄金标准**是**混合检索（Hybrid Search）**：BM25 和向量检索双路并行，结果通过 RRF（Reciprocal Rank Fusion，倒数排名融合）算法加权合并。RRF 的基本原理是将每条结果在各自检索方式中的排名取倒数后求和，从而既照顾到 BM25 精准命中，也兼顾向量检索的语义泛化，达到最佳的综合召回率。

### 5.2 混合检索架构

```
用户 Query
    ├── [向量检索] → 语义匹配 → Top N
    ├── [BM25 关键词检索] → 精确匹配 → Top N
    └── [RRF 融合排序] → 合并结果 → Top 50~100
```

- **Step 1 海选**：BM25 + 向量检索双路并发，同时追求速度和召回率
- **RRF（Reciprocal Rank Fusion）**：将两路结果的排名融合，洗牌混合

**代码示例：BM25 + 向量混合检索（LangChain EnsembleRetriever）**

```python
from langchain.retrievers import BM25Retriever, EnsembleRetriever
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

# BM25 关键词检索（精确匹配专有名词/错误代码/型号）
bm25_retriever = BM25Retriever.from_texts([
    "错误代码 503：服务器暂时不可用",
    "错误代码 404：页面未找到",
])
bm25_retriever.k = 5

# 向量语义检索（理解近义词和上下文）
vectorstore = Chroma.from_texts([
    "服务器故障处理方案与排查步骤",
    "页面访问异常及网络连接排查",
], OpenAIEmbeddings())
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# RRF 融合排序：两路结果按排名加权融合
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.5, 0.5]  # BM25 和向量检索权重对半
)

docs = ensemble_retriever.get_relevant_documents("503 怎么解决")
# BM25 精确定位"503"错误码，向量检索定位"服务器故障"方案，互补
```

### 5.3 Rerank 重排序

Rerank 是 RAG 效果提升性价比最高、效果最立竿见影的手段。

| 阶段 | 角色 | 算法 | 目标 | 数量 |
|------|------|------|------|------|
| **粗筛（海选）** | 混合检索 | Bi-Encoder（分开编码） | 速度和召回率 | Top 50-100 |
| **精排（决赛）** | Rerank 重排序 | Cross-Encoder（拼接逐字计算） | 精准度 | Top 3-5 |

- **Embedding** 是 Bi-Encoder，query 和文档分开编码，速度快但精度有限
- **Rerank** 是 Cross-Encoder，query 和文档拼接在一起逐字计算相关性，精度高但速度慢
- **效果**：准确率提升 10-20 个百分点

**代码示例：Cohere Rerank / BGE Reranker 重排序**

```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CohereRerank
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

# Step 1: 初筛（海选）— 向量检索搜 Top 20（速度优先）
vectorstore = Chroma(embedding_function=OpenAIEmbeddings())
base_retriever = vectorstore.as_retriever(search_kwargs={"k": 20})

# Step 2: 精排（决赛）— Cross-Encoder 逐对打分
compressor = CohereRerank(
    model="rerank-english-v3.0",   # 也可用 BGE Reranker v2-m3
    top_n=5                         # 只保留最相关的 Top 5
)

# 组合：海选放宽标准 → Rerank 精确提纯
rerank_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)

docs = rerank_retriever.get_relevant_documents("你的问题")
print(f"Rerank 精选后保留 {len(docs)} 个高相关文档")
# 面试点：Embedding 是 Bi-Encoder 海选，Rerank 是 Cross-Encoder 决赛
```

> **⚠️ 面试追问**：Rerank 会增加延迟，值得吗？具体延迟增加多少？
>
> **答**：值，而且通常是 RAG 系统中**性价比最高的优化手段**，原因要从 Bi-Encoder 和 Cross-Encoder 的本质区别说起。
>
> **技术原理**：Embedding 检索使用的是 Bi-Encoder（双编码器）架构——Query 和 Document 分别独立编码为向量，然后计算余弦相似度。这种架构的优点是可以将文档向量提前计算好并建索引，检索时只需编码 Query 做一次向量搜索，速度快（毫秒级），但缺点是 Query 和 Document 没有交互，语义匹配精度有限。而 Rerank 使用的是 Cross-Encoder（交叉编码器）架构——将 Query 和 Document 拼接成一个文本对，一起送入模型计算相关性分数，模型可以逐字对比两边的内容，匹配精度远高于 Bi-Encoder。
>
> **延迟与收益分析**：Rerank 确实会增加几十到几百毫秒的延迟（取决于文档数量和模型大小），但这个代价换来的是海选阶段 Top 50-100 个文档被精细排序后只保留 Top 3-5 个高相关文档。具体收益如下：
> 1. **信噪比大幅提升**：砍掉 90% 以上的噪音文档，大模型输入的信息密度极高，生成质量显著提升。
> 2. **减少幻觉**：干净的上下文大大降低了模型被无关文档误导的概率，减少了后续迭代调试的人力成本。
> 3. **降低 Token 消耗**：输入给大模型的文本量大幅缩减，反而节省了推理成本和 LLM API 费用。
>
> **工程实践**：典型做法是海选阶段用较小的 Embedding 模型（如 BGE-small）快速检索 Top 50，精排阶段用 Cross-Encoder Rerank 模型（如 BGE-Reranker-v2-m3 或 Cohere Rerank）对 Top 50 逐对打分，最终取 Top 3-5 送入 LLM。全链路延迟增加约 100-300ms，但生成质量提升 10-20 个百分点，属于"用 300ms 换 20% 准确率"的划算买卖。
>
> **面试加分点**：还可以补充说明——Rerank 的最优插入位置是在混合检索（BM25 + 向量检索）之后、LLM 生成之前。同时可以提到 Rerank 模型的选型标准：要看 reranking latency 和 NDCG@10 指标的平衡，生产环境通常选择 110M-330M 参数级别的 Rerank 模型以达到性价比最优。

### 5.4 递归检索 / 多跳检索

- **递归检索（Recursive Retrieval）**：第一轮检索结果作为线索，第二轮基于线索继续检索，逐层深入
- **多跳检索（Multi-Hop Retrieval）**：将复杂问题拆解为多个子问题，每个子问题独立检索，最终汇总答案
- **适用场景**：需要跨文档推理的问题、需要多步信息链的问题

### 5.5 Graph RAG 与知识图谱增强检索

**核心思路**：将文档从「一堆碎肉」变为「一张网」。

| 维度 | 传统 RAG | Graph RAG |
|------|---------|-----------|
| 数据结构 | 文档碎片（chunks） | 实体 + 关系图谱 |
| 检索方式 | 向量相似度搜索 | 沿图谱链路遍历 + 向量检索 |
| 优势 | 快速找到相似文本 | 跨文档推理、顺藤摸瓜 |
| 代价 | 较低 | 更高（更贵、更慢） |
| 适用场景 | 事实性问答 | 需通读全文、理解关系的复杂问题 |

Graph RAG 的操作方式：
1. 提取实体（人名、地名、概念、事件）
2. 建立实体间关系（因果关系、从属关系、时序关系）
3. 形成知识图谱
4. 检索时沿图谱链路查找，实现结构化深度理解

**知识图谱构建的四层架构**

| 层级 | 操作 | 技术要点 |
|------|------|----------|
| 数据层 | 打通多元异构数据源 | 结构化数据（工商信息、司法判决）；半结构化数据（年报PDF、Word）；非结构化数据（新闻舆情爬取） |
| 知识层 | 抽取核心实体与关系 | 实体识别（Fin-BERT）；关系抽取（GNN 抽取控股/担保关系） |
| 图谱层 | 构建并存储知识图谱 | 图数据库选型（Neo4j 存储十亿级节点与边）；知识融合（VF2 图同构检测合并重复实体） |
| 应用层 | 风险监测与可视化 | 实时风险检测（Spark Streaming 处理舆情流）；交互式图谱可视化 |

**知识图谱在 RAG 中的作用**

1. **解决数据孤岛**：将分散在多系统的数据（工商、司法、税务、舆情）构建为统一语义网络。
2. **深度关联推理**：通过图游走（Graph Walk）技术，跨实体（企业、人物、事件）和关系（控股、担保、诉讼）拎出完整证据链，弥补纯向量检索无法处理的多跳关联。
3. **提升检索精度**：向量检索负责语义模糊匹配，图谱负责实体间硬逻辑连接，两者互补。

**行业案例：金融风控知识图谱**
- 数据源整合：工商总局企业信息 1.2 亿条 + 裁判文书网 5000 万案例 + 企业年报 + 新闻舆情
- 实体识别 F1 值 91.2%
- 图谱查询响应 < 200ms
- 人工 3 天的风险分析压缩至秒级响应
- 风险预警准确率 92%

### 5.6 文档问答召回率提升方案（全链路漏斗工程）

| 阶段 | 优化手段 | 效果 |
|------|----------|------|
| 数据接入层 | 智能语义切块替代固定字符数切块；滑动窗口保持上下文重叠 | 语义连续性提升 |
| 元数据增强 | 给文档打标签（时间、部门、业务模块），检索时先做元数据过滤 | 排除90%干扰信息 |
| 向量模型微调 | 用领域语料做对比学习微调 embedding 模型 | 专业术语理解能力提升 |
| Query 处理层 | 多路 query 改写（3-5个同义词版本并发检索）；HYDE 假设性回答 | 只要一路命中即胜出 |
| 查询拆解 | 复杂问题（如"某公司法人是谁"）拆解为多步检索（先找公司，再找法人） | 多跳推理能力增强 |
| 检索策略层 | 多路混合检索（向量 + BM25）；父子文档架构（小块检索，大块召回） | 既准又全 |
| Graph RAG | 引入知识图谱，通过实体和关系的图游走抓取关联信息 | 深层关联信息召回 |
| 重排与过滤层 | 交叉重排模型对50个文档重新打分，精选Top5 | 宽进严出，精确提纯 |
| 上下文压缩 | 剔除文档中的无关噪声，只留精华 | 缓解中间注意力丢失 |
| 闭环评估 | 盯着 Recall@K 等量化指标 | 避免蒙眼狂奔 |

**核心原则**：卓越的工程师从不迷信某个单一模型，而是用一套组合策略去对抗系统的不确定性。

### 5.7 语义鸿沟问题与解决方案

**语义鸿沟的本质**：用户查询意图与知识表示之间的语义断层，导致生成内容偏离需求。

**四大痛点**

| 痛点 | 描述 | 影响 |
|------|------|------|
| 数据孤岛 | 文档分散在 Word、PDF、数据库等不同系统，格式不一 | 跨源检索效率低 |
| 上下文断裂 | 检索到的文档片段分散，模型难以串联逻辑 | 回答缺乏连贯性 |
| 时效性矛盾 | 知识库未及时更新，推荐过时信息 | 决策风险高 |
| 跨模态适配 | 图片、表格、文本等异构数据语义不一致 | 多模态信息缺失 |

**四维解决方案**

| 方案 | 技术手段 | 效果指标 |
|------|----------|----------|
| 知识图谱融合 | 从数据源提取实体（疾病、药物、症状），构建关联图谱；图游走算法识别关键节点，推荐高置信度路径 | 跨源数据召回率提升37%，诊断准确率提升22% |
| 上下文感知检索 | 初始检索（BM25 召回 Top50）；语义重排序（计算查询与文档语义相似度 + 时间权重衰减）；多模态融合（影像特征与文本拼接） | 上下文完整度提升45% |
| 动态提示工程 | 结构化提示词设计 + 角色定义 + 步骤拆解 + 自洽校验（如"若推荐阿莫西林，检查患者过敏史"） | 误推荐率下降19%，指南通过率98% |
| 跨模态对齐 | CLIP 模型提取影像文本描述；影像特征与文本查询拼接生成复合 Embedding；优先返回同时匹配文本和影像的片段 | 影像相关病例诊断准确率提升31% |

**案例：医疗诊断 RAG 系统**——将多源异构数据（电子病历 HIS、医学文献 PubMed、药品本地库）通过知识图谱融合 + 动态检索 + 跨模态对齐，诊断准确率从78%提升至91%，日均处理病例数增加40%。

> **一句话速记**：向量检索找意思 + BM25 抠字眼 + Rerank 精选 + Graph RAG 顺藤摸瓜 + 语义鸿沟全维打通，五大检索手段层层递进。

## 六、高质量上下文构建

### 6.1 核心思路：三层过滤流水线

不要试图把大海塞给模型，要给模型一杯经过层层过滤的纯净水。整个流程像漏斗，一级一级过滤噪音。

| 层级 | 核心组件 | 作用 |
|------|---------|------|
| **第一层：数据准备** | 文档清洗 + 父子索引 + 混合存储 | 从源头保证数据质量 |
| **第二层：查询与检索** | Query Rewrite / HyDE + 混合检索 + Rerank | 核心降噪环节 |
| **第三层：生成优化** | Prompt 压缩 + U 型排列 | 最后一公里优化 |

### 6.2 上下文质量控制

- **以准换多**：不追求数量，通过重排序敢于丢弃垃圾，保留高信噪比的片段
- **结构化抗噪**：用父子索引，别让碎片化信息误导模型，保全程式化上下文
- **工程化降本**：在大模型出厂之前，先用廉价小模型（Embedding + Rerank）把脏活累活干完

### 6.3 内容缺失解决方案（四阶段全链路）

| 阶段 | 问题类型 | 核心方案 |
|------|---------|---------|
| **输入意图层** | 没搜对 | Query 重写 + 指代消解 + Multi-Query 多路并发扩展 |
| **数据与检索层** | 搜不到 | 混合检索 + Graph RAG + 语义分块 + 父子文档策略 |
| **上下文组装层** | 看漏了（中间迷失） | 强力重排序（Cross-Encoder）+ 上下文动态压缩 |
| **生成与兜底层** | 库真没有 | Self-RAG 自省 + 联网搜索补充 + 优雅降级（数据飞轮） |

**兜底策略详细说明**：

| 层级 | 方案 | 说明 |
|------|------|------|
| 自查 | Self-RAG 机制 | 模型生成前自省参考资料够不够，相关度低则触发下一步 |
| 补充 | 联网搜索 | 触发外部搜索获取实时信息做补充 |
| 兜底 | 优雅降级 | 没查到 A 但查到相近 B，询问用户是否需要；将缺失 Query 记入「数据飞轮」，推动业务人员补文档 |

### 6.4 上下文压缩

- **操作**：去掉"的得地"等无贡献的废话，只保留核心关键内容
- **收益**：节省 token 成本（部分技术能压缩一半 token）+ 回答更精准
- 可使用 `LangChain` / `LlamaIndex` 等工具的上下文压缩模块

### 6.5 U 型排列（Lost in the Middle 应对）

- **斯坦福发现**：大模型与人类似——开头记得住（Primacy）、结尾记得住（Recency）、**中间最易忘**
- **解决方案**：组装 Prompt 时，将 Rerank 分数最高（最关键）的片段放在**最前面**或**最后面**
- 好钢用在刀刃上，关键信息不要埋在中间

### 6.6 Long-Context vs RAG 成本对比

| 指标 | 长上下文（全量输入） | RAG |
|------|-------------------|-----|
| 预处理延迟 | ~15 秒 | 0.5 秒 |
| 单次成本 | ~5 元 | ~1 分钱 |
| 输入量 | 整本书 / 全知识库 | 最相关切片（约 1000 字） |
| 记忆效果 | 中间迷失 | 信息密度高 |

**三个反直觉的工程事实**

| 指标 | 纯长上下文方案 | RAG + 长上下文方案 |
|------|---------------|-------------------|
| 成本 | 每次重复计算全部文档的 Token 费用，输入量巨大且费用高昂 | 只检索必要信息，输入量减少99%，成本大幅降低 |
| 速度（首字延迟） | 超大文件输入需等待一分钟才能输出第一个字 | 只检索必要信息，等待时间控制在秒级 |
| 准确性（Lost in the Middle） | 只记得开头和结尾，中间内容遗忘或搞混 | 先筛选去粗取精，回答更准确 |

**场景选择决策**：

| 场景 | 选择 | 原因 |
|------|------|------|
| 需要理解全貌（小说剧情总结、三年财报分析） | **长上下文** | RAG 只能看到碎片，看不到森林 |
| 需要精准查找（员工手册某条款、故障代码查询） | **RAG** | 精准定位，快又准 |

**终极架构：Long RAG**

```
RAG（解决广度：初筛最相关的文档）→ 上下文构建（重排序+清洗，构建高质量 Prompt）→ 长窗口模型（解决深度：深度推理、跨段落逻辑理解）→ 最终回答
```

> **唯一的核心公式**：
> **RAG 解决了"知道得广"，Long Context 解决了"理解得深"。**
> **Long RAG = RAG（初筛高质量长内容）+ Long Context Model（精读深度推理）**

未来趋势既不是单纯的 RAG，也不是单纯的 Long Context，而是两者的融合。

> **⚠️ 面试追问**：模型窗口越来越大（1M+ tokens），RAG 最终会被淘汰吗？
>
> **答**：RAG 不会被淘汰，理由需要从经济账、物理限制和检索上限三个层面分析。
>
> **一、经济账（最关键）**：大模型的推理成本与输入 Token 数成正比。以 128K 窗口模型为例，每次推理输入 100 页文档的成本约是 RAG 方案（只输入最相关的 3-5 个文档片段）的 50-100 倍。即使模型窗口扩展到 1M+ tokens，把整个知识库（例如一个中型企业的 500 万页文档）全部塞进上下文的成本仍然不可接受——每次调用都要重复处理全部 tokens。RAG 通过检索只注入必要信息，输入量减少 99%，成本降低 2-3 个数量级。
>
> **二、物理限制**：无论模型窗口多大，物理存储上限永远大于窗口大小。一个企业的完整知识库可能包含数千万甚至数亿文档，存在窗口装不下和模型注意力机制无法有效处理超长文本的问题。即使模型窗口能装下一百万 tokens，Lost in the Middle（中间信息迷失）现象也会导致中间的文档被遗忘。RAG 预先缩小检索范围是解决这个注意力瓶颈的必要步骤。
>
> **三、检索的价值**：窗口再大，它也只是"被动等待输入"——没有一个筛选器告诉模型哪些信息重要、哪些是噪音。RAG 的检索链路（Embedding + BM25 + Rerank）本质上是一个主动的信息筛选系统，它保证喂给模型的都是高质量、高相关的片段。没有 RAG 的纯长上下文方案，模型就像在图书馆里不加筛选地一次性读完全部藏书，效率极低且容易混淆。
>
> **正确的方向——Long RAG 融合架构**：未来趋势既不是单纯的 RAG，也不是单纯的 Long Context，而是两者的融合。先用 RAG 做初筛（解决广度——从海量数据中找出最相关的文档），再用长上下文做精读（解决深度——对筛选出的长篇文档进行深度推理和跨段落逻辑理解）。两者是最佳拍档而非替代关系。参考公式：**RAG 解决"知道得广"，Long Context 解决"理解得深"，Long RAG = RAG + Long Context Model**。

> **一句话速记**：漏斗三层过滤——数据清洗保源头，Rerank 重排砍噪音，U 型排列保关键；Long RAG 才是未来。

## 七、RAG 幻觉解决方案

### 7.1 幻觉类型分类

RAG 的幻觉不是大模型自身的问题，而是整条链路的问题。面试官问"RAG 为什么会出现幻觉"，标准答案不是"模型爱胡说"，而是全链路治理方案。

**按产生环节分类**

| 幻觉类型 | 产生环节 | 表现 |
|---------|---------|------|
| **输入幻觉** | 用户提问 | 用户问题太模糊，检索方向偏离 |
| **检索幻觉** | 检索环节 | 搜到不相关或错误的文档 |
| **上下文幻觉** | 排序环节 | 噪音文档过多，模型被误导 |
| **生成幻觉** | 生成环节 | 模型预训练记忆与检索结果冲突，坚持旧知识或脑补 |
| **知识覆盖幻觉** | 数据准备 | 知识库里没有答案，模型为讨好用户编造 |

**按性质分类（补充视角）**

| 幻觉类型 | 定义 | 示例 |
|---------|------|------|
| 事实性幻觉 | 生成内容与客观事实不符 | 编造论文标题、作者、期刊，格式正规但内容全假 |
| 忠实性幻觉 | 偏离给定上下文 | 做摘要时加入原文没有的内容 |
| 推理幻觉 | 多步推理时跳步或逻辑断裂 | 算出一个漂亮但错误的答案 |

### 7.2 幻觉原因深度分析

| 原因 | 机理 | 本质 |
|------|------|------|
| 训练目标缺陷（根本原因） | 语言模型训练目标是预测下一个词的最大概率，学的是语言统计规律而非真实世界知识 | 追求"听起来合理"而非"真实准确" |
| 训练数据问题 | 互联网数据噪声多、知识截止日期限制、长尾领域数据覆盖不足 | 不知道的知识强行编造 |
| 元认知缺失 | 模型感知不到自己的知识边界 | 不会说"不知道"而是自信编答案 |
| RLHF 奖励偏差 | 人类标注偏好流畅自信的回答 | 模型学会言之凿凿地胡说 |
| 注意力机制局限（Lost in the Middle） | 超长文本中中间位置的关键信息注意力显著下降 | 关键证据被忽略 |
| 解码策略影响 | Temperature 过高导致随机性大；自回归解码错误累积 | 开头一错，步步错 |

### 7.3 12 大痛点及全链路解决方案

#### 第一阶段：检索阶段（痛点 1-3）

**痛点 1：缺少内容（答案不在知识库）**
- **现象**：用户提问的知识不在知识库中，模型为讨好用户编造答案
- **本质**：数据治理问题，非算法问题
- **方案**：
  - 源头数据清洗（Garbage In, Garbage Out）
  - Prompt 明确要求：若知识库没有相关信息，请回答「我不知道」，绝不编造
  - 建立数据飞轮，记录缺失 Query 推动业务补文档

**痛点 2：遗漏了排名靠前的文档**
- **现象**：答案在知识库中但检索没找到，因其排名靠后被截断
- **原因**：Chunk Size 或 Top K 参数没调好
- **方案**：超参数微调
  - 适当调大 Top K
  - 调整分块策略（如父子文档），优先保障召回率
  - 引入重排序：海选放宽标准 → 专业评委精细打分 → 只留前几名

**痛点 3：上下文限制（上下文塞入大量无关文档）**
- **现象**：为防遗漏调大 Top K 导致上下文塞入大量无关文档
- **后果**：烧钱 + 迷失中间效应——模型被噪音淹没找不到重点
- **方案**：引入**重排序**
  - 海选放宽标准（Top K 拉大）→ 重排序精细打分 → 只留最相关的 Top 5
  - 既保证召回率，又保证上下文纯净度

#### 第二阶段：生成阶段（痛点 4-5）

**痛点 4：提取失败（上下文明明有答案，但 AI 说找不到）**
- **原因**：文档太脏——HTML 标签、页眉页脚乱码、特殊符号等噪音掩盖关键信息
- **方案**：
  - 数据清洗 + 上下文压缩技术
  - 使用 `LangChain` / `LlamaIndex` 等工具挤干上下文水分
  - Prompt 中加少样本示例，教模型在乱码中找金子

**痛点 5：格式错误**
- **现象**：要求 AI 输出 JSON，结果多出解释文字或少括号，导致解析报错
- **方案**：强规则约束
  - LangChain 输出解析器 / Pydantic
  - OpenAI JSON Mode
  - 格式不对程序自动打回重写，直至格式正确

#### 第三阶段：生成质量（痛点 6-7）

**痛点 6：缺乏具体细节**
- **现象**：回答笼统——问电池续航具体多少毫安时，回答「续航很不错」
- **原因**：检索时仅匹配到概括性段落，丢失具体细节上下文
- **方案**：**从小到大检索策略（父子文档）**
  - 检索用句子粒度（精准匹配）
  - 喂模型时带上整个段落甚至整页（上下文完整）

**痛点 7：回答不完整**
- **现象**：对比类问题（对比 A 和 B 的优劣势）只回答了一半
- **原因**：系统只检索到其中一个实体的文档
- **方案**：**查询变换 + 分而治之**
  - 系统先思考，将复杂问题拆解为多个子问题
  - 分别检索，再汇总资料回答

#### 第四阶段：工程落地（痛点 8-12）

**痛点 8：数据摄入可扩展性差**
- **现象**：Demo 阶段 100 个文档运行飞快，上线后 100 万个文档索引构建需一周甚至内存溢出
- **方案**：**并行摄取 Pipeline**
  - 数据切片，利用多线程 / 分布式系统并行构建索引
  - LlamaIndex 的 `IngestionPipeline` 设置 `num_workers` 参数，处理速度最多提升 15 倍

**痛点 9：结构化数据问答难**
- **现象**：RAG 擅长处理文字，但面对 Excel 表格或数据库中的结构化数据效果差
- **方案**：
  - **Text-to-SQL**：用大模型将自然语言翻译成 SQL 语句直接查数据库
  - **Chain of Table Pack**：链式表格工具，通过思维链一步步处理表格数据
  - **混合自洽查询引擎**：文本推理 + Python 符号推理 + 多数投票聚合结果

**痛点 10：从复杂 PDF 中提取信息难**
- **现象**：企业 PDF 含双栏排版、图表、嵌套表格，简单转 text 工具拆得七零八落
- **方案**：
  - 多模态模型或专用解析器（如 LlamaParse）
  - 将 PDF 转为 HTML 保留格式，再用递归检索提取嵌入式表格
  - 最终将表格还原为 Markdown 格式供给 AI

**痛点 11：路由与稳定性**
- **现象**：系统同时包含向量库、SQL 数据库、联网搜索，不知道用哪个
- **方案**：
  - 训练分类器或利用 Agent 推理能力，自动判断走哪个数据源
  - **备用模型策略**：主模型故障（限流 / 宕机）时自动切换
    - Neutrino Router：智能调度十几种模型，主模型故障自动切换
    - OpenRouter：统一 API 一键接入各类模型，自动选低价稳定服务

**痛点 12：模型安全风险**
- **现象**：用户输入恶意提示或模型泄露敏感信息
- **方案**：**内容安全护栏**
  - 使用安全工具包（如 Llama Guard Moderator Pack）
  - 基于专用安全模型训练，先检查用户输入 → 再检查模型输出
  - 不安全内容直接拦截

### 7.4 七大方案全景与防幻觉全链路防线

**七大方案全景**

| 方案 | 机制 | 优点 | 缺点 |
|------|------|------|------|
| RAG 检索增强生成 | 从外部知识库检索内容注入上下文，让模型基于真实资料生成 | 可实时更新、结果可溯源、工程上最常用 | 依赖检索质量 |
| 思维链（Chain-of-Thought） | 让模型一步步写出推理过程 | 减少跳步错误，逻辑更严谨 | 推理速度变慢 |
| 不确定性感知训练 | 训练阶段加入大量"I don't know"样本 | 模型学会在知识边界处主动表示不确定 | 需要重新训练 |
| 自一致性投票（Self-Consistency） | 多次采样 + 多数投票 | 即插即用，无需改模型 | 推理成本成倍增加 |
| 事后检测与校验 | 生成后接一层校验，对接知识图谱或搜索引擎核查事实 | 兜底过滤网 | 增加延迟 |
| 改进 RLHF（事实性奖励） | 奖励模型不仅评价"喜欢不喜欢"，还要评价"准不准确" | 从训练根源改善，效果持久 | 迭代周期长 |
| 知识编辑 | 直接定位模型参数中存储错误知识的那一层，精准修改 | 无需重新训练 | 前沿方向，不够成熟 |

**RAG 防幻觉全链路防线**

1. **Query 改写（检索前）**：使用 HYDE 等策略将模糊提问转化为精准搜索关键词，从源头掐断"垃圾进垃圾出"。
2. **Graph RAG（检索中）**：知识图谱图游走，拎出一整条证据链，让模型有证有据，无机会胡说。
3. **重排序与上下文压缩（检索后）**：交叉编码器精选 Top5，prompt 压缩技术删除废话。信息浓度越高，幻觉概率越低。
4. **知识锚定（生成时）**：System Prompt 中写死"严格仅基于给定参考资料回答，资料未提及则大方承认不知道，严禁脑补"。将模型任务从"自由创作"强行扭转为"信息提取"。
5. **自我反思（生成后）**：引入 Self-RAG 反思机制，让模型生成后自己检查"我的回答有证据支持吗？有没有夹带私货？"

**三个加分观点**
1. 幻觉与涌现能力是一体两面——自由生成的能力本身就是幻觉的来源。
2. 模型更大不等于幻觉更少，知识密度和置信度校准才是关键。
3. 多模态同样有幻觉问题，视觉语言模型也会描述图里不存在的物体。

**终极认知**：大模型幻觉的本质是语言建模目标与真实世界知识之间的结构性偏差。这个矛盾在现有架构下无法根本消除，只能通过 RAG、RLHF、思维链、知识编辑等多种手段协同缓解。

### 7.5 幻觉检测与纠错

**检测方法**：

| 检测方法 | 操作 | 适用场景 |
|----------|------|----------|
| 事实核查（Factual Check） | 集成 ES 知识库，实时对比模型输出与权威数据库，未匹配则标记为疑似幻觉 | 金融条款、医疗知识等高精度要求场景 |
| 一致性验证（Consistency Check） | 使用多轮对话日志，检查前后回答逻辑一致性，矛盾率高于阈值则报警 | 客服助手、多轮问答 |
| 置信度评分（Confidence Score） | 对模型输出计算置信度，低于阈值时提示"根据现有资料无法确认" | 通用问答系统 |
| 专业术语校验器 | 基于领域知识图谱（疾病库、药品库、产品库）扫描转录文本中的实体和剂量，标记异常项 | 医疗转录、金融文档 |
| 自洽性推理（Self-Consistency） | 对同一问题多次采样，用多数投票选出最一致的答案 | 高可靠性要求的推理任务 |
| 忠实度评估（RAGAS） | 检查答案中的事实是否来源于检索文档（RAGAS 框架的 Faithfulness 指标） | RAG 系统通用评估 |
| 引用追溯 | 每句话标注来源文档和页码，用户可手动核查 | 所有场景 |

**纠错机制**：
- **Self-RAG**：模型在生成答案前自省参考资料够不够，相关度低则触发联网搜索或重新检索
- **Corrective RAG**：LLM 评估每个检索文档的相关性——完全相关则直接使用，模糊/错误则重写查询找准确信息
- **Self-Reflective RAG**：AI 自己反思自己的答案——判断是否检索 → 获取文档 → 生成初步答案 → 自我检查依据/逻辑/幻觉 → 不满意则重写

### 7.6 全链路幻觉治理五阶段速查表

| 阶段 | 核心问题 | 解决方案 |
|------|---------|---------|
| **一、数据准备** | Garbage In, Garbage Out | 语义切分 + 数据打标签（标注来源） |
| **二、用户提问** | 口语化问题与专业文档不匹配 | Query Rewriting + HyDE |
| **三、检索** | 向量检索对专有名词不敏感 | 混合检索（向量 + 关键词 BM25） |
| **四、排序** | 召回文档噪音太多 | Rerank 重排序（Cross-Encoder） |
| **五、生成** | 模型预训练记忆与检索结果冲突 | Prompt 约束 + 强制输出引用来源 |

### 7.7 面试四字口诀

> **净、改、排、控**

| 口诀 | 含义 | 阶段 |
|------|------|------|
| **净** | 源头数据要干净，语义切分做好 | 数据准备 |
| **改** | 用户问题要改写，让机器听得懂 | 提问阶段 |
| **排** | 混合检索 + 重排序，去粗取精 | 检索 + 排序 |
| **控** | Prompt 控制模型必须有理有据 | 生成阶段 |

> **一句话速记**：幻觉治理四字诀——净（数据干净）改（Query 改写）排（混合检索 + Rerank）控（Prompt 约束 + 溯源），12 大痛点逐一排查，Self-RAG 做最后守门员。解决幻觉不是一个 prompt 的事，而是全链路防线——从检索前的改写、检索中的重排、到生成时的约束，每一环都要扣死。

## 八、RAG 技术演进

### 8.1 四大演进阶段总览

```
Naive RAG → Advanced RAG → Modular RAG → Agentic RAG
（有没有）   （准不准）    （模块化）    （自主决策）
```

| 阶段 | 核心解决 | 核心手段 |
|------|---------|---------|
| **Naive RAG（朴素 RAG）** | 0→1 突破，解决**有没有**知识的问题 | 直接向量检索，简单线性流程 |
| **Advanced RAG（进阶 RAG）** | 工程精细化，解决检索**准不准**的问题 | 查询优化 + 混合检索 + 重排序 |
| **Modular RAG（模块化 RAG）** | 组件可插拔，灵活编排 | 各模块独立替换、组合编排 |
| **Agentic RAG（智能体 RAG）** | 自主规划，解决**深不深、全不全**的问题 | Agent 自主决策何时检索、如何检索、多轮迭代 |

### 8.2 Naive RAG（朴素 RAG）

- **流程**：用户提问 → 文档切块 → 向量检索 → 找最相似段落 → 扔给大模型回答
- **致命缺陷**：像只懂关键词匹配的实习生，检索错了 → 垃圾进垃圾出
- **唯一贡献**：实现了私有知识从无到有的突破

### 8.3 Advanced RAG（进阶 RAG）

- **核心逻辑**：打补丁——在检索前、中、后加入各种工程化手段提升精度
- **检索前优化**：Query 改写、HyDE
- **检索中优化**：混合检索（向量 + BM25）
- **检索后优化**：Rerank 重排序、上下文压缩
- **局限**：无法回答"总结 100 份合同中的所有风险点"这类通读全文型问题

### 8.4 Modular RAG（模块化 RAG）

- **核心思路**：将 RAG 流程拆分为可独立替换的模块（Rewrite、Search、Rerank、Generate 等）
- **优势**：灵活编排、组件可插拔、支持新增功能（如图谱、记忆模块）
- **与 Advanced RAG 的区别**：Advanced 是在线性流程上加补丁，Modular 是原生支持模块化组合

### 8.5 Agentic RAG（智能体 RAG）

Agentic RAG = RAG 的知识库外挂能力 + AI Agent 的自主决策能力。

**七大架构变体**：

| 变体 | 核心机制 | 适用场景 |
|------|---------|---------|
| **Agentic RAG 路由器** | Agent 判断查询类型，路由到不同处理引擎（SQL/向量/Web） | 多数据源系统 |
| **查询规划型（Query Planning）** | 查询规划器将复杂查询拆为多个子任务，分配不同引擎，最后合成 | 多步复杂查询 |
| **自适应 RAG（Adaptive RAG）** | 分类器判断查询复杂度 → 简单直接生成 / 普通检索 / 复杂多步 | 混合难度场景 |
| **纠错 RAG（Corrective RAG）** | LLM 评估检索文档相关性：相关→使用，不相关→重写再搜 | 检索质量不稳 |
| **自反射 RAG（Self-Reflective RAG）** | AI 自我反思答案，检查依据/逻辑/幻觉，不满意则重写 | 高精度要求场景 |
| **Speculative RAG** | 小型 Drafter 快速生成草稿 + 大型 Verifier 验证评分 | 速度 + 精度兼顾 |
| **Self-Route RAG** | LLM 判断上下文够不够 → 够则标准生成 / 不够则切长上下文模型 | 上下文长度决策 |

**代码示例：LangGraph 实现 Agentic RAG（简化）**

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

# 定义 Agent 状态（贯穿整个流程的数据结构）
class AgentState(TypedDict):
    question: str
    retrieved_docs: List[str]
    is_sufficient: bool  # 检索结果是否足够支撑回答
    answer: str

# 节点 1：混合检索（向量 + BM25）
def retrieve(state: AgentState) -> AgentState:
    docs = ensemble_retriever.get_relevant_documents(state["question"])
    state["retrieved_docs"] = [d.page_content for d in docs]
    return state

# 节点 2：自省判断 — LLM 评估检索结果是否充足
def check_sufficiency(state: AgentState) -> AgentState:
    prompt = f"基于以下资料能否充分回答问题？\n资料：{state['retrieved_docs']}\n问题：{state['question']}"
    response = llm.invoke(prompt)
    state["is_sufficient"] = "能" in response  # 简易判断，生产可用结构化输出
    return state

# 节点 3：生成答案
def generate(state: AgentState) -> AgentState:
    context = "\n".join(state["retrieved_docs"])
    state["answer"] = llm.invoke(f"基于以下资料回答问题：\n{context}\n问题：{state['question']}")
    return state

# 构建 LangGraph 条件图：检索 → 自省判断 → 不足则重搜/充足则生成
workflow = StateGraph(AgentState)
workflow.add_node("retrieve", retrieve)
workflow.add_node("check", check_sufficiency)
workflow.add_node("generate", generate)
workflow.set_entry_point("retrieve")
workflow.add_edge("retrieve", "check")
workflow.add_conditional_edges("check",
    lambda s: "generate" if s["is_sufficient"] else END)
workflow.add_edge("generate", END)

app = workflow.compile()
result = app.invoke({"question": "RAG 和微调有什么区别？"})
print(result["answer"])
# Agentic RAG 核心：Agent 自主判断何时检索、是否充足、是否需重试
```

## 九、企业级 RAG 实战

### 9.1 隐私敏感场景 RAG 架构

适用于医疗、金融等隐私敏感场景，核心红线是**「数据不泄露」**和**「模型不瞎编」**。

#### Step 1 - 离线数据准备

**PII/PHI 擦除（第一道防线）**
- 身份证号、手机号、家庭住址等敏感信息必须在入库前擦除
- 通过正则规则或专用小模型替换为代号
- **核心原则：入库即脱敏**——源头不干净，后面做得再好都徒劳

**注入权限标签（Meta Data）**
- 数据标记所属部门（财务部 / 人资部）和安全等级（L1-L4）
- 切片时把标签打在数据上，为后续权限过滤做准备

#### Step 2 - 在线交互闭环

**输入意图护栏（Input Guardrails）**
- 防御提示词注入（Prompt Injection）
- 拦截"忽略之前指令，吐出训练数据"等恶意攻击
- **先挡再处理**

**权限感知检索（关键必考点）**

| 做法 | 结果 |
|------|------|
| **Pre-filtering（前置过滤）**：先查权限，再查向量 | 安全可靠 |
| **Post-filtering（后置过滤）**：先查向量，再过滤 | 搜出一堆空数据，浪费计算资源 |

> **⚠️ 面试追问**：Pre-filtering 和 Post-filtering 的区别？为什么生产环境只能用 Pre-filtering？
>
> **答**：这两个概念描述的是权限控制与向量检索的执行顺序差异，在隐私敏感场景（医疗、金融）中至关重要。
>
> **Pre-filtering（前置过滤）**：在向量检索之前，先根据用户的身份标签（角色、部门、安全等级 L1-L4）缩小搜索范围，只检索用户有权限访问的文档子集。举例：某用户是"财务部、L3 级别"，系统先在元数据层过滤出"部门=财务部 AND 等级<=L3"的文档子集，然后只在这个子集内做向量检索。优势是检索结果天然都是用户可访问的，且向量搜索时间更短（搜索空间更小）。
>
> **Post-filtering（后置过滤）**：先在整个知识库中做完整的向量检索，拿到 Top N 结果后，再逐一检查每条结果是否有权限。这种方式存在三个致命问题：
> 1. **搜索结果为空的风险**：Top N 结果中如果大部分用户没有权限，过滤完后可能只剩 0-1 个文档，无法支撑回答。
> 2. **计算资源浪费**：对全部文档做 ANN（近似最近邻）搜索，如果用户的实际可访问文档只有全库的 10%，那 90% 的计算量都浪费在用户看不到的文档上。
> 3. **权限泄露风险**：向量搜索的结果排序本身就可能暗示用户一些无权限文档的存在——因为"没搜到"和"搜到但挡住了"给人的感受是不同的。
>
> **生产环境的强制要求**：必须用 Pre-filtering。在向量数据库（如 Pinecone、Milvus、Weaviate）中，可以通过**元数据过滤（Metadata Filtering）** 实现——给每个文档切片打上权限标签（部门、角色、安全等级），检索时在 ANN 搜索的 query 中附加过滤条件。例如在 Milvus 中可以用 `expr = "department == 'finance' && security_level <= 3"` 作为检索的过滤表达式，确保搜索引擎只在合法范围内执行向量匹配。

**私有化 LLM + 引用来源**
- 绝不能调用公网 API，数据出境即违规
- 每句话必须标注来源文档和页码——医疗场景中，不带引用的结果医生不敢信

#### Step 3 - 审计追溯

- 谁在什么时间问了什么、看了哪份文档
- 像黑匣子一样完整记录
- 应付监管机构合规审查的必备环节——提到此点即显专家段位

**面试满分总结——四个关键词**：

| 维度 | 原则 |
|------|------|
| 架构 | 私有化，物理隔离公网 |
| 数据 | 脱敏，PII 是红线 |
| 检索 | 控权，无权限过滤的 RAG 就是漏洞 |
| 输出 | 可溯源，每句话都要有出处 |

### 9.2 企业级混合 RAG 架构（结构化 + 非结构化）

解决 RAG 系统中结构化数据（SQL/表格）查询难题，构建智能路由 + Text-to-SQL + Agent 闭环。

#### 结构化查询三大坑

| 痛点 | 问题 | 后果 |
|------|------|------|
| **表太多** | 几千张表一次性全塞进 Prompt | 模型"CPU 过载"，幻觉满天飞 |
| **记不住业务逻辑** | 业务口径（如什么是"活跃用户"）模型不知道 | 答非所问 |
| **SQL 语法写不对** | 表名、字段名、函数名容易编错 | 执行报错 |

#### 智能路由：双轨制架构

```
用户提问
    |
智能路由（判断意图：查文档还是查数据库？）
    |--- 非结构化流 ---> 向量库（文档检索）
    |--- 结构化流 ----> Schema Linking + Few-shot + SQL生成 + 沙盒执行 ---> 数据报表
    |
聚合总结 Agent（将文档片段 + 数据报表融合为自然语言回答）
```

#### Schema RAG / Schema Linking：精简上下文

- **问题**：几百张表的字段全塞进 Prompt 太冗余
- **方案**：将表名、字段名和中文注释也做成向量存储。模型写 SQL 前，先去搜索与问题相关的表，只把最相关的 3-5 张表结构传入
- **效果**：精简上下文，减少幻觉，避免模型"失忆"

#### Few-shot 示例检索

- **问题**：模型不知道公司业务口径（如"活跃用户"的定义）
- **方案**：将过去写对的 SQL 例子存入库，模型写之前参考类似例子
- **效果**：准确率从 60% 提升到 90%+

#### Agent 闭环：自愈机制（面试高分点）

```
模型生成 SQL → 沙盒环境测试执行 → 报错则把错误信息和 SQL 一同返回给模型
→ 模型自我修正 → 直到跑通
```

- 解决 SQL 语法幻觉的终极武器

#### 安全防护

- **硬性规定一**：给大模型的数据库账号必须是严格的 **READ-ONLY（只读）**
- **硬性规定二**：SQL 执行前加拦截层（AST 解析或正则校验），发现有 DDL/DML 操作直接毙掉

> **面试核心总结**：解决结构化数据查询不是靠一个超级 Prompt，而是靠**Schema Linking + Few-shot 示例检索 + 沙盒自愈容错重试**的流程式智能体架构。

### 9.3 RAG 全链路优化（四大命门）

| 命门 | 问题 | 优化方案 |
|------|------|---------|
| **数据清洗与索引** | 语义破碎、PDF 解析乱码 | 语义切分 + 父子索引 + Markdown 中间格式 |
| **检索阶段** | 向量搜索精度不足 | 混合检索（向量 + BM25）+ Query 变换（HyDE） |
| **重排与压缩** | 检索噪声多、中间迷失 | Rerank 重排序 + 上下文压缩 + U 型排列 |
| **生成与评估** | 模型幻觉 | Prompt 约束 + 强制引用 + RAGAS 评估（忠实度 / 上下文精度 / 上下文召回率） |

**RAGAS 评估框架（核心指标）**：

| 指标 | 含义 |
|------|------|
| 忠实度（Faithfulness） | 答案是否根据文档写的，有无脑补 |
| 上下文精度（Context Precision） | 检索出来的文档是否与问题相关，相关文档排序是否靠前 |
| 上下文召回率（Context Recall） | 是否检索到包含答案的关键文档 |
| 答案相关性（Answer Relevancy） | 检索结果是否与问题相关 |

**代码示例：RAGAS 评估框架**

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,          # 忠实度：答案是否基于检索文档，有无脑补
    answer_relevancy,      # 答案相关性：检索结果是否与问题相关
    context_precision,     # 上下文精度：相关文档排序是否靠前
    context_recall,        # 上下文召回率：是否搜到所有关键文档
)
from datasets import Dataset

# 准备评估数据
data = {
    "question": ["什么是 RAG？", "Embedding 的作用是什么？"],
    "answer": ["RAG 是检索增强生成技术...", "Embedding 将文本转为向量..."],
    "contexts": [["RAG 将检索与生成相结合，利用外部知识..."], ["Embedding 模型负责将非结构化文本映射到向量空间..."]],
    "ground_truth": ["RAG 是检索增强生成技术，结合检索与生成...", "Embedding 将文本映射到高维向量空间..."]
}
dataset = Dataset.from_dict(data)

# 一键评估四大指标
result = evaluate(dataset=dataset, metrics=[
    faithfulness, answer_relevancy, context_precision, context_recall
])

print(result)
# 输出示例: {'faithfulness': 0.92, 'answer_relevancy': 0.88, ...}
```

### 9.4 LlamaIndex 在企业级 RAG 中的角色

LlamaIndex 是 RAG 系统的**数据操作系统和智能调度中枢**，而非简单的"数据连接器"。

**四层核心逻辑**

| 层级 | 角色 | 核心能力 |
|------|------|----------|
| 数据摄取与解析 | 数据翻译官 | 像素级解析复杂 PDF；对接企业数据库拉取结构化信息；Slack/Notion 等协作平台数据抓取；表格还原为带语义格式 |
| 结构化索引与存储 | 逻辑架构师 | 向量索引（模糊匹配）+ 树状索引（跨长文本层级摘要）+ 属性图谱（实体关系网络）；元数据提取器自动打标签 |
| 高级检索与工作流编排 | 智能指挥官 | 意图路由（判断是查具体数据还是全文总结）；从小到大检索（搜短特征片段 + 带周围背景）；事件驱动动态工作流：支持多步查询与自我修正 |
| 推理引擎与评估闭环 | 品控官 | 混合检索重排序；RAGAS 自动化评估（忠实度 Faithfulness、答案相关性 Answer Relevancy、上下文精度 Context Precision、上下文召回率 Context Recall） |

**代码示例：LlamaIndex 实现 RAG 全流程**

```python
# LlamaIndex 数据摄取 + 索引构建 + 检索生成
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# 1. 加载本地文档（支持 PDF/Word/TXT 等格式）
documents = SimpleDirectoryReader("data").load_data()

# 2. 自动切分 + Embedding + 构建向量索引
index = VectorStoreIndex.from_documents(documents)

# 3. 转换为查询引擎（内部完成检索 + 生成）
query_engine = index.as_query_engine()
response = query_engine.query("你的问题")
print(response)

# 进阶用法：自定义检索 + 后处理
from llama_index.core.retrievers import VectorIndexRetriever
from llama_index.core.postprocessor import SimilarityPostprocessor

retriever = VectorIndexRetriever(index=index, similarity_top_k=10)
nodes = retriever.retrieve("你的问题")

# 后处理：过滤低分文档，保留高信噪比片段
processor = SimilarityPostprocessor(similarity_cutoff=0.7)
filtered_nodes = processor.postprocess_nodes(nodes)
```

**面试总结三句话**
1. **数据大管家**：智能解析 + 多维度索引，解决复杂文档读不懂。
2. **逻辑架构师**：属性图谱 + 元数据提取，让检索具备深度推理能力。
3. **高级指挥官**：事件驱动 Workflow，把 RAG 从死板线性链路变成会思考、会自愈的代理系统。

> **一句话速记**：企业级 RAG 三座大山——隐私安全（PII 擦除 + Pre-filtering + 私有化）、混合数据（意图路由 + Schema RAG + Agent 纠错）、全链路优化（海选粗筛 + 决赛精排 + RAGAS 评估）。RAG 的上限不在于模型有多大，而在于数据流洗得有多干净，检索链路玩得有多精细。

### 关键技术对比

| 对比项 | 向量检索 | BM25 关键词检索 |

| 方式 | 按 500 字硬切 | 按标题、段落、语义边界切分 |
| 问题 | 句子被截断，语义丢失 | 无 |
| 推荐 | 不推荐 | 推荐 |

| 对比项 | RAG | 纯 Long Context |
|--------|-----|-----------------|
| 成本 | 低（输入量减少99%） | 高 |

### 面试四字诀

> **净、改、排、控**——幻觉治理全链路

| 口诀 | 含义 | 对应阶段 |
|------|------|---------|
| **净** | 源头数据要干净，语义切分做好 | 数据准备 |
| **改** | 用户问题要改写，让机器听得懂 | 提问阶段 |
| **排** | 混合检索 + 重排序，去粗取精 | 检索 + 排序 |
| **控** | Prompt 控制模型必须有理有据 | 生成阶段 |

### 面试答题框架

**通用框架（四层递进）**

1. **高度认知**：讲全局——改写解决语言风格不匹配、信息缺陷、对话连贯性三大痛点。
2. **技术深度**：列举具体策略——HYDE 反向生成法、任务拆解法、历史融合法。
3. **工程意识**：主动提延迟、漂移、成本三大工程困局及解决方案。
4. **闭环思维**：补充评估体系——召回率提升百分比、平均延迟、成本消耗等量化指标。

**RAG 通用满分回答框架**

1. **先说痛点**：表达不规范、语义不匹配、意图太复杂
2. **再说方案**：从数据准备 → 提问改写 → 混合检索 → 重排序 → 生成控制的完整链路
3. **最后谈权衡**：召回率 vs 延迟、成本 vs 效果、精度 vs 召回，永远有取舍
4. **升维收尾**：数据飞轮、A/B Testing、Agentic RAG 是未来方向

### 高频面试追问清单

| 追问 | 核心答案 |
|------|---------|
| RAG 为什么会出现幻觉？ | 不是模型爱胡说，而是全链路问题——数据脏、检索偏、排序噪、生成失控 |
| Multi-Query 和 Decomposition 的区别？ | Multi-Query 横向扩展取并集，Decomposition 纵向拆分子任务需逻辑组合 |
| 为什么有了向量检索还需要 BM25？ | 向量对数字/专有名词不敏感，BM25 负责精确匹配，两者互补 |
| Rerank 和 Embedding 的区别？ | Embedding（Bi-Encoder）海选，Rerank（Cross-Encoder）决赛 |
| Pre-filtering 和 Post-filtering？ | Pre-filtering 先查权限再搜，安全可靠；Post-filtering 先搜再删，浪费低效 |
| 长上下文会杀死 RAG 吗？ | 不会。物理存储 > 窗口上限，成本和延迟无法回避。RAG（广度）+ Long Context（深度）= Long RAG |
| Embedding 和大模型谁更重要？ | Embedding 决定下限（找得准），大模型决定上限（答得美）。70% 精力应花在检索链路上 |
| Agentic RAG 和传统 RAG 的区别？ | 传统 RAG 被动执行检索 + 生成，Agentic RAG 自主判断何时检索、如何检索、多轮迭代修正 |
| RAG 性能如何评估？ | RAGAS 指标：忠实度（Faithfulness）、答案相关性（Answer Relevancy）、上下文精度（Context Precision）、上下文召回率（Context Recall）；Recall@K |
| 数据更新怎么办？ | MD5 哈希校验 + Upset 覆盖更新；增量索引 |
| 如何处理表格/图片？ | 布局分析、表格转 Markdown、多模态大模型生成图片描述 |
| 多轮对话怎么做？ | 历史融合法改写；指代还原；多轮上下文管理 |

### 面试金句

> "RAG 的本质是用外部真实的非参数化知识去对冲大模型不稳定的参数化记忆，从根源剥离记忆负担。"

> "解决幻觉不是一个 prompt 的事，而是全链路防线——从检索前的改写、检索中的重排、到生成时的约束，每一环都要扣死。"

> "卓越的工程师从不迷信某个单一模型，而是用一套组合策略去对抗系统的不确定性。"

> "RAG 的上限不在于模型有多大，而在于你的数据流洗得有多干净，检索链路玩得有多精细。"

> "幻觉与涌现能力是一体两面——自由生成的能力本身就是幻觉的来源。"

> "系统越大，越不可能用一个超级模型解决所有问题。RAG 的本质是一种架构范式——将知识存储与推理能力解耦，让每个模块做自己最擅长的事。"
