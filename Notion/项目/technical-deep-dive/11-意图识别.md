## 12. 查询意图识别与路由

> 本项目的意图识别系统是一个 **规则优先 + LLM 兜底的两级架构**：第一级 "意图分类"（Intent Router）将自然语言查询映射为预定义意图标签；第二级 "检索路由"（Retrieval Router）基于意图 + 查询特征选择最优检索策略。两级之间通过 `query_analysis` 数据结构传递分类结果。

---

### 12.1 两级架构总览

```
用户 Query
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    第一级：意图分类 (Intent Router)                     │
│                                                                     │
│  输入: 自然语言查询                                                    │
│  输出: intent ∈ {policy_question, technical_question,                │
│                  troubleshooting, ticket_query, general_question}     │
│                                                                     │
│  策略: 正则强规则(<0.1ms) → 关键词词典(<1ms) → LLM 兜底(~200ms)                                        │
│  延迟: Pattern <0.1ms | Keyword <1ms | LLM ~200ms                   │
│  文件: modules/intent_router.py                                      │
└─────────────────────────────────────────────────────────────────────┘
    │
    │ query_analysis = {intent, matched_keywords, keywords, entities, classify_method}
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│                   第二级：检索路由 (Retrieval Router)                    │
│                                                                     │
│  输入: query + query_analysis                                       │
│  输出: RetrievalPlan {mode, weights, top_k, enabled_retrievers, ...} │
│                                                                     │
│  策略: 9 条纯规则（正则匹配 + 布尔逻辑），零 LLM 调用                      │
│  延迟: <1ms                                                          │
│  文件: rag/retrieval_router.py                                      │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
  进入 GraphRAGOrchestrator 执行检索
```

**核心设计哲学：** "规则优先保速度，LLM 兜底保覆盖，路由纯规则保确定性"。
- 意图分类：正则强规则和关键词词典覆盖 80%+ 常见查询（<1ms），仅低置信度时才调 LLM 兜底
- 检索路由只需要根据结构化特征（intent + entities + 正则模式）做策略选择 → 规则引擎更快更稳定

### 12.2 第一级：意图分类（Intent Router）

**文件：** `modules/intent_router.py`

**5 种预定义意图：**

| 意图 | 中文含义 | 典型查询 | 缓存特点 |
|------|---------|---------|---------|
| `policy_question` | 制度/政策查询 | "数据分类标准是什么？" | 文档可能定期更新 |
| `technical_question` | 技术问题 | "如何使用 API 网关？" | API 文档相对稳定 |
| `troubleshooting` | 故障排查 | "AUTH_401 错误怎么解决？" | 错误码答案可缓存 |
| `ticket_query` | 工单查询 | "TKT-001 处理进度？" | 动态数据，不缓存 |
| `general_question` | 通用/非知识库 | "你好" / "今天天气？" | 礼貌回复 |

#### 12.2.0 第 0 步：正则强规则（最高优先级，v2 新增）

```python
INTENT_PATTERNS = [
    # HTTP 错误码：4xx/5xx
    ("troubleshooting", "HTTP错误码", re.compile(r"(?:^|[\s,，。；;])(?:5\d{2}|4\d{2})\b")),
    # 错误代码：AUTH_401 / ERR_500 / ERROR_404 等
    ("troubleshooting", "错误代码", re.compile(r"\b(?:AUTH|ERR|ERROR|EXC|FAIL)[-_]\d+\b", re.IGNORECASE)),
    # 系统故障短语
    ("troubleshooting", "系统故障短语", re.compile(r"(?:挂了|宕机|崩溃|卡死|无法访问|访问不了|连不上)")),
    # 工单号模式：TKT-001 / WO-1234 / INC-12345 等
    ("ticket_query", "工单号模式", re.compile(r"\b(?:TKT|TICKET|WO|SR|REQ|INC|CHG)[-_]\d{2,6}\b", re.IGNORECASE)),
]
```

**为什么放在最前面：** 这些是**结构化、零歧义的模式**——一旦命中就是绝对确定的意图（HTTP 404 不可能是制度问题，TKT-001 不可能是技术问询）。提前判定可以：① 跳过词典打分（<0.1ms），② 避免词典误命中（如 "AUTH" 在 troubleshooting 词典里，但 "如何配置 AUTH" 应判 technical）。

**优先级顺序：** `troubleshooting > ticket_query`。一个 query 命中多条只取第一条。

**调参故事（v2 修复）：** v1 工单号正则 `r"\b[A-Z]{2,4}[-_]\d{2,6}\b"` 会误吞 `AUTH_401`（被判为工单），生产 case "用户反馈 AUTH_401 错误" 路由到了 ticket 工具。修复方案：把工单号正则收紧为**白名单常见前缀** `TKT|TICKET|WO|SR|REQ|INC|CHG`，错误码则用独立 pattern `AUTH|ERR|ERROR|EXC|FAIL` 前缀，两者互不干扰。

**命中时的返回结构：**
```python
{
    "intent": "troubleshooting",
    "matched_keywords": ["pattern:HTTP错误码"],   # 带 pattern: 前缀标识规则来源
    "entities": [...],
    "method": "pattern",                          # 区别于 "keyword" / "llm"
}
```

```python
def _classify_keyword(query: str) -> tuple[str, list[str], str]:
    """Score each intent by keyword hits.
    Returns: (intent, matched_keywords, confidence) where confidence is "high" | "low"
    """
    ql = query.lower()
    scores: dict[str, int] = {}
    matched: dict[str, list[str]] = {}

    for intent, keywords in INTENT_KEYWORDS.items():
        hits = [kw for kw in keywords if kw.lower() in ql]
        if hits:
            scores[intent] = len(hits)
            matched[intent] = hits

    if not scores:
        return "general_question", [], "low"

    ranked = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    best_intent, best_score = ranked[0]

    if len(ranked) == 1:
        if best_score >= MIN_KEYWORD_HITS_FOR_CONFIDENCE:  # = 2
            return best_intent, matched[best_intent], "high"
        return best_intent, matched[best_intent], "low"

    second_score = ranked[1][1]
    if best_score >= MIN_KEYWORD_HITS_FOR_CONFIDENCE and (best_score / second_score) >= CONFIDENCE_RATIO:  # CONFIDENCE_RATIO=2.0
        return best_intent, matched[best_intent], "high"

    return best_intent, matched[best_intent], "low"
```

**规则引擎的设计细节：**
- **评分制 + 置信度判定**：统计每个意图的关键词命中数，取最高分；同时根据 **命中数阈值（≥2）** 和 **best/second 比值（≥2.0）** 判定 `high` / `low` 置信度
- **置信度驱动 LLM 调用**：只有 `low` 置信度（如多意图打成平手、单意图仅命中 1 词）或 `general_question` 时才向上层透传 → 触发 LLM 兜底
- **完整命中列表传回**：返回 `matched_keywords` 列表（不是单字符串），下游 RetrievalRouter 可直接使用
- **大小写不敏感**：`query.lower()` + `kw.lower()` 统一处理中英文混合
- **关键词的中英文双覆盖**：`"工单"` 和 `"ticket"` 都作为 `ticket_query` 触发器
- **短查询拦截**：在 workflow 层（`classify_intent_node`），若 intent 为 `general_question` 且 query 长度 < 5，标记为 `unknown` → 触发 human_fallback

#### 12.2.2 第 2 步：LLM 兜底（仅在关键词低置信度时调用）

```python
def _try_llm(query: str) -> str | None:
    from enterprise_hybrid_rag.llm.provider_factory import get_llm_provider
    try:
        provider = get_llm_provider()
    except Exception:
        return None
    if provider.provider_name == "mock":
        return None  # mock provider 跳过 LLM 路径

    import asyncio
    prompt = (
        "你是一个企业级意图分类器。请将用户问题分类为以下意图之一：\n"
        "policy_question, technical_question, troubleshooting, ticket_query, general_question\n\n"
        f"用户问题: {query}\n\n请只返回意图标签。"
    )

    async def _call():
        return await provider.generate(prompt, temperature=0.0, max_tokens=32)

    try:
        try:
            loop = asyncio.get_running_loop()
        except RuntimeError:
            loop = None
        if loop is None:
            resp = asyncio.run(_call())
        else:
            # 异步环境（如 FastAPI）：丢到线程池跑同步事件循环，不阻塞主 loop
            import concurrent.futures
            with concurrent.futures.ThreadPoolExecutor(max_workers=1) as pool:
                future = pool.submit(asyncio.run, _call())
                resp = future.result(timeout=5.0)
        if resp.success and resp.content.strip().lower() in VALID_INTENTS:
            return resp.content.strip().lower()
    except Exception:
        pass
    return None
```

**关键设计：**
- `temperature=0.0`：保证分类结果确定性，同一 query 永远返回同一 intent
- `max_tokens=32`：仅需返回标签名称，极端受限减少延迟
- 返回值校验：LLM 输出必须在 `VALID_INTENTS` 集合中，且 `resp.success` 为 True，否则返回 `None`
- **Mock Provider 感知**：`provider_name == "mock"` 时直接跳过，零外部依赖即可完整调试
- **async 事件循环兼容（2026-06-09 修复）**：早期版本里用 `loop.is_running()` 判定就直接 return None，导致 FastAPI 这种永远 is_running=True 的环境里 LLM 路径完全失效。修复后改为：检测到事件循环已在运行时，把同步包装丢到 ThreadPoolExecutor 跑，5 秒超时，既不阻塞主 loop 也能真正调到 LLM

#### 12.2.3 三步策略的切换逻辑（规则优先，LLM 兜底）

```
analyze_intent(query):
    ├─ Step 0: _match_patterns(query)         # 正则强规则
    │   └─ 命中 → return {method="pattern", intent, matched=["pattern:规则名"]}
    │
    ├─ Step 1: _classify_keyword(query)       # 关键词词典 + 置信度评分
    │   └─ confidence=="high" 且 intent != general_question
    │      → return {method="keyword", intent, matched_keywords}
    │
    ├─ Step 2: _try_llm(query)                # 仅低置信度/ general_question 才调
    │   ├─ Mock provider / 异常 / 超时 → llm_intent=None
    │   └─ 返回 valid intent → llm_intent=<intent>
    │
    └─ Step 3: Resolve
        ├─ llm_intent 有效且非 general → 用 LLM 结果，method="llm"
        └─ 否则用关键词结果，method="keyword"
```

**三点保证：**
1. **降级安全**：LLM 路径的**任何异常**（超时、网络错误、返回非法标签、event loop 冲突）都被 `_try_llm` 内部 `except Exception` 吞掉返回 `None`，最终落回关键词结果，绝不抛异常
2. **开发友好**：Mock Provider 下直接跳过 LLM，规则路径始终可用，零外部依赖即可完整调试
3. **可对比性**：返回字典里的 `method` 字段（`"pattern" / "keyword" / "llm"`）可直接落日志做质量监控

### 12.3 第二级：检索路由（Retrieval Router）

**文件：** `rag/retrieval_router.py`

**核心原则：零 LLM 调用，纯规则引擎，单次路由 < 1ms。**

**输入数据结构 `query_analysis`（由 `classify_intent_node` 构建）：**
```python
{
    "intent": "troubleshooting",         # 意图标签
    "matched_keywords": ["错误", "解决"], # Intent Router 词典命中的关键词
    "keywords": ["错误", "解决", "API"],  # matched_keywords + 简单 token 去重合并
    "entities": ["AUTH_401", "SDK"],     # 实体抽取结果
    "classify_method": "keyword",        # "pattern" | "keyword" | "llm"
}
```

**输出数据结构 `RetrievalPlan`：**
```python
@dataclass
class RetrievalPlan:
    mode: str                       # parallel | graph_first | hybrid_only
    enabled_retrievers: list[str]   # ["keyword", "vector", "graph"]
    top_k: dict[str, int]           # {"keyword": 5, "vector": 5, "graph": 30}
    weights: dict[str, float]       # {"keyword": 0.3, "vector": 0.5, "graph": 0.2}
    graph_depth: int                # 图谱遍历深度 (1 or 2)
    need_query_expansion: bool      # 是否需要查询扩展
    need_second_stage_retrieval: bool  # 是否需要二阶段检索
    fallback_to_hybrid: bool        # 失败时是否降级到 hybrid_only
    reason: str                     # 人类可读的路由理由
```

#### 12.3.1 九条路由规则（优先级从高到低）

**规则执行流程：**
```
route(query, query_analysis):
    ├─ Rule 1: ENABLE_GRAPH_RAG=false? ───────────→ hybrid_only ❌
    ├─ Rule 2: Graph DB unavailable? ──────────────→ hybrid_only ❌
    ├─ 特征提取: _analyze_query(query, qa)
    ├─ Rule 3: _is_relational()? ──────────────────→ graph_first 🔗
    ├─ Rule 4: _is_exact()? ───────────────────────→ parallel (kw-heavy) 🎯
    ├─ Rule 5: _is_semantic()? ────────────────────→ parallel (vec-heavy) 🧠
    └─ Rule 6: 默认 ───────────────────────────────→ parallel (balanced) ⚖️
```

| 规则 | 触发条件 | 路由mode | 权重分配 | 典型场景 |
|------|---------|---------|---------|---------|
| **Rule 1** | `ENABLE_GRAPH_RAG=false` | `hybrid_only` | kw:0.4 vec:0.6 graph:0 | 全局关闭图谱 |
| **Rule 2** | Neo4j 不可用 / 初始化失败 | `hybrid_only` | kw:0.4 vec:0.6 graph:0 | DB 宕机 |
| **Rule 3** | 命中关系关键词 OR 多实体+询问关联 OR intent=relational | `graph_first` | kw:0.2 vec:0.3 graph:0.5 | "A 和 B 有什么关系？" |
| **Rule 4** | 错误码/API名/类名/函数名/配置项 OR intent=exact | `parallel` | kw:0.7 vec:0.3 graph:0 | "AUTH_401 错误" |
| **Rule 5** | 语义关键词（为什么/怎么/如何...）OR intent=semantic | `parallel` | kw:0.2 vec:0.6 graph:0.2 | "为什么白屏？" |
| **Rule 6** | 以上全不命中 | `parallel` | kw:0.3 vec:0.5 graph:0.2 | 默认均衡 |

#### 12.3.3 三大分类器的检测逻辑

**`_is_relational` — 关系推理检测：**
```python
def _is_relational(query, features, qa):
    # 3a: 命中关系关键词
    if features["has_relation_keywords"]:
        return True

    # 3b: 上游意图标签属于 relational 集合
    if features["intent_is_relational"]:
        return True

    # 3c: 多个实体 + 询问实体间联系
    if features["has_multi_entities"] and any(
        keyword in query
        for keyword in ["关系", "联系", "关联", "有关"]
    ):
        return True

    return False
```

**关系关键词正则（Rule 3 核心）：**
```python
_RELATIONAL_KEYWORDS = re.compile(
    r"关系|依赖|调用链|影响.*哪些|导致|上下游|"
    r"链路|生命周期|路径|关联|有关|涉及|联系|"
    r"相连|连接|触发|回调|监听|订阅|通知|"
    r"依赖关系|调用关系|什么关系|之间.*关系"
)
```

**`_is_exact` — 精确匹配检测：**
```python
def _is_exact(query, features, qa):
    # 错误码必然精确匹配
    if features["has_error_code"]:        return True
    # API名 且 非语义问题 → 精确
    if features["has_api_name"] and not features["has_semantic_keywords"]:
        return True
    # 类名/函数名/配置名 → 精确
    if features["has_class_name"]:        return True
    if features["has_function_name"]:     return True
    if features["has_config_name"]:       return True
    # 意图为 exact
    if features["intent_is_exact"]:       return True
    return False
```

**精确匹配正则集：**

| 模式 | 正则 | 示例 |
|------|------|------|
| 错误码 | `\b\d{4,10}\b\|\bERR[A-Z_]+\b\|\bAUTH_\w+\b` | `AUTH_401`, `9568321` |
| API名 | `@ohos\.\w+\|@\w+\.\w+\|\b\w+API\b` | `@ohos.app.ability` |
| 类名 | `\b[A-Z][a-z]+Ability\b\|\bclass\s+\w+` | `UIAbility` |
| 函数名 | `\bon\w+Create\b\|\bon\w+Destroy\b` | `onCreate()` |
| 配置项 | `\b[A-Z_]{3,30}\b\|module\.json` | `DATABASE_URL`, `module.json` |

**`_is_semantic` — 语义理解检测：**
```python
def _is_semantic(query, features, qa):
    # 语义关键词 且 不是错误码查询
    if features["has_semantic_keywords"] and not features["has_error_code"]:
        return True

    # 意图为 semantic/explanation
    if features["intent_is_semantic"]:
        return True

    # 长查询 + 无精确标记 + 含疑问词 → 语义倾向
    if (features["query_length"] > 20
        and not features["has_error_code"]
        and not features["has_api_name"]
        and any(kw in query for kw in ["什么是", "如何", "为什么"])):
        return True

    return False
```

**语义关键词正则（Rule 5 核心）：**
```python
_SEMANTIC_KEYWORDS = re.compile(
    r"为什么|怎么解决|可能原因|如何优化|区别|排查|怎么办|"
    r"如何|怎么|怎样|什么是|含义|原因|可能|建议|推荐|最佳实践|"
    r"有哪些|介绍|说明|解释"
)
```

#### 12.3.4 多实体检测

```python
_MULTI_ENTITY_PATTERN = re.compile(
    r"\b[A-Z][a-zA-Z]+\b.*(?:和|与|、|,|and).*\b[A-Z][a-zA-Z]+\b"
)
```

检测模式：**两个 CamelCase 实体名被中文/英文连接词分隔**。
例如：`"UIAbility 和 WindowStage 有什么关系？"` 被检测为多实体 → 触发关系路由。

### 12.4 意图 → 检索权重映射（2026-06-07 新增）

> **核心增强：** 将 Intent Router 的 5 种用户意图直接映射为三路检索（keyword/vector/graph）的权重分配，让意图分类结果充分驱动检索策略。

#### 12.4.1 设计动机

增强前，RetrievalRouter 的意图映射存在盲区——Intent Router 产出的 5 种意图中，只有 `technical_question` 被 `_INTENT_SEMANTIC` 匹配，其余 4 个（`policy_question`、`troubleshooting`、`ticket_query`、`general_question`）全部掉到正则兜底：

```
增强前:
  policy_question   ──→ 无匹配 ──→ 靠正则或默认 rule 6
  technical_question──→ _INTENT_SEMANTIC ✅
  troubleshooting   ──→ 无匹配 ──→ 靠正则或默认 rule 6
  ticket_query      ──→ 无匹配 ──→ 靠正则或默认 rule 6
  general_question  ──→ 无匹配 ──→ 靠正则或默认 rule 6

增强后:
  所有 5 种意图 ──→ 直接映射到专属权重 profile ✅
                    ──→ 与规则权重 70/30 融合
```

#### 12.4.2 意图权重映射表

```python
_INTENT_WEIGHT_PROFILES: dict[str, dict[str, float]] = {
    "policy_question":     {"keyword": 0.45, "vector": 0.45, "graph": 0.10},
    "technical_question":  {"keyword": 0.20, "vector": 0.55, "graph": 0.25},
    "troubleshooting":     {"keyword": 0.50, "vector": 0.30, "graph": 0.20},
    "ticket_query":        {"keyword": 0.80, "vector": 0.20, "graph": 0.00},
    "general_question":    {"keyword": 0.30, "vector": 0.50, "graph": 0.20},
}
```

**每种意图的权重设计理由：**

| 意图 | 主导检索器 | 权重点 | 设计逻辑 |
|------|-----------|--------|---------|
| `policy_question` | kw ≈ vec | kw:0.45 vec:0.45 | 制度名需精确匹配（关键词），内容语义需向量理解，制度间少有实体关系（graph 低） |
| `technical_question` | vec 主导 | vec:0.55 | API 用法是语义匹配密集场景，"如何使用"类问题向量效果最好；graph 保留因为 API 间有调用关系链 |
| `troubleshooting` | kw 主导 | kw:0.50 | 错误码必须精确匹配；故障描述可语义辅助；graph 保留因为错误码→原因→解决之间有实体关系 |
| `ticket_query` | kw 绝对主导 | kw:0.80 | 工单号是纯精确匹配场景；graph=0 因为工单号通常不在知识图谱中 |
| `general_question` | vec 均衡 | vec:0.50 | 无法预判用户关注点，向量语义理解覆盖面最广 |

#### 12.4.3 权重融合策略：70/30 规则

意图权重与规则权重的融合采用 **70% 规则 + 30% 意图** 策略：

```
最终权重 = 规则基础权重 × 0.7 + 意图 profile 权重 × 0.3
```

**为什么不直接用 100% 意图权重？**
- 规则权重捕获了查询的**结构特征**（如检测到错误码→关键词必须高），这是意图分类无法替代的
- 70/30 比例确保结构特征占主导，同时意图特征有实质影响
- 可通过修改融合比例在 `_make_*_parallel` 方法中调整

**效果验证 — 同一查询，不同意图 → 不同权重：**

```
Query: "API 调用返回 401 错误怎么办？"

意图=troubleshooting   → kw:0.64 vec:0.30 graph:0.06  (关键词主导)
意图=technical_question→ kw:0.55 vec:0.38 graph:0.07  (关键词降低，向量升高)
意图=policy_question   → kw:0.62 vec:0.34 graph:0.03  (关键词高，graph 低)
意图=general_question  → kw:0.58 vec:0.36 graph:0.06  (均衡)
```

```python
# → {"keyword": 0.50, "vector": 0.30, "graph": 0.20}
```

### 12.5 两级协作：规则优先意图分类 + 纯规则检索路由

这是整个意图识别系统最精妙的设计——**"分层处理"而非"二选一"：**

```
Intent Router (Pattern → Keyword → LLM)     Retrieval Router (纯规则)
─────────────────────────────────────       ────────────────────────
                 intent ────────────────────→ consumed by
                 entities ──────────────────→ _is_relational()
                 matched_keywords ──────────→ _is_exact()
                 keywords ──────────────────→ _is_semantic()
```

**为什么不让 Retrieval Router 也调用 LLM？**

| 维度 | LLM 路由 | 规则路由（本项目选择） |
|------|---------|---------------------|
| **延迟** | 200ms+ (网络 RTT + 生成) | <1ms (O(1) 正则 + 条件判断) |
| **确定性** | temperature > 0 产生随机性 | 同一 query 永远同一 plan |
| **可测试性** | 需 Eval 数据集 + LLM-as-Judge | 14 个纯函数单元测试（`tests/test_graph_router.py`） |
| **可调试性** | 需查看 LLM 日志追溯决策 | `reason` 字段直接写明决策路径 |
| **适用场景** | 仅作为关键词/正则覆盖不到的兜底（如本项目的意图分类） | 结构化特征 → 策略选择（如检索路由） |

**Intent Router 已经把 LLM 的不确定性吃在自己这一层（且仅在关键词低置信度时调），Retrieval Router 不需要再次面对自然语言的模糊性——** Intent Router 的输出（intent + matched_keywords + entities）已经将"自然语言的模糊性"转化为"结构化特征的确定性"，检索路由只需要用规则匹配这些特征即可。

### 12.6 意图识别在工作流中的位置

```
LangGraph 工作流：
  load_memory → check_permission → classify_intent ◀── 意图分类在这里
                                      │
                    ┌─────────────────┼─────────────────┐
                    ▼                 ▼                  ▼
            retrieve_knowledge    call_tools      human_fallback
                    │                 │
                    ▼                 ▼
              [RetrievalRouter    [Tool Orchestrator
               在 GraphRAG-       根据 intent 选择
               Orchestrator       工具集]
               内部调用]
```

**工作流节点 `classify_intent_node` 的核心逻辑：**
```python
async def classify_intent_node(state: PipelineState) -> dict[str, Any]:
    query = state.get("query", "")
    analysis = analyze_intent(query)          # ← 调用 Intent Router（全量分析）
    intent = analysis["intent"]

    # 短于 5 字符的 general_question → unknown → human_fallback
    if intent == "general_question" and len(query.strip()) < 5:
        intent = "unknown"

    # 确定路由方向和复杂度
    complexity = "moderate" if intent in ("troubleshooting", "policy_question") else "simple"
    route = "tools" if intent in _TOOL_INTENTS else "rag"
    # _TOOL_INTENTS = {"troubleshooting", "ticket_query"}

    # 构建给下游用的 query_analysis（matched_keywords + 简单 token 合并去重）
    matched_kw = analysis.get("matched_keywords", [])
    entities = analysis.get("entities", [])
    import re
    simple_tokens = [k for k in re.findall(r'[一-鿿\w]+', query) if len(k) >= 2][:10]

    query_analysis = {
        "intent": intent,
        "matched_keywords": matched_kw,
        "keywords": list({*matched_kw, *simple_tokens}),  # deduped merge
        "entities": entities,
        "classify_method": analysis.get("method", "keyword"),
    }

    if intent == "unknown":
        fb = _recovery.evaluate_failure(dict(state), fallback_type="unknown_intent")
        return {"intent": intent, "complexity": complexity, "route": route,
                "query_analysis": query_analysis, **fb}

    return {"intent": intent, "complexity": complexity, "route": route,
            "query_analysis": query_analysis}
```

**路由决策三向分流：**
- `route = "tools"` → `troubleshooting` 或 `ticket_query` → 先执行工具调用
- `route = "rag"` → `policy_question` / `technical_question` / `general_question` → 直接检索
- `intent = "unknown"` → 触发 human_fallback

### 12.7 工具选择中的意图驱动

```python
_TOOL_TRIGGERS: list[tuple[list[str], str]] = [
```

#### 12.8.1 路由决策可追溯

#### 12.8.2 意图分类质量监控

建议监控指标：
- `method_distribution`：pattern / keyword / llm 三种分类方法的占比（keyword 应占 80%+）
- `llm_fallback_rate`：LLM 兜底占比（过高说明词典覆盖不足）
- `unknown_intent_rate`：触发 unknown → human_fallback 的比率
- `intent_distribution`：各意图请求量分布

#### 12.8.3 调试技巧

**问题1：意图分类不准怎么办？**
→ 先看 `classify_method` 字段（pattern / keyword / llm）判断命中了哪一层
→ 如果是 keyword 层误判：调整 `INTENT_KEYWORDS` 增加/减少特定领域关键词
→ 如果是 keyword 低置信度且 LLM 也分错：考虑收集 100-200 条标注数据做 LLM 分类 fine-tuning
→ 如果是 pattern 误判：检查 `INTENT_PATTERNS` 正则是否过宽

**问题2：检索路由选错了 mode 怎么办？**
→ 路由不是"对/错"二元，而是"最优/次优"
→ Rule 6 的默认 parallel 模式仍然会三路检索，只是权重不是最优
→ 通过 trace 中的 `reason` 字段快速定位触发规则

##### Q1: 意图分类为什么要用规则优先 + LLM 兜底？[S]

**S — 情境：** 在 Enterprise Hybrid RAG 系统中，用户查询首先需要通过 Intent Router（`modules/intent_router.py`）分类为 5 种预定义意图之一（policy_question / technical_question / troubleshooting / ticket_query / general_question），以决定后续走 RAG 检索、工具调用还是人工兜底路径。意图分类的准确性直接影响整个系统的响应质量。

**T — 任务：** 设计一个规则优先、LLM 兜底的意图分类方案，使得 80%+ 的常见查询走快速路径（<1ms），低置信度或模糊查询才调 LLM。核心约束：① Mock Provider 模式下无真实 LLM 也能运行；② 分类延迟主流路径需 <1ms；③ FastAPI 异步环境中 LLM 调用不能阻塞事件循环。

**A — 行动：**
- **Step 0 正则强规则**：`_match_patterns()` 用 4 条正则（HTTP 错误码、AUTH_ERR 错误代码、系统故障短语、工单号模式）做最高优先级判定，<0.1ms，命中即返回 `method="pattern"`
- **Step 1 关键词词典**：`_classify_keyword()` 使用 100+ 词条（v2 扩展后）进行命中计数评分 + 置信度判定（阈值 ≥2 命中 + best/second ≥2.0），高置信度直接返回 `method="keyword"`，<1ms
- **Step 2 LLM 兜底**：`_try_llm()` 仅在关键词低置信度或 `general_question` 时调用，`temperature=0.0, max_tokens=32`，返回值必须在 `VALID_INTENTS` 中校验；异步环境用 ThreadPoolExecutor 跑，5 秒超时
- **返回结构**：`analyze_intent()` 返回 `{intent, matched_keywords, entities, method}`，method 字段记录 `"pattern"` / `"keyword"` / `"llm"`，便于后续质量对比

**R — 结果：** 
- Pattern 路径延迟 <0.1ms，Keyword 路径 <1ms，LLM 兜底 ~200ms
- v2 词典扩展后 80%+ 查询走快速路径，LLM 仅兜底模糊 case
- Mock 模式下 LLM 自动跳过，系统零外部依赖即可完整运行
- FastAPI 异步环境通过 ThreadPoolExecutor 调度 LLM，主 loop 不阻塞
- **项目评分：9/10** — 三步式分层架构 + 异步兼容修复 + Mock 友好，扣 1 分因为关键词覆盖度需持续人工维护

**📖 通用满分答案：** 设计分类器时评估三个维度：延迟要求、准确率要求、可用性要求。延迟敏感（<10ms）→ 规则；准确率敏感（>95%）+ 延迟可接受（200ms）→ LLM；两者都需要 → 规则优先 + LLM 兜底 = 本项目的方案。

##### Q2: 检索路由为什么用纯规则而不是 LLM？[S]

**S — 情境：** Intent Router 产出 query_analysis（含 intent + entities + keywords）后，Retrieval Router（`rag/retrieval_router.py`）需要据此生成 RetrievalPlan（含 mode、weights、top_k 等），决定使用 graph_first / parallel / hybrid_only 哪种检索策略。这是检索链路的关键决策点。

**T — 任务：** 在保证路由决策准确性的前提下，将路由延迟控制在检索总预算（P99 < 500ms）的极小占比。

**A — 行动：**
- **9 条纯规则引擎**：按优先级排列（Rule 1: 全局开关 → Rule 2: DB可用性 → Rule 3: 关系推理检测 → Rule 4: 精确匹配检测 → Rule 5: 语义理解检测 → Rule 6: 默认均衡），每条规则对应明确的正则模式
- **查询特征提取器** `_analyze_query()`：一次遍历提取 error_code、API名、类名、函数名、配置项、关系关键词、语义关键词、多实体等信号
- **三类检测器**：`_is_relational()`（关系推理）、`_is_exact()`（精确匹配）、`_is_semantic()`（语义理解），各含多个正则+意图组合条件
- **决策透明化**：每个 RetrievalPlan 的 `reason` 字段记录完整决策路径（如 "精确匹配类问题 → parallel（检测到错误码，检测到 API 名称）"）

**R — 结果：**
- 路由延迟 <1ms（纯正则+条件判断，O(1)复杂度）
- 14 个单元测试（`test_graph_router.py`）覆盖所有路由分支
- 同一 query 永远产生同一 RetrievalPlan（确定性，便于复现和回归）
- **项目评分：9/10** — 四个核心优势（延迟/确定性/可测试性/前置LLM已做语义理解），扣1分因为规则需持续维护正则模式库

**核心原则：把 LLM 仅用在"规则无法覆盖的语义理解"场景（意图分类的兜底），把规则用在"只需要做结构化决策"的地方（意图分类的主流路径 + 检索路由）。**

##### Q3: 如果意图分错了怎么办？[S]

**S — 情境：** 在生产环境中，用户查询的多样性和口语化表达可能导致 Intent Router 分类错误。例如 "API 调用失败怎么办？" 可能被规则引擎因"失败"关键词误判为 troubleshooting，而实际用户想了解 API 用法。

**T — 任务：** 确保单点意图分类错误不会导致系统级故障——即使分错，用户仍能获得可用的回答。

**A — 行动（三层防护）：**
1. **"最优/次优"而非"对/错"二元设计**：即使 intent 分错（如 troubleshooting → technical_question），系统仍走 RAG 检索路径，用户仍收到基于知识库的回答
2. **工具层关键词兜底**：Tool Orchestrator（`modules/tool_orchestrator.py`）的关键词触发器独立于意图——query 含 `"AUTH_401"` 时无论 intent 是什么都会触发 `get_error_code_detail`
3. **human_fallback 终局兜底**：unknown intent → human_fallback；检索无结果+重试耗尽 → human_fallback；校验失败+重试耗尽 → human_fallback

**R — 结果：**
- 意图分类错误不会导致"完全不相关"的响应（最坏情况是走次优路径而非最优路径）
- 排查流程清晰：trace → intent 值 + routing decision → 定位是 intent 问题还是 routing rule 问题
- **项目评分：8/10** — 三层防护设计保证了可靠性，扣分因为 optimal path selection 在边界 case 上仍依赖规则覆盖度
