### 21.0 项目立场修订（**2026-06-09 重要更新**）

> **背景：** 项目原名 `enterprise_agentic_rag`，包装为 Agentic RAG。经源码 trace 验证，实际是 **LangGraph 工作流 + 三路混合检索系统，无自治 Agent**（详见 CLAUDE.md 命名约定表与项目定位）。本附录原 §3、§4 中**大量 Agent 题与项目实际不符**，按"代码先行"铁律修订如下。
>
> **修订原则：** 不删除原题（面试官会问 → 你要能答），但**答题视角全部翻转**——从"我们用了 Agent 怎么做"改为"我们显式**不**用 Agent，理由是 X"。

#### 21.0.4 标准减法：哪些原题在我项目里"不该回答"

| 原题 | 不该硬答的原因 | 怎么回避 |
|------|---------------|---------|
| Multi-Agent 协作模式（3.5） | 项目无多 Agent | 答"我们项目用单工作流，但 Multi-Agent 概念可以讲" → 转向概念题 |
| Orchestrator Agent 任务分配（3.8） | 项目无 Orchestrator | 同上，转向"我们的工作流编排器" |
| Agent 死锁实战（3.6） | 项目无 Agent，无死锁 | 转向"我们的循环防护"（`can_retry`），并主动报告 baseline 发现的死循环 bug |
| ReAct 在项目中的应用（4.6） | 项目无 ReAct | 答"主路径不用，规划中的修复 Agent 用" → 引出 N7 |

> **核心心法：** 面试时不要硬答你项目里没有的东西。**主动承认 + 给出理由 + 引出你真做了的东西**——比硬答 100 分还高分。

| 题目 | 重要性 |
|------|--------|

| Self-Attention 的线性变换 (QKV)：为什么需要 QKV 线性变换？ | S |
| Self-Attention 中的 Softmax：为什么需要 Softmax？ | S |
| attention中数据shape是batch first还是seq_len first | S |

#### Q: Transformer 核心原理（合并回答）[S]

**本项目相关性：** 低。TECH DEEP DIVE 关注的是系统架构层（LangGraph、RAG 引擎），不涉及 Transformer 底层原理。


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：**

**Q/K/V 的本质：** 
- Q（Query）= 当前 token 的"查询意图"，即"我在找什么"
- K（Key）= 所有 token 的"被查询标签"，即"我能提供什么"
- V（Value）= 所有 token 的"实际内容"，即"我的语义是什么"
- Q 与 K 做点积 → 注意力分数矩阵 → Softmax → 与 V 加权求和 = 上下文感知的表征

**为什么需要 QKV 线性变换？** 线性变换 = 将所有 token 映射到多个不同的表示子空间。如果没有 QKV 变换（即 Q=K=V=输入 embedding），每个 token 的注意力分布会被自身的高相似度主导（对角线效应），无法灵活关注其他位置。QKV 变换解耦了"查询""被查""输出"三个角色。

**为什么除以 √dk？** 当 dk（Key 的维度）较大时，Q·K^T 的点积值方差增大（约等于 dk）。Softmax 在大值区域梯度接近 0（饱和区），除以 √dk 将点积值"拉回"到 Softmax 的敏感区，保证梯度有效传播。数学上：若 Q、K 各分量独立且均值为 0、方差为 1，则 Q·K^T 的均值为 0、方差为 dk。除以 √dk 将方差归一化为 1。

**简化代码：**
```python
def attention(Q, K, V, mask=None):
    d_k = Q.size(-1)
    scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(d_k)
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))
    weights = F.softmax(scores, dim=-1)
    return torch.matmul(weights, V)
```

#### Q: LoRA 原理与核心设计 [S]

**本项目相关性：** 低（项目定位为工程化 RAG 系统，不包含模型训练）。


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：**

**LoRA 核心思想：** 基于低秩分解——预训练权重 W₀ ∈ R^{d×k} 在微调时冻结，增量 ΔW = B·A 由两个低秩矩阵乘积表示，其中 B ∈ R^{d×r}、A ∈ R^{r×k}、r << min(d,k)。训练时只更新 A 和 B（参数量 < 1%），推理时 ΔW 可合并回 W₀ 无额外延迟。

**为什么 A 高斯初始化、B 全零初始化？** 确保训练开始时 ΔW = B·A = 0（因为 B=0），不破坏预训练模型的原始行为。A 用高斯初始化保证梯度能有效传播（如果 A 也是零则梯度为零无法训练）。

**r（秩）选择经验：**
- r=4：风格迁移、防过拟合、简单分类
- r=8~16：通用场景、多数任务性价比最优（r=8 通常达 Pareto 最优）
- r≥32：复杂任务（代码翻译、跨模态对齐）但边际收益递减
- 实践建议：先用 r=8 做基线，再按任务复杂度 ±50% 调整

**灾难性遗忘的应对（五种策略）：**
1. **LoRA 本身的隔离效应**：冻结 99% 参数，自然保留预训练知识
2. **数据混合**：训练数据中混入 15% 通用数据 + 85% 领域数据
3. **渐进式解冻**：Stage1 冻结前 N 层 → Stage2 全解冻低学习率
4. **EWC 正则化**：对预训练关键参数施加 Fisher 信息矩阵约束
5. **知识蒸馏**：原始模型作为 Teacher，最小化 KL 散度

| 题目 | 重要性 |

| 大模型推理加速：vLLM部署、KV Cache优化 | A |
| vLLM 的核心优势是什么？ | A |
| vLLM 框架的核心优化原理是什么？ | A |
| 推理时OOM排查方法？显存估算？ | A |
| 本地部署时，如何平衡 vLLM 上下文长度和显存占用？ | A |

#### Q: vLLM 核心原理与推理优化 [A]

**本项目相关性：** 低。TECH DEEP DIVE §12 涉及 Provider 抽象层（支持 vLLM 兼容接口），但不涉及推理加速细节。


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：**

**vLLM 三大核心优化：**

1. **PagedAttention（显存分页管理）：** 受 OS 虚拟内存启发，将 KV Cache 切成固定大小 Block（如每 Block 16 token），通过映射表拼接逻辑序列。显存利用率从 ~40% 提升至 95%+，消除碎片化导致的 OOM。

2. **Continuous Batching（连续批处理）：** 传统 static batching 需等整个 batch 全部完成。Continuous Batching 在每个 forward 后检查：已生成 EOS 的请求立即踢出 → 塞入新请求 → GPU 流水线永不停歇。吞吐提升 2-4x。

3. **高效显存管理 + KV Cache 共享：** 多个请求共享相同 System Prompt 的 KV Cache（物理只存一份），beam search 场景下 beam 之间共享 KV block。

**推理显存估算公式：**
```
模型权重 = 参数量 × 2 字节 (FP16)
KV Cache = 2 × hidden_size × num_layers × seq_len × 精度字节
总显存 ≈ 模型权重 + KV Cache + 激活值
```
例：7B 模型 FP16 + 2048 token → 权重 ~14GB + KV Cache ~1GB = ~15GB（不含 batch 开销）

**推理时 OOM 排查三板斧：**
1. 检查 batch_size 是否过大 → 减小 max_num_seqs
2. 检查 max_model_len 是否过大 → 设为实际需要的 1.2x 即可
3. 启用 KV Cache 量化 (INT8/INT4) → 显存减半

#### Q: 推理时 OOM 排查方法 [A]

**排查步骤：**
1. `nvidia-smi` 查看显存使用 → 确认是 OOM 还是碎片化
2. 检查 `max_model_len` 是否设置过大 → 设为实际需要长度的 1.2x
3. 检查 `max_num_seqs`（并发数）→ 减小并发
4. 检查 batch 中是否有超长序列拉高了所有请求的显存预分配
5. 启用 KV Cache 量化（INT8）→ 显存减半
6. 启用 PagedAttention（vLLM 默认）→ 消除碎片化

**模型量化后的精度损失排查：**
1. 在标准 benchmark 上对比 FP16 vs INT8/INT4 的困惑度变化
2. 在业务评测集上对比量化前后的准确率/Recall
3. 重点关注：数学推理、代码生成等对精度敏感的任务
4. 如果损失过大：① 降低量化程度（INT4→INT8）；② 使用 AWQ 等激活感知量化方案；③ 混合精度：关键层保持 FP16

## 26. Agent 进阶详细补充

### Q: ReAct vs Plan-and-Execute vs Reflection [S]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：**

| 范式 | 核心流程 | 适用场景 | 局限 |
|------|---------|---------|------|
| **ReAct** | Thought→Action→Observation 循环 | 多步工具调用、动态决策 | 步骤数不确定，延迟不可控 |
| **Plan-and-Execute** | 先规划完整步骤列表 → 再逐步执行 | 步骤相对确定、可预先规划 | 中间出错需整体重规划 |
| **Plan-Execute-Replan** | Plan→Execute→检查中间结果→按需Replan | 复杂任务且中间结果影响后续 | 实现复杂 |
| **Reflection** | 生成→自我评估→修正→再生成 | 需要高质量输出的写作/代码 | 成本翻倍 |

**ReAct vs Reflection 核心区别：** ReAct 依赖外部环境反馈（工具调用结果），Reflection 依赖内部自我评估（LLM 自己判断质量）。两者可结合：ReAct 获取信息 → Reflection 优化输出质量。

### Q: Agent 常见失败模式与稳定性治理 [S]

**结合项目回答（评分 8/10）：** 项目通过重试机制（retrieve=1, tool_call=2, verify=1）、最大步数限制→human_fallback、重复调用检测、分级降级（LLM 不可用→Mock，Milvus 不可用→MemoryVectorStore）来治理。

**Agent 常见失败模式：** ① 循环死锁（解法：重复检测+强制步数上限）；② 工具选择错误（解法：优化 Tool Description+意图前置分类）；③ 过早终止（解法：Verifier 校验+信息完整性判断）；④ 参数幻觉（解法：参数 schema 校验+正则提取）；⑤ 上下文溢出（解法：中间结果摘要压缩）。

### Q: Multi-Agent 系统如何防止死锁和无限循环？[S]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** ① Orchestrator 超时机制（120s 强制终止）；② Agent 间消息协议（Request/Response/Ack/Nack + timestamp+TTL）；③ 环形依赖检测（维护调用图检测 A→B→C→A）；④ 最大跳数限制 ≤ 10；⑤ 死信队列（超限任务转人工介入）。

## 27. Agentic Coding 与模型选型

### Q: Agentic Coding（Claude Code/Cursor）工作原理？[S]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** 核心流程：① 代码库索引（embedding+向量检索+AST 分析→构建代码关系图）；② 任务理解（LLM 分解为子任务）；③ 工具调用（read/write/search/run_command）；④ 执行循环（读代码→理解→改→运行测试→读错误→修复→循环）；⑤ 上下文管理（只加载相关文件到 context window）。

Claude Code vs Cursor：Claude Code 是 CLI Agent（终端驱动，全量文件操作权限）；Cursor 是 IDE Agent（编辑器内嵌，受限于编辑器上下文）。相同点：都做代码库索引+RAG 检索相关上下文。

### Q: 模型选型时如何做定性定量对比？[B]

**结合项目回答（评分 7/10）：** 项目通过 LLM Provider 抽象层（§12）支持多模型切换，但未做系统性的模型选型定量对比。


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** ① 确定评测维度（准确性/延迟(p50,p99)/成本/长上下文/工具调用能力）；② 构建评测集 100-200 条标注标准答案；③ 同 Prompt 同评测集各模型各跑一遍；④ 统计分析 Win rate + 效果/成本比 + 延迟分布；⑤ 决策矩阵加权评分结合业务约束做最终选择。

## 31. 提示工程延展问答

### Q: Prompt 注入检测和防护 [S/A 合并]


**🎯 S/T/A — 项目实战（情境/任务/行动）：**
**本项目答案（评分 8/10）：** 项目多层防护：工具安全分级+参数白名单、租户隔离、入库文档扫描。


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** 基于规则检测误报率低但漏报率高；基于 LLM 检测准确率高但成本大。实践中先用规则过滤（零成本），再用 LLM 处理边界。System Prompt 不应存放敏感信息。多轮对话后期遗忘 System Prompt 的缓解：每轮重新插入关键约束+定期 LLM 检查合规。

### Q: 结构化输出与 Prompt 评测 [A/B 合并]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** grammar-constrained decoding 通过构建 token 级状态机保证 100% JSON 格式正确，但可能降低生成质量。XML 对 LLM 更友好（训练数据中比例高、自然分隔），JSON 更适合程序解析。输出随机性处理：每个 case 采样 3-5 次取平均分+标准差。

### Q: JSON 字段含自由文本时特殊字符怎么防格式错误？[B]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** 三种方案：① 代码层面 `json.dumps(value)` 自动转义引号和反斜杠（最可靠）；② Prompt 中要求"自由文本字段用 <text><![CDATA[...]]></text> 包裹"；③ 对自由文本字段做 base64 编码后填入 JSON，应用层解码。方案①最常用，方案③可彻底避免转义问题但牺牲可读性。

### Q: 如何给模型注入特定角色知识？Role Prompting vs Fine-tuning？[A]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** 两种路径：① **System Prompt 角色定义**（轻量）：在 System Prompt 中描述角色背景、知识范围、说话风格。适合知识量小、不需精确记忆的场景。② **Fine-tuning 角色定制**（深度）：用该角色的真实/模拟对话数据微调模型。更逼真但成本高、更新不灵活。实践中先用 System Prompt 验证效果，确认需深度定制再微调。注意：角色知识注入后模型可能产生"角色越界"——在不应使用角色身份的场合也以该角色口吻回答。

### Q: 大模型角色扮演时的越界问题怎么防护？[S]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
**满分答案：** 越界的两种形式：① 角色身份泄漏到非角色场景（如普通对话中突然用 CEO 口吻）；② 角色场景中产生有害内容。防护策略：① System Prompt 中明确角色激活条件（"只有在用户明确提到 X 话题时才以 Y 角色回答"）；② 输出审核：检测回答是否出现了不应在当前场景出现的角色特征→拦截；③ 内容安全审核：角色场景中的输出同样过安全过滤；④ 角色隔离：不同角色使用不同的 session 或 conversation ID，角色状态不跨会话污染。

### Q: 路由分类器训练数据怎么构造？分错了怎么办？[A/S 合并]

**📖 通用满分答案（独立于项目，可迁移到其他面试）：**
