# 面试随身 A4 小抄

> **打印 A4 一张，面试时摊在桌上瞥一眼。所有数字来自真实评估，不允许虚标。**

---

## 一句话定位（30 秒能背）
**企业级 Hybrid RAG**：三路混合检索（BM25/向量/图谱）+ LangGraph 工作流 + 长尾 LLM Agent 兜底。**不是 Agentic RAG**——主路径决策权全在代码里。

---

## 关键数字（必背）

| 指标 | 数值 |
|------|------|
| 通过率 | 50% → **100%** |
| Recall@5 | 50% → **100%** |
| 转人工率 | 50% → **0%** |
| P95 延迟 | 6023ms → **3138ms** |
| 工具召回 F1 | 88.9% → **90%** |
| 意图分类延迟 | <1ms (rule) / ~200ms (LLM) |
| 代码量 | 5K+ Python |
| 评估用例 | 10 回归 + 10 工具选择 |

## 三铁律（为什么不用 Agent）

1. **权限 100% 拦截** → `check_permission` 静态边必走，不能让 LLM "倾向"调
2. **延迟可控** → 关键词 <1ms vs 全 LLM >200ms
3. **节点级可观测** → `Tracer.traced_node` 14 个节点埋点（load_memory / check_permission / classify_intent / retrieve_knowledge / rewrite_query / call_tools / build_context / generate_answer / verify_answer / finalize_answer / repair / human_fallback / final_refusal / save_memory），Agent 黑盒做不到

## 五意图权重表（必背一个）

| 意图 | keyword | vector | graph | 原因 |
|------|---------|--------|-------|------|
| ticket_query | **0.80** | 0.20 | 0.00 | 工单号精确匹配 |
| troubleshooting | 0.50 | 0.30 | 0.20 | 错误码精确 + 关系链 |
| policy_question | 0.45 | 0.45 | 0.10 | 制度名 + 语义 |
| technical_question | 0.20 | **0.55** | 0.25 | API 用法靠语义 |
| general_question | 0.30 | 0.50 | 0.20 | 均衡 |

**70/30 融合**：规则基线 70% + 意图 profile 30%。**注意：仅 graph_first / exact_parallel / semantic_parallel 三条规则路径做 70/30 融合；默认 parallel（Rule 6）直接用 intent profile 100%，不做融合。**

## 6 条路由规则（RetrievalRouter）

1. ENABLE_GRAPH_RAG=false → hybrid_only
2. Neo4j 不可用 → hybrid_only
3. 关系类（命中"关系/依赖/调用链"）→ graph_first
4. 精确匹配（错误码/API/类名）→ parallel keyword-heavy
5. 语义类（命中"为什么/怎么"）→ parallel vector-heavy
6. 默认 → parallel intent-aware

## RRF 融合公式
`score = Σ weight_i / (k + rank_i)`，k=60 防止 rank=1 一家独大。

## 真 bug 故事（必背）

1. **LangGraph 死循环** (`workflow.py:_after_verify`)：verify 失败时没调 `record_retry()`，`can_retry` 永远 True，触发 `GraphRecursionError`（10007 次）。修复：路由前必须先 `record_retry`。
2. **LLM 意图兜底死代码** (`intent_router._try_llm`)：用 `loop.is_running()` 判断异步上下文然后 return None，FastAPI 永远 is_running=True → 死代码。修复：丢线程池跑同步事件循环。
3. **中文 Jaccard 0 召回**：`_tokenize` 用 `text.lower().split()`，中文不分词。修复：CJK 字符级 + bigram。

## Agent 设计（口语化）

**ToolSelectionNode**：规则失败时降级，单次 LLM call，3s 超时，JSON 输出，不能选 destructive 工具，仍走 ToolExecutor 权限门。
**RepairDecisionNode**：verify failed 前最后救援，5s 超时，硬限 1 步，3 种 action（retry_retrieve / retry_with_alt_query / escalate），不能修权限/跳 verify。
**核心金句**：Agent 不放主路径，放长尾兜底——主路径 Workflow 保 SLA，Agent 提救回率。

## 12 步真实数据流（query="AUTH_401 怎么办"）

1. ainvoke → 2. load_memory → 3. check_permission（u002 有 knowledge_search）→ 4. classify_intent（正则强规则命中 `AUTH_401`（错误代码 pattern）→ troubleshooting, <0.1ms，不经关键词打分）→ 5. route=tools → 6. call_tools（必调 system_status + AUTH_401 → error_code_detail）→ 7. retrieve_knowledge（Rule 4 命中 → parallel keyword-heavy，权重 0.64/0.30/0.06）→ 8. RRF 融合 → 9. build_context（4096 token budget）→ 10. generate_answer → 11. verify_answer（HallucinationGuard 句子级 0.4 阈值）→ 12. finalize → save_memory

## 4 个高频追问的标准答案

**Q1: 为什么不全用 LLM 选工具？**
A: 规则 <1ms vs LLM ~800ms，80% query 规则就够，剩余 20% 兜底。三层考虑：性能/稳定性/成本。

**Q2: 怎么防 Agent 乱来？**
A: 五层防御：① prompt JSON schema ② parse 校验工具名 ③ required 参数有值 ④ 不暴露 destructive 工具 ⑤ 后续仍走 ToolExecutor 权限/敏感门。

**Q3: 评估通过率怎么算的？**
A: 跑 `scripts/eval_before_after.py`（4 组配置对比）+ `scripts/baseline_eval.py`（当前 v2 baseline，产出 `baseline_metrics.json`）。10 个回归用例。success_rate = (1 - need_human率)。每个失败 case 能讲根因。

**Q4: 这个项目和"伪 Agent"项目区别？**
A: 主动改名 Hybrid RAG（CLAUDE.md 命名约定表），承认是 Workflow——诚实 + 工程审美。
