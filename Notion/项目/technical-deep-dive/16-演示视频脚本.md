## 二、脚本（按时间轴）

### 段落 1 — 开场（0:00 - 0:30）

**画面**：架构图（technical-deep-dive/01-系统架构.md）全屏

**旁白**：
> 大家好，我演示一下我做的企业级混合检索 RAG 系统。
> 它不是 Agent，而是 LangGraph 编排的工作流：意图分类 → 三路检索（BM25 + 向量 + 知识图谱）→ 答案生成 → 校验。
> 设计原则是"关键词优先，LLM 只做兜底"。
> 接下来 5 分钟，我会用 3 个真实查询跑通端到端，再展示评估数据。

**操作**：
- 鼠标在架构图上画 3 个圈：**Intent Router** / **三路检索** / **Verifier**
- 0:25 切到终端

---

### 段落 2 — Query 1: 技术问题（0:30 - 1:30）

**画面**：终端，运行命令

**旁白**：
> 第一个查询：「如何重置密码？」这是个技术问题。

**操作**：
```bash
PYTHONPATH=src .venv/bin/python3 -c "
import asyncio, os
os.environ['LLM_PROVIDER'] = 'mock'
os.environ['EMBEDDING_PROVIDER'] = 'local'
os.environ['ENABLE_GRAPH_RAG'] = 'false'
from enterprise_hybrid_rag.graph.workflow import build_workflow
async def main():
    wf = build_workflow()
    out = await wf.ainvoke({'query':'如何重置密码？','user_id':'u001','session_id':'demo1'}, {'recursion_limit':50})
    print('意图:', out.get('intent'))
    print('检索文档数:', len(out.get('retrieved_docs', [])))
    print('verified:', out.get('verified'))
    print('答案:'); print(out.get('final_answer','')[:500])
asyncio.run(main())
" 2>/dev/null
```

**旁白**（边输出边讲）：
> 意图分类直接命中关键词——"密码"+"重置"+"如何"三个词，无需调 LLM，0.1 毫秒搞定。
> 检索回来 4 篇文档，校验通过，答案带引用。这条路径根本没烧任何 LLM token。

**强调截图**：终端输出 "意图: technical_question" 这行

---

### 段落 3 — Query 2: 故障排查 + 正则强规则（1:30 - 2:30）

**画面**：终端继续

**旁白**：
> 第二个查询：「我接入 SDK 时遇到 AUTH_401 错误怎么办？」这种结构化错误码，词典是覆盖不到的。

**操作**：
```bash
PYTHONPATH=src .venv/bin/python3 -c "
import sys; sys.path.insert(0, 'src')
import os
os.environ['LLM_PROVIDER'] = 'mock'
from enterprise_hybrid_rag.modules.intent_router import analyze_intent
r = analyze_intent('我接入 SDK 时遇到 AUTH_401 错误怎么办？')
print(r)
"
```

**旁白**（出现 `method=pattern, matched_keywords=['pattern:错误代码']`）：
> 看，意图分类走了正则强规则，命中 AUTH_401 这个错误代码模式，直接判 troubleshooting。
> 这是个真实的坑——v1 版本的正则太松，把 AUTH_401 误判成工单号 TKT-001 那种格式。
> 我修复时收紧了前缀白名单，只匹配 TKT/WO/SR/INC/CHG 这些常见工单前缀。
> 这就是为什么我说"代码先行"——必须用真实 case 验证规则边界。

**强调截图**：正则规则输出

---

### 段落 4 — Query 3: 跨意图歧义（2:30 - 3:30）

**画面**：终端继续

**旁白**：
> 第三个查询：「系统的访问控制策略是什么？」这是个策略问题。v1 这一条会因为词典覆盖不到而降级到 general_question。

**操作**：
```bash
PYTHONPATH=src .venv/bin/python3 -c "
import sys; sys.path.insert(0, 'src')
import os; os.environ['LLM_PROVIDER'] = 'mock'
from enterprise_hybrid_rag.modules.intent_router import analyze_intent
r = analyze_intent('系统的访问控制策略是什么？')
print(r)
"
```

**旁白**（输出 `intent=policy_question, matched=['策略','访问控制']`）：
> v2 我把词典从 11 个扩到 24 个 policy 词条，加了"访问控制"、"策略"、"机密"、"敏感数据"这些。
> 现在直接命中两个词，high confidence，照样不需要 LLM。

### 段落 5 — 评估数据对比（3:30 - 4:30）

**画面**：切换到 `data/eval/intent_v2_report.md` 滚动展示

**旁白**：
> 这套优化不是拍脑袋——我有评估飞轮支撑。

**操作**：
- 屏幕左半：v1 报告 `baseline_report.md`（注意：已被覆盖，提前截图保存）
- 屏幕右半：v2 报告 `intent_v2_report.md`
- 滚动到对比表

**强调旁白**：
> 意图准确率：70% → 100%
> 通过率：50% → 100%
> Recall@5：50% → 100%
> P95 延迟：6 秒 → 3.1 秒，砍掉了 48%
> 转人工率：50% → 0%

> 而且我跑了 14 条对抗性 case 防过拟合——边界、歧义、反例都覆盖到了，14/14 全过。

**强调截图**：v1/v2 指标对比表 / 24 case 全绿截图

---

### 段落 6 — 收尾（4:30 - 5:00）

**画面**：架构图回归

**旁白**：
> 总结一下：
> 第一，这个项目是 Hybrid RAG，不是 Agent，因为客服场景的稳定性、可解释性、延迟和成本都不允许 LLM 自治决策。
> 第二，意图分类、检索路由、工具选择全部"关键词优先，LLM 兜底"，这是三铁律。
> 第三，所有改动都有评估数据支撑，70 到 100、6 秒到 3 秒、零 LLM 调用——这都是真实跑出来的，不是 PPT 数字。

> Agent 只在两个地方存在：工具选择的 function calling 降级、和 verify 失败前的 repair 救援。**长尾兜底，不放主路径**。

> 谢谢观看。
