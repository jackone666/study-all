### 12.2 语义缓存（最高 ROI 优化）

> **原理：** 相同或相似的查询返回缓存答案，绕过检索+LLM 全链路。

#### 12.2.1 架构设计

```
用户 Query
  │
  ├─ Embedding 编码 → query_vector
  │
  ├─ Milvus 相似查询 → 搜索历史缓存
  │   └─ cosine_similarity(query_vector, cached_vector)
  │
  ├─ 相似度 > 0.95？
  │   ├─ Yes → 返回缓存答案（跳过检索+LLM）
  │   │        延迟: ~10ms（仅 Embedding + Milvus lookup）
  │   │
  │   └─ No → 正常 RAG 流程
  │            延迟: ~500-2000ms（全链路）
  │            └─ 结果写入缓存
```

#### 12.2.2 缓存键设计

```python
cache_key = hash(query_embedding + tenant_id + doc_version_hash)
```

**为什么需要三要素？**
- `query_embedding`：语义相似的查询共享缓存
- `tenant_id`：不同租户的知识库不同，同一 query 在不同租户下答案不同
- `doc_version_hash`：知识库更新后，旧缓存的答案可能过时，需要失效

#### 12.2.3 命中率预估与阈值调优

| 场景 | 预期命中率 | 原因 |
|------|-----------|------|
| 企业客服 FAQ | 25-35% | "如何重置密码"类高频问题 |
| 开发者 API 文档 | 15-25% | "如何使用认证接口"类重复查询 |
| 内部制度查询 | 10-20% | 查询多样性高，重复率较低 |
| 故障排查 | 5-10% | 错误码查询重复率低 |

**相似度阈值的确定方法：**
1. 收集 500 条历史 query
2. 人工标注"可互换"的 query pair（如"怎么重置密码"↔"如何修改密码"）
3. 计算这些 pair 的余弦相似度分布
4. 取 P95 作为阈值（覆盖 95% 的可互换 pair）

**成本节省计算：**
```
假设: QPS=10, 缓存命中率=20%, LLM 成本=$0.01/次
日节省 = 10 × 86400 × 20% × 0.01 = $1,728 → 实际按 10 QPS 不可能
更现实: 日均 5000 次请求 × 20% 命中 × $0.01 = $10/天 = $300/月
```

#### 12.2.4 缓存失效策略

| 触发条件 | 动作 |
|----------|------|
| 知识库文档更新 (`ingest_docs.py` 执行) | 递增 `doc_version` → 旧缓存自动失效 |
| 缓存满 (LRU, 10000 条上限) | 淘汰最久未访问的条目 |
| 缓存答案被 👎 3 次以上 | 标记为低质量，下次相同查询重新生成 |
| 缓存条目超过 7 天 | 自动过期 |

### 12.3 Token 成本优化

#### 12.3.1 模型分级调用

```
查询复杂度              推荐模型             成本/1M tokens   适用节点
─────────────────────────────────────────────────────────────────
意图分类 (32 tokens)    GPT-4o-mini / Qwen-Turbo   $0.15       Router
简单问答                 GPT-4o-mini               $0.15       Generator
复杂排障                 GPT-4o / Qwen-Max         $2.20       Generator
答案校验 (256 tokens)    GPT-4o-mini               $0.15       Verifier
摘要压缩 (200 tokens)    GPT-4o-mini                $0.15       SummaryMemory
```

**项目当前状态：** Router 已实现 `max_tokens=32` 的极小输出限制（§4.2），这是最有效的单点优化——意图分类的 LLM 调用成本降低了 ~60 倍。

**下一步优化：**
```python
# provider_factory.py — 按节点选择不同模型
def get_llm_provider(node: str = "default"):
    if node == "router":
        return QwenTurboProvider()      # 最便宜，最快
    elif node == "generator":
        return GPT4oMiniProvider()      # 性价比最优
    elif node == "verifier":
        return QwenTurboProvider()      # 校验也可以用便宜的
```

#### 12.3.2 Prompt 缓存（Anthropic/OpenAI 原生支持）

| 提供商 | 缓存机制 | 折扣 | 适用内容 |
|--------|---------|------|---------|
| Anthropic | Prompt Caching | 缓存写入 +25%，读取 -90% | System Prompt、长文档 |
| OpenAI | Automatic Caching | 命中 -50% | 重复的 prefix |
| DeepSeek | Disk Cache | 命中 -90% | 相同 prompt |

**项目适配策略：**
- System Prompt（角色定义 + 输出格式指令）放在 Prompt 最前面 → 每次请求的 prefix 相同 → 自动命中缓存
- 检索到的文档在 Prompt 最后（变化部分）→ 不影响 prefix 缓存命中

#### 12.3.3 Token 预算的三级分配

```python
# context/token_budget.py — Token 优先级分配
TOKEN_BUDGET = 4096
ALLOCATION = {
    "system_prompt":  256,   # 角色定义 + 输出格式（固定，可缓存）
    "query":          100,   # 用户问题（完整保留）
    "retrieved_docs": 2500,  # 检索文档（按 score 截断）
    "tool_results":   512,   # 工具执行结果
    "chat_history":   512,   # 对话历史（最近 3 轮）
    "citations":      216,   # 引用标注
}
# 总计: 256+100+2500+512+512+216 = 4096
```

#### 12.3.4 检索结果截断策略

```python
# 文档按 score 降序排列，逐文档填入直到 token 配额耗尽
remaining = budget.retrieved_docs
for doc in sorted_docs:
    token_count = estimate_tokens(doc["content"])
    if token_count <= remaining:
        context.append(doc)
        remaining -= token_count
    else:
        # 截断最后一个文档
        context.append(truncate(doc, remaining))
        break
```

**为什么不是简单的 Top-K？** 不同文档的 content 长度差异巨大（短的 50 tokens，长的 2000 tokens）。固定 Top-5 可能导致 token 用量波动 10 倍。Token 预算制让每次 LLM 调用的成本可预测。

### 12.4 延迟优化

#### 12.4.1 延迟分布分析

```
典型 RAG 请求的延迟分布（未缓存，真实 LLM）：

  Embedding 编码:    ████ 50ms
  向量检索 (Milvus): ██ 20ms
  关键词检索 (ES):   ██ 25ms
  图谱检索 (Neo4j):  ███ 80ms
  RRF 融合:          █ 5ms
  LLM 生成:          ████████████████████████████████ 1500ms  ← 瓶颈
  ──────────────────────────────────────────────────
  总计: ~1680ms
```

**LLM 生成占总延迟的 89%。** 优化 LLM 延迟是核心。

#### 12.4.2 并行化三路检索

```python
# 当前实现（graph_rag_orchestrator.py）
keyword_task = asyncio.ensure_future(keyword_search(query))
vector_task = asyncio.ensure_future(vector_search(query))
graph_task = asyncio.ensure_future(graph_search(query))

# 总延迟 = max(三路)，不是 sum
# 20 + 25 + 80 的串行 = 125ms
# 20 + 25 + 80 的并行 = max(20, 25, 80) = 80ms  ← 节省 36%
```

**效果：** 检索阶段延迟从 sum(各路) → max(各路)，节省 30-40%。

#### 12.4.3 流式输出 (SSE)

| 模式 | 首 token 延迟 | 用户感知 |
|------|-------------|---------|
| 当前（等完整响应） | ~1500ms | 白屏等待 |
| SSE 流式 | ~200ms | 立即看到第一个字 |

```python
# app/main.py — SSE 端点
@app.post("/chat/stream")
async def chat_stream(req: ChatRequest):
    async def event_generator():
        async for token in llm.generate_stream(prompt):
            yield f"data: {json.dumps({'token': token})}\n\n"
        yield "data: [DONE]\n\n"
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

### 12.5 检索性能优化

#### 12.5.1 Milvus 索引调优

| 参数 | 默认值 | 推荐值 | 影响 |
|------|--------|--------|------|
| `index_type` | IVF_FLAT | HNSW | 检索更快，构建更慢 |
| `nlist` | 128 | `sqrt(n)` | 太大→精度降，太小→速度降 |
| `M` (HNSW) | 16 | 32 | 更大→精度高但内存多 |
| `efConstruction` | 200 | 200 | 构建质量 |
| `ef` (搜索) | 64 | 128 | 更大→精度高但慢 |

```python
# milvus_store.py
index_params = {
    "metric_type": "COSINE",
    "index_type": "HNSW",
    "params": {"M": 32, "efConstruction": 200},
}
search_params = {"ef": 128}
```

#### 12.5.2 ES 索引优化

| 参数 | 开发环境 | 生产环境 |
|------|---------|---------|
| `number_of_shards` | 1 | 3 |
| `number_of_replicas` | 0 | 1 |
| `refresh_interval` | `1s` | `30s`（降低写入压力） |
| `index_options` | `positions` | `docs`（关闭词频和位置，减少索引体积 30%） |

**IK 分词优化：** 索引用 `ik_max_word`（细粒度→高召回），搜索用 `ik_smart`（粗粒度→高精度+更快）。

##### Q4: 怎么论证优化方案的价值？[A]


**📖 通用满分答案（独立于项目，可迁移到其他面试）：**

2. **预估收益**：基于技术原理计算上限——如语义缓存理论上限 = 重复查询占比 × LLM 成本
3. **A/B 验证**：灰度 5% 流量 → 对比实验组和对照组的延迟/成本/用户满意度
4. **决策**：收益 > 成本 × 2 → 全量；收益 ≈ 成本 → 观望；收益 < 成本 → 放弃
