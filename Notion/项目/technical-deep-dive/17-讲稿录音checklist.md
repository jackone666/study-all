# 讲稿录音自检清单

> **用法**：找一个安静房间，手机录音，**完整朗读三遍**，每遍按对应版本的 checklist 自评。
> **目标**：消除卡顿、消除"嗯/啊/那个"口语化，关键数字脱口而出。
> 三个版本累计录音时间约 25 分钟，自检+重录约 60-90 分钟搞定。

---

## 一、录音前 5 分钟暖嗓

```
1. 喝温水（不烫嘴）
2. 大声朗读以下绕口令各 3 遍：
   - "四是四，十是十，十四是十四，四十是四十"
   - "黑化肥发灰会挥发"
3. 数 1-30，每个数字咬清楚
4. 假装跟面试官打招呼：「您好，我先简单介绍一下我的项目」× 3 遍
```

## 二、版本 1（30 秒电梯版）— 朗读节拍标注

> **目标速度**：30 秒 ±3 秒。**关键钩子必须用 ⬆ 语调上扬强调**。

```
[呼吸] 我做了一个【企业级混合检索 RAG 系统】 ⬆ ：
       (重音在"混合检索")
[微停] 核心是把 BM25 关键词、向量检索、知识图谱 ── 三路融合 ⬆ ，
       (三路融合念慢，每个词清晰)
[微停] 用 LangGraph 编排工作流，【不靠 LLM 自主决策】 ⬆ 。
       (不靠 LLM 重音)
[呼吸] 这样能保证 ── P99 延迟可控、权限百分百拦截、节点级可观测 ⬆ ，
       (三个并列项每个稍停顿)
[微停] 也就是企业系统真正在意的【三件事】。
       (三件事念清晰)
[呼吸] 工作量大概是 5K 行 Python + 完整评估飞轮，
[微停] 目前 ── 意图准确率 100%、Recall@5 100%、P95 3.1 秒 ⬆ ，
       (新数字必须背熟！v2 数据)
[结束] 每个数字背后都有评估脚本支撑，不是 PPT 项目。
```

### ✅ 30 秒版自检 7 项

- [ ] 总时长 27-33 秒（超 35 秒说明语速过慢，节奏感会差）
- [ ] "**混合检索**" 4 个字清晰，没念成"hùnhé sōusuǒ"含糊
- [ ] "**三路融合**" 没念成"三路 RAG"（这是常见口误）
- [ ] "**意图准确率 100%**" 100 这个数字脱口而出，没卡壳
- [ ] "**P95 3.1 秒**" 这个数字脱口而出（不是"三秒多"）
- [ ] 没有 ≥3 次"嗯/啊/那个/就是"
- [ ] 结尾"不是 PPT 项目" 语气坚定，不上扬（陈述句不是疑问句）

### 红旗清单（出现任一就重录）

- [ ] 把"BM25"念成"BM 二十五"❌ 应念 "B M two five" 或 "BM 25"
- [ ] 把"LangGraph"念错 ❌ 正确：lan-graph，重音在前
- [ ] 把"通过率"和"准确率"念混
- [ ] 数字念错（30 秒讲稿里数字必须 100% 准确）

## 三、版本 2（2 分钟标准版）— 段落节拍标注

> **目标速度**：1 分 50 秒到 2 分 10 秒。**分 6 段**，段间深呼吸。

### 段落 1：定位（10s）
```
我做的项目叫 ⬆ Enterprise Hybrid RAG，
       (项目名重音)
是一个面向企业知识库的问答系统，
【显式不用自治 Agent】 ⬆ ，原因后面讲。
       (这句是钩子，故意留悬念)
```

### 段落 2：业务场景（15s）
```
[深呼吸]
业务场景：企业内部员工问── 
"我那张 TKT-001 工单到哪一步了"、
"AUTH_401 错误怎么解决"、
"数据分类标准是什么" ──
       (3 个例子节奏要整齐：4字+4字+4字感)
系统要在 2 秒内给出带引用的答案，
权限不够要拒绝，答不出来要转人工。
```

### 段落 3：架构三层（45s）— 速度最快的段
```
[深呼吸]
架构三层：

第一层是 LangGraph 工作流 ── 13 个节点 + 7 条条件路由，
确定性骨架，不让 LLM 决定流程。
       ("不让 LLM 决定流程"重音)

第二层是 三路混合检索引擎 ── 
BM25 走 Elasticsearch、向量走 Milvus、图谱走 Neo4j。
意图分类后驱动 RetrievalRouter 输出 RetrievalPlan，
每种意图独立权重 profile：
比如 ticket_query 关键词权重 0.8 ──工单号要精确匹配；
technical_question 向量权重 0.55 ──问 API 用法靠语义；
       (数字必须背熟，可以用"接近八成"等口语化降低门槛)
troubleshooting 关键词 0.5 加图谱 0.2 ──
错误码精确匹配加错误码-原因-解决方案的关系链推理。
三路结果用 weighted RRF 融合。

第三层是 韧性与可观测 ──
RecoveryManager 有界重试，Tracer 节点级埋点，
HallucinationGuard 句子级 0.4 阈值校验。
```

### 段落 4：为什么不用 Agent（25s）— 三铁律
```
[深呼吸]
为什么不用 Agent？企业场景三个硬要求：

第一，权限必须 100% 拦截 ──
check_permission 必须走，不能让 LLM "倾向于"调；
       ("倾向于"故意阴阳怪气强调)

第二，延迟必须可控 ──
意图分类关键词路径 <1 毫秒，全 LLM 路径 200+ 毫秒，
差两个数量级；
       ("两个数量级"重音)

第三，可观测必须节点级 ──
节点级 Tracer 才能定位 P99 是哪个节点拉的，
Agent 黑盒做不到。
```

### 段落 5：贡献和数据（25s）— v2 更新版
```
[深呼吸]
我的贡献和数据：

跑了 baseline 评估，10 个回归用例 ──
v1 通过率 50%，意图准确率 70%，Recall@5 50%；
       (v1 数字念慢)

我优化后 ── v2 通过率 100%，意图准确率 100%，Recall@5 100%，
P95 延迟从 6 秒降到 3.1 秒，砍掉 48%。
       (v2 数字念有力，强调对比)

关键改造是 ── 修了 3 个真 bug（verify 死循环、异步死代码、中文 Jaccard 0 召回），
扩了意图词典 + 加了正则强规则层，
加了 1 个 LLM function-calling 工具选择 Agent 做长尾兜底。
```

### 段落 6：收尾钩子（10s）
```
[深呼吸]
Agent 不放主路径 ── 放长尾兜底。
这个项目最大的差异化是 ── 数据先行 + 真 bug + 评估飞轮。
```

### ✅ 2 分钟版自检 10 项

- [ ] 总时长 105-130 秒（超 140 秒一定要砍段落 3 的细节）
- [ ] 6 段间能听出明显呼吸停顿（不是机关枪连射）
- [ ] 段落 3 的数字（0.8 / 0.55 / 0.5 + 0.2）背熟，没看稿
- [ ] 段落 4 的三个"第一/第二/第三"念出来层次感清晰
- [ ] 段落 5 的 v1 vs v2 对比，v1 数字念慢，v2 数字念有力（对比反差）
- [ ] "**RecoveryManager / HallucinationGuard / RetrievalRouter**" 三个英文类名不卡壳
- [ ] 没念错任何一个数据库名（Elasticsearch ≠ ElasticSearch 发音差不多但写法注意）
- [ ] "**LangGraph 工作流**" 这个词组连贯，不是逐字念
- [ ] 段落 6 收尾"长尾兜底"重音，给面试官留钩子
- [ ] 全程没有 ≥5 次"嗯/啊/那个"

### 红旗清单

- [ ] 段落 3 跑太快，听者根本记不住三层架构 → 慢一倍重录
- [ ] 段落 4 三铁律念得像背书 → 加入语气词 "这第一条很重要"
- [ ] 数字念错（比如 v2 通过率说成 90%）→ 立刻重录这段
- [ ] 段落 5 v1/v2 数字混了 → 用手势辅助（左手 v1，右手 v2）

## 四、版本 3（8 分钟深度版）— 段落级 checklist

8 分钟版太长，建议**分段录制再拼接**。每段 1-2 分钟，每段独立自检。

### 段落映射

| 段落 | 主题 | 时长 | 必备数据 / 行号 |
|------|------|------|----------------|
| 一 | 定位 | 30s | "Hybrid 指混合检索，不是混合 Agent" |
| 二 | 架构总览 | 60s | 5K 行、13 节点、7 条边、`workflow.py:661` |
| 三 | 真实 query trace | 4min | `intent_router.py:46`, `tool_orchestrator.py:47`, AUTH_401, TKT-001 |
| 四 | 评估飞轮 | 90s | v1: 70/50/50/6s → v2: 100/100/100/3.1s |
| 五 | 收尾差异化 | 30s | "三铁律 + 真 bug + 评估飞轮" |

### ✅ 8 分钟版自检 15 项

- [ ] 每段单独录制时长在目标 ±15% 内
- [ ] 段三的"5 个 step"trace 路径背得出文件名和函数名
- [ ] `analyze_intent` / `RetrievalRouter` / `tool_orchestrator` 三个核心函数能脱口而出
- [ ] 提到行号时听起来不像"显然记不清在编"（要么报准确数，要么说"在 modules 下"）
- [ ] AUTH_401 这个 case 能完整讲完不卡壳
- [ ] v1 → v2 优化故事有起承转合：发现问题（评估）→ 根因（词典差）→ 方案（扩词典+正则）→ 验证（24 case 全过）
- [ ] 段四里能提到一个具体 bug（比如 verify 死循环没调 record_retry）作为可信度证明
- [ ] 段五的"三铁律"和段四的"五铁律/四铁律"概念不混
- [ ] 全程没掉链子要看稿（如必须看稿，标记重录）
- [ ] 8 分钟末尾不疲惫、不喘气
- [ ] 拼接处自然，没有"上一段录的时候"这种穿帮
- [ ] 提到 LangGraph / FastAPI / RRF 时英文发音正确
- [ ] 整段没念错任何项目核心数字
- [ ] 8 分钟内主动给面试官留出至少 3 个钩子（"为什么这么设计 / 这一条还可以深挖"）
- [ ] 整段听感不像"背稿"，像"在讲一个自己亲手做的项目"

## 五、统一通病排查表

录完后回放找以下问题，**任一项都要重录对应段落**：

| 通病 | 自检方法 | 修复 |
|------|---------|------|
| 语速过快 | 自己 1.5× 倍速听都能听清 → 太快了 | 段落间加深呼吸 |
| 语速过慢 | 自己 0.75× 倍速听才舒服 → 太慢了 | 砍掉冗余形容词 |
| 嘴瓢 | 听到自己说错立刻笑了 → 暴露不熟练 | 单段重录 5 遍直到不出错 |
| 鼻音重 | 鼻塞感明显 | 喝水 / 换录音时段 |
| 麦克风噪音 | 背景有冰箱/空调嗡嗡 | 关电器，或加 Audacity 降噪 |
| 句尾上扬 | 陈述句听起来像疑问句 | 句尾故意往下压 |
| 无停顿 | 一口气念完整段 | 每个段落开头吸气可见 |
| 念错术语 | "BM25"念成"BM 二十五" | 标准发音录到顺口为止 |

## 六、终极检验：模拟面试场景

录完三版后，**强制做这一步**：

1. **手机录音** → 自己听一遍 → 给自己打分（1-10）
2. **打开第二个录音设备** → 朗读「请用 2 分钟介绍下你的项目」 → 录制
3. **对比** v2 「正式录音」和 v3「模拟面试朗读」
4. 如果差异 > 10%（卡顿增多、数字错位、节奏失控），说明**还不熟**

### 通过标准

- 不看稿，2 分钟版能完整复述，时长 ±15s
- 30 秒版能完整复述，时长 ±5s
- 关键数字（v1/v2 全套指标 + 三铁律）能任意挑一个秒答
- 任意一个钩子被深挖 30 秒内都能接住

## 七、自检完成后的归档

- [ ] 录音文件按 `v1_30s.m4a` / `v2_2min.m4a` / `v3_8min.m4a` 命名
- [ ] 自检评分表存档：每版打分 + 重录次数
- [ ] 暴露的"死结"列表（哪些段落怎么练都不顺）→ 加入面试前 1 小时复盘清单

---

> **最后的提醒**：录音自检的目的**不是把稿子背完美**，而是**让你在面试现场能轻松抓住任何一个钩子展开**。
> 如果某段你录了 5 遍还不顺，说明你没真懂 — 回去再读一遍源码，再录。
