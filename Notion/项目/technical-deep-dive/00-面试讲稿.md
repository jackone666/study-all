## 版本一：30 秒电梯版

> 我做了一个**企业级混合检索 RAG 系统**：核心是把 BM25 关键词、向量检索、知识图谱三路融合，用 LangGraph 编排工作流，**不靠 LLM 自主决策**，而是用意图分类驱动检索权重和工具调用。
>
> 这样做能保证 P99 延迟可控、权限百分百拦截、节点级可观测——也就是企业系统真正在意的三件事。

## 版本二：2 分钟标准版

> 我做的项目叫 **Enterprise Hybrid RAG**，是一个面向企业知识库的问答系统，**显式不用自治 Agent**，原因后面讲。
>
> **业务场景：** 企业内部员工问"我那张 TKT-001 工单到哪一步了"、"AUTH_401 错误怎么解决"、"数据分类标准是什么"，系统要在 2 秒内给出带引用的答案，权限不够要拒绝，答不出来要转人工。
>
> **架构三层：**
>
> 第一层是 **LangGraph 工作流**——`StateGraph` 静态边把 load_memory → check_permission → classify_intent → 检索 → 工具 → 生成 → 校验 → 落库串起来，路由函数 `_after_*` 决定走哪条边。这个是确定性骨架，不让 LLM 决定流程。
>
> 第二层是 **三路混合检索引擎**——BM25 走 Elasticsearch、向量走 Milvus、图谱走 Neo4j。意图分类后驱动 `RetrievalRouter` 输出 `RetrievalPlan`，每种意图有独立的权重 profile：比如 `ticket_query` 关键词权重 0.8，因为工单号要精确匹配；`technical_question` 向量权重 0.55，因为问 API 用法靠语义；`troubleshooting` 关键词 0.5 加图谱 0.2，因为错误码精确匹配 + 错误码-原因-解决方案的关系链推理。三路结果用 weighted RRF 融合。
>
> 第三层是 **韧性与可观测**——`RecoveryManager` 做有界重试（检索失败改写 query 1 次、工具失败重试 2 次），失败兜底转人工；`Tracer` 给每个节点埋点；`AnswerVerifier` 拿 `HallucinationGuard` 做句子级语义相似度校验，低于 0.4 拒答。
>
> **为什么不用 Agent？** 企业场景三个硬要求：
> 1. **权限必须 100% 拦截**——`check_permission` 必须走，不能让 LLM "倾向于"调
> 2. **延迟必须可控**——意图分类关键词路径 <1ms，全 LLM 路径 >200ms，差两个数量级
> 3. **可观测必须节点级**——节点级 Tracer 才能定位 P99 是哪个节点拉的，Agent 黑盒做不到
>
> **我的贡献和数据：** 跑了 baseline 评估，10 个回归用例 v1 通过率 50%，意图准确率 70%，Recall@5 50%。
> 我做了三件事推到现在：
> 1. 修了 3 个真 bug——verify 死循环（`_after_verify` 漏调 `record_retry`，触发了 LangGraph 递归上限）、LLM 兜底异步死代码（`_try_llm` 用 `loop.is_running` 后跳过，FastAPI 环境永远不生效）、中文 Jaccard 0 召回（`retrievers.py` 用 `split` 不分中文）。
> 2. 加了 2 个 Agent 兜底——ToolSelectionNode（LLM function-calling 工具选择）+ RepairDecisionNode（verify 失败前最后救援）。两个都不在主路径上。
> 3. 扩了意图词典 + 加正则强规则层，把意图准确率从 70% 推到 100%（24/24 case 全对，0 LLM 调用）。
>
> v2 数据：通过率 100%、意图准确率 100%、Recall@5 100%、P95 3.1 秒，转人工率 0%。
> **Agent 不放主路径，放长尾兜底**——主路径 Workflow 保 SLA，Agent 提救回率。

**用途：** 简历项目段落口述、面试官"介绍下你的项目"开场。

## 版本三：8 分钟深度版（含真实数据流 trace）

> 我详细讲一下这个项目。

### 一、定位（30 秒）

> Enterprise Hybrid RAG，企业知识库问答系统。**Hybrid 指的是混合检索，不是混合 Agent**——这一点很重要，因为很多面试项目叫"Agentic RAG"，但深挖发现是 if-else 串起来的工作流。我这个项目源码就是 LangGraph 工作流，**所以我用 Hybrid RAG 而不是 Agentic RAG 描述它**——这是诚实，也是差异化。

### 二、架构总览（1 分钟）

> 整体 16K 行 Python（含空行注释，src/ 下），分四层：
>
> - **接入层**：FastAPI + React 前端
> - **编排层**：LangGraph `StateGraph`（`graph/workflow.py` 第 761 行 `build_workflow()`），14 个节点 + 7 条条件路由
> - **检索层**：三路检索 + RetrievalRouter（`rag/retrieval_router.py` 第 200 行 `RetrievalRouter.route()`）+ 三路 RRF 融合（`rag/fusion.py`）
> - **基础设施**：PostgreSQL + Redis + Milvus + Elasticsearch + Neo4j + MinIO 全套 Docker Compose

### 三、一个真实 query 完整数据流（4 分钟，核心部分）

> 我用一个 query "我接入 SDK 时遇到 AUTH_401 错误怎么办" 把全链路 trace 一遍——这个是我 baseline 评估第 4 个用例，通过的：
>
> **Step 1 - 入口：** `workflow.ainvoke({"query": "...", "user_id": "u002", "session_id": "..."})`
>
> **Step 2 - load_memory：** `MemoryManager.load_memory_context()` 拿 session 短期记忆 + 长期 user_profile + summary
>
> **Step 3 - check_permission：** `check_permission_sync("u002")` 返回 role + permissions，路由函数 `_after_permission` 判断 `knowledge_search` 在不在 permissions 里，不在就走 `final_refusal`——**这是确定性硬路由，Agent 系统做不到这种确定性**
>
> **Step 4 - classify_intent：** `analyze_intent()`（`modules/intent_router.py` 第 124 行）
>   - 正则强规则 Step 0：query 中的 AUTH_401 命中错误代码正则 `r'\b(?:AUTH|ERR|ERROR|EXC|FAIL)[-_]\d+\b'` → 直接判 troubleshooting，无需 keyword 打分也无需 LLM
>   - 返回：`{intent: "troubleshooting", matched_keywords: ["pattern:错误代码"], entities: ["AUTH_401"], method: "pattern"}`
>   - **耗时 <0.1ms**——这是 v2 新增的规则层，比 keyword 还快
>   - 然后在 `classify_intent_node`（workflow.py 第 95 行）拼出完整的 `query_analysis` 写进 state
>
> **Step 5 - 路由 `_after_classify`：** intent=`troubleshooting` ∈ `_TOOL_INTENTS` → route=`tools` → 走 `call_tools`
>
> **Step 6 - call_tools：** `tool_orchestrator._select_tools()`（`modules/tool_orchestrator.py` 第 78 行）
>   - intent=`troubleshooting` 触发规则：必调 `get_system_status` + 检测到 `AUTH_401` 错误码 → 加调 `get_error_code_detail({error_code: "AUTH_401"})`
>   - `ToolExecutor.execute()` 跑工具，权限不够会拒（`adapters/permission_adapter.py`），敏感工具进 `pending_tool_confirmations`
>
> **Step 7 - 路由 `_after_call_tools`：** 工具有错就 retry（最多 2 次，`RetryPolicy.tool_call.max_retries=2`），没错走 `retrieve_knowledge`
>
> **Step 8 - retrieve_knowledge：** `GraphRAGOrchestrator.retrieve()`（`rag/graph_rag_orchestrator.py` 第 68 行）
>   - 因为 state 里已经有 `query_analysis`，**不重复调 classify_intent**——这是一个我自己 trace 出来的优化点，CLAUDE.md 里专门记了这个数据流
>   - 调 `RetrievalRouter.route()`，因为 query 命中 `AUTH_401` 错误码 → Rule 4 命中 → `_make_exact_parallel()`（retrieval_router.py 第 471 行）
>   - 70/30 融合：规则基线 `{kw:0.7, vec:0.3, graph:0}` × 70% + 意图 profile `{kw:0.5, vec:0.3, graph:0.2}` × 30% → 最终 `{kw:0.64, vec:0.30, graph:0.06}`
>   - asyncio.gather 并发跑三路检索 → weighted RRF 融合（`fusion.py` 第 13 行 `weighted_rrf_fusion()`，公式 `score = Σ weight_i / (k + rank_i)`，k=60 防止 rank=1 一家独大）→ rerank → top-5
>
> **Step 9 - build_context：** `ContextManager.build_context()` 拼 prompt，做 token budget 截断（4096 token 上限）
>
> **Step 10 - generate_answer：** `generate_answer()` 出 draft + citations
>
> **Step 11 - verify_answer：** `verify_answer()` 走 `HallucinationGuard`（`modules/hallucination_guard.py`）
>   - 句子级 TF-IDF + bigram 相似度 < 0.4 阈值就拒答
>   - 这个阈值 baseline 是 0.6（生产 sbert 版本），mock 环境降到 0.4 是因为 TF-IDF 信号弱
>
> **Step 12 - 路由 `_after_verify`：** verified=true → finalize；false → `can_retry` 为 True 则重试 build_context，否则走 repair agent 或 `human_fallback`。**这里曾经有一个真 bug**：`workflow.py` 的 `_after_verify` 逻辑里 `record_retry()` 没被调用，导致 retry counter 不递增、`can_retry` 永远 True，触发 LangGraph `GraphRecursionError`。我跑 baseline 时第一次 ainvoke 卡在 10007 次递归才报错——后来在第 417 行加了 `record_retry` 修复（注释标注 BUG fix 2026-06-09）
>
> **Step 13 - finalize + save_memory：** 写 PostgreSQL `qa_logs`、metrics 入 Prometheus、session checkpoint 保存

### 四、关键技术决策（2 分钟）

我挑三个最值得讲的：

**决策 1：意图分类用正则强规则 + 关键词 + LLM 兜底，而不是全 LLM**
> - v2 引入 Step 0 正则强规则层（4 条 pattern：HTTP错误码、错误代码、系统故障短语、工单号），<0.1ms 命中即定
> - 没命中正则才走关键词打分：`modules/intent_router.py` 第 202 行 `_classify_keyword`，用命中数排序 + 二级 ratio ≥ 2 判定置信度
> - 关键词低置信才进 LLM 兜底（第 246 行 `_try_llm`）
> - v1 这里有 bug：旧实现用 `loop.is_running()` 判断异步上下文然后直接 return None——结果在 FastAPI 这种永远 is_running=True 的环境里 LLM 路径根本没生效。v2 改成在 ThreadPoolExecutor 里跑 sync 调用（第 292 行 `future.result(timeout=5.0)`）
> - 收益：意图准确率从 70%（v1 关键词漏命中 + LLM 死代码）拉到 100%（v2 24/24 case，且 0 次 LLM 实际调用），平均延迟 <1ms

**决策 2：三路融合用 weighted RRF 而不是 score 加权**
> - 不同检索器分数不可比（BM25 是 0-100，cosine 是 0-1，图谱是路径数）
> - RRF 只用 rank，天然归一化
> - 权重 = 信源可信度，由意图驱动（见 `_INTENT_WEIGHT_PROFILES` 第 108 行）
> - 还做了 70/30 融合：规则基线 70% + 意图 profile 30%，避免一刀切

**决策 3：失败处理用静态有界重试 + 转人工，而不是让 LLM 自己决定**
> - `RecoveryManager` 注册每个节点的 `max_retries`（检索 1、工具 2、生成 1、校验 1）
> - 路由函数显式调 `can_retry()` 判断，超限走 `human_fallback`
> - 收益：失败路径**可预测**，不会出现"Agent 自己决定无限重试烧 token"
> - 风险：策略死板，长尾 case 救不回——这就是加 RepairDecisionNode 的地方（verify 失败但重试用尽时，让 LLM 看上下文尝试修复，但仍不放在主路径）
