## **1、OSI 七层模型是什么？每一层的作用是什么？**
OSI（Open System Interconnection，开放系统互连）七层模型是国际标准化组织（ISO）制定的**网络通信分层标准**，通过分层解耦降低网络设计复杂度，从下到上分为 7 层：

| **层级** | **核心作用** | **典型协议 / 设备** | **数据单元** |
| --- | --- | --- | --- |
| **应用层** | 直接为应用程序提供网络服务，定义应用程序间通信的格式和规则，面向用户业务。 | HTTP、HTTPS、FTP、SMTP、DNS | 报文（Message） |
| **表示层** | 负责**数据格式转换、加密解密、压缩解压缩**，确保不同系统应用层数据互通。 | JPEG、SSL/TLS、ASCII | 报文 |
| **会话层** | 建立、管理、终止**进程间的会话连接**，控制会话同步（如断点续传标记）。 | RPC、NetBIOS | 报文 |
| **传输层** | 提供**端到端（进程到进程）** 的通信服务，核心是可靠传输和流量控制。 | TCP、UDP | 报文段（Segment）/ 用户数据报（Datagram） |
| **网络层** | 负责**跨网络的路由与转发**，为分组选择最佳路径，实现不同网络间的互联。 | IP、ICMP、ARP、OSPF | 分组 / IP 数据报（Packet） |
| **数据链路层** | 负责**相邻节点间的帧传输**，封装 IP 数据报为帧，处理 MAC 地址寻址、差错检测。 | 以太网协议、PPP、CSMA/CD | 帧（Frame） |
| **物理层** | 负责**物理介质上的比特流传输**，定义接口标准（如电压、接口类型），屏蔽硬件差异。 | RJ45、光纤、Wi-Fi 物理层规范 | 比特（Bit） |

**核心考点**：分层的意义是**解耦、标准化、易维护**，上层无需关心下层实现细节（如应用层不用管物理层是光纤还是网线）。
## **2、TCP/IP 四层模型是什么？每一层的作用是什么？**
TCP/IP 是**实际互联网采用的协议栈**，比 OSI 更简洁，分为四层，与 OSI 七层的对应关系是：**应用层对应 OSI 上三层，网络接口层对应 OSI 下两层**。

| **层级** | **核心作用** | **典型协议 / 设备** |
| --- | --- | --- |
| **应用层** | 整合 OSI 应用层、表示层、会话层功能，为应用提供网络服务，定义应用通信规则。 | HTTP、HTTPS、FTP、DNS、SMTP |
| **传输层** | 端到端进程通信，提供可靠 / 不可靠传输，区分不同应用进程（通过端口号）。 | TCP、UDP |
| **网络层** | 跨网络路由转发，IP 地址寻址，实现不同网络互联，核心是 IP 协议。 | IP、ICMP、ARP、BGP |
| **网络接口层** | 整合 OSI 数据链路层 + 物理层功能，负责相邻节点的帧传输和比特流收发。 | 以太网协议、PPP、Wi-Fi |

**核心区别**：OSI 是**理论模型**，结构严谨但复杂；TCP/IP 是**实用模型**，简洁高效，贴合互联网实际部署。
## **3、从输入 URL 到页面展示到底发生了什么？（面试必问）**
这是**全链路网络流程题**，需按顺序说清每一步核心操作，细分如下：
1. **URL 解析**：浏览器解析 URL，提取**协议（http/https）、域名（如**[**www.baidu.com**](https://www.baidu.com/)**）、路径（如 /index.html）、端口（默认 80/443）**。
2. **DNS 域名解析**：将域名转换为 IP 地址，流程为 `本地DNS缓存 → 本地DNS服务器 → 根DNS → 顶级DNS → 权威DNS`，最终返回目标 IP（递归 + 迭代查询）。
3. **建立 TCP 连接（HTTPS 需额外 TLS 握手）**
    - HTTP：客户端与服务器进行**TCP 三次握手**，建立可靠连接。
    - HTTPS：三次握手后，增加**TLS 握手**（协商加密套件、交换随机数、验证证书、生成会话密钥），建立加密通道。
4. **发送 HTTP 请求**：浏览器构建请求报文（请求行 + 请求头 + 请求体），通过 TCP 连接发送给服务器（如 GET /index.html HTTP/1.1）。
5. **服务器处理请求并返回响应**：服务器接收请求，解析 URL 路径，处理业务逻辑（如查询数据库、读取静态文件），构建响应报文（状态行 + 响应头 + 响应体）返回。
6. **浏览器解析并渲染页面**
    - 解析 HTML：构建 DOM 树；
    - 解析 CSS：构建 CSSOM 树，与 DOM 树合并生成**渲染树**；
    - 布局（Layout）：计算元素位置和大小；
    - 绘制（Paint）：将渲染树绘制到屏幕；
    - 资源加载：解析 HTML 中的图片、JS、CSS 等资源 URL，**重复 1-5 步**请求这些资源。
7. **关闭 TCP 连接**：若为 HTTP 短连接，数据传输完成后进行**TCP 四次挥手**；若为长连接（Connection: keep-alive），则复用连接处理后续请求。

**核心考点**：DNS 缓存机制、TCP 三次握手 / 四次挥手、HTTPS TLS 握手、浏览器渲染流程。
## **4、HTTP 状态码有哪些？（按类别记忆）**
HTTP 状态码是服务器对请求的处理结果反馈，分为 5 类，**重点记忆常见状态码**：

| **类别** | **含义** | **常见状态码 & 核心场景** |
| --- | --- | --- |
| 1xx | 信息性状态码 | `100 Continue`：客户端继续发送请求体（用于 POST 大请求） |
| 2xx | 成功状态码 | `200 OK`：请求成功；`201 Created`：资源创建成功（如 POST 新增用户）；`204 No Content`：请求成功但无响应体 |
| 3xx | 重定向状态码 | `301 Moved Permanently`：永久重定向（域名变更）；`302 Found`：临时重定向；`304 Not Modified`：资源未修改，使用缓存 |
| 4xx | 客户端错误状态码 | `400 Bad Request`：请求参数错误；`401 Unauthorized`：未登录；`403 Forbidden`：权限不足；`404 Not Found`：资源不存在；`405 Method Not Allowed`：请求方法不支持 |
| 5xx | 服务端错误状态码 | `500 Internal Server Error`：服务器内部错误；`502 Bad Gateway`：网关错误（如 Nginx 连不上后端）；`503 Service Unavailable`：服务器过载 / 维护；`504 Gateway Timeout`：网关超时 |

**核心考点**：304 的缓存逻辑、401 与 403 的区别、502 与 504 的场景差异。
## **5、HTTP 是不保存状态的协议，如何保存用户状态？**
HTTP 是**无状态协议**，服务器无法识别连续请求的用户身份，需通过以下三种方案解决：
### **方案一：Session + Cookie（主流方案）**
8. 用户登录时，服务器验证账号密码，生成**Session 对象**（存储用户信息，如 userid、权限），并分配唯一`SessionID`。
9. 服务器通过响应头 `Set-Cookie: JSESSIONID=xxx`，将 SessionID 发送给浏览器。
10. 浏览器将 SessionID 保存为 Cookie，后续请求时自动携带 `Cookie: JSESSIONID=xxx`。
11. 服务器通过 SessionID 找到对应的 Session 对象，识别用户身份。

**注意事项**：
- Session 存储位置：① 服务器内存（简单但不支持分布式）；② 分布式缓存（如 Redis，主流方案，支持集群）；③ 数据库（性能差，不推荐）。
- 安全优化：Cookie 设置 `HttpOnly`（防止 JS 窃取）、`Secure`（仅 HTTPS 传输）、`SameSite`（防止 CSRF）。

### **方案二：URL 重写（Cookie 禁用时的备选方案）**
将 SessionID 直接拼接在 URL 中作为参数，如 `http://example.com/page?sessionid=xxx`，服务器解析 URL 获取 SessionID。**缺点**：URL 暴露 SessionID，安全性低；URL 长度有限制。
### **方案三：Token 认证（如 JWT，分布式系统首选）**
12. 用户登录成功后，服务器生成**加密 Token**（如 JWT，包含用户信息 + 签名），返回给客户端。
13. 客户端将 Token 保存在 localStorage/sessionStorage，后续请求通过请求头 `Authorization: Bearer <token>` 携带。
14. 服务器验证 Token 签名，解析用户信息，无需存储 Token（无状态）。

**核心区别**：Session 是**服务端存储**，Token 是**客户端存储**；Token 更适合分布式系统，无需共享 Session。
## **6、URI 和 URL 的区别是什么？**
- **URI（Uniform Resource Identifier）：统一资源标识符**核心是**唯一标识一个资源**，不管用什么方式定位。URI 是一个抽象概念，包含 URL 和 URN 两类。
    - 例子：`urn:isbn:9787115546081`（用 ISBN 标识一本书，不涉及地址）。
- **URL（Uniform Resource Locator）：统一资源定位符**是 URI 的**子集**，不仅标识资源，还指明**如何访问该资源**（通过协议 + 地址）。
    - 例子：`https://www.baidu.com/index.html`（通过 HTTPS 协议访问百度首页）。
- **核心结论**：**所有 URL 都是 URI，但不是所有 URI 都是 URL**。面试回答时，只需说清 “URI 是标识，URL 是标识 + 定位” 即可。

## **7、WebSocket 是什么？应用场景有哪些？**
WebSocket 是**基于 TCP 的全双工通信协议**，通过一次握手建立持久连接，客户端和服务器可**双向主动发送数据**，解决 HTTP 单向通信的痛点。
**核心应用场景**：
- 实时消息推送：如聊天软件、社交平台通知；
- 实时协作：如多人在线文档编辑（腾讯文档）；
- 实时数据展示：如股票行情、监控大屏；
- 在线游戏：如多人对战游戏（低延迟需求）。

## **8、WebSocket 和 HTTP 有什么区别？**
两者都是基于 TCP 的应用层协议，但设计目标和通信模式完全不同：

| **特性** | **HTTP** | **WebSocket** |
| --- | --- | --- |
| 通信方向 | 单向通信（客户端请求→服务器响应） | 全双工通信（双方可同时收发） |
| 连接方式 | 短连接 / 长连接（keep-alive） | 持久连接（一次握手，长期有效） |
| 协议标识 | `http://`/`https://` | `ws://`/`wss://`（加密版） |
| 头部开销 | 每次请求携带完整 HTTP 头部，开销大 | 握手时带头部，后续通信头部极小 |
| 服务端主动推送 | 不支持（需轮询模拟） | 原生支持（核心优势） |
| 适用场景 | 普通网页请求、资源获取 | 实时通信、消息推送 |

**核心考点**：WebSocket 握手时会发送 `Upgrade: websocket` 请求头，要求服务器升级协议。
## **9、WebSocket 与短轮询、长轮询的区别？**
三者都是为了解决**服务器向客户端实时推送数据**的问题，核心差异在实现方式和效率：

| **特性** | **短轮询（Short Polling）** | **长轮询（Long Polling）** | **WebSocket** |
| --- | --- | --- | --- |
| 实现原理 | 客户端每隔固定时间（如 5s）发 HTTP 请求，服务器立即响应 | 客户端发请求，服务器无数据则挂起连接，有数据 / 超时再响应，客户端收到后立即重连 | 一次握手建立 TCP 长连接，双向主动通信 |
| 实时性 | 低（延迟 = 轮询间隔） | 中（有数据立即推送） | 高（毫秒级延迟） |
| 资源开销 | 高（频繁建立 / 关闭 TCP 连接，空请求多） | 中（减少空请求，但仍需反复重连） | 低（持久连接，无冗余请求） |
| 服务器压力 | 大（高并发下请求量爆炸） | 中（连接挂起消耗线程资源） | 小（单连接可处理双向通信） |
| 实现复杂度 | 低（普通 HTTP 请求即可） | 中（需服务器支持挂起连接） | 高（需服务器和客户端支持协议） |

## **10、TCP 与 UDP 的区别（面试重中之重）**
TCP 和 UDP 是传输层两大核心协议，区别可从**连接性、可靠性、效率**等维度记忆：

| **特性** | **TCP** | **UDP** |
| --- | --- | --- |
| 连接性 | 面向连接（三次握手建立，四次挥手关闭） | 无连接（直接发送数据包） |
| 可靠性 | 可靠传输（序列号、确认应答、超时重传、流量控制、拥塞控制） | 不可靠传输（尽最大努力交付，不保证送达和有序） |
| 状态性 | 有状态（维护连接状态、序列号、窗口大小） | 无状态（发送后不关心后续） |
| 传输形式 | 面向字节流（无消息边界，可拆分 / 合并数据） | 面向报文（保留消息边界，应用给多大就发多大） |
| 首部开销 | 20~60 字节（可变） | 固定 8 字节（极简） |
| 通信方式 | 仅支持单播 | 支持单播、多播、广播 |
| 适用场景 | 可靠性优先：文件传输（FTP）、网页（HTTP）、邮件（SMTP） | 实时性优先：音视频直播、在线游戏、DNS 查询 |

## **11、什么时候选择 TCP，什么时候选 UDP？**
选择的核心是**权衡可靠性和实时性**：
- **选 TCP 的场景**：数据不能丢失、必须有序到达。例如：文件传输（丢包会导致文件损坏）、电商支付（数据错误会导致交易异常）、网页加载（资源缺失会导致页面错乱）。
- **选 UDP 的场景**：允许少量丢包，但要求低延迟。例如：视频直播（丢几个帧不影响观看）、在线游戏（延迟高比丢包更致命）、DNS 查询（快比准更重要，丢包可重查）。
- **特殊情况**：若需 “UDP 的实时性 + TCP 的可靠性”，可在应用层实现可靠机制（如 RTP 协议 + 重传机制）。

## **12、基于 TCP/UDP 的协议有哪些？**
传输层协议为应用层提供服务，需记住常见协议的底层依赖：
- **基于 TCP 的协议**（可靠性优先）：HTTP/HTTPS（网页）、FTP（文件传输）、SMTP/POP3/IMAP（邮件）、SSH（远程登录）、Telnet。
- **基于 UDP 的协议**（实时性优先）：DNS（域名解析）、DHCP（IP 地址分配）、RTP（音视频传输）、SNMP（网络管理）、TFTP（简单文件传输）。

## **13、TCP 三次握手的过程？为什么是三次不是两次 / 四次？**
### **（1）三次握手的核心目的**
确认**双方的发送和接收能力都正常**，并同步初始序列号（SEQ），为后续可靠传输奠定基础。
### **（2）三次握手详细过程**

| **握手次数** | **发起方** | **报文标志** | **核心动作** | **状态变化** |
| --- | --- | --- | --- | --- |
| 第一次握手 | 客户端 | SYN（同步） | 发送 `SYN=1，SEQ=x`（x 为随机初始序列号），请求建立连接 | 客户端→`SYN_SENT` |
| 第二次握手 | 服务端 | SYN+ACK（同步 + 确认） | 回复 `SYN=1，SEQ=y；ACK=1，确认号=x+1`（确认收到客户端的 SYN，同时发送自己的 SYN） | 服务端→`SYN_RCVD`；客户端收到后确认 “自己发收正常，服务端发收正常” |
| 第三次握手 | 客户端 | ACK（确认） | 发送 `ACK=1，确认号=y+1`（确认收到服务端的 SYN） | 客户端→`ESTABLISHED`；服务端收到后→`ESTABLISHED`，连接建立 |

### **（3）为什么是三次不是两次？**
- **两次握手的问题**：无法确认客户端的接收能力。若服务端的 SYN+ACK 报文丢失，客户端不知道服务端已准备好，服务端会一直等待客户端的 ACK，造成资源浪费。
- **四次握手的问题**：三次握手已能完成双向能力确认，四次会增加不必要的网络开销，降低效率。

### **（4）面试加分点**
三次握手的**第三次报文可以携带数据**（客户端发送 ACK 时，可同时发送 HTTP 请求）
## **14、TCP 四次挥手的过程？为什么是四次不是三次？**
TCP 是全双工通信，关闭连接时需**分别关闭两个方向的传输通道**，因此需要四次挥手：
### **（1）四次挥手详细过程**

| **挥手次数** | **发起方** | **报文标志** | **核心动作** | **状态变化** |
| --- | --- | --- | --- | --- |
| 第一次挥手 | 客户端 | FIN（结束） | 发送 `FIN=1，SEQ=x`（表示客户端无数据要发，请求关闭客户端→服务端的通道） | 客户端→`FIN_WAIT_1` |
| 第二次挥手 | 服务端 | ACK（确认） | 发送 `ACK=1，确认号=x+1`（确认收到 FIN，此时客户端→服务端通道半关闭） | 服务端→`CLOSE_WAIT`；客户端收到后→`FIN_WAIT_2` |
| 第三次挥手 | 服务端 | FIN（结束） | 服务端数据发送完毕，发送 `FIN=1，SEQ=y`（请求关闭服务端→客户端的通道） | 服务端→`LAST_ACK` |
| 第四次挥手 | 客户端 | ACK（确认） | 发送 `ACK=1，确认号=y+1`（确认收到 FIN） | 客户端→`TIME_WAIT`；服务端收到后→`CLOSED` |

### **（2）为什么是四次不是三次？**
TCP 全双工特性决定了**关闭通道需要双向确认**。服务端收到客户端的 FIN 时，可能还有数据要发，因此需要先回复 ACK 确认关闭客户端通道，等数据发完再发 FIN 关闭自己的通道 —— 这两步无法合并，因此必须四次挥手。
### **（3）核心考点：TIME_WAIT 状态**
- 客户端第四次挥手后进入 `TIME_WAIT` 状态，持续 **2MSL**（MSL 是报文最大生存时间，默认 1-2 分钟）。
- 作用：① 确保最后一个 ACK 送达（服务端没收到会重发 FIN）；② 避免旧连接的报文干扰新连接（2MSL 足够让旧报文失效）。

## **15、HTTP 基于 TCP 还是 UDP？**
- **HTTP/1.0、HTTP/1.1、HTTP/2.0**：**基于 TCP 协议**，因为需要可靠传输（网页资源不能丢包）。
- **HTTP/3.0**：**基于 UDP 协议**，通过 QUIC 协议实现可靠传输，解决 TCP 队头阻塞问题，降低连接延迟。

**核心考点**：HTTP/3 改用 UDP 的原因是**TCP 队头阻塞**—— 一个 TCP 连接中，若一个数据包丢失，后续所有数据包都要等待重传，而 UDP 的 QUIC 协议支持**多路复用 + 独立重传**，大幅提升性能。
## **16、TCP 如何保证传输的可靠性？（面试核心）**
TCP 通过**六大机制**实现可靠传输，可按 “防丢、防错、防乱、防拥塞” 的逻辑记忆：
15. **序列号与确认应答**：每个字节都有序列号，接收方收到后发送 ACK 确认（确认号 = 期望收到的下一字节序号），未收到则重传。
16. **超时重传**：发送方发送报文后启动定时器，超时未收到 ACK 则重传（超时时间动态调整，基于 RTT）。
17. **流量控制**：通过**滑动窗口机制**，接收方告知自己的接收窗口大小（rwnd），发送方按 rwnd 控制发送速率，避免接收方缓冲区溢出。
18. **拥塞控制**：通过**慢启动、拥塞避免、快速重传、快速恢复**四大算法，根据网络状况动态调整拥塞窗口（cwnd），避免网络拥塞。
19. **校验和**：TCP 报文头和数据都带校验和，接收方校验失败则丢弃报文，触发重传。
20. **乱序重排与去重**：接收方按序列号重组报文，丢弃重复报文。

**核心考点**：流量控制是**端到端**（针对接收方缓冲区），拥塞控制是**全局性**（针对网络状况）；实际发送窗口 = min (cwnd, rwnd)。
## **17、TCP 的拥塞控制是怎么实现的？**
拥塞控制是 TCP 的核心难点，目的是**防止发送方速率过快导致网络拥塞**，核心是维护两个参数：**拥塞窗口（cwnd）** 和**慢开始门限（ssthresh）**。
### **（1）四大核心算法**

| **算法** | **触发条件** | **核心逻辑** |
| --- | --- | --- |
| **慢开始** | 连接建立 / 超时重传后 | cwnd 从 1 开始，每收到一个 ACK 就翻倍（指数增长），直到 cwnd=ssthresh，进入拥塞避免 |
| **拥塞避免** | cwnd >= ssthresh | cwnd 每经过一个 RTT 增加 1（线性增长），缓慢提升速率，避免拥塞 |
| **快速重传** | 收到 3 个重复 ACK | 无需等待超时，直接重传丢失的报文段（减少重传延迟） |
| **快速恢复** | 触发快速重传后 | 将 ssthresh 设为当前 cwnd 的一半，cwnd=ssthresh，直接进入拥塞避免（避免流量骤降） |

### **（2）特殊情况：超时重传（严重拥塞）**
若发送方超时未收到 ACK，判定为网络严重拥塞：
- 将 ssthresh 设为当前 cwnd 的一半；
- cwnd 重置为 1，重新进入慢开始阶段。

## **18、ARQ 协议了解吗？**
ARQ（Automatic Repeat-reQuest，自动重传请求）是**实现可靠传输的核心协议**，分为两类，适用于不同场景：
### **（1）停止等待 ARQ（简单但效率低）**
- **核心逻辑**：发送方发完一个分组后，停止发送，等待接收方 ACK 确认；超时未收到则重传。
- **优点**：实现简单；
- **缺点**：信道利用率极低（发送方大部分时间在等待）。
- **适用场景**：低带宽、低延迟的链路（如早期卫星通信）。

### **（2）连续 ARQ（高效，TCP 采用）**
- **核心逻辑**：发送方维护**发送窗口**，窗口内的分组可连续发送，无需等待 ACK；接收方采用**累计确认**（对按序到达的最后一个分组发送 ACK）。
- **优点**：信道利用率高；
- **缺点**：若中间分组丢失，需重传窗口内所有后续分组（称为**回退 N 步**，Go-Back-N）。
- **优化方案**：选择性重传 ARQ（只重传丢失的分组，不重传后续分组，TCP 的 SACK 选项实现此功能）。

**核心考点**：TCP 的重传机制就是基于**连续 ARQ + 选择性重传**实现的。
## **19、超时重传如何实现？超时重传时间怎么确定？**
超时重传是 ARQ 协议的核心，关键是**确定合理的重传超时时间（RTO）**：
21. **核心参数定义**
    - **RTT**：往返时间，数据包从发送到收到 ACK 的时间；
    - **RTO**：重传超时时间，超过此时间未收到 ACK 则重传。
22. **RTO 的确定方法**RTO 不能直接等于 RTT（网络波动会导致 RTT 变化），TCP 采用**加权移动平均（EWMA）** 算法计算**平滑往返时间（SRTT）**：*SRTT*=(1−*α*)×*SRTT*+*α*×*RTT*其中 *α* 是权重（通常取 0.125），目的是降低瞬时 RTT 波动的影响。最终 RTO = SRTT + 4×RTT 偏差（RTTVAR），确保 RTO 略大于正常 RTT，减少不必要的重传。

## **20、ARP 协议详解（网络层核心）**
ARP（Address Resolution Protocol，地址解析协议）的核心功能是**将 IP 地址解析为 MAC 地址**，解决 “网络层 IP 地址” 到 “数据链路层 MAC 地址” 的映射问题。
### **（1）ARP 工作流程（以局域网内通信为例）**
假设主机 A（IP：192.168.1.1）要给主机 B（IP：192.168.1.2）发送数据：
23. 主机 A 查询**本地 ARP 缓存表**，若有 B 的 IP-MAC 映射，直接使用；
24. 若无，主机 A 发送**ARP 请求广播包**（目标 MAC 为全 F：FF:FF:FF:FF:FF:FF），内容为 “谁是 192.168.1.2？请告诉 192.168.1.1 的 MAC 地址”；
25. 局域网内所有主机收到广播包，只有主机 B 匹配 IP 地址，发送**ARP 响应单播包**，内容为 “192.168.1.2 的 MAC 是 xx:xx:xx:xx:xx:xx”；
26. 主机 A 收到响应，将 IP-MAC 映射存入 ARP 缓存表（默认存活时间 2 分钟），后续通信直接使用 MAC 地址。

### **（2）ARP 跨网段通信（网关转发）**
若主机 A 要访问外网主机 C（IP：2.2.2.2），流程如下：
27. 主机 A 发现 C 不在本网段，将数据包发送给**网关**（IP：192.168.1.100）；
28. 主机 A 通过 ARP 解析网关的 MAC 地址，将数据包发送给网关；
29. 网关通过路由表找到 C 的路径，转发数据包（网关与外网通信时，会再次使用 ARP 解析下一跳的 MAC 地址）。

### **（3）ARP 欺骗原理与防范**
- **欺骗原理**：攻击者发送伪造的 ARP 响应包，篡改目标主机的 ARP 缓存表，将网关的 IP 映射到攻击者的 MAC 地址，劫持流量（中间人攻击）。
- **防范方法**：① 静态绑定 IP-MAC 映射；② 开启 ARP 防火墙；③ 交换机开启 DHCP Snooping，过滤伪造 ARP 包。

### **21、什么是 TCP 粘包 / 拆包？原因和解决方法？**
- **粘包**：多个数据包被合并成一个大数据包发送，接收方无法区分边界；
- **拆包**：一个大数据包被拆分成多个小数据包发送，接收方需要重组。

### **（1）产生原因**
30. TCP 是**字节流协议**，无消息边界；
31. 发送方的 Nagle 算法（合并小数据包发送，提高效率）；
32. 接收方的缓冲区大小限制（缓冲区未满时，不会立即向上层交付数据）。

### **（2）解决方法（核心是添加消息边界）**
33. **固定长度**：每个数据包长度固定，接收方按固定长度读取；
34. **添加分隔符**：在数据包末尾添加特殊分隔符（如`\r\n`）；
35. **消息头 + 消息体**：消息头包含数据包总长度，接收方先读长度再读数据（主流方案）。

### **22、HTTPS 的加密过程？TLS 握手步骤？**
HTTPS = HTTP + TLS/SSL，核心是**非对称加密协商对称密钥，对称加密传输数据**，TLS 握手步骤如下：
36. **客户端问候**：发送支持的 TLS 版本、加密套件列表、随机数`Client Random`；
37. **服务端问候**：选择 TLS 版本和加密套件，发送随机数`Server Random`、**服务器证书**（含公钥）；
38. **客户端验证证书**：验证证书合法性（通过 CA 根证书），生成**预主密钥**，用服务器公钥加密后发送；
39. **服务端解密预主密钥**：用私钥解密得到预主密钥；
40. **生成会话密钥**：双方用 `Client Random + Server Random + Pre-Master Secret` 生成**相同的对称会话密钥**；
41. **双向验证就绪**：双方发送加密的`Finished`报文，验证加密通道可用，后续 HTTP 数据用会话密钥加密传输。

**核心考点**：非对称加密（慢，用于协商密钥）+ 对称加密（快，用于传输数据）的组合，兼顾安全性和效率。
