# Java并发

> 重要性等级：S=至尊（必考，频率>80%） A=高级（高频，频率50%-80%） B=基础（中频，频率30%-50%） C=普通（低频，<30%）

## 1、基础知识

### 1.1 进程和线程区别？ [重要性:A]
**进程是程序的一次执行过程，是系统运行程序的基本单位，也是资源分配的基本单位。**一个进程在其执行的过程中可以产生多个线程。与进程不同的是**同类的多个线程共享进程的堆和方法区资源，但每个线程有自己的程序计数器、虚拟机栈和本地方法栈**，所以系统在产生一个线程，或是在各个线程之间做切换工作时，负担要比进程小得多，也正因为如此，**线程也被称为轻量级进程**。
### 1.2 说说线程的生命周期和状态？ [重要性:S]
Java 线程在运行的生命周期中的指定时刻只可能处于下面 6 种不同状态的其中一个状态：
- **NEW**: 初始状态，线程被创建出来但没有被调用 start () 。
- **RUNNABLE**: 运行状态，线程被调用了 start () 等待运行的状态。
- **BLOCKED**：阻塞状态，需要等待锁释放。
- **WAITING**：等待状态，表示该线程需要等待其他线程做出一些特定动作（通知或中断）。
- **TIME_WAITING**：超时等待状态，可以在指定的时间后自行返回而不是像 WAITING 那样一直等待。
- **TERMINATED**：终止状态，表示该线程已经运行完毕。

### 1.3 乐观锁和悲观锁 [重要性:A]
### 1.4 什么是悲观锁？ [重要性:A]
**悲观锁总是假设最坏的情况，认为共享资源每次被访问的时候就会出现问题**(比如共享数据被修改)，所以每次在获取资源操作的时候都会上锁，这样其他线程想拿到这个资源就会阻塞直到锁被上一个持有者释放。也就是说，共享资源每次只给一个线程使用，其它线程阻塞，用完后再把资源转让给其它线程。**像 Java 中 synchronized 和 ReentrantLock 等独占锁就是悲观锁思想的实现**。
高并发的场景下，激烈的锁竞争会造成线程阻塞，大量阻塞线程会导致系统的上下文切换，增加系统的性能开销。并且，悲观锁还可能会存在死锁问题，影响代码的正常运行。
### 1.5 什么是乐观锁？ [重要性:A]
**乐观锁总是假设最好的情况，认为共享资源每次被访问的时候不会出现问题**，线程可以不停地执行，无需加锁也无需等待，只是在提交修改的时候去验证对应的资源（也就是数据）是否被其它线程修改了（具体方法可以使用版本号机制或 CAS 算法）。
但是，如果冲突频繁发生（写占比非常多的情况），会频繁失败和重试，这样同样会非常影响性能，导致 CPU 飙升。
- **悲观锁通常多用于写比较多的情况（多写场景，竞争激烈）**，这样可以避免频繁失败和重试影响性能，悲观锁的开销是固定的。不过，如果乐观锁解决了频繁失败和重试这个问题的话（比如 LongAdder），也是可以考虑使用乐观锁的，要视实际情况而定。
- **乐观锁通常多用于写比较少的情况（多读场景，竞争较少）**，这样可以避免频繁加锁影响性能。不过，乐观锁主要针对的对象是单个共享变量（参考 java.util.concurrent.atomic 包下面的原子变量类）。

### 1.6 怎么实现乐观锁 [重要性:S]
**版本号机制**
**一般是在数据表中加上一个数据版本号 version 字段，表示数据被修改的次数。当数据被修改时，version 值会加一。** 当线程 A 要更新数据值时，在读取数据的同时也会读取 version 值，**在提交更新时，若刚才读取到的 version 值为当前数据库中的 version 值相等时才更新，否则重试更新操作，直到更新成功**。
**CAS 算法**
**CAS 的全称是 Compare And Swap（比较与交换） ，用于实现乐观锁，被广泛应用于各大框架中。** CAS 的思想很简单，**就是用一个预期值和要更新的变量值进行比较，两值相等才会进行更新**。
**ABA 问题**
如果一个变量 V 初次读取的时候是 A 值，并且在准备赋值的时候检查到它仍然是 A 值，那我们就能说明它的值没有被其他线程修改过了吗？很明显是不能的，因为在这段时间它的值可能被改为其他值，然后又改回 A，那 CAS 操作就会误认为它从来没有被修改过。**这个问题被称为 CAS 操作的 "ABA" 问题**。
**ABA 问题的解决思路是在变量前面追加上版本号或者时间戳。**
### 1.7 synchronized 底层原理了解吗？ [重要性:S]
**synchronized 同步语句块的实现使用的是 monitorenter 和 monitorexit 指令**，其中 monitorenter 指令指向同步代码块的开始位置，monitorexit 指令则指明同步代码块的结束位置。
当执行 monitorenter 指令时，**线程试图获取锁也就是获取 对象监视器 monitor 的持有权**。
在执行 monitorenter 时，会尝试获取对象的锁，**如果锁的计数器为 0 则表示锁可以被获取，获取后将锁计数器设为 1 也就是加 1**。**对象锁的拥有者线程才可以执行 monitorexit 指令来释放锁**。在执行 monitorexit 指令后，将锁计数器设为 0，表明锁被释放，其他线程可以尝试获取锁。
### 1.8 ⭐️synchronized 和 volatile 有什么区别？ [重要性:S]
**synchronized 关键字和 volatile 关键字是两个互补的存在，而不是对立的存在！**
- **volatile 关键字是线程同步的轻量级实现，所以 volatile 性能肯定比 synchronized 关键字要好 。但是 volatile 关键字只能用于变量而 synchronized 关键字可以修饰方法以及代码块 。**
- **volatile 关键字能保证数据的可见性，但不能保证数据的原子性。synchronized 关键字两者都能保证。**
- **volatile 关键字主要用于解决变量在多个线程之间的可见性，而 synchronized 关键字解决的是多个线程之间访问资源的同步性。**

### 1.9 ReentrantLock 是什么？ [重要性:A]
**ReentrantLock 实现了 Lock 接口，是一个可重入且独占式的锁，和 synchronized 关键字类似。** 不过，**ReentrantLock 更灵活、更强大，增加了轮询、超时、中断、公平锁和非公平锁等高级功能**。
### 1.10 公平锁和非公平锁有什么区别？ [重要性:A]
- **公平锁**: 锁被释放之后，先申请的线程先得到锁。性能较差一些，因为公平锁为了保证时间上的绝对顺序，上下文切换更频繁。
- **非公平锁**：锁被释放之后，后申请的线程可能会先获取到锁，是随机或者按照其他优先级排序的。性能更好，但可能会导致某些线程永远无法获取到锁。

### 1.11 synchronized 和 ReentrantLock 有什么区别？ [重要性:S]
i. **两者都是可重入锁**可重入锁 也叫递归锁，指的是线程可以再次获取自己的内部锁。比如一个线程获得了某个对象的锁，此时这个对象锁还没有释放，当其再次想要获取这个对象的锁的时候还是可以获取的，如果是不可重入锁的话，就会造成死锁。
ii. **synchronized 依赖于 JVM 而 ReentrantLock 依赖于 API  **synchronized 是依赖于 JVM 实现的，前面我们也讲到了 虚拟机团队在 JDK1.6 为 synchronized 关键字进行了很多优化，但是这些优化都是在虚拟机层面实现的，并没有直接暴露给我们。ReentrantLock 是 JDK 层面实现的（也就是 API 层面，需要 lock () 和 unlock () 方法配合 try/finally 语句块来完成）
iii. **ReentrantLock 比 synchronized 增加了一些高级功能**
- **等待可中断**: ReentrantLock 提供了一种能够中断等待锁的线程的机制，通过 lock.lockInterruptibly () 来实现这个机制。也就是说当前线程在等待获取锁的过程中，如果其他线程中断当前线程「 interrupt () 」，当前线程就会抛出 InterruptedException 异常，可以捕捉该异常进行相应处理。
- **可实现公平锁**: ReentrantLock 可以指定是公平锁还是非公平锁。而 synchronized 只能是非公平锁。所谓的公平锁就是先等待的线程先获得锁。ReentrantLock 默认情况是非公平的，可以通过 ReentrantLock 类的 ReentrantLock (boolean fair) 构造方法来指定是否是公平的。
- **可实现选择性通知（锁可以绑定多个条件）**: synchronized 关键字与 wait () 和 notify ()/notifyAll () 方法相结合可以实现等待 / 通知机制。ReentrantLock 类当然也可以实现，但是需要借助于 Condition 接口与 newCondition () 方法。
- **支持超时**：ReentrantLock 提供了 tryLock (timeout) 的方法，可以指定等待获取锁的最长等待时间，如果超过了等待时间，就会获取锁失败，不会一直等待。

### 1.12 可中断锁和不可中断锁有什么区别？ [重要性:B]
- **可中断锁**：获取锁的过程中可以被中断，不需要一直等到获取锁之后 才能进行其他逻辑处理。ReentrantLock 就属于是可中断锁。
- **不可中断锁**：一旦线程申请了锁，就只能等到拿到锁以后才能进行其他的逻辑处理。 synchronized 就属于是不可中断锁。

### 1.13 ThreadLocal [重要性:S]
**ThreadLocal 类允许每个线程绑定自己的值，可以将其形象地比喻为一个 “存放数据的盒子”。每个线程都有自己独立的盒子，用于存储私有数据，确保不同线程之间的数据互不干扰，从而避免了线程安全问题。**
### 1.14 ThreadLocal 原理了解吗？ [重要性:S]
**变量是放在了当前线程的 ThreadLocalMap 中，并不是存在 ThreadLocal 上，ThreadLocal 可以理解为只是 ThreadLocalMap 的封装，传递了变量值。** ThrealLocal 类中可以通过 Thread.currentThread () 获取到当前线程对象后，直接通过 getMap (Thread t) 可以访问到该线程的 ThreadLocalMap 对象。
**每个 Thread 中都具备一个 ThreadLocalMap，而 ThreadLocalMap 可以存储以 ThreadLocal 为 key ，Object 对象为 value 的键值对。**比如我们在同一个线程中声明了两个 ThreadLocal 对象的话， Thread 内部都是使用仅有的那 ThreadLocalMap 存放数据的，ThreadLocalMap 的 key 就是 ThreadLocal 对象，value 就是 ThreadLocal 对象调用 set 方法设置的值。
### 1.15 死锁 [重要性:A]
### 1.16 什么是线程死锁？ [重要性:A]
**线程死锁描述的是这样一种情况：多个线程同时被阻塞，它们中的一个或者全部都在等待某个资源被释放。由于线程被无限期地阻塞，因此程序不可能正常终止。**
如下图所示，线程 A 持有资源 2，线程 B 持有资源 1，他们同时都想申请对方的资源，所以这两个线程就会互相等待而进入死锁状态。
### 1.17 如何检测死锁？ [重要性:A]
1. 使用 `jmap`、`jstack` 等命令查看 JVM 线程栈和堆内存情况。如果有死锁，`jstack` 的输出中通常会出现 `Found one Java-level deadlock:` 这类提示。
2. 打开 `jconsole` 并连接目标程序，然后进入线程界面选择“检测死锁”。
### 1.18 产生死锁的四个必要条件： [重要性:A]
20. **互斥条件**：该资源任意一个时刻只由一个线程占用。
21. **请求与保持条件**：一个线程因请求资源而阻塞时，对已获得的资源保持不放。
22. **不剥夺条件**：线程已获得的资源在未使用完之前不能被其他线程强行剥夺，只有自己使用完毕后才释放资源。
23. **循环等待条件**：若干线程之间形成一种头尾相接的循环等待资源关系。

### 1.19 如何预防和避免线程死锁？ [重要性:A]
如何预防死锁？ **破坏死锁的产生的必要条件即可**：
24. **破坏请求与保持条件**：一次性申请所有的资源。
25. **破坏不剥夺条件**：占用部分资源的线程进一步申请其他资源时，如果申请不到，可以主动释放它占有的资源。
26. **破坏循环等待条件**：靠按序申请资源来预防。按某一顺序申请资源，释放资源则反序释放。破坏循环等待条件。

### 1.20 单核 CPU 上运行多个线程效率一定会高吗？ [重要性:B]
单核 CPU 同时运行多个线程的效率是否会高，**取决于线程的类型和任务的性质**。一般来说，有两种类型的线程：
27. **CPU 密集型**：CPU 密集型的线程主要进行计算和逻辑处理，需要占用大量的 CPU 资源。
28. **IO 密集型**：IO 密集型的线程主要进行输入输出操作，如读写文件、网络通信等，需要等待 IO 设备的响应，而不占用太多的 Cpu.

对于单核 CPU 来说，**如果任务是 CPU 密集型的，那么开很多线程会影响效率；如果任务是 IO 密集型的，那么开很多线程会提高效率**。当然，这里的 “很多” 也要适度，不能超过系统能够承受的上限 PU 资源。
### 1.21 ThreadLocal 内存泄露问题是怎么导致的？ [重要性:S]
**ThreadLocal 内存泄漏的根本原因在于其内部实现机制。**通过上面的内容我们已经知道：每个线程维护一个名为 ThreadLocalMap 的 map。 当你使用 ThreadLocal 存储值时，实际上是将值存储在当前线程的 ThreadLocalMap 中，其中 ThreadLocal 实例本身作为 key，而你要存储的值作为 value。
**ThreadLocalMap 的 key 和 value 引用机制**：
- **key 是弱引用**：  这意味着，如果 ThreadLocal 实例不再被任何强引用指向，垃圾回收器会在下次 GC 时回收该实例，导致 ThreadLocalMap 中对应的 key 变为 null。
-  **value 是强引用**：即使 key 被 GC 回收，value 仍然被 ThreadLocalMap.Entry 强引用存在，无法被 GC 回收。

当 ThreadLocal 实例失去强引用后，其对应的 value 仍然存在于 ThreadLocalMap 中，因为 Entry 对象强引用了它。**如果线程持续存活（例如线程池中的线程），ThreadLocalMap 也会一直存在，导致 key 为 null 的 entry 无法被垃圾回收，即会造成内存泄漏。**
也就是说，**内存泄漏的发生需要同时满足两个条件**：
29. ThreadLocal 实例不再被强引用；
30. 线程持续存活，导致 ThreadLocalMap 长期存在。

**如何避免内存泄漏的发生？**
31. **在使用完 ThreadLocal 后，务必调用 remove () 方法。 这是最安全和最推荐的做法。** remove () 方法会从 ThreadLocalMap 中显式地移除对应的 entry，彻底解决内存泄漏的风险。 即使将 ThreadLocal 定义为 static final，也强烈建议在每次使用后调用 remove ()。
32. **在线程池等线程复用的场景下，使用 try-finally 块可以确保即使发生异常，remove () 方法也一定会被执行。**

### 1.22 ThreadLocalMap 的 key 和 value 引用机制设计原因 [重要性:A]
ThreadLocalMap 对 key（ThreadLocal 实例）采用**弱引用**、对 value（线程私有数据）采用**强引用**的设计，是 JDK 开发团队在「内存占用」和「数据安全性」之间做出的权衡，核心目的是**避免 ThreadLocal 实例本身的内存泄漏**，同时尽可能保证线程私有数据的有效性，具体原因拆解如下：
**一、key 采用弱引用的核心原因：避免 ThreadLocal 实例内存泄漏**
**1. 解决 ThreadLocal 生命周期与线程生命周期不匹配的问题**
线程（尤其是线程池中的核心线程）的生命周期通常远长于 ThreadLocal 实例的生命周期：
若 ThreadLocal 是局部变量（如方法内创建），当方法执行完毕，ThreadLocal 实例的强引用会被销毁；
若 ThreadLocalMap 的 key 是**强引用**，则 ThreadLocal 实例会被 ThreadLocalMap 强引用持有，即使外部已无引用，GC 也无法回收该 ThreadLocal 实例，导致 ThreadLocal 实例内存泄漏。
**2. 弱引用的特性适配 ThreadLocal 的使用场景**
弱引用的特点是：只要发生 GC，无论内存是否充足，弱引用指向的对象都会被回收。将 key 设为弱引用后，当 ThreadLocal 实例失去外部强引用时，GC 会回收该 ThreadLocal 实例，ThreadLocalMap 中对应的 key 会变为 null，此时 JVM 可识别该 entry 为 “无效 entry”，后续通过 ThreadLocalMap 的「清理机制」（如 get/set/remove 时的 expungeStaleEntry 方法）清理这些 key 为 null 的 entry，从而避免 ThreadLocal 实例本身的内存泄漏。
**二、value 采用强引用的核心原因：保证线程私有数据的有效性**
**1. 避免数据被意外回收，保证业务逻辑正确性**
value 是线程绑定的私有数据（如用户上下文、数据库连接），是 ThreadLocal 的核心使用目标：
若 value 采用弱引用，当外部无其他引用时，GC 会直接回收 value，即使 ThreadLocal 实例仍在使用，线程也会丢失关键数据，导致业务逻辑异常；
强引用可确保 value 仅在「线程销毁」或「主动调用 ThreadLocal.remove ()」时被回收，符合 ThreadLocal “线程私有数据存储” 的设计初衷。
**2. 平衡 “内存泄漏风险” 与 “易用性”**
虽然 value 强引用可能导致「key 为 null 但 value 仍存在」的内存泄漏（即 ThreadLocal 内存泄漏的核心场景），但 JDK 认为：
这种泄漏是 “可避免的”：开发者只需在使用完 ThreadLocal 后调用 remove () 方法，即可显式清理 value；
若为了避免该泄漏而将 value 设为弱引用，会导致 ThreadLocal 失去实际使用价值（数据随时可能被回收），得不偿失。
**三、设计权衡的补充说明**
**为什么不直接将 value 也设为弱引用？**弱引用的价值是 “自动回收无外部引用的对象”，但 ThreadLocal 存储的 value 本身就是线程私有数据，外部通常无其他引用（唯一引用就是 ThreadLocalMap），若设为弱引用，value 会被 GC 直接回收，ThreadLocal 无法实现 “线程绑定数据” 的核心功能。
**为什么不通过其他方式（如软引用）设计 key？**软引用的回收规则是 “内存不足时才回收”，无法解决 ThreadLocal 实例在内存充足时的泄漏问题；而弱引用的 “GC 必回收” 特性，能最快释放 ThreadLocal 实例占用的内存，是最适配的选择。
**JDK 对 value 泄漏的兜底措施**ThreadLocalMap 内置了「过期 entry 清理机制」：在调用 get ()、set ()、remove () 方法时，会扫描并清理 key 为 null 的 entry，释放其对应的 value 引用，尽可能降低 value 泄漏的风险（但无法完全避免，仍需开发者主动调用 remove ()）。
**总结**
ThreadLocalMap 的引用机制设计核心是：
对 key 用弱引用：优先保证 ThreadLocal 实例本身不泄漏，是 “基础层” 的内存安全保障；
对 value 用强引用：优先保证业务数据的有效性，将 “避免 value 泄漏” 的责任部分交给开发者（主动 remove ()），是 “功能层” 的实用性保障。这种设计既最大化利用了 GC 的自动回收能力，又兼顾了 ThreadLocal 的核心使用场景，是 “技术约束” 和 “业务需求” 平衡的结果。

## 2、线程池基础

### 2.1 如何跨线程传递 ThreadLocal 的值？ [重要性:A]
1、简称 TTL是阿里巴巴开源的工具类，继承并加强了 InheritableThreadLocal 类，**可以在线程池的场景下支持 ThreadLocal 值传递**
2、**InheritableThreadLocal**：InheritableThreadLocal 是 JDK1.2 提供的工具，继承自 ThreadLocal 。**使用 InheritableThreadLocal 时，会在创建子线程时，令子线程继承父线程中的 ThreadLocal 值，但是无法支持线程池场景下的 ThreadLocal 值传递。**
### 2.2 如何创建线程池？ [重要性:S]
**方式一：通过 ThreadPoolExecutor 构造函数直接创建 (推荐)**这是最推荐的方式，因为它允许开发者明确指定线程池的核心参数，对线程池的运行行为有更精细的控制，从而避免资源耗尽的风险。
**方式二：通过 Executors 工具类创建 (不推荐用于生产环境)**通过 Executors 工具类可以创建多种类型的线程池，包括：
- **FixedThreadPool**：固定线程数量的线程池。该线程池中的线程数量始终不变。当有一个新的任务提交时，线程池中若有空闲线程，则立即执行。若没有，则新的任务会被暂存在一个任务队列中，待有线程空闲时，便处理在任务队列中的任务。
- **SingleThreadExecutor**： 只有一个线程的线程池。若多余一个任务被提交到该线程池，任务会被保存在一个任务队列中，待线程空闲，按先入先出的顺序执行队列中的任务。
- **CachedThreadPool**： 可根据实际情况调整线程数量的线程池。线程池的线程数量不确定，但若有空闲线程可以复用，则会优先使用可复用的线程。若所有线程均在工作，又有新的任务提交，则会创建新的线程处理任务。所有线程在当前任务执行完毕后，将返回线程池进行复用。
- **ScheduledThreadPool**：给定的延迟后运行任务或者定期执行任务的线程池

### 2.3 为什么不推荐使用内置线程池？ [重要性:S]
**《阿里巴巴 Java 开发手册》中强制线程池不允许使用 Executors 去创建，而是通过 ThreadPoolExecutor 构造函数的方式，这样的处理方式让写的同学更加明确线程池的运行规则，规避资源耗尽的风险（使用有界队列，控制线程创建数量）**
Executors 返回线程池对象的弊端如下 (后文会详细介绍到)：
- **FixedThreadPool 和 SingleThreadExecutor**: 使用的是阻塞队列 LinkedBlockingQueue，任务队列最大长度为 Integer.MAX_VALUE，可以看作是无界的，可能堆积大量的请求，从而导致 OOM。
- **CachedThreadPool**: 使用的是同步队列 SynchronousQueue, 允许创建的线程数量为 Integer.MAX_VALUE ，如果任务数量过多且执行速度较慢，可能会创建大量的线程，从而导致 OOM。
- **ScheduledThreadPool 和 SingleThreadScheduledExecutor**: 使用的无界的延迟阻塞队列 DelayedWorkQueue，任务队列最大长度为 Integer.MAX_VALUE, 可能堆积大量的请求，从而导致 OOM。

### 2.4 线程池常见参数有哪些？如何解释？ [重要性:S]
**ThreadPoolExecutor 3 个最重要的参数**：
- **corePoolSize**: 任务队列未达到队列容量时，最大可以同时运行的线程数量。
- **maximumPoolSize**: 任务队列中存放的任务达到队列容量的时候，当前可以同时运行的线程数量变为最大线程数。
- **workQueue**: 新任务来的时候会先判断当前运行的线程数量是否达到核心线程数，如果达到的话，新任务就会被存放在队列中。

**ThreadPoolExecutor 其他常见参数**:
- **keepAliveTime**: 当线程池中的线程数量大于 corePoolSize ，即有非核心线程（线程池中核心线程以外的线程）时，这些非核心线程空闲后不会立即销毁，而是会等待，直到等待的时间超过了 keepAliveTime 才会被回收销毁。
- **unit**: keepAliveTime 参数的时间单位。
- **threadFactory**:executor 创建新线程的时候会用到。
- **handler**: 拒绝策略（后面会单独详细介绍一下）

### 2.5 线程池的核心线程会被回收吗？ [重要性:A]
**ThreadPoolExecutor 默认不会回收核心线程，即使它们已经空闲了。** 这是为了减少创建线程的开销，因为核心线程通常是要长期保持活跃的。但是，如果线程池是被用于周期性使用的场景，且频率不高（周期之间有明显的空闲时间），**可以考虑将 allowCoreThreadTimeOut (boolean value) 方法的参数设置为 true，这样就会回收空闲（时间间隔由 keepAliveTime 指定）的核心线程了。**
### 2.6 核心线程空闲时处于什么状态？ [重要性:B]
核心线程空闲时，其状态分为以下两种情况：
- **设置了核心线程的存活时间**：核心线程在空闲时，会处于 WAITING 状态，等待获取任务。如果阻塞等待的时间超过了核心线程存活时间，则该线程会退出工作，将该线程从线程池的工作线程集合中移除，线程状态变为 TERMINATED 状态。
- **没有设置核心线程的存活时间**：核心线程在空闲时，会一直处于 WAITING 状态，等待获取任务，核心线程会一直存活在线程池中。

当队列中有可用任务时，会唤醒被阻塞的线程，线程的状态会由 WAITING 状态变为 RUNNABLE 状态，之后去执行对应任务。
### 2.7 线程池的拒绝策略有哪些？ [重要性:S]
**线程池的拒绝策略，是指当线程池的核心线程已满、工作队列已满、最大线程数也已满（即线程池达到饱和状态）时，对新提交任务的处理规则。Java 中默认提供 4 种拒绝策略，均实现了 RejectedExecutionHandler 接口**，具体解析如下：
33. **AbortPolicy（默认策略）**核心逻辑：直接抛出 RejectedExecutionException 异常，明确告知任务被拒绝，中断任务提交流程。适用场景：要求 “任务必须被处理” 或 “不允许任务丢失” 的场景（如核心业务逻辑），通过异常捕获可及时感知线程池饱和，进而做降级、重试等处理。特点：fail-fast（快速失败），不静默丢弃任务，便于问题排查，但需手动处理异常。
34. **CallerRunsPolicy（调用者执行策略）**核心逻辑：不丢弃任务，也不抛出异常，而是将任务回退给提交任务的调用线程（如主线程）执行。适用场景：任务处理压力不大、允许调用线程 “分担” 执行压力的场景（如非核心异步任务），可避免任务丢失，同时减缓新任务提交速度（调用线程执行任务时，无法继续提交新任务，间接起到 “限流” 作用）。特点：缓冲式处理，无任务丢失，但可能阻塞调用线程，影响调用者的正常逻辑。
35. **DiscardPolicy（静默丢弃策略）**核心逻辑：直接丢弃新提交的任务，不抛出任何异常，也不做任何处理，任务 “无声消失”。适用场景：任务无足轻重、丢失不影响业务的场景（如日志采集、非核心统计任务），无需感知线程池饱和，允许静默丢弃。特点：静默失败，无异常但有任务丢失风险，需确保业务能容忍任务丢失。
36. **DiscardOldestPolicy（丢弃最旧任务策略）**核心逻辑：丢弃工作队列中 “最旧” 的任务（即队列头部等待最久的任务），然后将新提交的任务加入队列，尝试再次提交。适用场景：任务有 “时效性”，新任务比旧任务更重要的场景（如实时数据处理，旧数据过期后无意义，优先保留新数据）。特点：替换式丢弃，避免新任务丢失，但会丢失队列中等待最久的任务，需确保旧任务可被替代。

**自定义拒绝策略**除默认策略外，可通过实现 RejectedExecutionHandler 接口自定义逻辑，例如：任务持久化（将拒绝的任务存入数据库 / 消息队列，后续重试）；日志告警（记录被拒绝的任务信息，触发监控告警）；降级处理（返回默认结果，而非执行真实任务）。
### 2.8 线程池处理任务的流程了解吗？ [重要性:S]
**1. 当新任务提交时，判断核心线程池是否已满（当前线程数 < corePoolSize）**，如果未满，则创建核心线程执行任务；如果已满，进入下一步。**2. 判断任务队列是否已满**，如果未满，则将任务加入队列等待执行；如果已满，进入下一步。**3. 判断线程池是否已满（当前线程数 < maximumPoolSize）**，如果未满，则创建非核心线程执行任务；如果已满，进入下一步。**4. 执行拒绝策略**，处理无法提交的任务。
### 2.9 线程池中线程异常后，销毁还是复用？ [重要性:A]
**使用 execute () 时，未捕获异常导致线程终止，线程池创建新线程替代；使用 submit () 时，异常被封装在 Future 中，线程继续复用。**
### 2.10 如何设定线程池的大小？ [重要性:S]
有一个简单并且适用面比较广的公式：
- **CPU 密集型任务 (N+1)**： 这种任务消耗的主要是 CPU 资源，可以将线程数设置为 N（CPU 核心数）+1。比 CPU 核心数多出来的一个线程是为了防止线程偶发的缺页中断，或者其它原因导致的任务暂停而带来的影响。一旦任务暂停，CPU 就会处于空闲状态，而在这种情况下多出来的一个线程就可以充分利用 CPU 的空闲时间。
- **I/O 密集型任务 (2N)**： 这种任务应用起来，系统会用大部分的时间来处理 I/O 交互，而线程在处理 I/O 的时间段内不会占用 CPU 来处理，这时就可以将 CPU 交出给其它线程使用。因此在 I/O 密集型任务的应用中，我们可以多配置一些线程，具体的计算方法是 2N。

### 2.11 如何设计一个能够根据任务的优先级来执行的线程池？ [重要性:A]
这是一个常见的面试问题，本质其实还是在考察求职者对于线程池以及阻塞队列的掌握。
**假如我们需要实现一个优先级任务线程池的话，那可以考虑使用 PriorityBlockingQueue （优先级阻塞队列）作为任务队列（ThreadPoolExecutor 的构造函数有一个 workQueue 参数可以传入任务队列）。**
**PriorityBlockingQueue 是一个支持优先级的无界阻塞队列，可以看作是线程安全的 PriorityQueue，两者底层都是使用小顶堆形式的二叉堆，即值最小的元素优先出队。不过，PriorityQueue 不支持阻塞操作。**
**要想让 PriorityBlockingQueue 实现对任务的排序，传入其中的任务必须是具备排序能力的，方式有两种**：
37. 提交到线程池的任务实现 Comparable 接口，并重写 compareTo 方法来指定任务之间的优先级比较规则。
38. 创建 PriorityBlockingQueue 时传入一个 Comparator 对象来指定任务之间的排序规则 (推荐)。

不过，这存在一些风险和问题，比如：
- **PriorityBlockingQueue 是无界的，可能堆积大量的请求，从而导致 OOM。**
- **可能会导致饥饿问题，即低优先级的任务长时间得不到执行。**
- **由于需要对队列中的元素进行排序操作以及保证线程安全（并发控制采用的是可重入锁 ReentrantLock），因此会降低性能。**

对于 OOM 这个问题的解决比较简单粗暴，**就是继承 PriorityBlockingQueue 并重写一下 offer 方法 (入队) 的逻辑，当插入的元素数量超过指定值就返回 false 。**
饥饿问题这个可以通过优化设计来解决（比较麻烦），比如等待时间过长的任务会被移除并重新添加到队列中，但是优先级会被提升。

## 3、补充知识点
### 3.1 你是如何理解线程安全的？ [重要性:S]
**①、原⼦性**：⼀个操作要么完全执⾏，要么完全不执⾏，不会出现中间状态。
可以通过同步关键字 synchronized 或原⼦操作，如 AtomicInteger 来保证原⼦性。
**②、可⻅性**：当⼀个线程修改了共享变量，其他线程能够⽴即看到变化。
可以通过 volatile 关键字来保证可⻅性。
**③、有序性**：要确保线程不会因为死锁、饥饿、活锁等问题导致⽆法继续执⾏。
### 3.2 如何理解协程？ [重要性:C]
协程被视为⽐线程更轻量级的并发单元，可以在单线程中实现并发执⾏，由我们开发者显式调度。
协程是在⽤户态进⾏调度的，避免了线程切换时的内核态开销。
Java ⾃身是不⽀持携程的，我们可以使⽤ Quasar、Kotlin 等框架来实现协程
### 3.3 线程间是如何进⾏通信的？ [重要性:B]
原则上可以通过消息传递和共享内存两种⽅法来实现。Java 采⽤的是共享内存的并发模型。
⽤⼀句话来概括就是：共享变量存储在主内存中，每个线程的私有本地内存，存储的是这个共享变量的副本。
线程 A 与线程 B 之间如果要通信，通常需要经历两个步骤：
1. 线程 A 把本地内存 A 中的共享变量副本刷新到主内存中。
2. 线程 B 从主内存中读取线程 A 刷新过的共享变量，再同步到自己的工作内存中。
### 3.4 说说线程有⼏种创建⽅式？ [重要性:B]
有三种，分别是继承 Thread 类、实现 Runnable 接⼝、实现 Callable 接⼝。
### 3.5 ⼀个 8G 内存的系统最多能创建多少个线程? [重要性:C]
理论上⼤约 8000 个。
创建线程的时候，⾄少需要分配⼀个虚拟机栈，在 64 位操作系统中，默认⼤⼩为 1M，因此⼀个线程⼤约需要 1M
的内存。
但 JVM、操作系统本身的运⾏就要占⼀定的内存空间，所以实际上可以创建的线程数远⽐ 8000 少。
### 3.6 调⽤ start ⽅法时会执⾏ run ⽅法，那怎么不直接调⽤ run⽅法？ [重要性:B]
**调用 `start()` 会创建一个新的线程，并异步执行 `run()` 方法中的代码。**
直接调用 `run()` 方法只是一个普通的同步方法调用，所有代码都在当前线程中执行，不会创建新线程，因此也就达不到多线程并发的目的。
也就是说，调⽤ start() ⽅法会通知 JVM，去调⽤底层的线程调度机制来启动新线程。
调⽤ start() 后，线程进⼊就绪状态，等待操作系统调度；⼀旦调度执⾏，线程会执⾏其 run() ⽅法中的代码；
### 3.7 线程有哪些常⽤的调度⽅法？ [重要性:B]
⽐如说 start ⽅法⽤于启动线程并让操作系统调度执⾏；sleep ⽅法⽤于让当前线程休眠⼀段时间；wait ⽅法会让当前线程等待，notify 会唤醒⼀个等待的线程。

### 3.8 说说wait⽅法和notify⽅法？ [重要性:A]
1、当线程 A 调⽤共享对象的 wait() ⽅法时，线程 A 会被阻塞挂起，直到：
  ** 1、线程 B 调⽤了共享对象的 notify() ⽅法或者 notifyAll() ⽅法；
   2、其他线程调⽤线程 A 的 interrupt() ⽅法，导致线程 A 抛出 InterruptedException 异常。**
2、线程 A 调⽤共享对象的 wait(timeout)⽅法后，没有在指定的 timeout 时间内被其它线程唤醒，那么这个⽅法会因为超时⽽返回。
3、当线程 A 调⽤共享对象的 notify() ⽅法后，会唤醒⼀个在这个共享对象上调⽤ wait 系列⽅法被挂起的线程。共享对象上可能会有多个线程在等待，具体唤醒哪个线程是随机的。
4、如果调⽤的是 notifyAll ⽅法，会唤醒所有在这个共享变量上调⽤ wait 系列⽅法⽽被挂起的线程
### 3.9 说说 sleep ⽅法？ [重要性:A]
当线程 A 调⽤了 Thread 的 sleep ⽅法后，线程 A 会暂时让出指定时间的执⾏权。
指定的睡眠时间到了后该⽅法会正常返回，接着参与 CPU 调度，获取到 CPU 资源后可以继续执⾏。
### 3.10 说说yield [重要性:C]
yield() ⽅法的⽬的是让当前线程让出 CPU 使⽤权，回到就绪状态。但是线程调度器可能会忽略。礼貌性让步，但是没有其他优先级的线程可能自己继续执行；而sleep肯定会睡眠到规定的时间；
**重点：wait() 会释放锁，这是它与 yield() 和 sleep()主要区别**
### 3.11 说说interrupt⽅法？ [重要性:B]
interrupt() ⽅法⽤于通知线程停⽌，但不会直接终⽌线程，需要线程⾃⾏处理中断标志。
常与 isInterrupted() 或 Thread.interrupted() 配合使⽤。
```javascript

Thread thread = new Thread(() -> {
while (!Thread.currentThread().isInterrupted()) {
System.out.println("Running");
}
System.out.println("Interrupted");
});
thread.start();
thread.interrupt(); // 中断线程
```

### 3.14 什么是线程上下⽂切换？ [重要性:B]
线程上下⽂切换是指 CPU 从⼀个线程切换到另⼀个线程执⾏时的过程。
在线程切换的过程中，CPU 需要保存当前线程的执⾏状态，并加载下⼀个线程的上下⽂。
之所以要这样，是因为 CPU 在同⼀时刻只能执⾏⼀个线程，为了实现多线程并发执⾏，需要不断地在多个线程之间切换。
为了让⽤户感觉多个线程是在同时执⾏的， CPU 资源的分配采⽤了时间⽚轮转的⽅式，线程在时间⽚内占⽤ CPU执⾏任务。当线程使⽤完时间⽚后，就会让出 CPU 让其他线程占⽤。
### 3.15 线程可以被多核调度吗？ [重要性:B]
多核处理器提供了并⾏执⾏多个线程的能⼒。每个核⼼可以独⽴执⾏⼀个或多个线程，操作系统的任务调度器会根据策略和算法，如优先级调度、轮转调度等，决定哪个线程何时在哪个核⼼上运⾏。
### 3.16 守护线程了解吗？ [重要性:C]
了解，守护线程是⼀种特殊的线程，它的作⽤是为其他线程提供服务。
Java 中的线程分为两类，⼀种是守护线程，另外⼀种是⽤户线程。
JVM 启动时会调⽤ main ⽅法，main ⽅法所在的线程就是⼀个⽤户线程。在 JVM 内部，同时还启动了很多守护线程，⽐如垃圾回收线程。
**守护线程和⽤户线程有什么区别呢？**
区别之⼀是当最后⼀个⾮守护线程束时， JVM 会正常退出，不管当前是否存在守护线程，也就是说守护线程是否结束并不影响 JVM 退出。
换⽽⾔之，只要有⼀个⽤户线程还没结束，正常情况下 JVM 就不会退出。
### 3.17 线程间有哪些通信⽅式？ [重要性:B]
线程之间传递信息的⽅式有多种，⽐如说使⽤ volatile 和 synchronized 关键字共享对象、使⽤ wait() 和notify() ⽅法实现⽣产者-消费者模式、使⽤ Exchanger 进⾏数据交换、使⽤ Condition 实现线程间的协调等
### 3.18 简单说说 volatile 和 synchronized 的使⽤⽅式？ [重要性:A]
多个线程可以通过 `volatile` 和 `synchronized` 关键字访问和修改同一个对象，从而实现信息传递。
关键字 `volatile` 可以用来修饰成员变量，告知程序任何对该变量的访问都需要从共享内存中获取，并同步刷新回共享内存，从而保证可见性并防止指令重排；
关键字 synchronized 可以修饰⽅法，或者同步代码块，确保多个线程在同⼀个时刻只有⼀个线程在执⾏⽅法或代码块。保证原子性和可见性；
### 3.19 wait() 和 notify() ⽅法的使⽤⽅式了解吗？ [重要性:A]
⼀个线程调⽤共享对象的 wait() ⽅法时，它会进⼊该对象的等待池，释放已经持有的锁，进⼊等待状态。
⼀个线程调⽤ notify() ⽅法时，它会唤醒在该对象等待池中等待的⼀个线程，使其进⼊锁池，等待获取锁。
Condition 也提供了类似的⽅法， await() 负责阻塞、 signal() 和 signalAll() 负责通知。
通常与锁 ReentrantLock ⼀起使⽤，为线程提供了⼀种等待某个条件成真的机制，并允许其他线程在该条件变化时通知等待线程。
### 3.20 请说说 sleep 和 wait 的区别？（补充） [重要性:S]
sleep 会让当前线程休眠，不需要获取对象锁，属于 Thread 类的⽅法；wait 会让获得对象锁的线程等待，要提前获得对象锁，属于 Object 类的⽅法。
①、所属类不同
sleep() ⽅法专属于 Thread 类。
wait() ⽅法专属于 Object 类。
②、锁⾏为不同
如果⼀个线程在持有某个对象锁时调⽤了 sleep ⽅法，它在睡眠期间仍然会持有这个锁，而wait会释放持有的锁；
③、使⽤条件不同
sleep() ⽅法可以在任何地⽅被调⽤。
wait() ⽅法必须在同步代码块或同步⽅法中被调⽤，这是因为调⽤ wait() ⽅法的前提是当前线程必须持有对象的锁。否则会抛出 IllegalMonitorStateException 异常。
④、唤醒⽅式不同
**调⽤ sleep ⽅法后，**线程会进⼊ TIMED_WAITING 状态，即在指定的时间内暂停执⾏。当指定的时间结束后，线程会⾃动恢复到 RUNNABLE 状态，等待 CPU 调度再次执⾏。
**调⽤ wait ⽅法后，**线程会进⼊ WAITING 状态，直到有其他线程在同⼀对象上调⽤ notify 或 notifyAll ⽅法，线程才会从 WAITING 状态转变为 RUNNABLE 状态，准备再次获得 CPU 的执⾏权。**但是wait超时后会开始尝试获取锁，不用被动唤醒；**
### 3.21 怎么保证线程安全？（补充） [重要性:S]
线程安全是指在并发环境下，多个线程访问共享资源时，程序能够正确地执⾏，⽽不会出现数据不⼀致的问题。
**为了保证线程安全，可以使⽤ synchronized 关键字对⽅法加锁，对代码块加锁。**线程在执⾏同步⽅法、同步代码块时，会获取类锁或者对象锁，其他线程就会阻塞并等待锁。

**如果需要保证变量的内存可⻅性，可以使⽤ volatile 关键字。**
**对于简单的原⼦变量操作，还可以使⽤ Atomic 原⼦类。**
**对于线程独⽴的数据，可以使⽤ ThreadLocal 来为每个线程提供专属的变量副本。**
**对于需要并发容器的地⽅，可以使⽤ ConcurrentHashMap、CopyOnWriteArrayList 等。**

## 4、Java 内存模型
### 4.1 说⼀下你对 Java 内存模型的理解？ [重要性:S]
**Java 内存模型是 Java 虚拟机规范中定义的⼀个抽象模型，⽤来描述多线程环境中共享变量的内存可⻅性。**
![[image 121.png]]
**共享变量存储在 主内存 中，每个线程都有⼀个私有的 本地内存 ，存储了共享变量的副本。**
- 当一个线程更改了本地内存中共享变量的副本，它需要通过 JVM 刷新到主内存中，以确保其他线程可以看到这些更改。
- 当一个线程需要读取共享变量时，它一般会先从本地内存中读取；如果副本已经过时，JVM 会将主内存中的最新值刷新到本地内存。

### 4.2 为什么线程要⽤⾃⼰的内存？ [重要性:A]
线程从主内存拷⻉变量到⼯作内存，可以减少 CPU 访问 RAM 的开销。
每个线程都有⾃⼰的变量副本，可以避免多个线程同时修改共享变量导致的数据冲突。
### 4.3 i++是原⼦操作吗？ [重要性:S]
不是，它包括三个步骤：
1. 从内存中读取 i 的值。
2. 对 i 进⾏加 1 操作。
3. 将新的值写回内存。
### 4.4 说说你对原⼦性、可⻅性、有序性的理解？ [重要性:S]
**原⼦性**要求⼀个操作是不可分割的，要么全部执⾏成功，要么完全不执⾏。
举个例⼦：就⽐如说 count++ 就不是⼀个原⼦操作，它包括读取 count 的值、加 1、写回 count 三个步骤，所以需要加锁或者使⽤ AtomicInteger代替 int 来保证原⼦性。
**可⻅性**要求⼀个线程对共享变量的修改，能够被其他线程及时看⻅。
**有序性**是指程序执⾏的顺序是否按照代码编写的顺序执⾏。
在单线程环境下，代码能够准确⽆误地按照编写顺序执⾏。但在多线程环境下，CPU 和编译器可能会进⾏指令重排，代码的执⾏顺序因此会发⽣变化。
原⼦性保证操作不可中断，可⻅性保证变量修改后线程能看到最新值，有序性保证代码执⾏顺序⼀致，可以通过volatile（有序、可见）、synchronized 和 CAS （原子性）机制来保证这些特性。
### 4.5 happens-before 了解吗？ [重要性:S]
Happens-Before 是 Java 内存模型定义的⼀种保证线程间可⻅性和有序性的规则。
**如果操作 A Happens-Before 操作 B，那么：**
**1. 操作 A 的结果对操作 B 可⻅。**
**2. 操作 A 在时间上先于操作 B 执⾏。**
**换句话说，如果 A Happens-Before B，那么 A 的修改必须对 B 可⻅，并且 B 不能重排序到 A 之前。**
JMM 规定了 6 种 Happens-Before 规则，满⾜这些规则的操作不会被重排序，并且保证了数据的可⻅性。
①、程序顺序规则：单线程内，代码按顺序执⾏；⽐如 a = 1; b = 2;，a 先于 b 执⾏。
②、监视器锁定规则： unlock() Happens-Before lock()；⽐如 synchronized 释放锁后，获取锁的线程能够
看到最新的数据。
### 4.6 volatile 了解吗？ [重要性:S]
第⼀，保证可⻅性，线程修改 volatile 变量后，其他线程能够⽴即看到最新值；第⼆，防⽌指令重排，volatile 变量的写⼊不会被重排序到它之前的代码。

volatile 关键字⽤于修饰变量，确保该变量的更新操作对所有线程是可⻅的，即⼀旦某个线程修改了 volatile 变量，其他线程会⽴即看到最新的值。
synchronized 关键字⽤于修饰⽅法或代码块，确保同⼀时刻只有⼀个线程能够执⾏该⽅法或代码块，从⽽实现互斥访问。

synchronized 在 JDK 1.6 之后，进⾏了锁优化，增加了偏向锁、轻量级锁，重量级锁，⼤⼤提升了 synchronized 的性能。
### 4.11 synchronized 上锁的对象是什么？ [重要性:A]

### 4.12 synchronized 的实现原理了解吗？ [重要性:S]
**synchronized 依赖 JVM 内部的 Monitor 对象来实现线程同步。使⽤的时候不⽤⼿动去 lock 和 unlock，JVM 会⾃动加锁和解锁。**
synchronized 加锁代码块时，JVM 会通过 monitorenter、 monitorexit 两个指令来实现同步：
`前者表示线程正在尝试获取 lock 对象的 Monitor；`
`后者表示线程执⾏完了同步代码块，正在释放锁。`
### 4.13 你对 Monitor 了解多少？ [重要性:A]
Monitor 是 JVM 内置的同步机制，每个对象在内存中都有⼀个对象头——Mark Word，⽤于存储锁的状态，以及Monitor 对象的指针。
synchronized 依赖对象头的 Mark Word 进⾏状态管理，⽀持⽆锁、偏向锁、轻量级锁，以及重量级锁。
### 4.14 synchronized 怎么保证可⻅性？ [重要性:A]
通过两步操作（和volatile一样）：
1、加锁时，线程必须从主内存读取最新数据。
2、释放锁时，线程必须将修改的数据刷回主内存，这样其他线程获取锁后，就能看到最新的数据。
### 4.15 synchronized 怎么保证有序性？ [重要性:A]
synchronized 通过 JVM 指令 monitorenter 和 monitorexit，来确保加锁代码块内的指令不会被重排。
### 4.16 synchronized 锁升级了解吗？ [重要性:S]
JDK 1.6 的时候，为了提升 synchronized 的性能，引⼊了锁升级机制，从低开销的锁逐步升级到⾼开销的锁，以最⼤程度减少锁的竞争。
没有线程竞争时，就使⽤低开销的“偏向锁”，此时没有额外的 CAS 操作；轻度竞争时，使⽤“轻量级锁”，采⽤ CAS⾃旋，避免线程阻塞；只有在重度竞争时，才使⽤“重量级锁”，由 Monitor 机制实现，需要线程阻塞。
### 4.17 了解 synchronized 四种锁状态吗？ [重要性:S]
**1、⽆锁状态，**对象未被锁定，Mark Word 存储对象的哈希码等信息。
**2、偏向锁：**当⼀个线程第⼀次获取锁时，JVM 会在对象头的 Mark Word 记录这个线程 ID，下次进⼊synchronized 时，如果还是同⼀个线程，可以直接执⾏，⽆需额外加锁。

**3、轻量级锁：**当多个线程尝试获取锁但不是同⼀个时段，偏向锁会升级为轻量级锁，等待锁的线程通过 CAS ⾃旋避免进⼊阻塞状态。
**4、重量级锁：**如果⾃旋失败，锁会升级为重量级锁，等待锁的线程会进⼊阻塞状态，等待监视器 Monitor 进⾏调度（Monitor 的线程阻塞和唤醒依赖操作系统的 mutex）。
### 4.18 详细解释下锁升级的过程： [重要性:A]
**①、从⽆锁到偏向锁：**
当⼀个线程⾸次访问同步代码时，如果此对象处于⽆锁状态且偏向锁未被禁⽤，JVM 会将该对象头的锁标记改为偏向锁状态，并记录当前线程 ID。此时，对象头中的 Mark Word 中存储了持有偏向锁的线程 ID。
**②、偏向锁到轻量级锁：**
如果另⼀个线程尝试获取这个已被偏向的锁，JVM 会检查当前持有偏向锁的线程是否活跃。如果持有偏向锁的线程不活跃，可以将锁偏向给新的线程；否则撤销偏向锁，升级为轻量级锁。
当有两个或以上线程竞争同⼀个偏向锁时，偏向锁模式不再有效，此时偏向锁会被撤销，对象的锁状态会升级为轻量级锁。
**③、轻量级锁到重量级锁：**
轻量级锁通过⾃旋来等待锁释放。如果⾃旋超过预定次数（⾃旋次数是可调的，并且是⾃适应的，失败次数多⾃旋次数就少），表明锁竞争激烈。
当⾃旋多次失败，或者有线程在等待队列中等待相同的轻量级锁时，轻量级锁会升级为重量级锁。在这种情况下，JVM 会在操作系统层⾯创建⼀个互斥锁——Mutex（Monitor 的线程阻塞和唤醒依赖操作系统的 mutex），所有进⼀步尝试获取该锁的线程将会被阻塞，直到锁被释放。
### 4.19 synchronized 和 ReentrantLock 的区别了解吗？ [重要性:S]
回答：
**1、synchronized 由 JVM 内部的 Monitor 机制实现，ReentrantLock基于 AQS 实现。**
**2、synchronized 可以⾃动加锁和解锁，ReentrantLock 需要⼿动 lock() 和 unlock()**。
3、**ReentrantLock支持公平锁和非公平锁选择，而synchronized只为非公平锁**
4、**ReentrantLock支持超时中断等，但是synchroniczed不支持**
### 4.20 如果⾯试官还想知道更多，可以继续回答： [重要性:A]
①、ReentrantLock 可以实现多路选择通知，绑定多个 Condition，⽽ synchronized 只能通过 wait 和 notify 唤醒，属于单路通知；
②、synchronized 可以在⽅法和代码块上加锁，ReentrantLock 只能在代码块上加锁，但可以指定是公平锁还是⾮公平锁。
③、ReentrantLock 提供了⼀种能够中断等待锁的线程机制，通过lock.lockInterruptibly() 来实现。
### 4.21 并发量⼤的情况下，使⽤ synchronized 还是 ReentrantLock？ [重要性:A]
我更倾向于 ReentrantLock，因为：
1、ReentrantLock 提供了超时和公平锁等特性，可以应对更复杂的并发场景。
2、ReentrantLock 允许更细粒度的锁控制（主动释放，tryLock 避免阻塞），能有效减少锁竞争。
3、ReentrantLock ⽀持条件变量 Condition，可以实现⽐ synchronized 更友好的线程间通信机制。
### 4.22 AQS 了解多少？ [重要性:S]
AQS 的思想是，如果被请求的共享资源处于空闲状态，则当前线程成功获取锁；否则，将当前线程加⼊到等待队列中，当其他线程释放锁时，从等待队列中挑选⼀个线程，把锁分配给它。
### 4.23 具体代码原理： [重要性:A]
1、状态 state 由 volatile 变量修饰，⽤于保证多线程之间的可⻅性；
2、同步队列由内部定义的 Node 类实现，每个 Node 包含了等待状态、前后节点、线程的引⽤等，是⼀个先进先出的双向链表。
AQS 支持两种同步方式：
- 独占模式：每次只能有一个线程持有锁，例如 ReentrantLock。
- 共享模式：多个线程可以同时获取锁，例如 Semaphore 和 CountDownLatch。
核心方法包括：
- `acquire`：获取锁，失败则进入等待队列；
- `release`：释放锁，唤醒等待队列中的线程；
- `acquireShared`：共享模式获取锁；
- `releaseShared`：共享模式释放锁。
AQS 使⽤⼀个 CLH 队列来维护等待线程，CLH 是三个作者 Craig、Landin 和 Hagersten 的⾸字⺟缩写，是⼀种基于链表的⾃旋锁。
在 CLH 中，当⼀个线程尝试获取锁失败后，会被添加到队列的尾部并⾃旋，等待前⼀个节点的线程释放锁。
CLH 的优点是，假设有 100 个线程在等待锁，锁释放之后，只会通知队列中的第⼀个线程去竞争锁。避免同时唤醒⼤量线程，浪费 CPU 资源。
### 4.24 说说 ReentrantLock 的实现原理？ [重要性:S]
**ReentrantLock 是基于 AQS 实现的 可重⼊排他锁，使⽤ CAS 尝试获取锁，失败的话，会进⼊ CLH 阻塞队列，⽀持公平锁、⾮公平锁，可以中断、超时等待。**
1、内部通过⼀个计数器 state 来跟踪锁的状态和持有次数。当线程调⽤ lock() ⽅法获取锁时，ReentrantLock 会检查 state 的值，如果为 0，通过 CAS 修改为 1，表示成功加锁。否则根据当前线程的公平性策略，加⼊到等待队列中。

2、线程⾸次获取锁时，state 值设为 1；如果同⼀个线程再次获取锁时，state 加 1；每释放⼀次锁，state 减 1。
3、当线程调⽤ unlock() ⽅法时，ReentrantLock 会将持有锁的 state 减 1，如果 state = 0，则释放锁，并唤醒等待队列中的线程来竞争锁。
### 4.25 CAS 了解多少？ [重要性:S]
**CAS 是⼀种乐观锁，⽤于⽐较⼀个变量的当前值是否等于预期值，如果相等，则更新值，否则重试。**
**在 CAS 中，有三个值：
V：要更新的变量(var)
E：预期值(expected)
N：新值(new)**
先判断 V 是否等于 E，如果等于，将 V 的值设置为 N；如果不等，说明已经有其它线程更新了 V，当前线程就放弃更新。
这个⽐较和替换的操作需要是原⼦的，不可中断的。Java 中的 CAS 是由 Unsafe 类实现的。AtomicInteger 类的 compareAndSet 就是⼀个 CAS ⽅法；

## 5、线程池进阶
### 5.1 什么是线程池？ [重要性:S]
线程池是⽤来管理和复⽤线程的⼯具，它可以减少线程的创建和销毁开销。
在 Java 中，ThreadPoolExecutor 是线程池的核⼼实现，它通过核⼼线程数、最⼤线程数、任务队列和拒绝策略来控制线程的创建和执⾏。
### 5.2 说⼀下线程池的⼯作流程？ [重要性:S]
任务提交 → 核⼼线程执⾏ → 任务队列缓存 → ⾮核⼼线程执⾏ → 拒绝策略处理。
### 5.3 线程池的主要参数有哪些？ [重要性:S]
**①、**corePoolSize：核⼼线程数，⻓期存活，执⾏任务的主⼒。
**②、**maximumPoolSize：线程池允许的最⼤线程数。
**③、**workQueue：任务队列，存储等待执⾏的任务。
**④、**handler：拒绝策略，任务超载时的处理⽅式。也就是线程数达到 maximumPoolSiz，任务队列也满了的时候，就会触发拒绝策略。
**⑤、**threadFactory：线程⼯⼚，⽤于创建线程，可⾃定义线程名。
**⑥、**keepAliveTime：⾮核⼼线程的存活时间，空闲时间超过该值就销毁。
**⑦、**unit：keepAliveTime 参数的时间单位：
### 5.4 能简单说⼀下参数之间的关系吗？ [重要性:A]
⼀句话：任务优先使⽤核⼼线程执⾏，满了进⼊等待队列，队列满了启⽤⾮核⼼线程备⽤，线程池达到最⼤线程数量后触发拒绝策略，⾮核⼼线程的空闲时间超过存活时间就被回收。
### 5.5 线程池的拒绝策略有哪些？ [重要性:S]
有四种：
**AbortPolicy**：默认的拒绝策略，会抛 RejectedExecutionException 异常。
**CallerRunsPolicy**：让提交任务的线程⾃⼰来执⾏这个任务，也就是调⽤ execute ⽅法的线程。
**DiscardOldestPolicy**：等待队列会丢弃队列中最⽼的⼀个任务，也就是队列中等待最久的任务，然后尝试重新提交被拒绝的任务。
**DiscardPolicy**：丢弃被拒绝的任务，不做任何处理也不抛出异常。
58.**线程池有哪⼏种阻塞队列？**
常⽤的有五种，有界队列 ArrayBlockingQueue；⽆界队列 LinkedBlockingQueue；优先级队列PriorityBlockingQueue；延迟队列 DelayQueue；同步队列 SynchronousQueue。
①、**ArrayBlockingQueue**：⼀个有界的先进先出的阻塞队列，底层是⼀个数组，适合固定⼤⼩的线程池。
②、**LinkedBlockingQueue**：底层是链表，如果不指定⼤⼩，默认⼤⼩是 Integer.MAX_VALUE，⼏乎相当于⼀个⽆界队列。
③、**PriorityBlockingQueue**：⼀个⽀持优先级排序的⽆界阻塞队列。任务按照其⾃然顺序或 Comparator 来排序。
④、**DelayQueue**：类似于 PriorityBlockingQueue，由⼆叉堆实现的⽆界优先级阻塞队列。
⑤、**SynchronousQueue**：每个插⼊操作必须等待另⼀个线程的移除操作，同样，任何⼀个移除操作都必须等待另⼀个线程的插⼊操作。
### 5.6 线程池提交 execute 和 submit 有什么区别？ [重要性:A]
execute ⽅法没有返回值，适⽤于不关⼼结果和异常的简单任务。
submit 有返回值，适⽤于需要获取结果或处理异常的场景。
### 5.7 线程池怎么关闭知道吗？ [重要性:A]
可以调⽤线程池的 shutdown或 shutdownNow⽅法来关闭线程池。
**shutdown** 不会⽴即停⽌线程池，⽽是等待所有任务执⾏完毕后再关闭线程池。
**shutdownNow** 会尝试通过⼀系列动作来停⽌线程池，包括停⽌接收外部提交的任务、忽略队列⾥等待的任务、尝试将正在跑的任务 interrupt 中断。
需要注意的是，shutdownNow 不会真正终⽌正在运⾏的任务，只是给任务线程发送 interrupt 信号，任务是否能真正终⽌取决于线程是否响应 InterruptedException。
### 5.8 如何知道你设置的线程数多了还是少了？ [重要性:A]
可以通过监控和调试来判断线程数是多还是少。
⽐如说通过 top 命令观察 CPU 的使⽤率，如果 CPU 使⽤率较低，可能是线程数过少；如果 CPU 使⽤率接近100%，但吞吐量未提升，可能是线程数过多。
### 5.9 有哪⼏种常⻅的线程池？ [重要性:S]
主要有四种：
**1、固定⼤⼩的线程池 **Executors.newFixedThreadPool(int nThreads);，适合⽤于任务数量确定，且对线程数有明确要求的场景。例如，IO 密集型任务、数据库连接池等。
**2、缓存线程池 Executors.newCachedThreadPool();**，适⽤于短时间内任务量波动较⼤的场景。例如，短时间内有⼤量的⽂件处理任务或⽹络请求。
**3、定时任务线程池 Executors.newScheduledThreadPool(int corePoolSize)**;，适⽤于需要定时执⾏任务的场景。例如，定时发送邮件、定时备份数据等。
**4、单线程线程池 Executors.newSingleThreadExecutor();，**适⽤于需要按顺序执⾏任务的场景。例如，⽇志记录、⽂件处理等。
### 5.10 能说⼀下四种常⻅线程池的原理吗？ [重要性:A]
**不管是 FixedThreadPool、CachedThreadPool，还是 SingleThreadExecutor 和**
**ScheduledThreadPoolExecutor，它们本质上都是 ThreadPoolExecutor 的不同配置**。
### 5.11 说说固定⼤⼩线程池的原理？ [重要性:B]
线程池⼤⼩是固定的， corePoolSize == maximumPoolSize，默认使⽤LinkedBlockingQueue 作为阻塞队列，适⽤于任务量稳定的场景，如数据库连接池、RPC 处理等。
新任务提交时，如果线程池有空闲线程，直接执⾏；如果没有，任务进⼊ LinkedBlockingQueue 等待。缺点是任务队列默认⽆界，可能导致任务堆积，甚⾄ OOM。
### 5.12 说说缓存线程池的原理？ [重要性:B]
线程池⼤⼩不固定， corePoolSize = 0， maximumPoolSize = Integer.MAX_VALUE。空闲线程超过 60 秒会被销毁，使⽤ SynchronousQueue 作为阻塞队列，适⽤于短时间内有⼤量任务的场景。
提交任务时，如果线程池没有空闲线程，直接新建线程执⾏任务；如果有，复⽤线程执⾏任务。线程空闲 60 秒后销毁，减少资源占⽤。缺点是线程数没有上限，在⾼并发情况下可能导致 OOM。
### 5.13 说说单线程线程池的原理？ [重要性:B]
线程池只有 1 个线程，保证任务按提交顺序执⾏，使⽤ LinkedBlockingQueue 作为阻塞队列，适⽤于需要按顺序执⾏任务的场景。
始终只创建 1 个线程，新任务必须等待前⼀个任务完成后才能执⾏，其他任务都被放⼊ LinkedBlockingQueue 排队执⾏。缺点是⽆法并⾏处理任务。
### 5.14 说说定时任务线程池的原理？ [重要性:B]
定时任务线程池的⼤⼩可配置，⽀持定时 & 周期性任务执⾏，使⽤ DelayedWorkQueue 作为阻塞队列，适⽤于周期性执⾏任务的场景。
缺点是，如果任务执⾏时间 > 设定时间间隔，scheduleAtFixedRate 可能会导致任务堆积。

### 5.16 能说⼀下线程池有⼏种状态吗？ [重要性:A]
状态由 RUNNING → SHUTDOWN → STOP → TIDYING → TERMINATED 依次流转。
![[image 136.png]]
RUNNING 状态的线程池可以接收新任务，并处理阻塞队列中的任务；
SHUTDOWN 状态的线程池不会接收新任务，但会处理阻塞队列中的任务；
STOP 状态的线程池不会接收新任务，也不会处理阻塞队列中的任务，并且会尝试中断正在执⾏的任务；
TIDYING 状态表示所有任务已经终⽌；
TERMINATED 状态表示线程池完全关闭，所有线程销毁。
### 5.17 线程池如何实现参数的动态修改？ [重要性:B]
线程池提供的 setter ⽅法就可以在运⾏时动态修改参数，⽐如说 setCorePoolSize 可以⽤来修改核⼼线程数、setMaximumPoolSize 可以⽤来修改最⼤线程数。
需要注意的是，调⽤ setCorePoolSize() 时如果新的核⼼线程数⽐原来的⼤，线程池会创建新的线程；如果更⼩，线程池不会⽴即销毁多余的线程，除⾮有空闲线程超过 keepAliveTime。
### 5.18 线程池调优了解吗？（补充） [重要性:A]
- **先确定业务类型**
    - 通过压测或监控观察线程是 CPU bound 还是 IO bound。
    - 查看线程栈：大量 park、wait → IO 密集；大量用户代码 → CPU 密集。
- **设置合理的核心线程数**
    - 从 CPU核数 × 2 开始试（IO 密集）。
    - 观察 CPU 使用率：如果长期 < 70%，可增加核心线程；>90% 且队列积压，可减少。
- **选择合适的队列**
    - **无界队列**（LinkedBlockingQueue）：简单，但高峰期可能导致内存 OOM（任务堆积）。
    - **有界队列**（ArrayBlockingQueue）：推荐！控制背压，队列满时触发拒绝策略。
    - **SynchronousQueue**：适合线程快速消费场景（如 Netty 的 EventLoop）。
- **设置最大线程数**
    - 一般设为核心线程数的 2~4 倍。
    - 观察高峰期线程数是否达到 maximum：如果经常达到，说明核心太小或队列太小。

### 5.19 ⼿写⼀个数据库连接池，可以吗？ [重要性:C]
可以的，我的思路是这样的：数据库连接池主要是为了避免每次操作数据库时都去创建连接，因为那样很浪费资源。所以我打算在初始化时预先创建好固定数量的连接，然后把它们放到⼀个线程安全的容器⾥，后续有请求的时候就从队列⾥拿，使⽤完后再归还到队列中。
### 5.20 线程池执⾏中断电了应该怎么处理？ [重要性:C]
线程池本身只能在内存中进⾏任务调度，并不会持久化，⼀旦断电，线程池⾥的所有任务和状态都会丢失。
我会考虑以下⼏个⽅⾯：
**第⼀，持久化任务。**可以将任务持久化到数据库或者消息队列中，等电恢复后再重新执⾏。
**第⼆，任务幂等性，**需要保证任务是幂等的，也就是⽆论执⾏多少次，结果都⼀致。
**第三，恢复策略。**当系统重启时，应该有⼀个恢复流程：检测上次是否有未完成的任务，将这些任务重新加载到线程池中执⾏，确保断电前的⼯作能够恢复。
