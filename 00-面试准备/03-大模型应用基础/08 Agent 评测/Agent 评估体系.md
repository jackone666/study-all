##### 1、基础题：Agent 评估体系有哪些核心维度？ ⭐⭐

**考察要点**：任务完成率、推理路径正确性、效率指标、安全性

Agent 评估主要从四个维度衡量。**任务完成率（Task Success Rate）**看 Agent 能否完成任务，"完成"要细化为结构化验收标准，用代码自动校验。**推理路径正确性（Trajectory Accuracy）**记录每步 `(Thought, Action, Observation)` 三元组，与标准路径对比，因为结果正确不代表过程合理。**效率维度**包括 Token 消耗（成本）、步骤数（规划能力）、时延（用户体验）三个子指标。**安全性**看拒绝率和降级率，特别是高风险工具调用场景必须有明确的拒绝机制。实践中将四个维度纳入 CI 流程，每次变更自动跑评估，用加权综合分决定是否上线。

##### 2、进阶题：使用 LLM-as-Judge 评估 Agent 输出时，如何设计评判标准和规避常见偏差？ ⭐⭐⭐

**考察要点**：位置偏差、冗长偏差、自我偏差、成对比较 vs 单独打分、多 Judge 投票、人工校准基准集

**1️⃣ Common Answer**

重点总结（便于面试记忆）：
- 三类核心偏差及规避
- 成对比较 vs 单独打分的组合策略
- 多 Judge 投票 + Golden Dataset 闭环
- 位置偏差：成对比较时 Judge 倾向选排在前面的答案 → 把两答案顺序各跑一遍，结果不一致的单独处理
- 冗长偏差：Judge 倾向给更长格式更丰富的答案打高分 → Prompt 里把"内容质量"和"表达质量"拆成两个维度分开打分
- 自我偏差：GPT-4 当 Judge 时倾向给 GPT-4 输出打高分 → 混用不同 Judge（GPT-4 + Claude），或在 Prompt 里隐藏输出来源

**2️⃣ Impressive Answer**

LLM-as-Judge 用不好，评估结果本身就不可信。我从偏差类型、方法选择、工程保障三个角度来说。
1. **三类核心偏差及规避**
  - **位置偏差**：成对比较时 Judge 倾向选排在前面的答案 → 把两答案顺序各跑一遍，结果不一致的单独处理
  - **冗长偏差**：Judge 倾向给更长格式更丰富的答案打高分 → Prompt 里把"内容质量"和"表达质量"拆成两个维度分开打分
  - **自我偏差**：GPT-4 当 Judge 时倾向给 GPT-4 输出打高分 → 混用不同 Judge（GPT-4 + Claude），或在 Prompt 里隐藏输出来源
1. **成对比较 vs 单独打分的组合策略**
  - 单独打分（Pointwise）效率高，适合批量初筛
  - 成对比较（A/B）对细微差别更敏感，用于分数接近的样本精确判断
  - 实践：先批量单独打分，再对接近样本做成对比较
1. **多 Judge 投票 + Golden Dataset 闭环**
  - 同一样本用不同 Prompt 模板、不同 Judge 模型各打一次，取中位数或投票，三个 Judge 组合可降低约 30% 偏差率
  - 维护 200-500 条人工精标基准集，定期校准 Judge 打分分布；若与人工标注相关性低于 0.8，重新调整 Judge Prompt 或更换 Judge 模型

##### 3、场景题：如果评估发现 Agent 准确率提升但 Token 消耗翻倍，你会怎么处理？ ⭐⭐⭐

**考察要点**：效率与效果权衡、成本控制、评估指标设计、优化策略

**1️⃣ Common Answer**

重点总结（便于面试记忆）：
- 先量化权衡，建立决策标准
- 定位 Token 暴增的根因
- 引入成本约束作为评估护栏
- 计算"准确率收益 / Token 成本增量"的性价比比值，而不是单看哪个指标变好了
- 制定加权综合分（如准确率权重 0.6、Token 效率权重 0.4），让迭代决策有数字依据而非凭感觉
- 用 Trajectory 追踪找到消耗异常的步骤，常见原因：Prompt 冗余、不必要的多轮工具调用、思维链过长

**2️⃣ Impressive Answer**

这个问题本质是评估指标设计和优化策略的结合，我从三个层面处理。
1. **先量化权衡，建立决策标准**
  - 计算"准确率收益 / Token 成本增量"的性价比比值，而不是单看哪个指标变好了
  - 制定加权综合分（如准确率权重 0.6、Token 效率权重 0.4），让迭代决策有数字依据而非凭感觉
1. **定位 Token 暴增的根因**
  - 用 Trajectory 追踪找到消耗异常的步骤，常见原因：Prompt 冗余、不必要的多轮工具调用、思维链过长
  - 对症下药：精简 System Prompt、用 Few-shot 替换长描述、为简单子任务引入更小的模型路由
1. **引入成本约束作为评估护栏**
  - 在 CI 评估里增加"Token 消耗不得超过基线 X%"的硬约束，和准确率指标并列
  - 生产环境设置每次调用的 Token 预算上限，触发则截断并降级处理，防止失控
