# Q：让你设计一个企业智能客服 Agent，你会怎么做？

## **需求澄清**

好的，我先确认一下需求哈。这个客服 Agent 主要面向什么场景？是电商售后还是 SaaS 产品支持？大概要支撑多少 QPS？这些会影响我后面的技术选型。

假设是电商场景，日均 100 万咨询量，峰值 QPS 2000 左右，那我先把这道题的核心挑战点一下——其实就三件事：**准确理解用户意图、稳定调用工具、遇到搞不定的情况要能平滑转人工**。

## **整体架构**

我画个分层架构给你看：

![客服Agent架构图竖版1_水印.png](img_587bbe3f0f3c9450.png)

**最上层是接入层**，负责对接各种渠道——网页 IM、微信公众号、小程序、APP 内客服，甚至电话语音转文字。这一层主要做协议适配和流量控制，用 Nginx + Gateway 就行。

**往下是对话管理层**，这是核心。它负责多轮对话的上下文维护，包括：

- 会话状态机（用户在哪个环节）

- 上下文窗口管理（历史对话保留多少轮）

- 对话策略（什么时候追问、什么时候结束）

这里我用 Redis Cluster 存会话状态，每个会话一个 key，用 Hash 结构存 userId、sessionId、当前意图、历史对话摘要。历史对话不会全量存，用 LLM 做摘要压缩，保留最近 5 轮完整对话 + 之前的摘要，这样能把 token 控制在 2000 以内。

**再往下是意图识别层**，这是准确性的关键。我用的是**多模型级联方案**：

- 第一层用**规则引擎 + Embedding 相似度匹配**做粗筛，把高频简单问题（查订单、物流、退换货）直接路由到规则引擎（用 Qwen2.5-0.5B 做 Embedding，离线算好意图模板的向量，在线余弦相似度匹配，延迟 < 10ms）

- 第二层用通用大模型（比如 Qwen-14B）做意图分类和槽位提取

- 第三层针对复杂场景（比如纠纷处理）用微调过的领域模型

这里有个关键的 Trade-off 要聊一下——为什么不用单一模型？因为成本和响应速度。简单问题占 60% 以上，用规则引擎 50ms 响应，成本几乎为零；复杂问题才上大模型，这样整体 P99 延迟能控制在 1.5s 内，成本也能压下去。

**然后是工具调用层**，这一层是 Agent 的"手脚"。核心工具有：

- 订单查询（调用订单服务 RPC）

- 物流追踪（对接快递公司 API）

- 退款操作（调用交易系统）

- 工单系统（创建、更新、查询）

- 知识库检索（向量数据库 + 关键词检索混合）

工具调用我用了 **Function Calling** 的方案，给 LLM 一个工具清单，让它自己决定调用哪个。这里有个坑——LLM 可能会胡乱调用，所以我加了个**工具调用守卫**：对写操作（退款、改地址）强制要求二次确认，对读操作做参数校验，比如订单号格式不对就直接拒绝。

**最底层是数据层**，包括向量数据库（存知识库）、关系数据库（存工单、用户画像）、缓存层（Redis）。

## **关键技术**

**意图识别怎么搞准确？**

我用了**意图消歧 + 槽位填充**的组合拳。比如用户说"退款"，可能有 5 种意图：申请退款、查询退款进度、取消退款、退款到账时间、退款失败原因。我会先识别出是"退款"这个大类，然后追问"您是想申请退款还是查询退款进度？"——这就是主动澄清。

槽位填充也很重要。用户说"我昨天买的衣服想退"，系统要能提取出：时间=昨天、商品=衣服、意图=退款。这里我用的是**LLM 结构化抽取**方案——直接 prompt 让模型输出 JSON，比 BERT+CRF 更灵活，不需要单独训练，新增槽位只需改 prompt，准确率在 95% 以上。

**多轮对话怎么管理？**

我用的是**状态机 + 上下文摘要**的方案。定义了 20 多个对话状态，比如 `WAITING_ORDER_INFO`、`CONFIRMING_REFUND`、`HUMAN_HANDOVER`。每个状态有对应的触发条件和动作。

比如用户在 `CONFIRMING_REFUND` 状态说"算了不退了"，系统就切换到 `DIALOGUE_END` 状态，记录日志。

上下文摘要我设计了一个**滑动窗口机制**：每 3 轮对话做一次摘要，保留最近 5 轮完整对话。这样既能保持连贯性，又不会让 token 爆掉。

**什么时候转人工？**

这是面试官最爱问的点。我设计了**三级触发机制**：

1. **显式触发**：用户说"转人工"、"我要投诉"

2. **隐式触发**：连续 3 轮对话用户满意度评分低于 3 分，或者 LLM 自信度低于 60%

3. **风险触发**：涉及投诉、法律风险、金额超过 5000 元的情况

转人工时我会把对话摘要、当前状态、用户画像一起推给客服坐席系统，让客服能快速接手。

**工单怎么对接？**

我做了**双向同步**：

- Agent 创建工单时，调用工单系统 API，带上对话摘要和用户意图

- 工单状态变更时，通过 MQ 通知 Agent，更新用户会话状态

- 用户再次咨询时，Agent 能查到历史工单，避免重复创建

这里有个细节——工单创建要异步化，不能阻塞对话流程。我用了消息队列，创建工单请求先入队列，后台消费，给用户返回"工单已创建，编号 12345"。

## **扩展性和边界情况**

**扩展性方面**，我做了几件事：

- 工具层插件化，新增工具只要注册一下就行

- 意图识别模型可热更新，不用重启服务

- 多租户支持，不同企业可以有独立的意图配置和工具权限

**边界情况我也考虑到了**：

- 用户说脏话：触发敏感词过滤，先警告，再严重就转人工

- 连续追问同一个问题：检测到重复意图，直接给出答案，不再走完整流程

- 超时无响应：30 分钟无操作自动结束会话，下次来算新会话

- 上下文丢失：用户换设备登录，根据 userId 恢复会话状态

## **面试官可能追问**

**追问 1：如果 LLM 响应太慢怎么办？**

我做了三层降级。第一层，超时 1.5s 就返回"正在处理，请稍候"；第二层，用缓存的历史相似问题直接回复；第三层，转人工。另外，我会对高频问题做预计算，提前生成回复模板。

**追问 2：如何评估客服效果？**

我设计了三个指标：

- **解决率**：不需要转人工的比例，目标是 70%

- **满意度**：用户打分，平均分要 4.2 以上

- **响应时间**：P99 延迟小于 1.5s

另外，我会抽样人工质检，每周复盘 badcase，优化意图识别模型和对话策略。

**追问 3：知识库怎么维护？**

我做了**知识闭环**。客服人工处理的对话，如果发现是知识库缺失，会标注出来，定期审核后加入知识库。另外，用户反馈"没解决"的对话，也会自动触发知识更新流程。

## **一句话总结**

这个架构的核心是**分层解耦 + 渐进式降级**。简单问题走规则，复杂问题上模型，搞不定就转人工。最关键的还是**数据飞轮**——用真实对话数据不断优化意图识别和对话策略，越用越聪明。

# **Q：让你设计一个 Deep Research Agent 怎么做**

## **需求澄清**

面试官，在开始设计之前，我想先确认几个关键点。这个 Deep Research Agent 的目标用户是谁？是普通用户还是专业研究员？研究深度要求到什么程度？是做一个快速概览，还是需要深挖到学术级别的细节？还有，用户对实时性有要求吗？

另外，用户会怎么使用这个 Agent？是一次性提交一个主题就等着结果，还是希望能在过程中看到进度、甚至能干预方向？这些细节会影响我在架构上的取舍。

![DeepResearch.png](img_6fe5bba5b805d21c.png)

## **核心挑战**

这道题其实挺有意思的，核心难点不在于搜索本身，而在于怎么让 Agent 知道"什么时候该停下来"。搜索是无止境的，你总能找到更多信息，但用户的时间是有限的。

第二个难点是**多源验证**。互联网上信息质量参差不齐，同一个事实在不同来源可能相互矛盾，Agent 需要有能力判断哪些信源更可靠，哪些是谣言或过时信息。这不仅仅是简单的交叉验证，还需要建立信源的可信度模型。

第三个难点是**上下文管理**。Deep Research 的过程可能会持续几分钟甚至更久，涉及几十次搜索和上百页的内容，如何在一个对话窗口内有效管理这些信息，既不丢失关键细节，又不超过模型的上下文窗口，这是个很现实的工程问题。

## **架构设计**

我倾向于用 **Orchestrator-Worker 的模式**来做，具体来说就是 Planner + Searcher + Analyzer + Writer 这四个 Agent 协作。

**Planner Agent** 负责把用户的研究主题拆解成可执行的子任务。比如用户问"2025 年 AI Agent 的发展趋势"，Planner 会先分析这个主题的维度：技术突破、商业应用、监管政策、开源生态等等，然后生成一个搜索计划。这个计划不是静态的，Planner 会根据中间结果动态调整，如果发现某个方向信息不足，就会增加搜索任务。

**Searcher Agent** 专门负责执行搜索。它需要理解不同信息源的特点：学术论文用 arXiv 或 Google Scholar，新闻用实时新闻 API，行业报告可能需要特定的数据库。Searcher 还要做去重，避免反复搜到相同的内容。

**Analyzer Agent** 是信息聚合的核心。它要做三件事：一是**事实提取**，从搜索结果中抽取出关键事实；二是**信源评估**，判断信息的可靠性；三是**矛盾检测**，当不同来源信息冲突时，标记出来并尝试找到更权威的来源。

**Writer Agent** 负责最终报告的生成。它不是简单地把 Analyzer 的输出拼起来，而是要有逻辑地组织内容，形成连贯的叙述。Writer 还要考虑用户的阅读体验，比如用图表可视化数据，用案例说明观点。

这四个 Agent 之间通过一个共享的**任务队列**和**状态存储**协作。Planner 把任务放入队列，Searcher 消费任务并更新状态，Analyzer 分析结果，Writer 最终产出报告。整个过程是流式的，用户可以实时看到进度。

## **可能的追问**

### 追问：你认为项目的技术细节在哪里？

**搜索策略**上，我不会只用一种搜索方式。对于事实类问题，用精确搜索；对于概念类问题，用语义搜索；对于趋势类问题，可能需要时间序列搜索。关键是要建立一个搜索策略的决策树，根据任务类型自动选择最合适的方式。

**信息聚合**这块，我打算用 **RAG + 知识图谱**的混合方案。RAG 用于快速检索相关内容，知识图谱用于建立事实之间的关联。比如研究某个技术，知识图谱可以把它的历史发展、关键人物、相关公司都串联起来，帮助 Agent 理解全貌。

**上下文管理**是个大问题。我的方案是分层存储：短期记忆用向量数据库，存储最近搜索的内容；中期记忆用摘要，定期把旧内容压缩成摘要；长期记忆用知识图谱，存储经过验证的结构化知识。这样可以在有限窗口内最大化信息密度。

**Streaming 中间结果**对用户体验很重要。我会设计一个事件流机制，每当一个 Searcher 完成任务、Analyzer 得出新发现，就推送事件给前端，让用户看到实时进度。比如"正在搜索学术论文...发现 12 篇相关论文...正在分析矛盾观点..."这样的进度反馈。

### 追问：你在**扩展性上做了什么处理**

如果用户量上来，Planner、Searcher、Analyzer、Writer 这四个 Agent 都可以水平扩展，特别是 Searcher，因为它是计算密集型的。任务队列可以用 Kafka 或者 Redis Streams 来做，保证高吞吐。

另一个扩展点是**多模态支持**。现在的设计主要处理文本，但很多研究主题需要图表、视频、数据集。可以在 Analyzer 阶段加入多模态处理能力，比如从论文中提取图表，从视频中提取关键帧。

还有一个有趣的方向是**个性化**。不同用户对研究报告的偏好不同，有人喜欢详细的技术细节，有人更关注商业影响。可以在 Planner 阶段根据用户画像调整搜索策略，在 Writer 阶段根据用户偏好调整报告风格。

### **追问 1：如果搜索结果质量很差怎么办？**

Searcher 会优先选择高质量信源，比如学术期刊、权威媒体、官方报告。Analyzer 有信源评估机制，会给每个来源打分。如果整体质量都不高，Planner 会调整搜索策略，比如换个关键词、换个时间范围，或者尝试不同的信息源类型。极端情况下，如果实在找不到可靠信息，系统会诚实地告诉用户"这个主题目前缺乏可靠信息"，而不是编造内容。

### **追问 2：怎么防止 Agent 陷入搜索死循环？**

这是个好问题。我会设计几个停止条件：一是时间限制，比如整个研究过程不超过 5 分钟；二是信息饱和度，如果连续几次搜索都没有新信息，就认为已经足够；三是任务完成度，Planner 最初设定的子任务都完成后就停止。还有一个兜底机制，如果搜索次数超过阈值，强制触发停止并生成当前已有的报告。

### **追问 3：怎么保证报告的客观性？**

客观性来自几个方面：一是多源验证，不依赖单一信息源；二是信源透明，报告中会标注每个事实的来源；三是矛盾呈现，如果不同来源有冲突，会把双方观点都列出来，而不是武断地选一边。另外，Writer Agent 在生成报告时，会刻意避免情绪化表达，保持中立语气。

### **追问 4：如果用户中途想调整研究方向怎么办？**

这需要支持动态干预。用户可以在任何时候给 Planner 发送新的指令，比如"多关注商业应用，少关注技术细节"。Planner 收到后会重新评估当前任务队列，调整优先级，甚至取消一些不相关的任务。已经完成的任务不会被浪费，因为它们的结果仍然可能有用，只是权重会降低。
