# 1、静态方法为什么不能调用非静态成员？ [重要性:A]
这个需要结合 JVM 的相关知识，主要原因如下：
- **静态方法属于类**：在类加载的时候就会分配内存，可以通过类名直接访问。而非静态成员属于实例对象，只有在对象实例化之后才存在，需要通过类的实例对象去访问。
- **生命周期不同**：在类的非静态成员不存在的时候静态方法就已经存在了，此时调用在内存中还不存在的非静态成员，属于非法操作。

# 2、== 和 equals 的区别 [重要性:S]
> **💡 核心区别**：`==` 比较的是地址，`equals()` 比较的是内容（如果被重写）

## == 运算符
- **基本数据类型**：比较的是值
- **引用数据类型**：比较的是内存地址（是否为同一个对象）

## equals() 方法
- **类没有覆盖 equals() 方法**：等价于使用 `==` 比较，使用的是 Object 类默认的 equals() 方法
- **类覆盖了 equals() 方法**：一般用来比较两个对象的内容是否相等

**注意**：equals() 不能判断基本类型,因为基本类型没有方法。
# 3、hashCode() 与 equals() [重要性:S]
> **面试高频问题**："为什么重写 equals 时必须重写 hashCode 方法？"

## hashCode 介绍
`hashCode()` 的作用是获取**哈希码**（也称为散列码），返回一个 int 整数。这个哈希码用于确定对象在哈希表中的索引位置。
- `hashCode()` 定义在 JDK 的 Object 类中，意味着 Java 中的任何类都包含此方法
- 散列表存储键值对（key-value），能根据"键"快速检索出对应的"值"

## 为什么要有 hashCode？
以 **HashSet 如何检查重复**为例说明：
1. 当你把对象加入 HashSet 时，HashSet 会先计算对象的 hashCode 值来判断加入位置
2. 将 hashCode 与已加入对象的 hashCode 作比较
3. 如果没有相符的 hashCode，假设对象没有重复
4. 如果发现相同的 hashCode，再调用 `equals()` 方法检查对象是否真的相同
5. 如果两者相同，HashSet 不会让其加入

**性能优势**：hashCode 比较很快，可以先比较 hashCode 快速排除不同对象，大大减少 equals() 的调用次数，提高执行速度。
## 为什么重写 equals 时必须重写 hashCode？
因为在集合中要先比较 hashCode：
- 如果不重写 hashCode，Object 类返回的是对象的地址值
- 即使两个对象内容相同，hashCode 也不一样
- 这会造成集合判断错误，认为两个内容相同的对象是不同的对象

## hashCode 相同的对象一定相等吗？
**不一定**。因为：
- hashCode() 使用的散列算法可能让多个对象返回相同的散列值（**哈希碰撞**）
- 越糟糕的散列算法越容易碰撞
- 这也与数据值域分布的特性有关

# 4、重载和重写的区别 [重要性:S]
> **方法签名**：方法名 + 参数列表，唯一确定一个方法

## 定义
- **重载（Overload）**：同样的方法名能够根据输入数据的不同，做出不同的处理
- **重写（Override）**：子类继承父类的相同方法，输入数据一样，但要做出有别于父类的响应时，覆盖父类方法

## 重载的特点
- 发生在同一个类中（或父类和子类之间）
- **方法名必须相同**
- **参数必须不同**（类型、个数或顺序不同）
- 方法返回值和访问修饰符可以不同

## 重写的特点
- 发生在子类
- **方法签名必须完全相同**（方法名 + 参数列表）
- 返回值类型、访问修饰符、异常抛出可以不同（但遵循"向下兼容"原则）

## 发生时机
- **重载**：发生在编译期间
- **重写**：发生在运行期，是多态的表现

# 5、深拷贝 vs 浅拷贝 [重要性:A]
## 浅拷贝
- 对基本数据类型进行**值传递**
- 对引用数据类型进行**引用传递**
- 结果：**两个引用指向同一个对象**

## 深拷贝
- 对基本数据类型进行**值传递**
- 对引用数据类型，**创建一个新的对象**，并复制其内容
- 结果：**两个引用指向不同对象，但对象内容相同**

# 6、面向对象三大特征 [重要性:A]
## 1. 封装
封装是指把一个对象的状态信息（属性）隐藏在对象内部，不允许外部对象直接访问对象的内部信息。
## 2. 继承
- 子类拥有父类对象所有的属性和方法（包括私有属性和私有方法）
- 但父类中的私有属性和方法子类无法访问，**只是拥有**
- **构造方法不能继承**

## 3. 多态
父类引用指向子类对象，根据子类的不同，表现出不同的状态。
# 7、String、StringBuffer 和 StringBuilder 的区别 [重要性:S]
## 可变性
**依据**：char 数组是否被 final 修饰
- **String**：不可变，每次修改都会创建新的对象，原对象不会改变
- **StringBuffer 和 StringBuilder**：可变，修改时不会创建新对象，直接操作原有对象

## 线程安全性
- **StringBuffer**：线程安全，其方法都被 synchronized 修饰
- **StringBuilder**：非线程安全，没有同步机制
- **String**：由于不可变，天然具备线程安全性

## 性能
- 单线程环境下，StringBuilder 性能优于 StringBuffer（避免了同步开销）
- 频繁修改字符串时，StringBuffer 和 StringBuilder 远优于 String（减少对象创建开销）

## 使用建议
- **String**：字符串不常修改或用于字符串常量时
- **StringBuffer**：多线程环境且需要修改字符串时
- **StringBuilder**：单线程环境且需要修改字符串时（性能更优）

**String 为什么不可变？**
- 安全性
- 便于实现字符串常量池

# 9、异常机制 [重要性:A]
## Exception 和 Error 的区别
在 Java 中，所有的异常都有一个共同的祖先：`java.lang.Throwable` 类。Throwable 类有两个重要的子类：
## Exception（异常）
程序本身可以处理的异常，可以通过 catch 来进行捕获。Exception 分为：
- **Checked Exception（受检查异常）**：必须处理
- **Unchecked Exception（不受检查异常）**：可以不处理

## Error（错误）
属于程序无法处理的错误，不建议通过 catch 捕获。例如：
- Java 虚拟机运行错误（VirtualMachineError）
- 虚拟机内存不够错误（OutOfMemoryError）
- 类定义错误（NoClassDefFoundError）

这些异常发生时，JVM 一般会选择线程终止。
# 10、try-catch-finally [重要性:A]
## 各部分作用
- **try 块**：用于捕获异常。其后可接零个或多个 catch 块，如果没有 catch 块，则必须跟一个 finally 块
- **catch 块**：用于处理 try 捕获到的异常
- **finally 块**：无论是否捕获或处理异常，finally 块里的语句都会被执行

## 执行顺序
当在 try 块或 catch 块中遇到 return 语句时：
1. finally 语句块将在方法返回之前被执行
2. finally 语句的返回值会**覆盖** try 语句块的返回值

**注意**：catch 和 finally 必须至少选一个。
# 11、多线程状态 [重要性:S]
Java 线程在运行的生命周期中的指定时刻只可能处于以下 6 种状态之一：
## 1. 初始（NEW）
新创建了一个线程对象，但还没有调用 `start()` 方法。
## 2. 运行（RUNNABLE）
Java 线程中将**就绪**（ready）和**运行中**（running）两种状态笼统地称为"运行"：
- **就绪状态（ready）**：线程对象创建后，其他线程调用了该对象的 `start()` 方法。该状态的线程位于可运行线程池中，等待被线程调度选中，获取 CPU 使用权
- **运行中状态（running）**：就绪状态的线程获得 CPU 时间片后的状态

## 3. 阻塞（BLOCKED）
表示线程阻塞于锁。
## 4. 等待（WAITING）
进入该状态的线程需要等待其他线程做出一些特定动作（通知或中断）。
## 5. 超时等待（TIMED_WAITING）
该状态不同于 WAITING，它可以在指定的时间后自行返回。
## 6. 终止（TERMINATED）
表示该线程已经执行完毕。
# 12、易错知识点：foreach 循环中的 remove/add 操作 [重要性:A]
在 `foreach` 循环（本质是迭代器遍历）中执行元素的 `remove/add` 操作，会直接抛出 `ConcurrentModificationException`（并发修改异常）。
## 核心问题
违反了迭代器的**快速失败（fail-fast）机制**。
## 原因分析
## 1. 迭代器的校验逻辑被打破
- `foreach` 依赖集合的迭代器（如 ArrayList 的 Itr）遍历元素
- 迭代器初始化时会记录集合的"修改次数"（modCount）
- 每次循环都会校验当前集合的 modCount 是否与初始化时一致
- 若执行 `remove/add`，会直接修改集合的 modCount
- 导致下一次校验时"次数不匹配"，迭代器立刻抛出异常

## 2. 遍历与修改的逻辑冲突
即使不抛异常（如某些非 fail-fast 集合），也可能出现逻辑错误：
- `remove` 后，集合长度缩短，迭代器可能跳过元素或重复遍历
- `add` 后，新增元素可能被遍历（或不被遍历），结果不可控，破坏遍历的"确定性"

## 正确做法
- **使用迭代器自身的 remove() 方法**：会同步更新迭代器的校验次数，避免异常
- **先记录后操作**：遍历时记录要操作的元素，遍历结束后再批量 `remove/add`（如用临时列表存储待删元素）

# 13、final 和 static 关键字 [重要性:S]
## final 关键字
final 关键字用来修饰类、方法和变量，具有以下特点：
- **修饰类**：final 修饰的类不能被继承，final 类中的所有成员方法都会被隐式地指定为 final 方法
- **修饰方法**：final 修饰的方法不能被重写
- **修饰变量**：final 修饰的变量是常量

## static 关键字
static 关键字主要有以下使用场景：
## 1. 修饰变量、方法和内部类
- 被 static 修饰的成员属于类，不属于单个对象
- 被类中所有对象共享
- 可以并且建议通过类名调用
- 存放在 Java 内存区域的方法区

## 2. 静态代码块
- 静态代码块定义在类中方法外
- **执行顺序**：静态代码块 → 非静态代码块 → 构造方法
- 该类不管创建多少对象，静态代码块只执行一次

# 16、多态解决了什么问题？实现原理是什么？ [重要性:S]
- 解决问题：同一接口或方法在不同类有不同实现，动态绑定父类引用指向的子类对象，方法调用决策延迟到运行时，提高代码灵活性；

# 18、final、finally、finalize 的区别？ [重要性:A]
- final：修饰符，可修饰类、方法、变量，限制其可变性；
- finally：异常处理的一部分，try 块后执行，无论是否抛出异常都会执行，用于释放资源（如关闭文件、数据库连接）；
- finalize：Object 类的方法，垃圾回收前自动调用，用于清理资源，Java 9 已弃用，不可显式调用。

# 21、String str1 = new String("abc") 和 String str2 = "abc" 的区别？ [重要性:S]
- `str2 = "abc"`：先检查字符串常量池，存在则引用池对象，不存在则创建并放入池后引用；
- `str1 = new String("abc")`：先检查常量池（无则创建），再在堆中创建新对象并初始化为池对象的副本；
- 示例：
`String s1 = "沉默王二";
String s2 = "沉默王二";
String s3 = new String("沉默王二");
System.out.println(s1 == s2); // 输出 true，引用常量池同一对象
System.out.println(s1 == s3); // 输出 false，s3 指向堆对象`

# 23、Java 中异常处理体系 [重要性:A]
- 顶层类：Throwable，包含两个子类：
    - Error：严重错误（如 OutOfMemoryError、StackOverflowError），程序无法处理；

- `throws`：方法声明可能抛出的异常

# 25、Java 中 IO 流分为几种？ [重要性:A]
按不同维度划分：
1. 数据流方向：输入流（InputStream/Reader）、输出流（OutputStream/Writer）；
2. 处理数据单位：字节流（处理二进制数据）、字符流（处理文本数据）；
3. 功能：节点流（直接连接数据源 / 目的地，如 FileInputStream）、处理流（包装节点流，如 BufferedInputStream）、管道流（线程间数据传输）；
4. 设计模式：装饰器模式。

# 26、既然有了字节流，为什么还要有字符流？ [重要性:A]
字节流转换为字符需 JVM 处理，耗时且易因编码问题乱码。字符流直接操作字符，方便文本处理（如查找字符串）；二进制文件（音视频）用字节流，文本文件用字符流更便捷

# 30、说一下你对注解的理解 [重要性:A]
注解本质是标记，可标记在类、方法、属性上，可设置属性值。编译或运行阶段可识别标记并执行相应逻辑（如 AOP 切点、Lombok 编译期生成代码）。
注解生命周期：
1. `RetentionPolicy.SOURCE`：编译期使用，不写入 class 文件（如 @Override）；
2. `RetentionPolicy.CLASS`：写入 class 文件，类加载时丢弃；
3. `RetentionPolicy.RUNTIME`：永久保存，可通过反射获取（如 @Autowired）。

# 31、什么是反射？应用？原理？ [重要性:S]
- 定义：运行时检查和操作类的字段、方法、构造方法等信息，动态创建对象、调用方法；
- 应用场景：Spring 框架 Bean 管理、动态代理（AOP 实现）、测试框架（如 JUnit 查找 @Test 方法）；
- 原理：JVM 类加载时，将类信息加载到方法区，反射通过 Class 类及 java.lang.reflect 包（Field、Method、Constructor）获取这些信息并操作。
