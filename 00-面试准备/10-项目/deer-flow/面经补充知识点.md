

## 一、统一项目叙事

### 1. 项目一句话怎么讲？

总答：小艺 Claw 是小艺侧的长任务 agentic workflow 能力，把原来偏指令式的小艺入口，扩展成能完成复杂任务的多步执行系统。

展开：
- 背景：小艺是鸿蒙系统级 AI 助手，不是一个独立小工具，入口覆盖语音、图文识别、服务建议、设备能力和多设备协同。
- 能力：用户给出复杂目标后，系统会做意图澄清、任务拆解、工具调用、检索、总结、校验和结果交付。
- 架构：底层是 lead agent / supervisor 负责规划和调度，sub agent 负责检索、工具调用、沙箱执行、文档处理、总结和校验。
- 价值：不是做一个 demo，而是把长任务 agent 能力产品化到小艺这种高频、系统级、亿级入口里。

本地实现依据：
- `lead_agent/agent.py` 的工厂入口会解析运行时配置、选择模型、加载工具、组装中间件、渲染 prompt，并创建 LangGraph agent。
- `runtime/runs/worker.py` 负责把用户消息放进后台 run，注入 `thread_id/run_id/app_config/user_id` 等 runtime context，再流式执行 agent graph。
- `runtime/runs/manager.py` 用 RunRecord 管理 pending/running/success/error 等状态，并记录 token、消息数、最后回复等运行信息。

小艺包装说法：
- “底层是通用 agent runtime 思路，我做的是把它适配到小艺场景：入口、权限、工具、端云、上下文、评测和体验都要重新设计。”

### 2. 面试里哪些话不能说？

总答：不能说“只是包装”，要说“基于通用 agent runtime 做了小艺场景的工程化和产品化适配”。

展开：
- 不说：只是套了一层 UI。
- 要说：做了任务分层路由、工具接入、上下文管理、沙箱执行、记忆策略、评测闭环和小艺业务适配。
- 不说：整个底层框架都是我从 0 到 1 发明。
- 要说：底层范式参考成熟 agent runtime，个人重点在小艺场景的流程裁剪、模块适配和体验优化。

本地实现依据：
- 工厂模块里有明确的运行时配置解析、模型解析、工具加载、技能白名单、延迟工具、middleware 链和 agent graph 创建流程。
- 这些机制说明它不是简单 prompt，而是一个完整 agent runtime。

小艺包装说法：
- “我重点讲自己负责的小艺侧适配：用户意图、工具选择、结果结构化、评测、时延和稳定性。”

## 二、请求链路与整体架构

### 1. 一个用户消息从发送到返回，经历哪些步骤？

总答：链路可以分成“入口接收 - run 创建 - agent 构造 - 图执行 - 流式返回 - 状态落库”六步。

展开：
- 入口接收：前端或小艺入口把用户消息发到 Gateway。
- run 创建：RunManager 创建 RunRecord，状态从 pending 进入 running，并启动后台 worker。
- agent 构造：worker 调用 lead agent 工厂，解析模型、模式、子 agent 开关、工具和中间件。
- 图执行：LangGraph agent 进入多轮 LLM/tool loop，必要时调用子 agent、沙箱、MCP 工具。
- 流式返回：StreamBridge 把 messages、values、updates 等事件流式推给前端。
- 状态落库：RunManager / RunJournal 汇总 token、消息、最终状态、事件和错误。

本地实现依据：
- `runtime/runs/worker.py`：`run_agent()` 安装 runtime context，调用 agent factory，执行 `graph.astream()`，发布 stream 事件。
- `runtime/runs/manager.py`：`RunRecord` 记录 run_id、thread_id、status、abort_event、token、message_count、last_ai_message。
- `lead_agent/agent.py`：`make_lead_agent()` 是 agent graph 构造入口。

小艺包装说法：
- “在小艺侧我会把这个链路包装成用户可感知的进度：正在理解、正在检索、正在调用工具、正在整理结论，而不是让用户等黑盒。”

### 2. 核心架构怎么讲？

总答：整体是 Gateway + Run Manager + Lead Agent Graph + Sub Agent + Sandbox/MCP Tools + Memory/State 的架构。

展开：
- Gateway：负责接收请求、鉴权、兼容 LangGraph API、管理线程和 run。
- Run Manager：负责 run 的生命周期、取消、状态、持久化快照和 token 统计。
- Lead Agent：负责主流程规划、工具调度、中间件链和最终输出。
- Sub Agent：负责复杂子任务，有自己的模型、工具、技能、状态和执行结果。
- Sandbox/MCP：把文件、命令、外部工具封装为可控工具能力。
- Memory/State：负责当前线程、任务状态、长期记忆和摘要压缩。

本地实现依据：
- `docs/INTERVIEW_PREP.md` 总结了 Gateway、Run Manager、Lead Agent、Sub-Agent、Sandbox、MCP Tools 的架构。
- `subagents/executor.py` 显示子 agent 是独立 agent graph，不是普通工具函数。
- `config.example.yaml` 明确有 subagents、skills、summarization、memory、checkpoint、store 等配置块。

小艺包装说法：
- “小艺 Claw 可以理解成把这套 agent runtime 接到小艺入口：小艺负责用户入口和设备上下文，云侧 runtime 负责复杂任务执行。”

## 三、Lead Agent 与中间件链

### 1. Lead Agent 是怎么构造的？

总答：Lead Agent 不是直接起一个模型，而是经过“配置解析 - 模型选择 - 工具加载 - 中间件组装 - prompt 渲染 - graph 创建”的完整工厂流程。

展开：
- 配置解析：从 `configurable` 和 `context` 合并出 model、thinking、plan_mode、subagent_enabled、agent_name、user_id 等运行参数。
- 模型选择：请求指定模型优先，其次 agent 配置，再退回全局默认；不支持 thinking 时自动降级。
- 工具加载：加载 sandbox、MCP、community、subagent task 等工具，并按技能白名单过滤。
- 中间件组装：根据配置决定是否启用 summarization、todo、memory、vision、tool search、subagent limit、loop detection 等。
- graph 创建：最终调用 `create_agent()`，传入 model、tools、middleware、system_prompt 和 ThreadState。

本地实现依据：
- `lead_agent/agent.py` 的 `_make_lead_agent()` 明确分成运行时参数、模型选择、追踪回调、技能过滤、工具加载、延迟工具、middleware、prompt、create_agent。
- `_resolve_model_name()` 体现模型选择回退。
- `_get_runtime_config()` 合并 `configurable` 和 `context`。

小艺包装说法：
- “小艺侧可以把用户请求带来的设备上下文、权限状态、模型选择和是否进入复杂任务模式，都放进 runtime context，由工厂构造出适合本次任务的 agent。”

### 2. 中间件链为什么重要？

总答：中间件链把横切能力从 agent loop 里拆出来，保证上下文、记忆、工具、视觉、子 agent、循环检测等能力按固定顺序执行。

展开：
- 依赖顺序：ThreadData 先创建线程目录，Uploads 依赖线程数据，Sandbox 依赖线程/文件环境。
- 上下文顺序：DynamicContext 注入日期/记忆，Summarization 尽早压缩历史，Todo 在 plan 模式下提供任务追踪。
- 结果顺序：TokenUsage 记录成本，Title 生成标题，Memory 在 Title 后更新，避免标题污染记忆。
- 工具顺序：ViewImage 在模型前注入图片，DeferredToolFilter 控制大工具 schema，SubagentLimit 限制子 agent 调用。
- 安全顺序：LoopDetection 防止重复工具调用，SafetyFinish 处理安全终止，Clarification 最后拦截澄清工具。

本地实现依据：
- `_build_middlewares()` 明确列出中间件顺序和条件启用逻辑。
- `build_lead_runtime_middlewares()` 提供 ThreadData、Uploads、Sandbox、工具错误处理等基础运行时中间件。
- 代码注释明确说明 DynamicContext 不放 system prompt，而是隐藏 HumanMessage 注入，以提高前缀缓存命中率。

小艺包装说法：
- “小艺是大流量入口，中间件链的价值是把复杂能力模块化：哪些影响性能、哪些影响安全、哪些影响用户体验，都能单独控制和评测。”

### 3. plan mode / Todo 的作用是什么？

总答：plan mode 是显式任务管理机制，用 todo 帮 agent 跟踪复杂多步目标，避免长任务执行中丢步骤。

展开：
- 使用场景：3 步以上的复杂任务、非平凡目标、用户明确要求跟踪、多任务列表、动态规划任务。
- 不使用场景：简单问答、单一工具调用、纯对话、方法明确的小任务。
- 执行要求：每次完成一个步骤立即标记 completed，始终保持一个 in_progress，计划可随中间结果调整。
- 用户价值：让复杂任务可见、可追踪，而不是 agent 在后台黑盒执行。

本地实现依据：
- `_create_todo_list_middleware()` 的 prompt 明确要求“复杂任务才使用”“实时更新”“一个 in_progress”“不要批量完成”。
- `_build_middlewares()` 只在 `is_plan_mode=true` 时插入 TodoMiddleware。

小艺包装说法：
- “在小艺侧，todo 不一定原样展示给用户，但可以转成进度提示，比如正在检索、正在分析、正在生成。”

## 四、Sub Agent 与任务委派

### 1. Sub Agent 和 Tool 有什么区别？

总答：Tool 是原子能力，Sub Agent 是一个完整 agent graph，适合需要多步推理和多轮工具调用的子任务。

展开：
- Tool：一次调用、输入输出明确，如读文件、执行命令、调用 API。
- Sub Agent：有自己的模型、prompt、工具集、技能、状态和循环。
- 使用场景：复杂检索、数据处理、代码/文件分析、需要多步推理的任务。
- 风险控制：sub agent 要有超时、最大轮次、工具白名单和取消机制。

本地实现依据：
- `subagents/executor.py` 中 SubagentExecutor 会创建独立 agent，并通过 `agent.astream()` 执行。
- `SubagentResult` 有 task_id、trace_id、status、result、error、ai_messages、token_usage_records、cancel_event。
- `SubagentStatus` 包括 pending、running、completed、failed、cancelled、timed_out。

小艺包装说法：
- “对用户来说它是一个任务；对系统来说复杂任务会被拆给不同子 agent，保证专项能力和上下文隔离。”

### 2. Sub Agent 怎么执行？

总答：Sub Agent 的执行流程是“过滤工具 - 加载技能 - 构造初始状态 - 创建子 agent - 流式执行 - 收集结果和 token”。

展开：
- 工具过滤：按 subagent 配置的 allowlist/denylist 过滤工具。
- 技能加载：按配置加载全部、禁用或白名单技能，并作为 SystemMessage 注入。
- 状态继承：父 agent 的 sandbox_state、thread_data、thread_id 可以传给子 agent。
- 执行控制：设置 recursion_limit、timeout_seconds、cancel_event。
- 结果收集：捕获 AI messages、最终 AIMessage、token usage 和错误状态。

本地实现依据：
- `_filter_tools()` 实现工具 allowlist/denylist。
- `_load_skills()` 支持 `None` 加载全部、`[]` 不加载、列表白名单。
- `_build_initial_state()` 把 system prompt、skill messages 和 task 合成初始 messages，并传递 sandbox/thread_data。
- `_aexecute()` 使用 `agent.astream(... stream_mode="values")` 流式执行，收集 AIMessage 和 token usage。

小艺包装说法：
- “小艺侧可以把检索、文档处理、工具执行拆给不同子 agent；但每个子 agent 只拿必要上下文和工具权限。”

### 3. Sub Agent 并发怎么控制？

总答：并发通过配置和中间件双重控制，避免模型一轮生成过多子任务导致成本和时延失控。

展开：
- 配置层：subagent 有全局 timeout、max_turns、每类 agent 的覆盖配置。
- 中间件层：SubagentLimitMiddleware 限制单轮最多几个 task 调用。
- 执行层：ThreadPoolExecutor / 隔离事件循环支撑后台执行。
- 状态层：每个 SubagentResult 独立记录状态和取消信号。

本地实现依据：
- `config.example.yaml` 里 subagents 默认 timeout 900s，general-purpose/bash 可分别配置 timeout/max_turns/model/skills。
- `_build_middlewares()` 在 `subagent_enabled=true` 时加入 SubagentLimitMiddleware，默认 max_concurrent_subagents 为 3。
- `subagents/executor.py` 使用 scheduler pool、持久隔离 event loop、cancel_event、timeout_seconds。

小艺包装说法：
- “小艺入口不能无限并行，必须控制子任务数量和超时，否则用户体验和成本都会失控。”

## 五、工具体系、MCP 与延迟工具

### 1. 工具是怎么接入和选择的？

总答：工具不是直接全部暴露给模型，而是经过工具加载、技能白名单过滤、延迟工具分离和运行期提升。

展开：
- 工具来源：sandbox、本地内置工具、MCP 工具、community 工具、subagent task 工具。
- 过滤机制：技能可以声明 allowed-tools，Agent 只保留当前技能允许的工具。
- 延迟绑定：大体积 MCP schema 默认不直接绑定给模型，通过 tool_search 按需发现。
- 失败策略：如果启用延迟工具但延迟集合异常为空，代码选择 fail-closed，不静默丢工具。

本地实现依据：
- `_make_lead_agent()` 调用 `get_available_tools()` 加载工具，再用 `filter_tools_by_skill_allowed_tools()` 过滤。
- `_assemble_deferred()` 把 MCP 工具从活跃列表分离到 deferred set，并追加 tool_search。
- `_assemble_deferred()` 对异常情况抛 RuntimeError，避免 MCP schema 静默丢失。
- `DeferredToolFilterMiddleware` 控制模型调用时工具 schema 的隐藏/提升。

小艺包装说法：
- “小艺可接入大量系统 API 和服务能力，但不能把所有工具 schema 都塞给模型。要先候选召回，再按需提升工具。”

### 2. API 很多时怎么选对工具？

总答：不能让模型在所有 API 中自由选择，要通过工具 taxonomy、候选召回、schema 校验和评测闭环来控制。

展开：
- 工具注册：每个工具有清晰名称、能力边界、参数 schema、失败条件和示例。
- 候选召回：根据意图和槽位筛出少量候选工具。
- 参数校验：模型填参后做 schema 校验，必要时让用户确认。
- 评测闭环：把误调用 case 回流到工具描述、候选策略和测试集。

本地实现依据：
- 工具策略过滤由 skill allowed-tools 驱动。
- 延迟工具机制说明工具规模很大时，上下文成本是实际问题。
- ToolErrorHandlingMiddleware / runtime middlewares 说明工具错误是统一处理的横切关注点。

小艺包装说法：
- “小艺侧 API 很多，核心不是工具越多越好，而是工具边界和候选选择足够清楚。”

## 六、Memory、Context 与 Summary

### 1. memory 怎么构建？

总答：要区分 workflow memory 和小艺产品侧 memory；前者服务一次长任务推进，后者服务用户个性化和跨会话体验。

展开：
- workflow memory：保存当前 task、plan、step 状态、工具结果、检索证据、阶段性 summary。
- 产品侧 memory：保存用户偏好、常用服务、设备上下文、长期事实。
- 写入边界：不是所有中间推理都入 memory，只有稳定、有复用价值、用户允许的信息才进入长期记忆。
- 使用边界：workflow memory 负责“任务继续执行”，产品侧 memory 负责“下次更懂用户”。

本地实现依据：
- `MemoryMiddleware` 在 `after_agent` 阶段触发，说明长期记忆不是每个 step 同步写，而是 agent 完成后入队。
- `filter_messages_for_memory()` 只保留用户输入和最终 AI 回复，工具调用过程被过滤。
- `MemoryUpdateQueue` 用 `(thread_id, user_id, agent_name)` 作为合并键，说明记忆按线程、用户、agent 隔离。
- `MemoryUpdater` 通过 LLM 提取 facts，持久化到 memory storage，支持 confidence、category、manual fact 等字段。
- `config.example.yaml` memory 配置包含 enabled、storage_path、debounce_seconds、max_facts、fact_confidence_threshold、injection_enabled、max_injection_tokens。

小艺包装说法：
- “面试里不要把 memory 讲成一个大池子。要讲清楚短期任务状态、summary 压缩和长期用户记忆是三件不同的事。”

### 2. memory 更新时机是什么？

总答：长期 memory 在 agent 完成后异步入队更新，摘要压缩前会强制 flush，避免历史被压缩前没保存。

展开：
- 正常路径：Agent 完成一轮后，MemoryMiddleware 过滤消息并入队。
- 去抖策略：同一 thread/user/agent 在去抖窗口内多次更新会合并，减少 LLM 调用。
- 紧急路径：summarization 发生前触发 memory flush，防止即将被压缩的消息还没进入记忆。
- 置信策略：MemoryUpdater 用 LLM 对新旧记忆做对比，提取新增 facts、删除过期 facts，并带 confidence。

本地实现依据：
- `MemoryMiddleware.after_agent()` 检查 memory.enabled、thread_id、messages、人类消息和 AI 回复后才入队。
- `MemoryUpdateQueue.add()` 默认去抖；`add_nowait()` 支持立即处理。
- `memory_flush_hook` 被挂到 summarization middleware 的 before_summarization hooks。
- `MemoryUpdater.update_memory()` 加载当前 memory、格式化会话、调用 MEMORY_UPDATE_PROMPT、解析 JSON、应用更新、保存。

小艺包装说法：
- “高频入口不能每轮都调用 LLM 更新长期记忆，所以要异步、去抖、合并，还要在摘要前兜底。”

### 3. summary 是怎么做的？

总答：summary 是上下文压缩层，触发条件是上下文接近阈值，目标是保留近期对话和关键任务状态。

展开：
- 触发：按 token、消息数或模型上下文比例触发；配置里默认 token 阈值。
- 保留：压缩后保留最近若干条消息，避免当前任务断层。
- 保护：最近加载的 skill 文件可以排除在摘要外，防止压缩后丢失技能指令。
- 风险：summary 只能替代部分上下文，不能替代原始证据和结构化状态。

本地实现依据：
- `config.example.yaml` summarization 默认 enabled，trigger token 32000，keep 最近 10 条消息。
- `_create_summarization_middleware()` 构造独立 summary model，并打 `middleware:summarize` 标签，便于 token 归因。
- summarization 配置包含 `preserve_recent_skill_count`、`preserve_recent_skill_tokens`、`skill_file_read_tool_names`。
- summary 前会执行 memory flush hook。

小艺包装说法：
- “小艺侧长任务不能无限堆上下文。summary 是为了控制成本和稳定性，但关键证据要结构化保存或可回查。”

### 4. plan 从 123 调整到 123456，旧记忆会不会影响结果？

总答：会，所以要维护 active plan 和状态版本，旧 plan 只能作为历史记录，不能继续驱动执行。

展开：
- 当前状态：保存 active task、active plan、已执行 step、工具结果和关键约束。
- 历史记录：保留旧 plan 的变更原因，但不直接参与下一步决策。
- summary 策略：summary 要强调“当前有效计划”，不要把旧计划和新计划并列成同等事实。
- 校验策略：执行前检查当前 step 是否属于 active plan。

本地实现依据：
- TodoMiddleware 要求实时更新任务状态，完成后立即标记，始终维护 in_progress。
- ThreadState / runtime context 保存 thread_id、run_id、messages、sandbox、thread_data 等运行状态。
- Summary 和 memory 是分层机制：summary 压缩历史，memory 存稳定事实，不能把旧 plan 当长期事实。

小艺包装说法：
- “用户多轮改目标很常见，所以小艺 Claw 要把‘当前任务状态’和‘历史对话’分开。”

### 5. 跨 session 怎么恢复？

总答：跨 session 恢复不是恢复完整聊天记录，而是恢复任务 summary、关键状态、必要记忆和权限上下文。

展开：
- 任务层：恢复 task goal、active plan、step 进度、工具结果摘要。
- 记忆层：注入与当前用户/agent 相关的长期事实。
- 时间层：把相对时间转换成绝对时间和时区。
- 权限层：重新确认已过期或高风险授权。

本地实现依据：
- worker 会把 `thread_id/run_id/app_config/user_id` 注入 runtime context。
- RunManager / checkpointer / store 负责运行状态和线程状态恢复。
- Memory storage 按 user_id/agent_name 隔离加载，DynamicContextMiddleware 负责动态上下文注入。

小艺包装说法：
- “小艺跨设备/跨天继续任务时，要恢复的是有效状态，不是把历史对话全塞回模型。”

## 七、RAG、检索与 Citation

### 1. 检索应该怎么设计？

总答：检索不是简单向量召回，而是 query rewrite、混合召回、rerank、证据压缩和 citation 绑定的链路。

展开：
- query rewrite：把口语化问题改成适合检索的表达。
- 混合召回：关键词/结构化过滤保证精确，向量检索补召回。
- rerank：按相关性、可信度和时效性排序。
- 证据压缩：保留关键片段，避免撑爆上下文。
- citation：关键结论绑定证据来源。

本地实现依据：
- 工具体系支持 MCP/外部工具接入，检索能力可以作为工具或 skill 接入。
- DeferredToolFilterMiddleware 说明外部工具 schema 可能很大，需要按需发现和提升。
- Summary/Memory 机制说明检索结果进入长任务后要做压缩和状态管理。

小艺包装说法：
- “小艺 Claw 不是单纯 RAG，而是 RAG + 工具 + 多步任务编排。检索结果必须服务最终任务，不是召回越多越好。”

### 2. citation 怎么设计？

总答：citation 的核心是让关键结论可追溯，而不是在回答末尾贴几个来源。

展开：
- 生成前：证据片段和候选结论绑定。
- 生成中：要求模型基于证据回答。
- 生成后：校验无证据支撑的结论，删除或降级表达。
- 展示层：用户不一定看完整过程，但关键出处要能展开。

本地实现依据：
- Result/ToolMessage/EventStore 的设计支持保留工具执行结果和消息事件。
- Summary 不应该替代原始证据，关键证据需要结构化或可回查。
- RunJournal / event store 思路能为结果追溯提供事件记录。

小艺包装说法：
- “小艺侧用户更关心可信度，所以 citation 不只是技术功能，也是降低不信任感的产品设计。”

## 八、评测体系与数据飞轮

### 1. agent 怎么评测？

总答：agent 评测要拆链路，不只看最终答案是否通顺。

展开：
- 任务成功率：用户目标是否完成。
- plan 质量：拆解是否合理，是否有遗漏或冗余。
- 工具准确率：工具是否选对，参数是否正确。
- 检索/citation：证据是否相关，是否支撑结论。
- 体验指标：首 token 时间、总耗时、取消率、追问率。
- 安全指标：权限、越权调用、敏感信息处理是否正确。

本地实现依据：
- TokenUsageMiddleware 和 SubagentTokenCollector 记录不同调用方的 token 成本。
- RunRecord 汇总 lead_agent_tokens、subagent_tokens、middleware_tokens、llm_call_count、message_count。
- LoopDetectionMiddleware、防止重复工具调用；SubagentLimitMiddleware 限制子 agent 数量。
- RunManager 记录 status、error、abort_event，支撑失败归因。

小艺包装说法：
- “小艺侧评测不能只看模型回答，要看任务是否完成、工具是否调对、用户是否愿意继续用。”

### 2. 数据飞轮怎么做？

总答：数据飞轮是把线上失败 case 分类回流到离线评测集和模块优化中。

展开：
- 采集：显性反馈、取消、追问、复制/执行、投诉、错误日志。
- 分类：意图错、工具错、检索弱、生成不可靠、时延过长、权限失败。
- 修复：分别改 prompt、tool schema、检索策略、路由策略、交互设计。
- 回归：高频失败 case 加入固定评测集，每次上线前跑回归。

本地实现依据：
- RunRecord 和 event stream 保存运行状态、错误、token、最后消息。
- Memory 检测 correction/reinforcement 信号，可用于识别用户纠偏和正反馈。
- Feedback/RunEvent/ThreadMeta 这类持久化模型在本地文档中作为运行历史和用户反馈基础。

小艺包装说法：
- “小艺入口量大，不能靠主观感觉迭代。要把失败链路拆成模块，再回流成评测集。”

## 九、沙箱、权限与安全

### 1. 沙箱解决什么问题？

总答：沙箱用于隔离高风险执行，比如命令、文件、代码和外部工具操作。

展开：
- 文件隔离：读写文件时限制在指定工作区。
- 命令隔离：命令在本地/Docker/K8s 沙箱执行，避免影响宿主环境。
- 路径安全：防止路径穿越和访问敏感路径。
- 审计：记录沙箱操作，便于排查和安全审计。

本地实现依据：
- 文档里 Sandbox 抽象包括 execute_command、read_file、download_file、list_dir、write_file、str_replace_editor。
- Provider 模式支持 local、Docker、K8s/provisioner。
- middleware 链里 sandbox 基础设施在前面创建，工具执行依赖 sandbox state。

小艺包装说法：
- “小艺侧涉及设备和用户数据，所有高风险执行都要先隔离、授权、审计。”

### 2. 权限和澄清怎么区分？

总答：澄清解决“用户想要什么”，权限解决“系统能不能做、是否允许做”。

展开：
- 澄清：目标、对象、约束不明确时提问。
- 权限：涉及文件、设备、消息、系统 API 时确认授权。
- 拒绝：安全或合规不允许时拒绝，并给安全替代方案。
- 状态：用户确认或拒绝都要写回 task state。

本地实现依据：
- ClarificationMiddleware 在 middleware 链最后注册，作为最终工具调用拦截点。
- Runtime context 注入 user_id/thread_id/run_id，使权限和上下文可传递。
- SafetyFinishReasonMiddleware 处理安全终止情况下的不完整 tool_call。

小艺包装说法：
- “小艺 Claw 不能为了完成任务越权执行，澄清和授权要分清楚。”

## 十、Prompt、Skill 与配置化

### 1. prompt 怎么管理？

总答：prompt 要按 agent、任务和中间件配置化管理，并与评测结果绑定。

展开：
- 分类：planner prompt、tool prompt、summary prompt、memory prompt、skill prompt。
- 版本：每次 prompt 变更要关联评测集和效果。
- 注入：系统 prompt 放稳定内容，动态上下文通过消息注入，减少缓存失效。
- 边界：prompt 负责约束行为，但不能替代 schema 校验和工具权限。

本地实现依据：
- `apply_prompt_template()` 在 agent 创建时渲染系统提示。
- DynamicContextMiddleware 以隐藏 HumanMessage 注入动态上下文，而不是改 system prompt。
- MemoryUpdater 使用 MEMORY_UPDATE_PROMPT 更新记忆。
- TodoMiddleware 有独立 system_prompt 和 tool_description。

小艺包装说法：
- “小艺侧 prompt 不是写一段话，而是和工具 schema、memory、评测集、灰度发布一起管理。”

### 2. skill 有什么作用？

总答：skill 是把某类任务的知识、流程和工具策略沉淀下来，让 agent 执行更稳定。

展开：
- 任务知识：给 agent 明确领域流程和注意事项。
- 工具策略：skill 可以限制 allowed-tools，减少错误调用。
- 子 agent 隔离：不同 sub agent 可加载不同技能白名单。
- 上下文控制：summary 压缩时保留最近读过的 skill 文件，避免丢指令。

本地实现依据：
- `_load_enabled_skills_for_tool_policy()` 加载启用技能并按白名单过滤。
- `filter_tools_by_skill_allowed_tools()` 按技能 allowed-tools 过滤工具。
- subagent executor 支持 `skills=None / [] / whitelist` 三种加载方式。
- summarization 配置支持 preserve_recent_skill_count / tokens / per_skill。

小艺包装说法：
- “小艺侧可以把不同业务场景做成 skill，比如出行、办公、文档整理、设备联动，每个 skill 有自己的工具边界。”

## 十一、行业对比与小艺差异

### 0. 业界统一做法是什么？

总答：业界主流 agent 系统基本都收敛到“编排器 + 专项 agent/tool + 状态记忆 + 上下文压缩 + 护栏 + tracing/eval”的分层架构。小艺 Claw 的讲法也应该往这个统一范式靠，而不是只讲某个具体框架或 prompt。

最佳实现模板：
- 第一层是入口层：先判断用户请求是普通问答、工具型任务，还是需要多步执行的长任务。普通问答走轻链路，长任务才进入 agentic workflow。
- 第二层是编排层：由 lead agent / supervisor 维护任务目标、阶段状态和全局约束，子 agent 只负责局部任务，避免多个 agent 同时争夺全局决策权。
- 第三层是工具层：工具必须有 schema、权限、超时、错误处理和审计，不把所有工具一次性塞进上下文，而是按场景动态检索和授权。
- 第四层是状态层：短期状态服务当前 run，长期 memory 服务用户偏好和跨会话连续性，两者隔离存储、隔离更新、隔离注入。
- 第五层是上下文层：长任务用 summary / compaction 控制 token，把关键决策、已完成步骤、待办事项、工具结果摘要留下，把冗余过程压掉。
- 第六层是安全与观测层：每个 run 都要能追踪 LLM 调用、工具调用、子 agent 调用、token 成本、失败原因和安全终止原因，便于评测和线上排障。

展开：
- 编排层：用 lead agent / supervisor / orchestrator 负责理解目标、拆解任务、分配子任务、聚合结果。OpenAI Agents SDK 把 agent 定义为能规划、调用工具、协作和保持状态的应用，LlamaIndex 也把 agent 定义为能拆复杂问题、选择工具、规划任务、存储已完成任务的推理决策引擎。
- 工具层：把外部能力做成可声明的 tool，包括工具名、参数 schema、能力边界、失败方式和权限要求。工具不应该全部无脑暴露给模型，而是按场景、角色、权限和上下文窗口做筛选。
- 状态层：把短期 workflow state 和长期 memory 分开。LangGraph 官方把 checkpointer 用于 thread-scoped 短期状态，把 store 用于跨线程长期数据；这和我们文档里“workflow memory 不等于长期用户记忆”的说法一致。
- 上下文层：长任务不能无限堆消息，需要 summarization / compaction。Anthropic 的 context engineering 文章提到会压缩历史，保留架构决策、未解决问题、实现细节等关键内容，同时丢弃冗余工具输出。
- 护栏层：复杂 agent 要有权限确认、输入/输出校验、工具边界、安全终止、循环检测和人工介入机制。OpenAI Agents SDK 也把 handoffs、guardrails、tracing 作为生产 agent 的核心能力。
- 可观测层：生产级 agent 必须能追踪 LLM 调用、工具调用、handoff、guardrail 和自定义事件。OpenAI Agents SDK tracing 明确强调 tracing 用于调试、可视化和生产监控。

本地实现依据：
- 编排层：`lead_agent/agent.py` 用 lead agent 工厂构造主 agent graph，运行期解析 model、plan_mode、subagent_enabled、agent_name 等配置。
- 工具层：工具经 `get_available_tools()` 加载，再经 skill allowed-tools 过滤；MCP 工具通过 deferred tool / tool_search 避免 schema 直接撑爆上下文。
- 状态层：`runtime/runs/worker.py` 注入 `thread_id/run_id/user_id/app_config`；`RunManager` 管 run 状态；Memory 模块按 `(thread_id, user_id, agent_name)` 合并和隔离。
- 上下文层：SummarizationMiddleware 按 token 阈值压缩历史，并保留最近消息和最近加载的 skill 文件；压缩前触发 memory flush。
- 护栏层：ClarificationMiddleware、SafetyFinishReasonMiddleware、LoopDetectionMiddleware、SubagentLimitMiddleware、SandboxMiddleware 分别处理澄清、安全终止、循环、子 agent 数量和隔离执行。
- 可观测层：RunRecord 记录 token、LLM 调用、消息数和最后回复；SubagentTokenCollector 区分子 agent token；summary 模型打 `middleware:summarize` 标签，便于成本归因。

小艺包装说法：
- “我们的设计没有把小艺 Claw 做成单个大 prompt，而是按业界生产 agent 的通用范式拆层：入口路由、任务编排、工具权限、状态记忆、上下文压缩、护栏、追踪评测。这样才能支撑小艺这种系统级高频入口。”
- “如果面试官问最佳实践，我会强调两点：第一，agent 不是越自主越好，而是要把自主性关在可观测、可恢复、可授权的边界里；第二，系统级助手不能只追求一次任务做完，还要考虑用户隐私、设备权限、低时延和失败后的可解释恢复。”

### 1. 和 Deep Research 类能力怎么对比？

总答：共同点是 long-horizon task，不是单轮问答；不同点是小艺要跑在系统级入口里。

展开：
- 共同点：任务拆解、多步检索、工具使用、综合分析、引用校验。
- 小艺差异：用户请求更口语化，设备/应用上下文更多，权限边界更复杂。
- 工程约束：小艺入口要求低时延、高稳定、可恢复、可解释。
- 产品约束：用户不一定懂 agent，结果要像助手自然完成任务。

本地实现依据：
- 本地 runtime 具备 long task 的 run 状态、stream、subagent、summary、memory、tool schema 控制等能力。
- 小艺包装层需要把这些能力转成面向普通用户的自然交互。

小艺包装说法：
- “小艺 Claw 不是只做报告生成，而是把长任务能力嵌入小艺入口，让用户自然触发和追问。”

### 2. 和 Claude Code / open claw 怎么对比？

总答：它们都强调工具调用和上下文管理，但小艺 Claw 面向普通用户和设备生态，不是只面向开发者。

展开：
- Claude Code：偏开发者，围绕代码仓库、项目规则、终端和文件。
- open claw：偏 agentic coding / workflow，强调工具执行和长任务。
- 小艺 Claw：偏系统级用户入口，强调语音/多模态、设备上下文、权限、服务 API 和用户体验。
- 差异：小艺更需要轻重任务分流和权限审计，不能所有任务都进入重流程。

本地实现依据：
- 本地 runtime 支持 sandbox、subagent、skills、MCP、memory、summarization、tool search，和编码 agent 能力有相似基础。
- 小艺包装层要增加用户入口、设备状态、系统 API、权限和体验控制。

小艺包装说法：
- “可以借鉴开发者 agent 的 runtime 思路，但小艺侧的核心是普通用户可用、稳定、可信、低打扰。”

## 十二、补漏题与加厚题

### 1. 用户意图模糊时怎么澄清？

总答：不是所有模糊请求都直接交给模型猜，生产系统里要先判断“能否安全默认、是否影响结果、是否涉及权限”，再决定追问、给选项、或先执行低风险步骤。

展开：
- 可安全默认：比如用户说“帮我整理一下”，可以先按通用结构整理，同时说明默认口径。
- 需要澄清：目标、对象、时间范围、输出格式、权限动作不明确时，要先问。
- 给选项而不是开放追问：小艺场景里用户更偏口语交互，最好给 2-3 个可选项，降低交互成本。
- 渐进式澄清：先问阻塞任务的问题，不一次问一串。
- 澄清后要写回当前 run state，避免下一轮又丢失约束。

本地实现依据：
- ClarificationMiddleware 负责在 agent 继续执行前插入澄清判断。
- worker 会把 `thread_id/run_id/user_id/app_config` 注入运行上下文，澄清后的约束能留在同一 run/thread。
- Todo/plan 能把澄清后的目标拆成可执行步骤。

小艺包装说法：
- “小艺侧澄清不是为了显得聪明，而是为了避免在用户意图不明确时越权或做错。我的设计是能默认就默认，影响结果或权限的才追问，并且尽量用选择题式澄清。”

### 2. 用户否定回答或修改目标时，流程怎么处理？

总答：用户否定不是异常，而是 agent workflow 的常态输入。系统要把它识别成 correction signal，更新当前目标、废弃过期 plan，并避免旧结论继续影响后续结果。

展开：
- 先识别否定类型：是事实纠错、偏好变更、任务目标变化，还是对结果不满意。
- 再处理状态：如果只是补充约束，更新 plan；如果目标变化大，要重建 plan。
- 对 memory 要谨慎：不是所有否定都写长期记忆，只有稳定偏好和事实才沉淀。
- 对工具结果要标记失效：旧工具调用结果不能直接复用，需要绑定版本或阶段。
- 对用户要明确响应：“我按新的目标重新规划”，减少用户觉得系统固执。

本地实现依据：
- MemoryMiddleware 会检测 correction/reinforcement 信号，但通过队列和 updater 做结构化更新，不是把每句话都持久化。
- TodoMiddleware 支持把任务状态变成显式待办，目标变化时可以重排。
- Summarization 前触发 memory flush，避免旧上下文压缩后丢失关键纠错信息。

小艺包装说法：
- “用户说‘不是这个意思’时，我不会把它当失败，而是当作一次目标重写。小艺 Claw 要能把当前 run 的 plan 更新掉，同时避免把临时否定误写成长期用户画像。”

### 3. 用户问超纲问题或系统做不了，怎么处理？

总答：超纲问题要走“识别边界 - 给可做范围 - 降级方案 - 必要时拒绝或转人工”的链路，不能让模型硬编。

展开：
- 能力超纲：没有对应工具或数据源，就说明限制，并给可替代路径。
- 权限超纲：涉及设备控制、账号、支付、隐私数据，要先授权。
- 安全超纲：违反安全策略时拒绝，并给安全替代建议。
- 上下文超纲：信息不足时先澄清，不把猜测包装成事实。
- 系统异常：工具失败、模型超时、沙箱失败，要可恢复、可重试、可解释。

本地实现依据：
- Guardrail/Sandbox/SafetyFinishReason 负责权限、安全和终止原因控制。
- LLM error handling middleware 会把模型侧 quota、限制、异常转成可解释错误。
- tool 调用本身有错误包装，agent 可以换工具或回退。

小艺包装说法：
- “系统级助手最怕一本正经地胡说。我的处理原则是：能做的继续做，不能做的说明边界，需要权限的先授权，失败的给恢复路径。”

### 4. 整体有几个模型？大小模型怎么分工？

总答：面试里不要死背具体模型数量，应该讲“模型路由策略”。复杂推理、计划、总结用强模型；分类、改写、轻量提取、格式化用小模型；高风险动作再加规则或评审模型。

展开：
- 主 agent 模型：负责复杂任务理解、计划、工具选择和最终综合。
- summary 模型：负责压缩历史，要求稳定、便宜、上下文能力够。
- memory 更新模型：负责从对话里提炼事实和偏好，重点是结构化和保守更新。
- 工具选择/路由模型：可以用小模型或规则召回，减少把大量工具塞进主模型。
- 安全/评测模型：用于输出质量、风险判断、离线评测或线上抽检。
- 小模型适合高频低风险步骤，大模型适合低频高复杂步骤。

本地实现依据：
- `models/factory.py` 统一通过配置创建模型，并支持按 `model_name` 解析。
- lead agent 创建时会解析请求指定模型，找不到则 fallback 到默认模型。
- summary 单独创建模型，并打独立 token 归因标签。
- config 支持 subagent 继承主模型或指定模型。

小艺包装说法：
- “小艺 Claw 不应该所有请求都打到最贵最强模型。我的设计是入口先分流：轻任务走轻模型或规则，长任务走强模型编排，summary/memory/评测这些后台步骤再用性价比模型。”

### 5. 微调怎么做？项目里有没有必要微调？

总答：这个项目优先不是微调，而是先做 prompt、tool schema、检索、memory、eval 数据闭环。微调适合稳定、高频、边界清楚的子任务，不适合一开始就拿来解决所有 agent 问题。

展开：
- 先不用微调解决的问题：工具选错、权限边界不清、上下文污染、数据缺失，这些应该先改系统。
- 可以微调的场景：意图分类、query rewrite、结构化抽取、格式遵循、业务术语理解。
- 数据来源：线上失败 case、用户纠错、人工标注、灰度对比。
- 训练前提：有稳定标签体系、评测集和回归集。
- 上线方式：小流量灰度，和 prompt 版本、模型版本、工具版本一起记录。

本地实现依据：
- 当前 runtime 已经有 feedback、run event、token usage、trace 这类数据基础。
- tool_search、prompt、skill、memory 都是可以先迭代的系统层杠杆。
- 真要微调，也应该先从可评测的小任务开始，而不是直接微调整个 agent。

小艺包装说法：
- “我不会把微调当万能药。小艺这种系统级入口，第一优先级是工具边界、上下文和评测闭环；微调用在高频稳定子任务上，比如意图识别和结构化抽取。”

### 6. ReAct、CoT、Plan 模式怎么理解？什么时候用？

总答：CoT 是模型内部推理方式，ReAct 是“推理 + 行动”的工具调用范式，Plan 是显式任务管理。生产 agent 里更推荐显式 plan + 可观测 tool/action，而不是只依赖隐式思考。

展开：
- CoT：适合复杂推理，但生产上不一定暴露完整思考链，重点看结果和可解释摘要。
- ReAct：适合边想边查、边调用工具的任务，比如检索、代码、文件处理。
- Plan：适合长任务，先拆步骤，再逐步执行和更新。
- 简单问答不需要 plan，否则增加延迟和成本。
- 长任务、跨工具、跨多轮、需要恢复的任务必须有 plan 或 todo。

本地实现依据：
- Lead agent 用 graph 进入 LLM/tool loop，本质支持 ReAct 风格执行。
- TodoMiddleware 把计划显式化，适合长任务和用户追问。
- `is_plan_mode` 是运行期配置，可以按任务复杂度启用。

小艺包装说法：
- “我会把 plan 用在小艺侧的长任务，比如整理资料、跨应用操作、连续生成；普通问答不强行 plan。这样既能控制成本，也能让用户看到任务进度。”

### 7. Supervisor 和子 agent 怎么划分职责？

总答：Supervisor 负责全局目标和最终质量，子 agent 负责局部任务。边界不能反过来，否则多个子 agent 都在做全局决策，会导致冲突、重复调用工具和结果不可控。

展开：
- Supervisor：理解用户目标、拆任务、分配任务、控制权限、整合结果、判断是否结束。
- 子 agent：拿到明确任务包，在限定工具和上下文内完成子任务。
- 子 agent 不应该修改全局目标，也不应该无限创建新的子 agent。
- 子 agent 的输出要回到 supervisor，由 supervisor 做冲突消解和最终表达。
- 子 agent 适合并行检索、代码执行、文件分析、独立专题研究。

本地实现依据：
- subagent executor 会构造独立 initial state，只传 task、system prompt、skill messages、sandbox/thread data。
- SubagentStatus 有 pending/running/completed/failed/cancelled/timed_out。
- SubagentLimitMiddleware 控制子 agent 数量，避免无限委派。
- 子 agent token 通过 SubagentTokenCollector 独立归因。

小艺包装说法：
- “小艺 Claw 里主 agent 像项目经理，子 agent 像专项执行者。主 agent 保证用户目标不丢，子 agent 只在受限上下文里完成局部任务。”

### 8. 子 agent 如何隔离上下文？

总答：子 agent 不能共享主 agent 的全部上下文，而是由主 agent 生成任务包，只传必要目标、约束、工具和少量背景。这样能降低 token、减少污染、保护隐私。

展开：
- 任务隔离：子 agent 只知道自己要解决的问题。
- 工具隔离：通过 allowlist/denylist 限定可用工具。
- memory 隔离：长期记忆按用户、线程、agent 名称隔离。
- sandbox 隔离：文件、命令、网络等能力在隔离环境执行。
- 结果隔离：子 agent 返回结果、错误、token 和关键消息，不把完整中间过程全灌回主上下文。

本地实现依据：
- `_filter_tools` 支持 allowlist/denylist。
- `_load_skills` 支持 `None / [] / whitelist` 三种加载方式。
- `_build_initial_state` 只组装子任务需要的 prompt、skill、task 和运行态。
- Memory key 使用 `(thread_id, user_id, agent_name)`，天然支持隔离。

小艺包装说法：
- “小艺有大量隐私和设备上下文，所以子 agent 不能随便拿全量上下文。我的做法是任务包最小化、工具白名单、memory 分域隔离。”

### 9. 端侧和云侧怎么通信？怎么保持长链路？

总答：长任务适合用“请求创建 run + 流式事件返回 + 可恢复查询”的协议。端侧负责展示和用户交互，云侧负责执行和状态管理，中间用 run_id/thread_id 维持会话连续性。

展开：
- 创建阶段：端侧带上 thread_id、用户输入、配置和权限信息，请求云侧创建 run。
- 执行阶段：云侧异步执行 agent graph，通过 SSE/流式事件返回 token、工具调用、状态更新和结束事件。
- 恢复阶段：端侧断线后可用 run_id 查询 run 状态、历史消息和事件。
- 取消阶段：端侧可以发送 cancel，云侧通过 abort_event/cancel_event 协作终止。
- 体验层：端侧不需要等最终答案才更新，可以展示“正在检索/正在调用工具/正在总结”。

本地实现依据：
- Gateway 有 `/runs/stream` 和 thread run 生命周期接口。
- `StreamBridge` 解耦 agent 生产事件和 HTTP SSE 消费。
- client 侧事件和 LangGraph stream mode 对齐，包括 messages、values、custom、end。
- RunManager 维护 run 状态，支持 cancel/join/list/get。

小艺包装说法：
- “小艺端侧更像交互壳和权限入口，云侧 runtime 负责长任务执行。协议上用 thread_id 保持上下文，用 run_id 追踪一次任务，用流式事件改善体感延迟。”

### 10. API 很多时，怎么精准选择工具？

总答：工具选择不能只靠主模型在几百个 schema 里硬挑，应该先做工具分层、动态召回、权限过滤，再把少量候选工具交给模型决策。

展开：
- 工具目录：每个工具有名称、描述、参数 schema、适用场景、风险级别。
- 粗召回：按用户意图、关键词、skill、业务域召回候选工具。
- 权限过滤：没有授权、当前场景不能用、风险过高的工具先剔除。
- 上下文控制：只把候选工具 schema 注入主模型，避免 token 爆炸。
- 失败回退：工具结果为空或错误时，允许换工具、澄清或降级。

本地实现依据：
- `get_available_tools()` 统一加载工具。
- skill allowed-tools 会二次过滤工具。
- deferred tool / tool_search 先提供工具搜索入口，需要时再绑定真实工具。
- deferred 工具为空时有 fail-closed 保护，避免模型以为能用但实际不可用。

小艺包装说法：
- “小艺生态里 API 会非常多，我不会让模型一次看到全部 API，而是先按场景召回和权限过滤，只暴露少量候选工具给主 agent。”

### 11. RAG 用哪种检索？向量、关键词还是混合？

总答：实际项目里优先用混合检索：关键词保证精确匹配，向量保证语义召回，再加 rerank 和业务过滤。单纯向量检索在专有名词、编号、时间、接口名上容易漏。

展开：
- 向量检索：适合语义相似、自然语言问题。
- 关键词检索：适合专有名词、错误码、接口名、文档标题、数字。
- 混合检索：先并行召回，再 rerank，兼顾召回率和准确率。
- 结构化过滤：按用户、时间、文档类型、权限、业务域过滤。
- 多跳任务：检索结果不直接给用户，要服务后续工具调用和综合推理。

本地实现依据：
- 当前工具体系支持 web_search、web_fetch、image_search 等搜索类工具。
- file conversion 支持把文件转成 Markdown，适合后续文本检索和引用。
- citation 设计可以把检索来源和最终答案绑定，降低幻觉。

小艺包装说法：
- “小艺侧 RAG 不应该只做向量库问答。我会用混合检索，先保证找得到，再通过 rerank、权限过滤和 citation 保证可信。”

### 12. 检索对象是文字为主，还是多模态？

总答：第一阶段以文字为主，因为文字最容易切分、索引、引用和评测；但小艺场景天然会走向多模态，需要把图片、PDF、截图、语音转成可检索的结构化表示。

展开：
- 文本文档：直接切分、索引、引用。
- PDF/Office：先转换成 Markdown，保留标题、段落、表格结构。
- 图片/截图：先 OCR/视觉模型抽取文字和视觉描述，再进入索引。
- 语音：先 ASR 成文本，再结合时间戳和说话人信息。
- 多模态引用：不能只说“见图片”，要能定位到图片、页码、区域或时间片段。

本地实现依据：
- uploads 和 file_conversion 支持文件进入 agent 上下文。
- image_search/tool 和 view image 能力说明系统已经考虑图像入口。
- summary 和 citation 机制可以把长文档、多轮上下文压缩为可引用信息。

小艺包装说法：
- “小艺是系统级助手，用户可能发截图、语音、文档，所以底层要能把多模态输入转成结构化上下文。但第一优先级还是把文本链路做稳。”

### 13. 多目标或多个相似对象时怎么确认用户想要哪一个？

总答：多个候选目标时不要擅自选，要先排序再澄清。排序依据是上下文相关性、最近使用、用户历史偏好、权限可用性和业务优先级。

展开：
- 候选生成：识别可能对象，比如多个文件、多个联系人、多个设备、多个 API。
- 候选排序：结合当前上下文、最近操作、命名匹配、权限状态。
- 低风险默认：如果置信度很高，可以执行并说明对象。
- 高风险澄清：涉及删除、支付、发送、设备控制必须确认。
- 澄清表达：给用户展示候选列表，而不是问“你指哪个”。

本地实现依据：
- ThreadDataMiddleware/UploadsMiddleware 能把线程数据和上传文件纳入上下文。
- ClarificationMiddleware 可在目标不唯一时阻断执行。
- Guardrail 能对高风险工具调用做二次校验。

小艺包装说法：
- “比如用户说‘把那个文档发给他’，小艺侧要先解析文档和联系人候选。低风险可默认，高风险必须让用户确认具体对象。”

### 14. 出海场景下，英文输出质量不好怎么优化？

总答：出海不是简单翻译，要从语言、文化、工具、评测和用户反馈一起优化。尤其 agent 输出要符合当地表达习惯和任务预期。

展开：
- 语言层：使用目标语言 prompt、术语表、风格指南。
- 数据层：补充目标地区真实 query、失败 case 和人工标注。
- 工具层：检索源、地图、服务 API 要切到当地可用源。
- 评测层：用 native speaker 或本地化 LLM judge 评估自然度、准确性、礼貌边界。
- 产品层：不同地区对隐私、权限、推荐语气的接受度不同。
- 灰度层：按地区、语言、场景灰度，不一次全量。

本地实现依据：
- prompt/skill 是配置化的，可以按地区和场景加载不同指令。
- tool config 支持不同搜索 provider 和参数。
- feedback/run trace 可以收集地区化失败样本，回流到评测集。

小艺包装说法：
- “出海优化不是把中文答案翻译成英文，而是构建本地化 agent：本地数据、本地工具、本地评测、本地表达习惯。”

### 15. 整个流程耗时多久？怎么关注实时性？

总答：agent 长任务不能只看最终耗时，要拆成首 token 时间、工具等待时间、子任务耗时、总完成时间和用户可感知进度。优化目标是“可感知快”和“最终可靠”同时满足。

展开：
- 首响延迟：用户发起后多久看到系统开始响应。
- 流式输出：先返回计划/状态，再逐步返回结果。
- 工具耗时：搜索、文件解析、沙箱、外部 API 分别统计。
- 子 agent 耗时：并行化独立任务，但要控制数量和超时。
- 总耗时：长任务可接受更久，但必须显示进度和可取消。
- 成本指标：token、调用次数、失败重试次数要纳入评估。

本地实现依据：
- StreamBridge/SSE 支持流式事件。
- RunRecord 记录 token、LLM 调用、消息数和状态。
- SubagentStatus 支持 timed_out/cancelled。
- TokenUsageMiddleware 和 SubagentTokenCollector 能做成本归因。

小艺包装说法：
- “小艺用户不一定愿意等黑盒跑完，所以我会优先保证首响和过程可见：先展示计划，再流式更新执行状态，长任务支持取消和恢复。”

### 16. 上线一个 skill 前怎么评测？

总答：skill 上线前要测“能不能触发、会不会误触发、工具边界是否正确、输出质量是否稳定、是否引入安全风险”，不能只测几条 happy path。

展开：
- 触发评测：该触发时能触发，不该触发时不触发。
- 工具评测：allowed-tools 是否最小可用，是否漏工具或多给危险工具。
- 任务评测：用真实 case 测完成率、步骤正确率、结果质量。
- 回归评测：新 skill 不能影响已有 skill 和通用 agent。
- 安全评测：权限、隐私、越权、注入攻击。
- 灰度评测：小流量上线，观察失败率、用户反馈、token 和延迟。

本地实现依据：
- skill storage 支持 enabled_only 和 skill whitelist。
- skill allowed-tools 控制工具边界。
- subagent executor 支持按 skill whitelist 加载。
- run trace、feedback、token usage 可做上线后观测。

小艺包装说法：
- “skill 上线不是写完提示词就结束，我会把它当一个可发布能力：有触发集、回归集、工具白名单、安全 case 和灰度指标。”

### 17. Prompt 本地调得好，怎么判断线上也好？

总答：prompt 不能靠感觉上线，要有离线评测、A/B 灰度、线上监控和失败回流。尤其 agent prompt 改动会影响工具调用路径，必须看完整 trajectory。

展开：
- 离线集：覆盖高频问题、边界问题、失败样本、权限样本。
- 指标：任务完成率、工具选择准确率、格式遵循、幻觉率、澄清率、用户满意度。
- trajectory：不仅看最终答案，也看中间是否乱调工具、是否多花 token。
- A/B：同模型同工具下比较 prompt 版本。
- 线上回流：差评、取消、重复追问、人工纠错进入回归集。

本地实现依据：
- RunJournal/event store 能保留完整事件链。
- feedback 表按 run 记录用户反馈。
- token usage 能比较 prompt 改动后的成本变化。
- summary/message history 的修复思路说明系统重视完整历史和反馈对齐。

小艺包装说法：
- “prompt 线上好不好，不是看 demo 顺不顺，而是看真实用户任务完成率、工具路径、成本、延迟和差评回流。”

### 18. 用户侧评价体系怎么设计？

总答：用户侧评价要分“结果质量、过程体验、效率收益、信任安全”四类，不只看点赞率。

展开：
- 结果质量：是否解决问题、准确性、完整性、引用可信度。
- 过程体验：澄清是否少、等待是否可接受、进度是否清楚。
- 效率收益：节省时间、减少操作步骤、替代人工流程。
- 信任安全：权限是否清晰、是否越界、失败是否可解释。
- 行为指标：重试率、追问率、取消率、任务完成率、二次使用率。
- 主观指标：满意度、NPS、人工标注质量分。

本地实现依据：
- feedback 路由支持按 run 写入和统计反馈。
- RunManager 和 event store 能关联一次任务的状态、消息、工具和反馈。
- token/latency/error 可以和质量指标一起做 dashboard。

小艺包装说法：
- “小艺是高频入口，评价不能只看回答是否漂亮，还要看用户是否少操作、少等待、敢授权、愿意下次继续用。”

### 19. 数据飞轮回流后，怎么证明对产品有效？

总答：数据飞轮有效性要用“回流样本 - 修复方案 - 回归评测 - 线上指标变化”证明，而不是说收集了数据就算闭环。

展开：
- 采集：线上失败、用户差评、人工纠错、工具失败 trace。
- 标注：标出失败类型，如意图错、工具错、知识错、权限错、表达错。
- 修复：对应到 prompt、tool schema、memory、RAG、模型或产品交互。
- 回归：修复前后的离线集分数对比。
- 线上：灰度观察完成率、差评率、取消率、成本和延迟。
- 防退化：把失败 case 固化进长期回归集。

本地实现依据：
- run event、feedback、token usage 提供回流数据基础。
- Prompt/skill/tool config 都是可迭代对象。
- eval suite 可以把失败 trace 转成固定 case。

小艺包装说法：
- “我会把数据飞轮讲成工程闭环：线上失败不是只做统计，而是要变成回归 case，推动 prompt、工具、检索、memory 的下一轮迭代。”

### 20. 内部提效工作和 AI coding 怎么展开？

总答：内部提效不能只靠让大家“多用 AI”，要选高频、重复、有明确验收标准的流程，做成模板、skill 或工具链，再用指标证明收益。

展开：
- 选场景：代码 review、测试生成、日志分析、文档总结、需求拆解、接口联调。
- 做模板：沉淀 prompt、上下文收集方式、输出格式和验收清单。
- 接工具：代码仓库、终端、CI、文档、缺陷系统。
- 控风险：权限隔离、敏感信息过滤、人工确认。
- 量效果：节省时间、缺陷发现率、测试覆盖率、PR 周期、采纳率。
- 推广方式：先找小团队试点，再把成功 case 产品化。

本地实现依据：
- skill 机制适合把团队最佳实践沉淀成可复用能力。
- sandbox 支持隔离执行命令和文件操作。
- event/feedback 可以衡量内部工具真实使用效果。

小艺包装说法：
- “AI coding 提效要从流程入手，不是发一个聊天入口。我的思路是把高频研发动作做成 skill 和工具链，再用 PR 周期、缺陷率、采纳率衡量。”

### 21. vibe coding 有哪些指标？平时怎么用？

总答：vibe coding 不能只看生成了多少代码，要看从想法到可运行结果的速度，以及代码质量、可维护性和人工接管成本。

展开：
- 速度指标：需求到首版 demo 时间、一次修改耗时、PR 周期。
- 质量指标：测试通过率、静态检查通过率、线上缺陷率、review 问题数。
- 采纳指标：AI 生成代码采纳率、人工重写率、返工次数。
- 复杂度指标：是否引入不必要抽象、是否符合项目规范。
- 体验指标：开发者满意度、上下文准备成本、调试成本。
- 适合场景：原型、脚手架、测试、迁移、重复 CRUD；不适合无边界的核心架构直接放飞。

本地实现依据：
- sandbox + tools 可以支持代码执行、文件检查和测试。
- run trace 能记录从需求到执行的路径。
- feedback 和 token usage 可以衡量效率与成本。

小艺包装说法：
- “vibe coding 的价值不是代码行数，而是让想法更快变成可验证结果。指标上我会同时看速度、质量和人工接管成本。”

## 十三、基础技术与算法题

### 1. Go 协程调度模型

总答：Go 使用 GMP 调度模型，把大量 goroutine 映射到较少 OS thread 上执行。

展开：
- G：goroutine，代表用户态轻量任务。
- M：machine，对应 OS thread。
- P：processor，持有本地运行队列，是调度 G 的执行资源。
- 机制：本地队列、全局队列、work stealing、系统调用阻塞处理和抢占式调度。

### 2. 大量 close wait 的原因

总答：close wait 表示对端已关闭连接，本端应用没有及时 close socket。

展开：
- 代码问题：异常路径没有关闭连接。
- 资源泄漏：连接池或 goroutine 卡住。
- 读写阻塞：应用层读写逻辑没有退出。
- 排查：看连接生命周期、异常处理、超时配置和 goroutine 堆栈。

### 3. LC 33 搜索旋转排序数组

总答：用二分。每次判断左半边还是右半边有序，再判断 target 落在哪一边。

展开：
- 如果 `nums[l] <= nums[mid]`，说明左半边有序。
- 如果 `nums[l] <= target < nums[mid]`，收缩右边；否则去右半边。
- 反之右半边有序，判断 `nums[mid] < target <= nums[r]`。
- 时间复杂度 `O(log n)`，空间复杂度 `O(1)`。

面试提醒：
- 先问是否有重复元素。LC33 默认无重复；如果有重复，会退化成另一类题。

### 4. LC 792 匹配子序列的单词个数

总答：如果 words 很多，不要每个单词都扫一遍 s。可以预处理 s 中每个字符出现的位置，再对每个 word 用二分找下一个位置。

展开：
- 建 `pos[char] = [index1, index2...]`。
- 对 word 的每个字符，在对应列表里找大于当前 index 的第一个位置。
- 找不到则不是子序列。
- 复杂度约为 `O(|s| + sum(len(word) * log |s|))`。

面试提醒：
- 如果数据量小，双指针也可以；但面试官问这题一般想看批量查询优化。

### 5. LC 3 最长无重复子串

总答：滑动窗口 + 哈希表记录字符最近位置。

展开：
- 维护窗口左边界 `left`。
- 遍历右边界 `right`，如果字符出现过且位置在窗口内，移动 `left`。
- 每次更新最大长度。
- 时间复杂度 `O(n)`，空间复杂度 `O(字符集大小)`。

面试提醒：
- 更新 `left` 时要取 `max(left, last[c] + 1)`，避免左边界回退。

### 6. LC 35 搜索插入位置

总答：标准 lower_bound，找第一个 `>= target` 的位置。

展开：
- 左闭右开或左闭右闭都可以，但边界要统一。
- 如果 target 存在，返回它的位置。
- 如果不存在，返回最终插入点。
- 时间复杂度 `O(log n)`。

面试提醒：
- 这题简单，但容易暴露二分边界不熟。建议固定背一种模板。

### 7. LC 1004 最大连续 1 的个数 III

总答：滑动窗口，窗口内最多允许 `k` 个 0。

展开：
- 右指针扩展，遇到 0 就增加 zero count。
- 如果 zero count 大于 k，左指针右移直到窗口合法。
- 每次窗口合法时更新最大长度。
- 时间复杂度 `O(n)`。

面试提醒：
- 可以把它抽象成“最多替换 k 个字符后最长连续区间”。

### 8. LC 88 变种：合并多个有序数组/日志

总答：两个有序数组可以从后往前双指针；多个有序数组适合用小顶堆做 k 路归并。

展开：
- 两数组原地合并：从末尾开始放最大值，避免覆盖未处理元素。
- 多数组合并：堆里放每个数组当前元素，弹出最小值后推进该数组指针。
- 如果是日志按时间合并，堆元素就是 `(timestamp, source_id, index)`。
- 复杂度：总元素数为 N、数组数为 k，则 `O(N log k)`。

面试提醒：
- 如果追问“快速搜索某个时间段有多少日志”，合并后用二分找 `[start, end]` 边界，查询复杂度 `O(log N)`。
