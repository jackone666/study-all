## 一、网络模型

### 1. OSI 七层模型是什么？每一层的作用是什么？ [重要性:A]

OSI（Open System Interconnection，开放系统互连）七层模型是国际标准化组织（ISO）制定的**网络通信分层标准**，通过分层解耦降低网络设计复杂度，从下到上分为 7 层：

| **层级** | **核心作用** | **典型协议 / 设备** | **数据单元** |
| --- | --- | --- | --- |
| **应用层** | 直接为应用程序提供网络服务，定义应用程序间通信的格式和规则，面向用户业务。 | HTTP、HTTPS、FTP、SMTP、DNS | 报文（Message） |
| **表示层** | 负责**数据格式转换、加密解密、压缩解压缩**，确保不同系统应用层数据互通。 | JPEG、SSL/TLS、ASCII | 报文 |
| **会话层** | 建立、管理、终止**进程间的会话连接**，控制会话同步（如断点续传标记）。 | RPC、NetBIOS | 报文 |
| **传输层** | 提供**端到端（进程到进程）** 的通信服务，核心是可靠传输和流量控制。 | TCP、UDP | 报文段（Segment）/ 用户数据报（Datagram） |
| **网络层** | 负责**跨网络的路由与转发**，为分组选择最佳路径，实现不同网络间的互联。 | IP、ICMP、OSPF（ARP 位于 IP 与链路层之间，分层归属按教材口径略有差异） | 分组 / IP 数据报（Packet） |
| **数据链路层** | 负责**相邻节点间的帧传输**，封装 IP 数据报为帧，处理 MAC 地址寻址、差错检测。 | 以太网协议、PPP、CSMA/CD | 帧（Frame） |
| **物理层** | 负责**物理介质上的比特流传输**，定义接口标准（如电压、接口类型），屏蔽硬件差异。 | RJ45、光纤、Wi-Fi 物理层规范 | 比特（Bit） |

> **记忆口诀：物 → 链 → 网 → 传 → 会 → 表 → 应**（从下到上顺序）。谐音记忆：**"物联网传会表应"** = 想让"物联网"传数据，得先"会话→表示→应用"。或者用一句话：**"物理链路网传输，会话表示应用层"**。

### 2. TCP/IP 四层模型是什么？每一层的作用是什么？ [重要性:A]

TCP/IP 是**实际互联网采用的协议栈**，比 OSI 更简洁，分为四层，与 OSI 七层的对应关系是：**应用层对应 OSI 上三层，网络接口层对应 OSI 下两层**。

| **层级** | **核心作用** | **典型协议 / 设备** |
| --- | --- | --- |
| **应用层** | 整合 OSI 应用层、表示层、会话层功能，为应用提供网络服务，定义应用通信规则。 | HTTP、HTTPS、FTP、DNS、SMTP |
| **传输层** | 端到端进程通信，提供可靠 / 不可靠传输，区分不同应用进程（通过端口号）。 | TCP、UDP |
| **网络层** | 跨网络路由转发，IP 地址寻址，实现不同网络互联，核心是 IP 协议。 | IP、ICMP、BGP（ARP 常作为网络接口/链路相关协议讨论） |
| **网络接口层** | 整合 OSI 数据链路层 + 物理层功能，负责相邻节点的帧传输和比特流收发。 | 以太网协议、PPP、Wi-Fi |

> **记忆口诀：接 → 网 → 传 → 应**（从下到上）。谐音：**"接网传应"**（接网传音——接上网络传输声音）。一句话：**"OSI 是理论课代表，TCP/IP 是实战程序员"**。

### 3. 说一下每一层对应的网络协议有哪些？ [重要性:A]

> **记忆口诀：不同层级用不同协议**——应用层 HTTP/DNS，传输层 TCP/UDP，网络层 IP/ICMP，链路层走以太网。ARP 用来把 IP 地址解析成 MAC 地址，严格分层时处在网络层和链路层之间，面试时说明口径即可。

### 4. 数据在各层之间是怎么传输的？ [重要性:A]

对于发送方而言，从上到下层层包装；对于接收方而言，从下到上层层解开包装。

> **记忆口诀：往下走加头打包，往上走拆头解包**——像俄罗斯套娃，每一层套一层壳，到了对面再一层层拆开。

## 二、DNS

### 5. 说说 DNS 的解析过程？ [重要性:A]

![[image 3.png]]

**假设你要查询 [www.baidu.com](http://www.baidu.com/) 的 IP 地址：**
1. 首先会查找浏览器的缓存，看看是否能找到 [www.baidu.com](http://www.baidu.com/) 对应的 IP，如果找到则直接返回，否则进行下一步。
2. 将请求发往**本地 DNS 服务器**，如果查找到也直接返回，否则继续进行下一步。
3. 本地 DNS 服务器向**根域名服务器**发送请求，根域名服务器返回负责 **com** 的顶级域名服务器的 IP 地址列表。
4. 本地 DNS 服务器再向其中一个负责 **com** 的顶级域名服务器发送一个请求，返回负责 [baidu.com](http://baidu.com/) 的权限域名服务器的 IP 地址列表。
5. 本地 DNS 服务器再向其中一个权威域名服务器发送一个请求，返回 [www.baidu.com](http://www.baidu.com/) 所对应的 IP 地址。

> **记忆口诀：浏览器缓存 → 本地 DNS → 根 DNS → 顶级 DNS → 权威 DNS**。关键词串联：**"缓本根顶权"**（缓存→本地→根→顶级→权威）。一句话：**层层递归，逐级缩小范围，最终问到 IP**。

## 三、HTTP

### 6. 从浏览器地址栏输入 URL 到显示主页的过程？（面试必问） [重要性:A]

1. **URL 解析**：浏览器解析 URL，提取**协议（http/https）、域名（如 [www.baidu.com](https://www.baidu.com/)）、路径（如 /index.html）、端口（默认 80/443）**。
2. **DNS 解析**：将域名解析成对应的 IP 地址。流程为 `本地DNS缓存 → 本地DNS服务器 → 根DNS → 顶级DNS → 权威DNS`，最终返回目标 IP（递归 + 迭代查询）。
3. **TCP 连接**：与服务器通过三次握手，建立可靠连接并向服务器发送 HTTP 请求。
   - HTTPS 情况：三次握手后，增加 **TLS 握手**（协商加密套件、交换随机数、验证证书、生成会话密钥），建立加密通道。
4. **发送 HTTP 请求**：浏览器构建请求报文（请求行 + 请求头 + 请求体），通过 TCP 连接发送给服务器。
5. **服务器处理请求并返回响应**：服务器接收请求，解析 URL 路径，处理业务逻辑（如查询数据库、读取静态文件），构建响应报文（状态行 + 响应头 + 响应体）返回。
6. **浏览器解析并渲染页面**。
7. **断开连接**：若为 HTTP 短连接，数据传输完成后进行**TCP 四次挥手**；若为长连接（Connection: keep-alive），则复用连接处理后续请求。

#### 各个过程都使用了哪些协议？ [重要性:A]

> **记忆口诀：解析 DNS 连 TCP，发请求等响应，渲染页面再挥手**。关键词串联：**"URL 解析→DNS→TCP→请求→响应→渲染→挥手"**。面试时按这 7 步流畅说出来，就是满分答案！

### 7. 说说 HTTP 常用的状态码及其含义 [重要性:A]

HTTP 状态码可以先按类别记：
- **1xx**：信息性状态码
- **2xx**：成功状态码
- **3xx**：重定向状态码
- **4xx**：客户端错误状态码
- **5xx**：服务端错误状态码

常用状态码（面试和实际排查都常见）：
![[image 5.png]]

| **类别** | **含义** | **常见状态码 & 核心场景** |
| --- | --- | --- |
| 1xx | 信息性状态码 | `100 Continue`：客户端继续发送请求体（用于 POST 大请求） |
| 2xx | 成功状态码 | `200 OK`：请求成功；`201 Created`：资源创建成功（如 POST 新增用户）；`204 No Content`：请求成功但无响应体 |
| 3xx | 重定向状态码 | `301 Moved Permanently`：永久重定向（域名变更）；`302 Found`：临时重定向；`304 Not Modified`：资源未修改，使用缓存 |
| 4xx | 客户端错误状态码 | `400 Bad Request`：请求参数错误；`401 Unauthorized`：未登录；`403 Forbidden`：权限不足；`404 Not Found`：资源不存在；`405 Method Not Allowed`：请求方法不支持 |
| 5xx | 服务端错误状态码 | `500 Internal Server Error`：服务器内部错误；`502 Bad Gateway`：网关错误（如 Nginx 连不上后端）；`503 Service Unavailable`：服务器过载 / 维护；`504 Gateway Timeout`：网关超时 |

> **记忆口诀：1 等 2 成 3 重 4 错 5 挂**。谐音：**"要等、二成、三重、四错、五误"**。面试重点记：**200 成功、301/302 重定向、304 缓存、401/403 权限、404 找不到、500/502/503/504 服务器问题**。

### 8. HTTP 缓存机制是怎样的？（强缓存 vs 协商缓存） [重要性:A]

HTTP 缓存分为**强缓存**和**协商缓存**两级，浏览器先查强缓存，命中则直接用，未命中再走协商缓存。

### 9. 说一下 301 和 302 的区别？ [重要性:A]

- **301（永久重定向）**：请求的资源已被永久移动到新位置。服务器会在响应中返回新的资源地址（Location）。
- **302（临时重定向）**：服务器临时从另一个地址响应资源，但客户端后续仍应使用原地址。

> **记忆口诀：301 是永久搬家换地址，302 是临时出差人不在**。面试考点：**301 浏览器会缓存新地址，302 每次还是先找原地址**。

### 10. HTTP 有哪些请求方式？ [重要性:A]

POST、DELETE、PUT、GET 的含义分别对应我们最熟悉的增、删、改、查。

> **记忆口诀：增 POST、删 DELETE、改 PUT、查 GET**。后三个的首字母 DPG 可以记成 **"D-P-G = 删-改-查"**。

### 12. HTTP 请求的过程与原理？ [重要性:A]

1. 每个服务器都有一个进程，它不断监听 TCP 端口 80，以便发现是否有浏览器向它发出连接建立请求。
2. 监听到连接请求后，就会建立 TCP 连接。
3. 浏览器向服务器发出浏览某个页面的请求，服务器返回所请求的页面作为响应。
4. 最后，释放 TCP 连接。

> **记忆口诀：监听 80 口→TCP 连→发请求→收响应→断连接**。四个关键字：**"监连请断"**。

### 13. 说一下 HTTP 的报文结构？ [重要性:A]

HTTP 报文有两种，HTTP 请求报文和 HTTP 响应报文：
- **请求报文**：请求行（方法 + URL + 版本）+ 请求头 + 空行 + 请求体
- **响应报文**：状态行（版本 + 状态码 + 状态描述）+ 响应头 + 空行 + 响应体

> **记忆口诀：行 + 头 + 空行 + 体**。就像写信：**第一行写收件人（请求行/状态行），接着写附加说明（头），空一行，最后正文（体）**。

### 15. URI 和 URL 有什么区别? [重要性:A]

- **URI（Uniform Resource Identifier，统一资源标识符）**：核心是**唯一标识一个资源**，不管用什么方式定位。URI 是一个抽象概念，包含 URL 和 URN 两类。
    - 例子：`urn:isbn:9787115546081`（用 ISBN 标识一本书，不涉及地址）。
- **URL（Uniform Resource Locator，统一资源定位符）**：是 URI 的**子集**，不仅标识资源，还指明**如何访问该资源**（通过协议 + 地址）。
    - 例子：`https://www.baidu.com/index.html`（通过 HTTPS 协议访问百度首页）。

> **记忆口诀：URI 是身份证号（标识你是谁），URL 是家庭住址（标识你在哪）**。**所有 URL 都是 URI，但不是所有 URI 都是 URL**——能定位的 URI 才是 URL。

### 16. 说一下 HTTP/1.0、1.1、2.0 的区别？ [重要性:A]

关键需要记住：**HTTP/1.0** 默认是短连接，可以强制开启；**HTTP/1.1** 默认长连接；**HTTP/2.0** 采用多路复用。

#### HTTP/1.0 [重要性:A]
- 默认使用**短连接**，每次请求都需要建立一个 TCP 连接。它可以设置 `Connection: keep-alive` 这个字段，强制开启长连接。

#### HTTP/1.1 [重要性:A]
- **引入了持久连接**，即 TCP 连接默认不关闭，可以被多个请求复用。
- **分块传输编码**，即服务端每产生一块数据，就发送一块，用"流模式"取代"缓存模式"。
- **管道机制（Pipelining）**，即在同一个 TCP 连接里面，客户端可以同时发送多个请求。
- **队头阻塞（Head-of-Line Blocking）问题**：虽然管道允许并行发请求，但**服务端响应必须按请求顺序返回**。如果第一个请求的响应很慢（如数据库查询），后面所有请求的响应都要排队等待，造成队头阻塞。这是 HTTP/1.1 的性能瓶颈，HTTP/2.0 的多路复用正是为了解决这个问题。

> HTTP/1.1 队头阻塞记忆：**"先进先出，第一个慢了全队都等"**。

#### HTTP/2.0 [重要性:A]
- **二进制协议**：1.1 版本的头信息是文本（ASCII 编码），数据体可以是文本或者二进制；2.0 中，头信息和数据体都是二进制。
- **完全多路复用**：在一个连接里，客户端和浏览器都可以同时发送多个请求或回应，而且不用按照顺序一一对应。
- **报头压缩**：HTTP/2.0 引入 HPACK，通过静态表、动态表和 Huffman 编码压缩头部，减少重复 Header 的传输开销。HTTP/3 中对应机制是 QPACK。
- **服务端推送**：允许服务器未经请求，主动向客户端发送资源。

> **记忆口诀：1.0 用完就断（短连接），1.1 连上不松（长连接 + 管道），2.0 一条车道多车跑（多路复用 + 二进制 + 头压缩 + 推送）**。关键词：**"1.0 短、1.1 长、2.0 多"**。

### 17. HTTP/3 了解吗？ [重要性:A]

**HTTP/3** 主要有两大变化：**传输层基于 UDP**、**使用 QUIC 保证 UDP 可靠性**。HTTP/2 存在的一些问题，比如重传等等，都是由于 TCP 本身的特性导致的，所以 HTTP/3 在 QUIC 的基础上进行发展而来。QUIC（Quick UDP Connections）直译为**快速 UDP 网络连接**，底层使用 UDP 进行数据传输。

#### HTTP/3 主要有这些特点： [重要性:A]
- **使用 UDP 作为传输层进行通信**。
- **安全性保证**：在 UDP 的基础上，QUIC 协议保证了 HTTP/3 的安全性，在传输的过程中就完成了 **TLS 加密握手**。
- **减少握手往返**：传统 HTTPS 需要先建立 TCP 连接，再进行 TLS 握手，通常需要多个 RTT；QUIC 将传输握手与 TLS 1.3 集成，首次连接通常 1-RTT，连接恢复时可支持 0-RTT，从而降低建连延迟。
- **解决行头阻塞（Head-of-Line Blocking）**：QUIC 有自己的一套机制可以保证传输的可靠性。当某个"流"发生丢包时，**只会阻塞这个流**，其他流不会受到影响（解决了 TCP 层面丢包导致的所有流阻塞问题）。

> **记忆口诀：HTTP/3 = UDP + QUIC，握手少 RTT，丢包只堵一条流**。一句话总结：**"受够了 TCP 的队头阻塞，干脆换 UDP 自己搞可靠传输"**。

### 18. HTTP 基于 TCP 还是 UDP？ [重要性:A]

- **HTTP/1.0、HTTP/1.1、HTTP/2.0**：**基于 TCP 协议**，因为需要可靠传输（网页资源不能丢包）。
- **HTTP/3.0**：**基于 UDP 协议**，通过 QUIC 协议实现可靠传输，解决 TCP 队头阻塞问题，降低连接延迟。

> **记忆口诀：前三个版本用 TCP，3.0 改用 UDP + QUIC**。面试说清楚：**"TCP 丢包会阻塞所有流，UDP+QUIC 丢包只阻塞一个流"**。

## 四、HTTPS

### 20. 说说 HTTP 与 HTTPS 有哪些区别？ [重要性:S]

- **安全性**：
  HTTP 是超文本传输协议，信息是**明文传输**，存在安全风险的问题。HTTPS 则解决了 HTTP 不安全的缺陷，在 TCP 和 HTTP 网络层之间加入了 **SSL/TLS 安全协议**，使得报文能够加密传输。
- **连接建立过程**：
  HTTP 连接建立相对简单，**TCP 三次握手**之后便可进行 HTTP 的报文传输。而 HTTPS 在 TCP 三次握手之后，还需进行 **SSL/TLS 的握手过程**，才可进入加密报文传输。
- **端口号**：
  HTTP 的默认端口号是 **80**，HTTPS 的默认端口号是 **443**。
- **证书体系**：
  HTTPS 协议需要向 **CA（证书权威机构）**申请数字证书，来保证服务器的身份是可信的。

> **记忆口诀：HTTP 明文 80 端口，HTTPS 加密 443 端口**。三句话区别：**"HTTP：明文 + TCP 三次握手 + 80 端口；HTTPS：加密 + TCP 三次握手 + TLS 握手 + 443 端口 + CA 证书"**。

### 21. 为什么要用 HTTPS？解决了哪些问题？ [重要性:S]

因为 HTTP 是明文传输，存在安全上的风险：
- **窃听风险**，比如通信链路上可以获取通信内容，用户账号被盗。
- **篡改风险**，比如强制植入垃圾广告，视觉污染。
- **冒充风险**，比如冒充淘宝网站，用户金钱损失。

**所以引入了 HTTPS，HTTPS 在 HTTP 与 TCP 层之间加入了 SSL/TLS 协议，可以很好地解决了这些风险：**
- **信息加密**：交互信息无法被窃取（防窃听）。
- **校验机制**：无法篡改通信内容，篡改了就不能正常显示（防篡改，通常使用数字签名/摘要算法）。
- **身份证书**：能证明淘宝是真淘宝（防冒充）。

所以 **SSL/TLS 协议是能保证通信是安全的**。

> **记忆口诀：三风险 → 三解决**。**"窃听→加密、篡改→校验、冒充→证书"**。一句话记住：**"HTTPS 就是 HTTP 穿上三件防弹衣：加密、签名、证书"**。

### 22. HTTPS 工作流程是怎样的？ [重要性:S]

这道题有几个要点：**公私钥、数字证书、加密、对称加密、非对称加密**。

HTTPS = HTTP + TLS/SSL，核心是**证书验证身份、密钥协商生成会话密钥、对称加密传输数据**。早期 TLS 可用 RSA 直接加密预主密钥，现代 TLS 主流使用 ECDHE/DHE 做密钥协商，以获得前向安全性；TLS 1.3 已移除 RSA 密钥交换。

#### 详细流程： [重要性:A]
1. **客户端发起 HTTPS 请求**，连接到服务端的 443 端口。
2. **服务端有一套数字证书**（证书内容有公钥、证书颁发机构、失效日期等）。
3. **服务端将自己的数字证书发送给客户端**（公钥在证书里面，私钥由服务器持有）。
4. **客户端收到数字证书之后，会验证证书的合法性**。如果证书验证通过，双方通过 ECDHE/DHE 等密钥交换算法协商出共享秘密。
5. **双方根据共享秘密和随机数派生会话密钥**，后续 HTTP 数据使用对称加密算法（如 AES-GCM、ChaCha20-Poly1305）加密传输。
6. **服务器私钥主要用于证明服务器身份**，而不是在现代 TLS 中直接解密客户端发来的会话密钥。
7. **服务器将加密后的密文返回到客户端**。
8. **客户端收到后，用自己的密钥对其进行对称解密**，得到服务器返回的数据。

#### TLS 握手步骤详解： [重要性:A]
1. **客户端问候（Client Hello）**：发送支持的 TLS 版本、加密套件列表、随机数 `Client Random`。
2. **服务端问候（Server Hello）**：选择 TLS 版本和加密套件，发送随机数 `Server Random`、**服务器证书**（含公钥）。
3. **客户端验证证书**：验证证书链、域名、有效期和吊销状态等，确认服务器身份可信。
4. **密钥交换**：双方交换 ECDHE/DHE 参数，各自计算出相同的共享秘密；早期 RSA 握手才是客户端生成 Pre-Master Secret 后用服务器公钥加密发送。
5. **生成会话密钥**：双方用随机数和共享秘密通过 KDF 派生出相同的对称会话密钥。
6. **双向验证就绪**：双方发送加密的 `Finished` 报文，验证加密通道可用，后续 HTTP 数据用会话密钥加密传输。

> **记忆口诀：证书验身份，ECDHE 协商密钥，对称加密传数据**。关键词串联：**"Client Hello → Server Hello + 证书 → 验证证书 → 密钥交换 → 派生会话密钥 → Finished 加密通信"**。面试加分：**"RSA 加密传预主密钥是早期 TLS 口径，现代 TLS 更常讲 ECDHE 和前向安全"**。

### 23. 客户端怎么去校验证书的合法性？ [重要性:A]

**首先，服务端的证书从哪来的呢？**
为了让服务端的公钥被大家信任，服务端的证书都是由 **CA**（**Certificate Authority**，证书认证机构）签名的，CA 就是网络世界里的公安局、公证中心，具有极高的可信度，所以由它来给各个公钥签名，信任的一方签发的证书，那必然证书也是被信任的。

![[image 7.png]]

**CA 签发证书的过程，如上图左边部分：**
- 首先 CA 会把持有者的公钥、用途、颁发者、有效时间等信息打成一个包，然后对这些信息进行 **Hash 计算**，得到一个 **Hash 值**；
- 然后 CA 会使用自己的**私钥**将该 Hash 值加密，生成 **Certificate Signature**，也就是 CA 对证书做了签名；
- 最后将 Certificate Signature 添加在文件证书上，形成**数字证书**。

**客户端校验服务端的数字证书的过程，如上图右边部分：**
- 首先客户端会使用同样的 **Hash 算法**获取该证书的 **Hash 值 H1**；
- 通常浏览器和操作系统中集成了 CA 的公钥信息，浏览器收到证书后可以使用 **CA 的公钥**解密 **Certificate Signature 内容**，得到一个 **Hash 值 H2**；
- 最后比较 **H1 和 H2**，如果值相同，则为可信赖的证书，否则则认为证书不可信。

**假如在 HTTPS 的通信过程中，中间人篡改了证书原文，由于他没有 CA 机构的私钥，所以 CA 公钥解密的内容就不一致。**

> **记忆口诀：CA 用私钥签名证书，客户端用 CA 公钥验证签名——H1（自己算） == H2（解密得）则可信**。一句话：**"CA 给证书盖私章，浏览器用 CA 的公章验真伪"**。

## 五、状态保持

### 26. 如何理解 HTTP 协议是无状态的？ [重要性:A]

这个**无状态**的**状态**指的是什么？是客户端的状态，所以字面意思，就是 HTTP 协议中服务端不会保存客户端的任何信息。
比如当浏览器第一次发送请求给服务器时，服务器响应了；如果同一个浏览器发起第二次请求给服务器时，它还是会响应，但是呢，服务器不知道你就是刚才的那个浏览器。

**常用的保持状态：Cookie 和 Session / Token。**

> **记忆口诀：HTTP 是金鱼——只有 7 秒记忆，每次请求都是"你谁啊？"**。解决方案：**"Cookie/Session/Token 帮你贴名牌，服务器一看就知道是你"**。

### 27. 说说 Session 和 Cookie 有什么联系和区别？ [重要性:A]

#### Session 和 Cookie 到底有什么不同呢？ [重要性:A]
- **存储位置不一样**，Cookie 保存在客户端，Session 保存在服务器端。
- **存储数据类型不一样**，Cookie 只能保存 ASCII，Session 可以存任意数据类型，一般情况下我们可以在 Session 中保持一些常用变量信息，比如说 UserId 等。
- **有效期不同**，Cookie 可设置为长时间保持，比如我们经常使用的默认登录功能，Session 一般有效时间较短，客户端关闭或者 Session 超时都会失效。
- **隐私策略不同**，Cookie 存储在客户端，比较容易遭到不法获取，早期有人将用户的登录名和密码存储在 Cookie 中导致信息被窃取；Session 存储在服务端，安全性相对 Cookie 要好一些。
- **存储大小不同**，单个 Cookie 保存的数据不能超过 4K，Session 可存储数据远高于 Cookie。

#### 分布式环境下 Session 怎么处理呢？ [重要性:A]
使用 Redis 缓存。

#### 方案一：Session + Cookie（主流方案） [重要性:A]
1. 用户登录时，服务器验证账号密码，生成 **Session 对象**（存储用户信息，如 userid、权限），并分配唯一 `SessionID`。
2. 服务器通过响应头 `Set-Cookie: JSESSIONID=xxx`，将 SessionID 发送给浏览器。
3. 浏览器将 SessionID 保存为 Cookie，后续请求时自动携带 `Cookie: JSESSIONID=xxx`。
4. 服务器通过 SessionID 找到对应的 Session 对象，识别用户身份。

**注意事项**：
- Session 存储位置：① 服务器内存（简单但不支持分布式）；② 分布式缓存（如 Redis，主流方案，支持集群）；③ 数据库（性能差，不推荐）。
- 安全优化：Cookie 设置 `HttpOnly`（防止 JS 窃取）、`Secure`（仅 HTTPS 传输）、`SameSite`（防止 CSRF）。

#### 方案三：Token 认证（如 JWT，分布式系统首选） [重要性:A]
1. 用户登录成功后，服务器生成**加密 Token**（如 JWT，包含用户信息 + 签名），返回给客户端。
2. 客户端将 Token 保存在 localStorage/sessionStorage，后续请求通过请求头 `Authorization: Bearer <token>` 携带。
3. 服务器验证 Token 签名，解析用户信息，无需存储 Token（无状态）。

> **记忆口诀：Cookie 留在客户端（像通行证），Session 存在服务端（像档案本），Token 自己带身份（像身份证）**。三个方案一句话对比：**"Session+Cookie 是传统方案，URL 重写是保底手段，Token/JWT 是分布式首选"**。五个区别记首字：**"位置、类型、有效期、隐私、大小"**。

## 六、代理

### 28. 正向代理与反向代理有什么区别？ [重要性:A]

正向代理和反向代理都是代理服务器，但**服务的对象不同**：

#### 正向代理（Forward Proxy） [重要性:A]
**代理的是客户端**，替客户端向目标服务器发起请求。
- **工作原理**：客户端知道目标服务器的存在，但通过正向代理访问。目标服务器不知道真正的客户端是谁，只知道代理的 IP。
- **典型场景**：
  - **翻墙/跨域访问**：公司内网通过正向代理访问外部网站。
  - **访问控制**：家长控制孩子能访问哪些网站。
  - **隐藏客户端身份**：匿名上网（如 VPN 的核心思想）。
- **例子**：用户 A → 代理服务器（10.0.0.1）→ 目标网站（www.google.com），目标网站以为请求来自 10.0.0.1，不知道是用户 A。

#### 反向代理（Reverse Proxy） [重要性:A]
**代理的是服务端**，替服务端接收客户端请求，并将请求转发到后端的真实服务器。
- **工作原理**：客户端不知道目标服务器的存在，只知道反向代理的地址。客户端以为自己在跟真实的服务器通信，实际上是由反向代理做了"中间人"。
- **典型场景**：
  - **负载均衡**：Nginx 做反向代理，将请求分发到多个后端服务器（如 3 台 Tomcat）。
  - **安全防护**：隐藏真实服务器 IP，防止 DDoS 攻击打到源站。
  - **统一入口**：微服务架构中，API 网关作为反向代理统一管理路由、认证、限流。
- **例子**：用户访问 www.example.com → Nginx 反向代理（43.1.1.1）→ 真实服务器（192.168.1.10），客户端不知道有 192.168.1.10 的存在。

| **对比维度** | **正向代理** | **反向代理** |
| --- | --- | --- |
| 代理对象 | **客户端**（替客户端发请求） | **服务端**（替服务端收请求） |
| 谁隐藏了身份 | 客户端身份被隐藏 | 服务端身份被隐藏 |
| 客户端知情 | 客户端**知道**目标服务器存在 | 客户端**不知道**真实服务器存在 |
| 典型用途 | 翻墙、匿名访问、访问控制 | 负载均衡、安全防护、统一入口 |
| 代表产品 | Shadowsocks、VPN 客户端 | Nginx、HAProxy、Kong |

> **记忆口诀：正向代理 = "帮我买个东西，卖家不知道我是谁"；反向代理 = "我找A买东西，结果是B在帮我服务，我不知道有B"**。一句话区分：**"正向代理隐藏客户端，反向代理隐藏服务端"**。面试话术：**"VPN 是正向代理帮客户端翻墙，Nginx 是反向代理帮服务器做负载均衡"**。

## 七、WebSocket

### 29. WebSocket 与 Socket 的区别？ [重要性:S]

- **Socket** 其实等于 **IP 地址 + 端口 + 协议**。具体来说，Socket 用于更好地进行网络编程。**Socket 是一套标准，它完成了对 TCP/IP 的高度封装，屏蔽网络细节，方便开发者。**
- **WebSocket** 是为解决 **HTTP 不支持全双工通信**（服务端无法主动推送数据）而出现的协议，属于应用层通信协议。

Socket 是一个持久化的协议接口，是伴随 H5 出现的**网络编程标准接口**。

> **记忆口诀：Socket 是编程接口（想怎么发就怎么发），WebSocket 是应用层协议（专门解决服务端主动推）**。一句话：**"Socket 是打电话的工具，WebSocket 是用这个工具打的一个特定类型的电话"**。

### 30. WebSocket 是什么？应用场景有哪些？ [重要性:S]

WebSocket 是**基于 TCP 的全双工通信协议**，通过一次握手建立持久连接，客户端和服务器可**双向主动发送数据**，解决 HTTP 单向通信的痛点。

**核心应用场景**：
- 实时消息推送：如聊天软件、社交平台通知；
- 实时协作：如多人在线文档编辑（腾讯文档）；
- 实时数据展示：如股票行情、监控大屏；
- 在线游戏：如多人对战游戏（低延迟需求）。

> **记忆口诀：一条连接、双向互推、持久在线**。应用场景可概括为：**"聊天、协作、行情、游戏"——四个场景都要实时**。

### 31. WebSocket 和 HTTP 有什么区别？ [重要性:S]

两者都是基于 TCP 的应用层协议，但设计目标和通信模式完全不同：

| **特性** | **HTTP** | **WebSocket** |
| --- | --- | --- |
| 通信方向 | 单向通信（客户端请求→服务器响应） | 全双工通信（双方可同时收发） |
| 连接方式 | 短连接 / 长连接（keep-alive） | 持久连接（一次握手，长期有效） |
| 协议标识 | `http://`/`https://` | `ws://`/`wss://`（加密版） |
| 头部开销 | 每次请求携带完整 HTTP 头部，开销大 | 握手时带头部，后续通信头部极小 |
| 服务端主动推送 | 不支持（需轮询模拟） | 原生支持（核心优势） |
| 适用场景 | 普通网页请求、资源获取 | 实时通信、消息推送 |

> **记忆口诀：HTTP 是"你问我答"，WebSocket 是"随时聊天"**。核心差异：**"HTTP 一问一答开销大，WS 握一次手一直聊"**。关键词：**Upgrade: websocket** 请求头升级协议。

### 32. WebSocket 与短轮询、长轮询的区别？ [重要性:S]

三者都是为了解决**服务器向客户端实时推送数据**的问题，核心差异在实现方式和效率：

| **特性** | **短轮询（Short Polling）** | **长轮询（Long Polling）** | **WebSocket** |
| --- | --- | --- | --- |
| 实现原理 | 客户端每隔固定时间（如 5s）发 HTTP 请求，服务器立即响应 | 客户端发请求，服务器无数据则挂起连接，有数据 / 超时再响应，客户端收到后立即重连 | 一次握手建立 TCP 长连接，双向主动通信 |
| 实时性 | 低（延迟 = 轮询间隔） | 中（有数据立即推送） | 高（毫秒级延迟） |
| 资源开销 | 高（频繁建立 / 关闭 TCP 连接，空请求多） | 中（减少空请求，但仍需反复重连） | 低（持久连接，无冗余请求） |
| 服务器压力 | 大（高并发下请求量爆炸） | 中（连接挂起消耗线程资源） | 小（单连接可处理双向通信） |
| 实现复杂度 | 低（普通 HTTP 请求即可） | 中（需服务器支持挂起连接） | 高（需服务器和客户端支持协议） |

> **记忆口诀：短轮询是"每隔几秒敲门问好了没"，长轮询是"敲了门站在门口等"，WebSocket 是"门一直开着，有事直接喊"**。一句话记住选择：**"追求实时性、低开销 → 选 WebSocket；简单场景不在乎资源 → 用短轮询"**。

## 八、传输层：TCP 与 UDP

### 33. TCP 与 UDP 的区别（面试重中之重） [重要性:S]

TCP 和 UDP 是传输层两大核心协议，区别可从**连接性、可靠性、效率**等维度记忆：

| **特性** | **TCP** | **UDP** |
| --- | --- | --- |
| 连接性 | 面向连接（三次握手建立，四次挥手关闭） | 无连接（直接发送数据包） |
| 可靠性 | 可靠传输（序列号、确认应答、超时重传、流量控制、拥塞控制） | 不可靠传输（尽最大努力交付，不保证送达和有序） |
| 状态性 | 有状态（维护连接状态、序列号、窗口大小） | 无状态（发送后不关心后续） |
| 传输形式 | 面向字节流（无消息边界，可拆分 / 合并数据） | 面向报文（保留消息边界，应用给多大就发多大） |
| 首部开销 | 20~60 字节（可变） | 固定 8 字节（极简） |
| 通信方式 | 仅支持单播 | 支持单播、多播、广播 |
| 适用场景 | 可靠性优先：文件传输（FTP）、网页（HTTP）、邮件（SMTP） | 实时性优先：音视频直播、在线游戏、DNS 查询 |

> **记忆口诀：TCP 是顺丰快递（要握手、要确认、不丢件、用箱子装），UDP 是飞鸽传书（直接扔、不管到不到、轻装上阵）**。七维度对比关键词：**"连-可-状-传-首-通-场"**（连接→可靠→状态→传输→首部→通信→场景）。

### 34. 什么时候选择 TCP，什么时候选 UDP？ [重要性:S]

选择的核心是**权衡可靠性和实时性**：
- **选 TCP 的场景**：数据不能丢失、必须有序到达。例如：文件传输（丢包会导致文件损坏）、电商支付（数据错误会导致交易异常）、网页加载（资源缺失会导致页面错乱）。
- **选 UDP 的场景**：允许少量丢包，但要求低延迟。例如：视频直播（丢几个帧不影响观看）、在线游戏（延迟高比丢包更致命）、DNS 查询（快比准更重要，丢包可重查）。
- **特殊情况**：若需 "UDP 的实时性 + TCP 的可靠性"，可在应用层实现可靠机制（如 RTP 协议 + 重传机制）。

> **记忆口诀：丢不得 → TCP，等不得 → UDP**。一句话判断：**"文件支付网页用 TCP，视频游戏 DNS 用 UDP"**。

### 35. 基于 TCP/UDP 的协议有哪些？ [重要性:S]

传输层协议为应用层提供服务，需记住常见协议的底层依赖：
- **基于 TCP 的协议**（可靠性优先）：HTTP/HTTPS（网页）、FTP（文件传输）、SMTP/POP3/IMAP（邮件）、SSH（远程登录）、Telnet。
- **基于 UDP 的协议**（实时性优先）：DNS（域名解析）、DHCP（IP 地址分配）、RTP（音视频传输）、SNMP（网络管理）、TFTP（简单文件传输）。

> **记忆口诀：TCP 协议群 = "上网（HTTP）传文件（FTP）发邮件（SMTP）远程登录（SSH）"；UDP 协议群 = "查域名（DNS）拿地址（DHCP）看视频（RTP）管网络（SNMP）"**。面试常问：**"DNS 用了 UDP（快），但大响应会切 TCP"**。

## 九、TCP 三次握手

### 36. 详细说一下 TCP 的三次握手机制 [重要性:S]

TCP 提供面向连接的服务，在传送数据前必须建立连接，TCP 连接是通过三次握手建立的。

![[image 8.png]]

#### 三次握手的过程： [重要性:S]
- **最开始**，客户端和服务端都处于 `CLOSE` 状态，服务端监听客户端的请求，进入 `LISTEN` 状态。
- **第一次握手（SYN）**：客户端发送连接请求（`SYN=1, seq=x`），发送完毕后，客户端进入 `SYN_SENT` 状态。
- **第二次握手（SYN+ACK）**：服务端确认连接（`SYN=1, ACK=1, seq=y, ACKnum=x+1`），发送完毕后，服务端进入 `SYN_RCVD` 状态。
- **第三次握手（ACK）**：客户端收到服务端的确认之后，再次向服务端确认（`ACK=1, ACKnum=y+1`），发送完毕后，客户端进入 `ESTABLISHED` 状态，当服务端接收到这个包时，也进入 `ESTABLISHED` 状态。

| **握手次数** | **发起方** | **报文标志** | **核心动作** | **状态变化** |
| --- | --- | --- | --- | --- |
| 第一次握手 | 客户端 | SYN（同步） | 发送 `SYN=1, SEQ=x`（x 为随机初始序列号），请求建立连接 | 客户端→`SYN_SENT` |
| 第二次握手 | 服务端 | SYN+ACK（同步 + 确认） | 回复 `SYN=1, SEQ=y；ACK=1，确认号=x+1`（确认收到客户端的 SYN，同时发送自己的 SYN） | 服务端→`SYN_RCVD` |
| 第三次握手 | 客户端 | ACK（确认） | 发送 `ACK=1，确认号=y+1`（确认收到服务端的 SYN） | 客户端→`ESTABLISHED`；服务端收到后→`ESTABLISHED`，连接建立 |

#### 三次握手的核心目的 [重要性:S]
确认**双方的发送和接收能力都正常**，并同步初始序列号（SEQ），为后续可靠传输奠定基础。

#### 为什么是三次不是两次/四次？ [重要性:S]
- **两次握手的问题**：无法确认客户端的接收能力。若服务端的 SYN+ACK 报文丢失，客户端不知道服务端已准备好，服务端会一直等待客户端的 ACK，造成资源浪费。
- **四次握手的问题**：三次握手已能完成双向能力确认，四次会增加不必要的网络开销，降低效率。

### 37. 三次握手中每一次没收到报文会发生什么情况？ [重要性:S]

- **第一次握手服务端未收到 SYN 报文**
  服务端不会进行任何的动作，而客户端由于一段时间内没有收到服务端发来的确认报文，等待一段时间后会**重新发送 SYN 报文**，如果仍然没有回应，会重复这个过程，直到发送次数超过**最大重传次数限制**，就会返回连接建立失败。
- **第二次握手客户端未收到服务端响应的 ACK 报文**
  客户端会继续重传，直到次数限制；而服务端此时会阻塞在 `accept()` 处，等待客户端发送 ACK 报文。
- **第三次握手服务端未收到客户端发送过来的 ACK 报文**
  服务端同样会采用类似客户端的**超时重传机制**，如果重试次数超过限制，则 `accept()` 调用返回 -1，服务端建立连接失败；而此时客户端认为自己已经建立连接成功，因此开始向服务端发送数据，但是服务端的 `accept()` 系统调用已经返回，此时不在监听状态，因此服务端接收到客户端发送来的数据时会发送 **RST 报文**给客户端，消除客户端单方面建立连接的状态。

> **记忆口诀：哪次没收到，就重传哪次的响应，重传超限就放弃**。场景速记：**"第一次丢→客户端重发 SYN；第二次丢→客户端重发 SYN；第三次丢→服务端重发 SYN+ACK，客户端还在傻等"**。

### 38. 第二次握手传回了 ACK，为什么还要传回 SYN？ [重要性:A]

`ACK` 是为了告诉客户端传来的数据已经接收无误。
而传回 `SYN` 是为了告诉客户端，服务端响应的确实是客户端发送的报文。

> **记忆口诀：ACK = "收到你的消息"，SYN = "这是我的回复，请确认"**。两者缺一不可——**只有 ACK 没有 SYN，服务端就没法发起自己的序列号同步**。

### 39. 第三次握手可以携带数据吗？ [重要性:S]

第 3 次握手是可以携带数据的。
此时客户端已经处于 `ESTABLISHED` 状态。对于客户端来说，它已经建立连接成功，并且确认服务端的接收和发送能力是正常的。
第一次握手不能携带数据是出于安全的考虑，因为如果允许携带数据，攻击者每次在 `SYN` 报文中携带大量数据，就会导致服务端消耗更多的时间和空间去处理这些报文，会造成 CPU 和内存的消耗。

> **记忆口诀：第三次可以带数据（搭便车），第一次绝对不能带（防攻击）**。面试加分：**"第三次握手时客户端可以顺手把 HTTP 请求发出去，省一次交互"**。

### 40. 说说半连接队列和 SYN Flood 攻击的关系？ [重要性:A]

- **半连接队列（SYN Queue）**：
    - 当服务端收到客户端的 `SYN` 包时，会将该连接放入半连接队列。
    - 此时服务端处于 `SYN_RECV` 状态，并向客户端发送 `SYN+ACK`。
- **全连接队列（Accept Queue）**：
    - 当服务端收到客户端最后的 `ACK` 确认包后，连接从半连接队列移出，进入全连接队列。
    - 此时服务端处于 `ESTABLISHED` 状态，等待应用程序调用 `accept()`。
- **SYN Flood 攻击的关系**：
    - **原理**：攻击者伪造大量不存在的 IP 地址向服务器发送 `SYN` 包。
    - **后果**：服务器发送 `SYN+ACK` 后无法收到回执（`ACK`），导致**半连接队列被这些"僵尸连接"占满**。
    - **影响**：正常的连接请求因为半连接队列溢出而被丢弃，导致拒绝服务（DoS）。

**常用的防御手段：**
- **SYN Cookies**：不直接在队列中存状态，而是通过算法生成一个 Cookie 回传，验证通过后再分配资源。
- **增大队列容量**：调整内核参数。
- **减少重试次数**：缩短等待超时连接的时间。

> **记忆口诀：半连接队列是"候选人等待区"，SYN Flood 就是派一堆假候选人塞满这个区，真候选人进不来**。最有效的防御：**SYN Cookies——不占队列，通过 Cookie 验证身份再分配资源**。

## 十、TCP 四次挥手

### 41. 说说 TCP 四次挥手的过程？ [重要性:S]

TCP 是全双工通信，关闭连接时需**分别关闭两个方向的传输通道**，因此需要四次挥手：

#### 四次挥手详细过程 [重要性:S]

| **挥手次数** | **发起方** | **报文标志** | **核心动作** | **状态变化** |
| --- | --- | --- | --- | --- |
| 第一次挥手 | 客户端 | FIN（结束） | 发送 `FIN=1, SEQ=x`（表示客户端无数据要发，请求关闭客户端→服务端的通道） | 客户端→`FIN_WAIT_1` |
| 第二次挥手 | 服务端 | ACK（确认） | 发送 `ACK=1，确认号=x+1`（确认收到 FIN，此时客户端→服务端通道半关闭） | 服务端→`CLOSE_WAIT`；客户端收到后→`FIN_WAIT_2` |
| 第三次挥手 | 服务端 | FIN（结束） | 服务端数据发送完毕，发送 `FIN=1，SEQ=y`（请求关闭服务端→客户端的通道） | 服务端→`LAST_ACK` |
| 第四次挥手 | 客户端 | ACK（确认） | 发送 `ACK=1，确认号=y+1`（确认收到 FIN） | 客户端→`TIME_WAIT`；服务端收到后→`CLOSED` |

#### 为什么是四次不是三次？ [重要性:S]
TCP 全双工特性决定了**关闭通道需要双向确认**。服务端收到客户端的 FIN 时，可能还有数据要发，因此需要先回复 ACK 确认关闭客户端通道，等数据发完再发 FIN 关闭自己的通道——这两步无法合并，因此必须四次挥手。

**核心逻辑解析：**
- **客户端的意图**：当客户端发送 `FIN` 包时，它只是在告诉服务端："我的数据发完了，我不再发送新数据了"。但是，此时客户端**仍然具有接收数据**的能力。
- **服务端的反应**：
    1. 服务端收到 `FIN` 后，先回一个 `ACK`。这表示"我知道你没数据发了"。
    2. **关键点**：此时服务端可能还有没处理完的数据要发给客户端。它需要先把手头的工作做完。
    3. 当服务端也准备好关闭连接时，它才会发送自己的 `FIN` 包，表示"我也没数据要发了，同意关闭"。
- **结论**：由于服务端的 `ACK`（收到确认）和 `FIN`（准备好关闭）通常不在同一个时刻发生，所以这两个包必须分开发送。这就导致了比三次握手多出了一次。

> **记忆口诀：FIN → ACK → FIN → ACK。场景模拟：A 说"我走了"（FIN），B 说"知道了"（ACK），B 收拾完说"我也走了"（FIN），A 说"好"（ACK）**。四次和三次的区别：**"握手时 SYN 和 ACK 可以合并，挥手时 ACK 和 FIN 不能合并——因为服务端还要发剩余数据"**。

### 42. TCP 四次挥手过程中，为什么需要等待 2MSL，才进入 CLOSED 关闭状态？ [重要性:S]

#### 为什么需要等待？ [重要性:S]
1. **为了保证客户端发送的最后一个 ACK 报文段能够到达服务端**。
   这个 ACK 报文段有可能丢失，从而使处在 **LAST-ACK** 状态的服务端就收不到对已发送的 **FIN + ACK** 报文段的确认。服务端会超时重传这个 FIN + ACK 报文段，而客户端就能在 2MSL 时间内（**超时 + 1MSL 传输**）收到这个重传的 FIN + ACK 报文段。接着客户端重传一次确认，重新启动 2MSL 计时器。最后，客户端和服务端都正常进入到 **CLOSED** 状态。
2. **防止已失效的连接请求报文段出现在本连接中**。
   客户端在发送完最后一个 ACK 报文段后，再经过时间 2MSL，就可以使本连接持续的时间内所产生的所有报文段都从网络中消失。这样就可以使下一个连接中不会出现这种旧的连接请求报文段。

#### 为什么等待的时间是 2MSL？ [重要性:S]
- **MSL** 是 **Maximum Segment Lifetime**（报文最大生存时间），它是任何报文在网络上存在的最长时间，超过这个时间报文将被丢弃。
- **TIME_WAIT** 等待 2 倍的 MSL，比较合理的解释是：网络中可能存在来自发送方的数据包，当这些发送方的数据包被接收方处理后会向对方发送响应，所以**一来一回需要等待 2 倍的时间**。

> **记忆口诀：等 2MSL = 怕最后一个 ACK 丢了 + 怕旧包污染新连接**。一句话理解：**"2MSL = 我的包过去（1MSL）+ 你的回复过来（1MSL），一来一回"**。

### 43. CLOSE-WAIT 和 TIME-WAIT 的状态和意义？ [重要性:A]

#### CLOSE-WAIT 状态有什么意义？ [重要性:A]
服务端收到客户端关闭连接的请求并确认之后，就会进入 CLOSE-WAIT 状态。此时服务端可能还有一些数据没有传输完成，因此不能立即关闭连接，而 CLOSE-WAIT 状态就是为了保证服务端在关闭连接之前将待发送的数据处理完。

#### TIME-WAIT 有什么意义？ [重要性:A]
TIME-WAIT 状态发生在第四次挥手，当客户端向服务端发送 ACK 确认报文后进入 TIME-WAIT 状态。
1. **防止旧连接的数据包**：如果客户端收到服务端的 FIN 报文之后立即关闭连接，但是此时服务端对应的端口并没有关闭，如果客户端在相同端口建立新的连接，可能会导致新连接收到旧连接残留的数据包，导致不可预料的异常发生。
2. **保证连接正确关闭**：假设客户端最后一次发送的 ACK 包在传输的时候丢失了，由于 TCP 协议的超时重传机制，服务端将重发 FIN 报文，如果客户端没有维持 TIME-WAIT 状态而直接关闭的话，当收到服务端重新发送的 FIN 包时，客户端就会使用 RST 包来响应服务端，导致服务端以为有错误发生，然而实际关闭连接过程是正常的。

> **记忆口诀：CLOSE-WAIT = "我还有数据要发，等等再关"（被动方等自己发完）；TIME-WAIT = "等 2MSL 确保你的 ACK 到了，顺便让旧包消失"（主动方等对方确认）**。

### 44. TIME_WAIT 状态过多会导致什么问题？怎么解决？  谁第一次主动谁time wait [重要性:A]

**TIME_WAIT 状态过多会导致什么问题？**
如果服务器有处于 TIME-WAIT 状态的 TCP，则说明是由服务器方主动发起的断开请求。过多的 TIME-WAIT 状态主要的危害有两种：
1. **内存资源占用**；
2. **对端口资源的占用**，一个 TCP 连接至少消耗一个本地端口。

**怎么解决 TIME_WAIT 状态过多？**
- 服务器可以设置 `SO_REUSEADDR` 套接字来通知内核，如果端口被占用，但是 TCP 连接位于 TIME_WAIT 状态时可以重用端口。
- 还可以使用**长连接**的方式来减少 TCP 的连接和断开，在长连接的业务里往往不需要考虑 TIME_WAIT 状态。

#### 一、TIME_WAIT 过多 [重要性:A]

**状态定位**：出现在**主动关闭连接的一方**（通常是客户端，但在爬虫或微服务调用中，服务端也可能作为调用方主动关闭）。

**产生原因**：
- **短连接频繁**：使用了大量的短连接（HTTP Short Connection），每次请求完成后立即关闭连接。
- **高并发压力**：在极短时间内产生大量连接，虽然每个连接都正常关闭，但由于 TCP 协议规定必须等待 **2MSL** 才能释放端口，导致状态堆积。
- **未开启连接池**：在微服务架构中，如果下游服务调用没有使用连接池，每次访问都会经历"建立-关闭"，迅速占满 65535 个端口。

**危害**：
- **端口耗尽**：新的连接请求因无法分配到可用端口而失败（Address already in use）。
- **资源占用**：虽然内存占用相对较小，但大量 socket 结构体会消耗系统内核资源。

**解决对策**：
- **启用 Keep-Alive**：改短连接为长连接，减少关闭频率。

#### 二、CLOSE_WAIT 过多 [重要性:A]

**状态定位**：出现在**被动关闭连接的一方**（通常是服务端）。

**产生原因**：
**核心结论：这是代码 Bug，不是系统参数问题。**
当对方（客户端）发送了 FIN 包请求关闭连接时，你的服务器内核回了 ACK。此时，内核在等**应用程序**调用 `socket.close()` 来发送最后的 FIN 包。如果程序没关，就一直卡在 CLOSE_WAIT。
- **资源未释放**：代码里忘记在 `finally` 块或异常处理中调用 `close()`。
- **响应过慢/线程池满了**：程序忙于处理其他逻辑，或者线程池排队太长，导致负责关闭连接的逻辑一直没执行。
- **死锁或阻塞**：程序在关闭连接前触发了某个阻塞操作（如等待数据库锁），导致连接一直悬挂。

**解决对策**：
- **检查代码**：排查所有的 `Socket`、`HttpClient`、`JDBC Connection` 等资源是否都有正确的关闭逻辑。

| **状态** | **谁会出现** | **主要原因** | **解决核心** |
| --- | --- | --- | --- |
| **TIME_WAIT** | 主动关闭方 | 短连接过多、并发过高 | 优化内核参数、使用长连接 |
| **CLOSE_WAIT** | 被动关闭方 | **代码没写 close()**、程序阻塞 | **修复代码 Bug** |

> **记忆口诀：TIME_WAIT 多是"主动方端口不够用"（系统参数问题），CLOSE_WAIT 多是"被动方代码没写 close"（程序 Bug）**。一句话记住：**"TIME_WAIT 调内核，CLOSE_WAIT 改代码"**。

### 45. 保活计时器有什么用？ [重要性:A]

**核心内容识别：**
- **设计初衷**：防止服务器在客户端发生故障（如突然掉电、崩溃）后，无休止地等待下去，白白浪费资源。
- **工作机制**：
    1. 服务器每收到一次数据，就重置计时器。
    2. 默认设置通常为 **2 小时**。
    3. 若 2 小时内没收到新数据，服务器会发送一个**探测报文段**。
    4. 随后每隔 **75 秒** 发送一次。
    5. 若连续发送 **10 个** 探测报文段仍无响应，服务器认为客户端故障，主动关闭连接。

![[image 9.png]]

> **记忆口诀：2 小时没动静 → 75 秒发一次探针 → 发 10 次没回应 → 断连**。一句话：**"两小时不吭声，开始喊你，喊 10 次不理就挂电话"**。

## 十一、TCP 可靠传输

### 46. TCP 是如何保证可靠性的？ [重要性:S]

**TCP 主要提供连接管理、校验和、序列号/确认应答、超时重传、最大消息长度、滑动窗口控制等方法实现了可靠性传输。**

TCP 通过**六大机制**实现可靠传输，可按"防丢、防错、防乱、防拥塞"的逻辑记忆：

#### 1. 连接管理 [重要性:A]
TCP 使用**三次握手**和**四次挥手**保证可靠地建立连接和释放连接。

#### 2. 校验和 [重要性:A]
TCP 将保持它首部和数据的校验和。这是一个**端到端**的校验和，目的是检测数据在传输过程中的任何变化。如果接收端的校验和有差错，TCP 将丢弃这个报文段和不确认收到此报文段。

#### 3. 序列号/确认应答（Sequence Number & ACK） [重要性:A]
TCP 为每个字节都编号。接收方收到后回复一个 ACK（确认号），告诉发送方："我收到了第 X 字节之前的所有数据，下次请发第 X 字节开始的内容"，解决**丢包**和**乱序**问题。

#### 4. 流量控制 [重要性:A]
TCP 连接的每一方都有固定大小的缓冲空间，TCP 的接收端只允许发送端发送接收端缓冲区能接纳的数据。当接收方来不及处理发送方的数据，能提示发送方降低发送的速率，防止包丢失。TCP 使用的流量控制协议是**可变大小的滑动窗口协议**。（TCP 利用滑动窗口实现流量控制）

**核心考点**：流量控制是**端到端**（针对接收方缓冲区），拥塞控制是**全局性**（针对网络状况）；实际发送窗口 = min(cwnd, rwnd)。

#### 5. 最大消息长度（MSS） [重要性:A]
在建立 TCP 连接的时候，双方约定一个最大的长度（MSS）作为发送的单位，重传的时候也是以这个单位来进行重传。理想的情况下是该长度的数据刚好不被网络层分块。

#### 6. 超时重传 [重要性:A]
超时重传是指发送出去的数据包到接收到确认包之间的时间，如果超过了这个时间会被认为是丢包了，需要重传。

#### 7. 拥塞控制 [重要性:S]
如果网络非常拥堵，此时再发送数据就会加重网络负担，那么发送的数据段很可能超过了最大生存时间也没有到达接收方，就会产生丢包问题。为此 TCP 引入**慢启动**机制，先发出少量数据，就像探路一样，先摸清当前的网络拥堵状态后，再决定按照多大的速度传送数据。

#### 8. 乱序重排与去重 [重要性:A]
接收方按序列号重组报文，丢弃重复报文。

> **记忆口诀：连接管理 + 校验和 + 序列号/ACK + 流量控制 + MSS + 超时重传 + 拥塞控制 + 乱序重排 = TCP 八大护法**。一句话总结：**"TCP 用三次握手建连接，校验和查错误，序列号排顺序，滑动窗口控流量，拥塞控制防拥堵，超时重传补丢包"**。

## 十二、TCP 核心机制

### 47. TCP 的拥塞控制是怎么实现的？ [重要性:A]

拥塞控制是 TCP 的核心难点，目的是**防止发送方速率过快导致网络拥塞**，核心是维护两个参数：**拥塞窗口（cwnd）** 和**慢开始门限（ssthresh）**。

#### 四大核心算法 [重要性:A]

| **算法** | **触发条件** | **核心逻辑** |
| --- | --- | --- |
| **慢开始** | 连接建立 / 超时重传后 | cwnd 从 1 开始，每收到一个 ACK 就翻倍（指数增长），直到 cwnd=ssthresh，进入拥塞避免 |
| **拥塞避免** | cwnd >= ssthresh | cwnd 每经过一个 RTT 增加 1（线性增长），缓慢提升速率，避免拥塞 |
| **快速重传** | 收到 3 个重复 ACK | 无需等待超时，直接重传丢失的报文段（减少重传延迟） |
| **快速恢复** | 触发快速重传后 | 将 ssthresh 设为当前 cwnd 的一半，cwnd=ssthresh，直接进入拥塞避免（避免流量骤降） |

#### 特殊情况：超时重传（严重拥塞） [重要性:A]
若发送方超时未收到 ACK，判定为网络严重拥塞：
- 将 ssthresh 设为当前 cwnd 的一半；
- cwnd 重置为 1，重新进入慢开始阶段。

> **记忆口诀：慢开始指数涨，拥塞避免线性涨，快重传不等超时，快恢复不归零**。四种算法关键词：**"慢：×2 翻倍；避：+1 慢升；重：3ACK 就传；恢：减半后线性"**。特殊情况：**"超时→严重拥塞→cwnd 归 1，从头再来"**。

### 50. 超时重传如何实现？超时重传时间怎么确定？ [重要性:A]

超时重传是 ARQ 协议的核心，关键是**确定合理的重传超时时间（RTO）**：

**核心参数定义**
- **RTT**：往返时间，数据包从发送到收到 ACK 的时间。
- **RTO**：重传超时时间，超过此时间未收到 ACK 则重传。

**RTO 的确定方法**
RTO 不能直接等于 RTT（网络波动会导致 RTT 变化），TCP 采用**加权移动平均（EWMA）** 算法计算**平滑往返时间（SRTT）**：
`SRTT = (1-α) × SRTT + α × RTT`
其中 α 是权重（通常取 0.125），目的是降低瞬时 RTT 波动的影响。
最终 `RTO = SRTT + 4 × RTT偏差（RTTVAR）`，确保 RTO 略大于正常 RTT，减少不必要的重传。

> **记忆口诀：RTO = 平滑 RTT + 4 倍抖动**。一句话：**"不直接用 RTT 设超时（网络会抖），用加权平均算出 SRTT，再留点余量（+4 倍偏差），避免冤枉重传"**。

### 51. 什么是 TCP 粘包/拆包？原因和解决方法？ [重要性:A]

- **粘包**：多个数据包被合并成一个大数据包发送，接收方无法区分边界。
- **拆包**：一个大数据包被拆分成多个小数据包发送，接收方需要重组。
