## 一、进程与线程

### 1. 进程和线程的区别？ [重要性:S]
- **定义**：进程是资源分配的基本单位，线程是 CPU 调度的基本单位。
- **资源占用**：进程有独立的地址空间、内存、文件描述符等；线程共享进程的资源，仅拥有自己的栈、寄存器和程序计数器。
- **开销**：进程创建 / 销毁、切换的开销大；线程的开销小。
- **通信方式**：进程间通信（IPC）需要借助管道、消息队列、共享内存等；线程间可直接读写进程数据段。

> **记忆口诀：进程是"包工头"（管资源、分活），线程是"工人"（干活、被调度）**。一句话区分：**"进程管资源，线程管执行"**。三个核心区别：**"进程独立空间开销大，线程共享空间开销小"**。

### 2. 协程和线程有什么区别？ [重要性:A]

| **特性** | **线程** | **协程** |
| --- | --- | --- |
| 调度方式 | 通常由 OS 内核抢占式调度 | 通常由语言运行时或事件循环调度，很多实现是协作式让出 |
| 切换开销 | 较大（涉及内核调度、上下文保存/恢复） | 通常更小，但具体取决于语言实现和运行时 |
| 栈大小 | 常见默认栈较大，可配置 | 多数实现更轻量，有的可动态增长 |
| 并发数量 | 受线程栈、内核调度和系统资源限制 | 更适合大量 IO 并发，但不是无限制 |
| 数据竞争 | 多线程共享内存，需加锁同步 | 单线程事件循环中的协程天然串行；多线程/多核运行时中的协程仍可能有数据竞争 |

**核心区别一句话：线程是操作系统调度的"轻量级进程"，协程是程序员调度的"轻量级线程"。**

**应用场景对比：**
- **线程**：CPU 密集型任务（计算密集）、需要多核并行、需要调用阻塞式系统接口。
- **协程**：IO 密集型任务（网络请求、文件读写）、高并发低延迟场景。注意 Go goroutine、Python asyncio、Kotlin coroutine、Java 虚拟线程的实现模型并不完全相同：goroutine 是 Go 运行时 M:N 调度，asyncio 是事件循环协作式调度，Java 虚拟线程由 JVM 调度但底层仍会映射到平台线程执行。

> **记忆口诀：线程多由 OS 调度，协程多由运行时调度；线程重但天然多核，协程轻但要看运行时是否能并行**。面试时不要把所有协程实现混成一种模型。

### 3. 进程的几种状态及转换？ [重要性:A]
- 基本状态：**就绪态**（等待 CPU 调度）、**运行态**（占用 CPU 执行）、**阻塞态**（等待资源或事件，如 IO）。
- 状态转换：
    - 就绪 → 运行：CPU 调度器选中该进程。
    - 运行 → 就绪：时间片用完，或被高优先级进程抢占。
    - 运行 → 阻塞：进程发起 IO 请求或等待同步信号。
    - 阻塞 → 就绪：IO 完成或等待的信号到达。

> **记忆口诀：三种状态——就绪等 CPU、运行占 CPU、阻塞等 IO**。状态转换：**"就绪→运行（被调度）、运行→就绪（时间片完）、运行→阻塞（等 IO）、阻塞→就绪（IO 完成）"**。可以想象成排队打饭：**"就绪=排队中，运行=在打饭，阻塞=饭卡没钱去充值"**。

### 4. CPU 调度算法有哪些？ [重要性:A]

操作系统的 CPU 调度算法决定了**就绪队列中的进程以什么顺序获得 CPU 时间片**，常见算法如下：

| **算法** | **核心逻辑** | **优点** | **缺点** |
| --- | --- | --- | --- |
| **FCFS（先来先服务）** | 按到达顺序依次执行，非抢占 | 实现简单，公平 | 平均等待时间长，短任务被长任务堵死（convoy effect） |
| **SJF（短作业优先）** | 优先执行预计运行时间最短的进程 | 平均等待时间最小（理论最优） | 无法预知运行时间，长任务可能**饥饿** |
| **优先级调度** | 高优先级进程先执行 | 优先级可控，适合实时系统 | 低优先级进程可能饥饿（可用"老化"解决——等待越久优先级越高） |
| **时间片轮转（RR）** | 每个进程分配固定时间片，轮流执行 | 响应快，交互式体验好 | 时间片大小难确定（太大退化为 FCFS，太小切换开销大） |
| **多级反馈队列（MLFQ/MFQ）** | 多个就绪队列，优先级从高到低，时间片从短到长，进程可在队列间升降级 | 兼顾 IO 密集（高优先级短时间片）和 CPU 密集（低优先级长时间片），是经典通用调度思想 | 实现复杂，参数配置敏感 |

> **记忆口诀：FCFS 先来先得、SJF 短工优先、RR 轮流坐庄、MLFQ 多级升降**。补充现代 Linux 口径：普通进程长期使用 CFS（Completely Fair Scheduler，尽量按虚拟运行时间公平分配 CPU），较新内核逐步引入 EEVDF；实时任务还有 SCHED_FIFO、SCHED_RR 等调度类。面试如果问“算法有哪些”讲经典算法，如果问“Linux 用什么”讲 CFS/调度类。

### 6. 进程间通信（IPC）有哪些方式？各自优缺点？ [重要性:A]

| **通信方式** | **优点** | **缺点** |
| --- | --- | --- |
| 管道 | 简单，匿名管道适用于父子进程 | 半双工，只能单向通信；数据在内存中，随进程结束消失 |
| 消息队列 | 独立于进程，可实现异步通信 | 有大小限制；数据拷贝开销 |
| 共享内存 | 最快的 IPC 方式，无数据拷贝 | 需要同步机制（如信号量）防止竞态 |
| 信号量 | 用于进程同步与互斥 | 只能传递信号，不能传递大量数据 |
| 套接字 | 可跨网络通信 | 开销较大，适用于不同主机进程 |

> **记忆口诀：五类 IPC——管道、消息队列、共享内存、信号量、套接字**。关键词串联：**"管道父子用，队列异步通，共享最快但要锁，信号量只传信号，Socket 跨机器"**。面试最高频：**"共享内存最快 + 信号量同步"**。

### 7. 什么是死锁？死锁的必要条件？如何避免死锁？ [重要性:S]
- **死锁定义**：多个进程因竞争资源而互相等待，且无法推进的状态。
- **必要条件**（4 个，缺一不可）：互斥条件、请求与保持条件、不可剥夺条件、循环等待条件。
- **解决方法**：
    - 预防：破坏任意一个必要条件（如一次性分配所有资源）。
    - 避免：银行家算法，动态判断资源分配是否安全。
    - 检测与解除：定期检测死锁，通过撤销进程或抢占资源解除。

> **记忆口诀：死锁四条件——互斥、请求与保持、不可剥夺、循环等待**。谐音记忆：**"互请不循"**（互斥、请求、不剥夺、循环）。解决三板斧：**"预防破坏条件，避免用银行家算法，检测就撤销进程"**。

### 8. 线程同步的方式有哪些？ [重要性:A]
- 互斥锁（Mutex）：保证同一时刻只有一个线程访问临界资源。
- 信号量（Semaphore）：可实现多个线程的同步与互斥，支持计数。
- 条件变量（Condition Variable）：用于线程间的信号传递，如生产者 - 消费者模型。
- 读写锁（Reader-Writer Lock）：区分读和写，读共享、写互斥，提高并发效率。

> **记忆口诀：四大同步利器——互斥锁、信号量、条件变量、读写锁**。一句话区分：**"互斥锁是一把钥匙一个人进，信号量是 N 把钥匙 N 个人进，条件变量是等通知再干活，读写锁是读不互斥写互斥"**。

## 二、内存管理

### 6. 分页和分段的区别？ [重要性:A]

| **维度** | **分页** | **分段** |
| --- | --- | --- |
| 目的 | 解决内存碎片，提高内存利用率 | 满足程序的模块化需求，便于共享和保护 |
| 单位 | 固定大小的页（如 4KB） | 可变大小的段（由程序逻辑决定） |
| 地址空间 | 一维地址空间（仅需页号 + 偏移量） | 二维地址空间（段号 + 段内偏移量） |
| 碎片 | 存在内部碎片（页内未用空间） | 存在外部碎片（段间未用空间） |

> **记忆口诀：分页 = 固定大小"格子"（防外部碎片），分段 = 按逻辑切"块"（方便共享保护）**。一句话：**"分页是给操作系统看的（管理内存），分段是给程序员看的（组织代码）"**。

### 8. 虚拟内存的原理和作用？ [重要性:A]
- **原理**：基于局部性原理，利用磁盘空间模拟内存，将进程的部分数据加载到内存，其余留在磁盘，需要时再调入。
- **作用**：
    - 扩大进程的可用内存空间，突破物理内存的限制。
    - 实现进程的内存隔离，提高安全性。
    - 允许多个进程共享内存，提高资源利用率。

> **记忆口诀：虚拟内存 = 用硬盘假装内存 + 局部性原理 + 按需调入**。一句话：**"物理内存不够？借硬盘空间来凑，要用到哪块就加载哪块"**。像图书馆书太多放不下，**"常用的放桌上（内存），不常用的放仓库（磁盘）"**。

### 9. 页面置换算法有哪些？ [重要性:A]
- **最佳置换算法（OPT）**：置换未来最久不使用的页面，理论最优但不可实现。
- **先进先出算法（FIFO）**：置换最先进入内存的页面，简单但会出现 Belady 异常（增加帧数，缺页率反而升高）。
- **最近最少使用算法（LRU）**：置换最近最少使用的页面，接近 OPT，需硬件支持记录使用时间。
- **时钟置换算法（Clock）**：基于 LRU 的近似，用位标记页面是否被使用，开销低。

> **记忆口诀：四种算法——OPT（理想）、FIFO（简单但有 Belady 异常）、LRU（最近最少用，最接近 OPT）、Clock（LRU 的简化版）**。面试高频：**"LRU 最常用，FIFO 要小心 Belady 异常——页框越多缺页越多，反直觉！"**。

### 10. 什么是内存碎片？如何解决？ [重要性:A]
- **内部碎片**：分配的内存大于进程所需，未使用的部分在分配单元内部。解决：分页管理（尽量匹配页大小）。
- **外部碎片**：内存中存在多个小的空闲块，总和足够但无法分配给连续需求的进程。解决：紧凑技术（移动进程合并空闲块）、分段加紧凑、分页管理。

> **记忆口诀：内部碎片 = "分多了用不完"（分页解决），外部碎片 = "零钱凑不整"（紧凑技术解决）**。类比：**"内部碎片是买大鞋空一截，外部碎片是散钱一堆买不起整件商品"**。

### 11. 什么是零拷贝（Zero-Copy）？为什么快？ [重要性:A]

零拷贝是一种**避免 CPU 将数据从内核空间拷贝到用户空间**的 IO 优化技术，核心是减少数据拷贝次数和上下文切换，常用于高性能网络服务（Kafka、Nginx、Netty）。

#### 零拷贝的实现方式（两种主流方案） [重要性:S]

**方案一：mmap + write（减少一次 CPU 拷贝）**
- 用 `mmap` 将文件映射到进程地址空间，**跳过"内核→用户"的 CPU 拷贝**
- 拷贝次数：3 次（2 次 DMA + 1 次 CPU 拷贝）
- 上下文切换：4 次（仍需要 read/write）

**方案二：sendfile（避免用户态拷贝，Linux 2.1+）**
- 通过一次 `sendfile` 系统调用完成文件到 Socket 的传输
- 拷贝次数：通常可减少到 2 次 DMA 相关传输，避免数据在内核态和用户态之间来回 CPU 拷贝；如果网卡支持 scatter-gather DMA，还可以进一步减少内核缓冲区到网卡缓冲区的数据复制
- 上下文切换：**2 次**（仅一次 sendfile 调用，两次切换）
- Nginx、Kafka 都使用此方式

>
> 磁盘 → ①内核缓冲区（DMA 拷贝）→ ②网卡（DMA 拷贝，数据描述符传递，非真正拷贝）
>

> **记忆口诀：传统 IO = 磁盘→内核→用户→Socket→网卡；sendfile = 数据不进用户态，少拷贝、少切换**。这里的“零拷贝”主要指避免 CPU 在用户态和内核态之间搬运数据，不代表物理链路上完全没有 DMA 或缓冲区移动。面试加分：**"Kafka、Nginx 常利用 sendfile/transferTo 等机制提升静态文件或日志传输性能"**。
