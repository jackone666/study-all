# 01-项目深挖学习文档

## 1. 项目一句话定位

这是一个面向鸿蒙开发者技术问答场景的企业级 Hybrid RAG 系统。系统基于 LangGraph StateGraph 编排固定问答工作流，将 BM25 关键词检索、向量检索、知识图谱检索和 LLM Wiki 知识沉淀层结合起来，解决开发文档、FAQ、技术规范、历史问答和最佳实践分散、难复用、难溯源的问题。

面试时不要说它是“多 Agent 系统”。更准确的说法是：

> 我们用的是固定工作流 + 条件路由 + 多模块协作。LLM 可以参与分类、生成和校验，但主链路不是让 LLM 自主规划，而是由 LangGraph StateGraph 管住流程边界。

## 2. 真实主链路

源码主链路在 `src/enterprise_hybrid_rag/graph/workflow.py`：

```text
load_memory
  -> check_permission
  -> classify_intent
  -> route_wiki
  -> call_tools / retrieve_knowledge
  -> build_context
  -> generate_answer
  -> verify_answer
  -> finalize_answer / repair / human_fallback
  -> save_memory
```

可以这样解释：

- `classify_intent_node` 调 `analyze_intent(query)`，产出 `intent / matched_keywords / entities / method`。
- workflow 层把这些字段补成 `query_analysis`，包括 `intent、matched_keywords、keywords、entities、classify_method`。
- `route_wiki` 做主题匹配、主题上下文构建和 query expansion。
- `retrieve_knowledge` 把 Wiki terms 合并进 `query_analysis`，再调用 `GraphRAGOrchestrator.retrieve()`。
- `GraphRAGOrchestrator` 调 `RetrievalRouter.route()` 生成检索计划，再执行 `parallel / graph_first / hybrid_only`。
- `verify_answer` 对答案做 LLM 校验或规则校验，失败后进入重试、repair 或人工兜底。

## 3. LangGraph StateGraph 为什么适合这个项目

面试官可能问：“为什么不用纯 Agent？”

回答重点：

- 技术问答系统的主流程相对固定：理解问题、检索证据、组织上下文、生成答案、校验答案。
- 企业知识库场景更看重稳定性、可观测性和权限边界，不适合让 LLM 自主决定任意步骤。
- StateGraph 的好处是节点边界清晰，每一步输入输出可测试，失败后可以做条件路由和降级。
- 这个项目里真正“动态”的地方在规则路由：意图分类、Wiki 主题匹配、检索权重选择、失败重试和人工兜底。

建议回答：

> 我们没有把它设计成自治 Agent，而是把 LLM 放在受控节点里。原因是企业知识问答更怕不可控行为，比如越权访问、无证据回答、错误工具调用。StateGraph 能把每个节点的输入输出固定下来，既方便做 trace，也方便失败时降级。

## 4. Hybrid RAG 检索架构

### 4.1 三路检索分别解决什么问题

| 检索源 | 适合问题 | 优点 | 局限 |
| --- | --- | --- | --- |
| BM25 关键词检索 | 错误码、API 名称、配置项、类名、函数名 | 精确匹配强、可解释 | 同义词和语义泛化弱 |
| 向量检索 | “为什么”“如何”“最佳实践”类语义问题 | 召回相似表达能力强 | 对错误码、短 query、专名不稳定 |
| 知识图谱检索 | 依赖关系、调用链、生命周期、上下游影响 | 适合关系推理和路径解释 | 构建成本高，图谱不完整时需要降级 |

面试时可以举例：

- “AUTH_401 报错怎么解决”更适合提高 BM25 权重。
- “为什么 ArkTS 页面状态变化后不刷新”更适合提高向量权重，同时保留图谱辅助。
- “生命周期和状态管理有什么关系”更适合走 graph_first。

### 4.2 检索路由数据流

源码依据：

- `src/enterprise_hybrid_rag/modules/intent_router.py`
- `src/enterprise_hybrid_rag/graph/workflow.py`
- `src/enterprise_hybrid_rag/rag/retrieval_router.py`
- `src/enterprise_hybrid_rag/rag/graph_rag_orchestrator.py`

数据流：

```text
query
  -> analyze_intent(query)
       returns {intent, matched_keywords, entities, method}
  -> workflow.classify_intent_node
       builds query_analysis
  -> retrieve_knowledge
       merges wiki topic terms into query_analysis
  -> GraphRAGOrchestrator.retrieve(query, query_analysis)
  -> RetrievalRouter.route(query, query_analysis)
       returns RetrievalPlan
  -> execute parallel / graph_first / hybrid_only
```

注意一个真实验证点：

用 query `ArkTS 生命周期和状态管理有什么关系，为什么页面不刷新？` 跑了一次，意图分类因为“状态”关键词被误打到 `ticket_query`，但检索路由命中“关系”关键词，最终仍走 `graph_first`。这说明系统有两层信号：意图分类可能有误差，但 retrieval router 还能基于 query pattern 做纠偏。

面试里可以把这讲成：

> 我们没有把所有判断压到单个意图分类器上，而是让 Router 同时看 intent、entities、keywords 和 query pattern。这样单个信号误判时，后面的检索路由仍有机会纠偏。

## 5. 意图驱动的动态权重路由

`RetrievalRouter` 的核心策略：

1. Graph RAG 关闭或不可用，直接 `hybrid_only`。
2. 关系类问题走 `graph_first`。
3. 精确匹配类问题走 `parallel`，提高关键词权重。
4. 语义解释类问题走 `parallel`，提高向量权重。
5. 默认问题走 `parallel`，按 intent profile 做平衡权重。

简历里写“动态调整 BM25、向量、知识图谱三路检索权重”，可以这样解释：

- `policy_question`：关键词和向量都重要，图谱权重低。
- `technical_question`：向量为主，关键词辅助，图谱中等。
- `troubleshooting`：关键词更高，因为错误码和报错文本很关键。
- `ticket_query`：关键词绝对主导，图谱基本不用。
- `general_question`：默认平衡。

源码里 exact / semantic / graph_first 会把规则基线和 intent profile 做 70/30 融合；默认 parallel 直接使用 intent profile。

## 6. 并行检索与降级机制

`GraphRAGOrchestrator._execute_parallel()` 会同时启动 keyword、vector 和 graph 检索：

```text
keyword task
vector task
graph task
  -> gather keyword/vector
  -> graph 失败不影响主链路
  -> graph 权重归零
  -> keyword/vector 重新归一化
  -> RRF 融合
  -> rerank
```

这样做的好处：

- 延迟取决于最慢的一路，而不是三路相加。
- 图谱服务不稳定时不阻塞问答。
- trace 中保留 `graph_failed / degraded_from / degraded_to`，方便线上归因。

面试回答关键词：

- “并行提升延迟上限”
- “单路失败不拖垮主链路”
- “权重重新归一化”
- “降级保留 trace，便于观测”

## 7. LLM Wiki 知识沉淀层

### 7.1 Wiki 单元结构

源码模型在 `src/enterprise_hybrid_rag/wiki/models.py`。

`WikiPage` 包括：

- `id`
- `title`
- `summary`
- `aliases`
- `tags`
- `parent`
- `children`
- `related_pages`
- `content_path`
- `source_doc_ids`

`WikiRelation` 包括：

- `source_id`
- `target_id`
- `relation_type`
- `weight`
- `description`

可以解释成：

> 普通 RAG 是 chunk 级检索，LLM Wiki 是主题级知识组织。它不是替代原始文档，而是在文档之上沉淀一层可导航、可复用、可溯源的主题索引。

### 7.2 主题匹配

`WikiTopicMatcher` 用四类信号匹配主题：

- exact：标题、别名、标签精确命中。
- BM25：对 Wiki page 的 title、summary、aliases、tags 做词项匹配。
- vector：对 Wiki summary 做向量召回。
- graph expansion：从命中主题扩展相关主题。

权重是：

```text
exact 0.4 + BM25 0.3 + vector 0.2 + graph 0.1
```

为什么这么设计：

- exact 权重最高，因为开发者问题里 API、框架名、错误码等专名很重要。
- BM25 适合主题页标题、标签、别名这种短文本。
- vector 弥补同义表达。
- graph expansion 带出关联主题，但不能过高，否则容易主题漂移。

### 7.3 查询扩展和上下文组织

`QueryExpander` 会把 query 与主题页的 title、aliases、tags、subtopics、related topics 合并，去重后生成 expanded query。

`wiki_context_to_doc()` 会把 WikiContext 包装成一个高优先级证据文档，复用原来的答案生成和校验链路。

这点可以面试时讲：

> 我们没有为了 Wiki 改造整个生成器，而是把 Wiki context 适配成 retrieved_doc，复用原来的 context manager、answer generator 和 verifier。这样改动小，风险低。

## 8. 主题级可信语义缓存

### 8.1 源码状态

当前源码里能看到：

- LLM Wiki 正向反馈沉淀：`WikiFeedbackIngestor.ingest_positive_feedback()`。
- Wiki 会保存 query、answer summary、source_doc_ids、aliases、tags、content_path。
- 近似文档去重使用 Redis 保存 dedup fingerprint。

当前没有看到完整的“Redis 可信语义缓存”实现文件。因此面试时建议按“生产最佳实践设计”来讲，不要说成完整落地代码。

### 8.2 最佳实践设计

缓存 key 建议包含：

```text
tenant_id
topic_id
query_embedding_hash / semantic_cluster_id
doc_version_hash
answer_version
permission_scope
```

缓存 value 建议包含：

```text
answer_summary
final_answer
citations
source_doc_ids
wiki_page_id
confidence
verified_at
expire_at
```

命中流程：

1. 先做 query embedding。
2. 在同 tenant、同 topic 下查相似 query。
3. 校验语义相似度是否超过阈值。
4. 校验 topic_id 是否一致。
5. 校验 doc_version_hash 是否一致，知识库更新后自动失效。
6. 校验权限范围，避免把有权限用户的问题缓存给无权限用户。
7. 命中后返回缓存答案和引用；低置信度或版本不一致则回源 RAG。

### 8.3 为什么普通语义缓存容易缓存错答案

普通语义缓存只看 query 相似度，会有三个风险：

- 语义相近但答案不同：例如“ArkTS 状态不刷新”和“组件生命周期不触发”很像，但解决方案可能不同。
- 知识版本过期：文档更新后，旧答案仍被缓存命中。
- 权限污染：不同租户或不同角色看到的答案可能不同。

主题级可信语义缓存的价值：

- 用 Wiki topic 限定语义空间，降低跨主题误命中。
- 用 verified answer 限定缓存来源，避免缓存未校验答案。
- 用 doc_version_hash 做失效控制，避免知识更新后返回旧答案。
- 用 citations 保留可追溯性，命中缓存也不是“裸答案”。

面试口径：

> 这块我会明确区分普通语义缓存和可信语义缓存。普通语义缓存只追求命中率，但生产问答系统更怕误命中，所以我们把 Wiki 主题、答案校验结果、引用来源和知识版本都放进缓存判断，牺牲一点命中率换可信度。

## 9. 近似文档去重

源码依据：

- `src/enterprise_hybrid_rag/rag/near_dedup.py`
- `src/enterprise_hybrid_rag/rag/ingestion.py`

两层去重：

1. SHA256 精确哈希去重。
2. embedding cosine similarity 近似去重，默认阈值 0.95。

Redis key：

```text
dedup:doc:{doc_id}
dedup:hash:{content_hash}
dedup:all
```

为什么要做：

- 减少重复文档入库，降低存储成本。
- 减少重复 chunk 在 top_k 中挤占位置，提高检索多样性。
- 减少同一答案被重复引用，提升上下文质量。

trade-off：

- 阈值过低会误删相似但不同的规范文档。
- 阈值过高只能去掉几乎重复文本，收益有限。
- 长文档全量 embedding 成本高，所以源码用 head + tail 采样生成文档级 fingerprint。

## 10. 五层幻觉控制机制

源码能支撑的部分：

- 检索为空或低分会触发 fallback。
- `verify_answer` 检查空答案、无文档、低相关噪音、低置信度、引用缺失、答案过短。
- `HallucinationGuard` 支持句子级相似度策略。
- verify 失败后进入 regenerate、repair 或 human_fallback。

最佳实践补充部分：

1. 检索证据质量评估  
   为什么做：没有高质量证据就不该强行生成。看 top score、平均分、source 多样性、是否有权威文档。

2. 事实锚定 Prompt  
   为什么做：要求模型只基于给定证据回答，无法确认就说明不确定，降低自由发挥。

3. 回答-证据相似度校验  
   为什么做：生成后的答案可能引入证据中没有的内容，句子级校验能发现局部幻觉。

4. 业务规则校验  
   为什么做：技术问答中有权限、版本、平台、API 废弃状态等业务规则，不能只靠语义相似度。

5. 低置信度拒答与二次生成/人工兜底  
   为什么做：生产系统不能为了“有答案”而牺牲可信度。低置信度时宁可拒答、改写 query 重试或转人工。

## 11. 面试主线总结

你可以把这个项目讲成四层：

1. Workflow 层：LangGraph 固定流程，控制权限、检索、生成、校验和兜底。
2. Retrieval 层：BM25 + Vector + Graph 三路混合，按意图和 query pattern 动态调权。
3. Wiki 层：把碎片知识沉淀成主题页，支持导航、扩展、上下文组织和正反馈沉淀。
4. Trust 层：可信缓存、去重、引用、相似度校验、拒答和人工兜底。

最强的一句话：

> 这个项目不是单纯把文档接进向量库，而是围绕“开发者问答可复用、可追溯、可降级、可校验”做了一套生产级 Hybrid RAG 链路。
