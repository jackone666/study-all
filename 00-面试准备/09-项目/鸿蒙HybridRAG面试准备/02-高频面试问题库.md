# 02-高频面试问题库

## 一、项目总览类

### 1. 你这个鸿蒙开发者问答系统解决了什么问题？

考察点：业务理解、项目价值、不是只会堆技术。

回答要点：
- 鸿蒙开发者问题分散在文档、FAQ、规范、历史问答和最佳实践里。
- 单纯向量 RAG 容易遇到专名召回不准、关系问题回答弱、答案不可溯源的问题。
- 系统用 Hybrid RAG + LLM Wiki + 答案校验，目标是提升知识复用、召回质量和回答可信度。

追问：
- 为什么不是直接用向量库？
- 为什么要加知识图谱？
- 为什么要做 Wiki？

### 2. 整体架构怎么讲？

考察点：能否画出架构和数据流。

回答要点：
- 入口是 LangGraph StateGraph。
- 前面做权限、记忆、意图分类。
- 中间做 Wiki 主题匹配和三路检索。
- 后面做上下文构造、生成、校验、失败重试和人工兜底。

追问：
- 每个节点输入输出是什么？
- 哪些节点会失败？怎么降级？
- 这个系统是不是 Agent？

### 3. 为什么说这是 Workflow，不是 Agent？

回答要点：
- 节点和边是固定的，LLM 不自主决定下一步。
- 动态性来自条件路由和规则权重，不是 LLM planning。
- 企业问答更需要可控、可测、可审计。

## 二、LangGraph / Workflow 类

### 4. 为什么选 LangGraph StateGraph？

回答要点：
- 多节点流程有明显状态传递。
- 需要条件路由，例如权限拒绝、无结果改写、校验失败重试。
- 需要可观测性，每个节点可以 trace latency、输入输出和失败原因。

### 5. State 里核心字段有哪些？

回答要点：
- query、intent、query_analysis。
- wiki_mode、wiki_context、wiki_expanded_query。
- retrieved_docs、reranked_docs、graph_paths。
- draft_answer、citations、verified、verification_reason。
- retry_count、fallback_reason、need_human。

### 6. 如果检索不到文档怎么办？

回答要点：
- 如果没有可用 docs，先判断 retry 次数。
- 可重试则 rewrite query 后重新 retrieve。
- 仍失败则 human_fallback。
- 低分文档也不能强行回答。

## 三、Hybrid RAG 检索类

### 7. BM25、向量、图谱分别解决什么问题？

回答要点：
- BM25：错误码、API、类名、配置项。
- 向量：语义相似、问法变化、最佳实践。
- 图谱：实体关系、调用链、依赖、生命周期。

追问：
- 如果三路结果冲突怎么办？
- 哪一路最重要？
- 为什么不是全部权重一样？

### 8. 三路检索怎么融合？

回答要点：
- parallel 模式下三路并发。
- 用 RRF 做融合，避免不同检索源分数尺度不一致。
- 融合后 rerank。
- graph 失败时 graph 权重归零，keyword/vector 重新归一化。

### 9. 为什么用 RRF？

回答要点：
- BM25、向量和图谱的原始分数不可直接比较。
- RRF 基于排名融合，对分数尺度不敏感。
- 实现简单、鲁棒，适合多路召回融合。

### 10. graph_first 和 parallel 的区别？

回答要点：
- parallel：keyword/vector/graph 同时跑，适合大多数问题。
- graph_first：先查图谱拿实体和路径，再用扩展 query 查 keyword/vector，适合关系类问题。
- graph_first 失败后降级到 hybrid_only。

### 11. 图谱检索不可用怎么办？

回答要点：
- Router 会检查 GraphRAG 是否开启和图数据库是否可用。
- 不可用时直接 hybrid_only。
- parallel 中 graph 失败不影响 keyword/vector。
- trace 记录 degraded_from/degraded_to。

## 四、动态检索路由类

### 12. 你的动态权重怎么设计？

回答要点：
- 先通过 intent 和 query pattern 判断问题类型。
- 关系问题提高 graph 权重。
- 精确问题提高 keyword 权重。
- 语义问题提高 vector 权重。
- 默认用 intent profile 平衡权重。

### 13. 意图分类错了怎么办？

回答要点：
- 不只依赖 intent。
- Router 同时看 query 原文、关键词、实体、错误码/API pattern。
- 即使 intent 误判，关系关键词、错误码等强规则仍能纠偏。

可举真实验证：
- `ArkTS 生命周期和状态管理有什么关系，为什么页面不刷新？`
- intent 可能误分，但 Router 命中“关系”后仍走 graph_first。

### 14. 为什么用规则路由，不直接让 LLM 判断检索策略？

回答要点：
- 检索策略要求稳定、低延迟、可解释。
- 规则路由能写入 reason，方便排查。
- LLM 可以作为低置信度增强，但不适合作为核心路由唯一依据。

### 15. 怎么评估路由效果？

回答要点：
- 离线 golden set：看 Recall@K、MRR、Hit@K。
- 按 query 类型分桶：错误码、API、语义解释、关系推理。
- 在线看 fallback_rate、无结果率、用户点赞率、引用点击率、延迟。

## 五、LLM Wiki 类

### 16. LLM Wiki 和普通文档 chunk 有什么区别？

回答要点：
- chunk 是原始文档切片，适合证据检索。
- Wiki 是主题级知识单元，包含摘要、别名、标签、关系和来源引用。
- Wiki 强调导航和复用，不替代原始证据。

### 17. Wiki 主题页怎么生成？

回答要点：
- 人工导入/离线构建业务文档。
- 用户点赞的高质量 QA 可以沉淀为 Wiki 候选。
- 只保存主题、摘要、source_doc_ids 和导航元数据，不复制原始 chunk。

### 18. Wiki 主题匹配怎么做？

回答要点：
- exact：标题、别名、标签。
- BM25：主题页文本。
- vector：summary 向量召回。
- graph expansion：关联主题扩展。
- 最终加权融合。

### 19. 为什么 Wiki 能提升回答可信度？

回答要点：
- 把高质量知识沉淀到主题层，减少重复问答。
- 主题页保留来源引用，方便溯源。
- 主题词、别名、标签可以增强 query expansion。
- 相关主题能补齐上下文。

### 20. Wiki 会不会引入错误知识？

回答要点：
- 只沉淀 verified 且非 human_fallback 的优质结果。
- 保留 source_doc_ids。
- 生产最佳实践是加人工审核、版本号、过期时间和质量分。

## 六、可信语义缓存类

### 21. 你说的可信语义缓存和普通 Redis 缓存有什么区别？

回答要点：
- 普通缓存看 exact key 或 query 相似度。
- 可信语义缓存要同时看 topic、语义相似度、知识版本、权限和答案校验状态。
- 缓存的不是任意答案，而是已验证的优质问答。

### 22. 为什么普通语义缓存会缓存错答案？

回答要点：
- 相似 query 不一定答案相同。
- 文档版本变化后旧答案可能过期。
- 不同租户/权限看到的内容不同。
- 所以不能只用 embedding similarity。

### 23. 缓存 key 怎么设计？

回答要点：
- tenant_id。
- topic_id。
- query semantic hash。
- doc_version_hash。
- permission_scope。
- answer_version。

### 24. 缓存命中流程怎么设计？

回答要点：
- 先做 query embedding。
- 限定同 tenant、同 topic。
- 相似度超过阈值。
- 校验 doc_version_hash。
- 校验权限。
- 返回答案和 citations。
- 任一校验不通过就回源 RAG。

### 25. 缓存如何失效？

回答要点：
- TTL。
- 文档版本变化触发版本失效。
- Wiki 页面更新触发 topic 缓存失效。
- 用户负反馈触发单条缓存降权或删除。

## 七、近似文档去重类

### 26. 为什么要做文档去重？

回答要点：
- 重复文档增加存储和索引成本。
- 重复 chunk 会挤占 top_k。
- 重复证据会让模型误以为某个信息很重要。

### 27. 你的去重怎么实现？

回答要点：
- 第一层 SHA256 exact hash。
- 第二层 embedding cosine similarity。
- Redis 保存 fingerprint，内存作为 cosine 计算缓存。
- 默认阈值 0.95。

### 28. 为什么不只用 SHA256？

回答要点：
- 轻微改写、格式变化、增删空格都会导致 hash 不同。
- embedding 能识别语义近似重复。

### 29. 为什么不只用 embedding？

回答要点：
- embedding 成本更高。
- exact hash 是 O(1) 快速路径。
- 两层组合能兼顾速度和召回。

### 30. 近似去重有什么风险？

回答要点：
- 阈值过低误删不同文档。
- 阈值过高收益不足。
- 需要保留人工恢复、白名单和抽样审计。

## 八、幻觉控制类

### 31. 你们怎么控制幻觉？

回答要点：
- 检索证据质量评估。
- 事实锚定 Prompt。
- 回答-证据相似度校验。
- 业务规则校验。
- 低置信度拒答与二次生成/人工兜底。

### 32. 回答-证据相似度怎么做？

回答要点：
- 句子级拆分答案。
- 每个句子找最相似证据。
- 取最弱句子作为整体风险。
- 生产建议用 bge/sbert，源码默认 TF-IDF + bigram 兜底。

### 33. 为什么要句子级，而不是整段平均？

回答要点：
- 整段平均会掩盖局部幻觉。
- 一句话编造 API 或参数也会造成严重错误。
- 句子级更严格，适合技术问答。

### 34. 什么时候拒答？

回答要点：
- 没有检索文档。
- 文档分数过低。
- 答案缺少引用。
- 回答和证据相似度低。
- 触发业务规则风险。

### 35. 拒答会不会影响用户体验？

回答要点：
- 会牺牲一部分覆盖率，但换来可信度。
- 拒答不是结束，可以给出补充问题建议、改写 query 重试或转人工。
- 企业技术问答里错误答案成本比拒答更高。

## 九、系统设计追问

### 36. 如果 QPS 提高 10 倍怎么优化？

回答要点：
- 缓存高频 verified QA。
- 检索并行化和超时控制。
- ES/Milvus/Neo4j 分别扩容。
- embedding 批处理。
- Rerank 限制候选数量。
- 低风险问题走轻量模型或模板答案。

### 37. 如果知识库每天更新，怎么保证缓存不返回旧答案？

回答要点：
- 文档 ingestion 生成 doc_version_hash。
- 缓存 key/value 绑定版本。
- 文档更新后按 topic/doc_id 批量失效。
- 命中缓存前校验版本。

### 38. 如果用户问的问题跨多个主题怎么办？

回答要点：
- TopicMatcher 返回 main_topic、sub_topics、related_topics。
- query expansion 合并多个主题词。
- 检索结果融合后 rerank。
- 如果主题置信度低，回到普通 Hybrid RAG。

### 39. 如果图谱里没有关系怎么办？

回答要点：
- graph_first 无结果会降级 hybrid_only。
- parallel 里 graph 无结果不影响 keyword/vector。
- 离线通过失败 case 补充实体和关系抽取规则。

### 40. 你觉得项目还有什么改进空间？

回答要点：
- 意图分类可以减少“状态”等歧义词误判。
- 可信缓存需要更完整的版本失效和负反馈淘汰。
- 幻觉校验可从 TF-IDF fallback 升级到领域 embedding。
- Wiki 沉淀可增加人工审核和质量分。
- 离线评估可补 RAGAS/DeepEval 或自研 faithfulness 指标。
