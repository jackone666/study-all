# 03-回答稿与追问拆解

## 1. 60 秒项目介绍

我负责的是一个面向鸿蒙开发者的企业级 Hybrid RAG 技术问答系统。这个场景的问题是开发知识分散在文档、FAQ、技术规范、历史问答和最佳实践里，普通向量 RAG 对错误码、API 专名、实体关系和答案可信度处理都不够稳定。

所以我们没有只做一个“文档切 chunk + 向量检索”的系统，而是基于 LangGraph StateGraph 编排固定问答工作流，把 BM25 关键词检索、向量检索和知识图谱检索结合起来。前面通过意图分类和规则路由识别问题类型，中间按 query 特征动态调整三路检索权重，后面通过上下文组织、答案生成、引用溯源和答案校验控制幻觉。

另外我们建设了一层 LLM Wiki，把高质量文档和历史问答沉淀成主题页、摘要、标签/别名、关联关系和来源引用，用来支持主题导航、查询扩展和可信缓存。整体目标是让开发者问答既能召回准，也能解释清楚答案依据。

## 2. 项目不是 Agent，怎么解释

如果面试官问：“你这个是不是 Agentic RAG？”

回答：

不是严格意义上的自治 Agent。我们项目核心是 Hybrid RAG Workflow。流程是固定的：权限校验、意图分析、Wiki 主题匹配、知识检索、上下文构造、答案生成、答案校验和兜底。LLM 可以参与生成和校验，也可以作为低置信度增强，但不会让 LLM 自主规划任意步骤。

这么设计是因为企业知识问答系统更重视稳定性、权限边界和可观测性。用 LangGraph StateGraph 可以把每个节点的输入输出固定下来，失败时走条件路由，比如无结果就改写 query，校验失败就二次生成或转人工。这样比纯 Agent 更可控。

## 3. Hybrid RAG 架构回答稿

面试官问：“为什么要 BM25 + 向量 + 图谱三路检索？”

回答：

因为鸿蒙开发者问题有几类典型形态。第一类是精确匹配，比如错误码、API 名称、类名、配置项，这类 BM25 更稳定。第二类是语义解释，比如“为什么页面不刷新”“如何优化启动速度”，这类向量检索能覆盖同义表达。第三类是关系问题，比如生命周期、状态管理、组件依赖之间的关系，这类需要知识图谱补充实体关系和路径解释。

所以我们的设计不是固定一路检索，而是根据 query 类型动态调整权重。精确问题提高关键词权重，语义问题提高向量权重，关系问题增强图谱检索。最后用 RRF 融合多路结果，避免不同检索源分数尺度不一致。

追问：“为什么不用向量检索一把梭？”

回答：

向量检索对自然语言语义很好，但对短 query、错误码、API 名、配置项不一定稳定。比如 `AUTH_401` 或 `module.json`，BM25 会比向量更可控。而图谱能回答实体关系问题，这是单纯向量相似度做不到的。

## 4. 动态权重路由回答稿

面试官问：“动态检索权重具体怎么做？”

回答：

上游 `analyze_intent` 会产出 intent、matched_keywords 和 entities，workflow 再补充 query tokens，形成 `query_analysis`。检索路由不是只看 intent，而是同时看 query 原文里的 pattern，比如错误码、API 名、函数名、关系关键词、语义关键词等。

路由规则大概分几层：

第一，Graph RAG 不可用就直接走 keyword + vector 的 hybrid_only。第二，命中关系类问题，比如“依赖、调用链、生命周期、关系”，走 graph_first。第三，命中错误码、API 名、类名等精确匹配信号，走 parallel 但提高 keyword 权重。第四，命中“为什么、如何、怎么解决”等语义问题，走 parallel 但提高 vector 权重。剩下的走默认 parallel，用 intent profile 平衡三路权重。

追问：“意图分类错了怎么办？”

回答：

我们没有把路由完全依赖意图分类。比如我用 `ArkTS 生命周期和状态管理有什么关系，为什么页面不刷新？` 做过验证，意图分类可能会因为“状态”这类词产生误判，但 Router 还能基于“关系”关键词走 graph_first。也就是说系统把 intent、entities、keywords 和 query pattern 都作为信号，单个信号错了不会直接导致整个检索策略失败。

## 5. LLM Wiki 回答稿

面试官问：“LLM Wiki 是什么，和普通 RAG 文档有什么区别？”

回答：

普通 RAG 的基本单位是 chunk，它是原始文档的切片，适合做证据检索。但技术问答里还有很多主题级知识，比如 ArkTS、状态管理、生命周期、权限申请、常见报错，这些知识不是单个 chunk 能完整表达的。

所以我们建设了 LLM Wiki，把业务文档、FAQ、规范、历史问答和最佳实践沉淀为主题页。一个主题页包含 title、summary、aliases、tags、关联关系和 source_doc_ids。它不替代原始文档，而是在原始文档之上构建一层主题导航和知识复用层。

用户提问时，系统可以先匹配 Wiki 主题，再把主题标题、别名、标签、子主题和相关主题注入 query expansion，同时把 Wiki context 包装成高优先级 evidence doc，复用原来的 RAG 生成和校验链路。

追问：“Wiki 怎么避免变成另一套脏知识库？”

回答：

首先 Wiki 只保存主题摘要、来源引用和导航元数据，不复制原始 chunk。其次用户点赞的问答要满足 verified 且没有进入人工兜底，才会沉淀为候选 Wiki 知识。生产上还应该加人工审核、质量分、版本号和过期机制，保证 Wiki 是高质量知识索引，而不是垃圾答案堆积。

## 6. 可信语义缓存回答稿

面试官问：“你们的 Redis 缓存怎么设计？和普通缓存有什么区别？”

回答：

这块我会分成两层讲。普通语义缓存通常只看 query embedding 相似度，超过阈值就返回历史答案。但技术问答里这样做风险很高，因为两个问题语义相似不代表答案相同，而且文档版本、权限范围和平台版本都可能影响答案。

所以我设计的是主题级可信语义缓存。缓存来源不是任意历史回答，而是 LLM Wiki 中已经验证过的优质问答。缓存命中时不只校验 query 相似度，还要校验 topic 是否一致、知识版本是否一致、权限范围是否一致，以及缓存答案是否有来源引用。只有这些都通过，才返回 Redis 里的答案和 citation。

追问：“为什么要放 topic_id？”

回答：

topic_id 是为了限制语义相似度的搜索空间。同一个 query 在不同主题下答案可能完全不同，比如“状态不更新”在 UI 状态管理和工单状态查询里含义不同。如果只看 embedding，很容易误命中。先限定 Wiki topic，再做相似度，可以降低跨主题误命中。

追问：“缓存怎么失效？”

回答：

我会用三种机制。第一是 TTL，避免长期不更新。第二是 doc_version_hash，文档更新后相关缓存自动失效。第三是负反馈淘汰，如果用户点踩或者校验失败，就降低缓存质量分或删除缓存。这样缓存不会只追求命中率，而是兼顾可信度。

## 7. 近似文档去重回答稿

面试官问：“近似文档去重怎么做？”

回答：

我用了两层去重。第一层是 SHA256 内容哈希，解决完全重复文档，速度快、成本低。第二层是 embedding 余弦相似度，解决轻微改写、格式变化、内容高度相似但 hash 不同的文档。默认阈值是 0.95。

去重 fingerprint 会保存到 Redis，比如 `dedup:doc:{doc_id}`、`dedup:hash:{content_hash}` 和 `dedup:all`。为了降低长文档 fingerprint 成本，文档级 embedding 不是全量内容，而是取 head + tail 采样，这能在成本和代表性之间做平衡。

追问：“为什么去重能提升检索质量？”

回答：

重复 chunk 会挤占 top_k，导致模型看到的证据多样性下降。比如 top5 里有 3 个来自同一份重复文档，模型上下文看起来很多，其实信息增量很低。去重后可以降低存储成本，也能让 top_k 容纳更多不同来源证据。

追问：“阈值怎么选？”

回答：

阈值要用标注样本调。过低会误删相似但不同的文档，过高只能去掉完全近似的文本。技术规范类文档建议阈值保守一些，比如 0.95，并保留白名单和抽样审计。

## 8. 五层幻觉控制回答稿

面试官问：“你们怎么控制幻觉？”

回答：

我把幻觉控制分成五层。

第一层是检索证据质量评估。没有检索到文档、文档分数都很低，或者证据来源太单一，就不应该强行回答。

第二层是事实锚定 Prompt。生成时要求模型只基于给定证据回答，无法确认的内容要说明不确定。

第三层是回答-证据相似度校验。我们有一个 HallucinationGuard，默认用句子级策略，把答案拆成句子，每个句子和检索证据计算相似度，取最弱句子作为风险点。源码里默认是 TF-IDF + bigram 兜底，生产上建议替换成 bge/sbert embedding。

第四层是业务规则校验。比如权限、平台版本、API 废弃状态、引用完整性，这些不能只靠语义相似度。

第五层是低置信度拒答和二次生成。如果校验不过，系统会重试 build_context/generate，必要时 repair 或 human_fallback。企业技术问答里，错误答案的成本比拒答更高。

追问：“为什么句子级校验？”

回答：

整段平均相似度容易掩盖局部幻觉。技术问答里可能只有一句话编了一个 API 参数，就会导致用户照着做出错。句子级校验更严格，能抓到最悬空的那一句。

## 9. 系统失败和降级回答稿

面试官问：“线上图谱服务挂了怎么办？”

回答：

分两种情况。如果 Router 一开始发现 Graph RAG 没开启或图数据库不可用，会直接走 hybrid_only，只用 keyword + vector。如果是在 parallel 模式下 graph 检索失败，graph_candidates 会置空，graph 权重归零，keyword 和 vector 权重重新归一化，主链路继续。如果 graph_first 失败或无结果，会降级到 hybrid_only。

同时 trace 会记录 degraded_from、degraded_to 和 graph_failed，后续可以用来分析到底是图谱稳定性问题，还是某类关系问题覆盖不足。

## 10. 项目反思回答稿

面试官问：“这个项目你觉得还有什么不足？”

回答：

我觉得有四个可以继续优化的点。

第一，意图分类词典还有歧义，比如“状态”既可能是工单状态，也可能是 UI 状态管理。后续可以把业务域词典拆开，结合上下文和实体类型做 disambiguation。

第二，可信语义缓存需要更完整的版本失效和负反馈淘汰机制。缓存不是命中率越高越好，技术问答更要控制误命中。

第三，幻觉校验现在有 TF-IDF + bigram 兜底，但生产上应该换成领域 embedding，并用标注数据校准阈值。

第四，Wiki 沉淀可以增加人工审核和质量分层，避免正反馈样本直接进入高置信知识层。

这个回答的好处是：你不是只说成功，也能讲系统边界、风险和下一步演进。

## 11. 行为面试 STAR 案例：动态路由

问题：“说一个你解决过的技术难题。”

回答框架：

S：鸿蒙开发者问答里 query 类型差异很大，错误码、API、语义解释、关系推理混在一起，单一路检索很难兼顾。

T：我的目标是设计一个稳定、可解释、可降级的检索路由，让不同类型问题走更合适的检索组合。

A：我把上游意图分类产出的 intent、entities、keywords 和 query pattern 结合起来，设计了关系类 graph_first、精确类 keyword-heavy、语义类 vector-heavy、默认 parallel 的路由策略。每个策略都生成 RetrievalPlan，里面包含 mode、enabled_retrievers、top_k、weights 和 reason。执行层通过 asyncio 并发三路检索，图谱失败时自动降级并记录 trace。

R：这样系统不再依赖单一路召回，错误码、语义问题和关系问题能走不同权重；同时单路失败不会拖垮主链路，线上排查时也能根据 route reason 和 degradation trace 定位问题。

## 12. 面试前速记

- 项目定位：Hybrid RAG Workflow，不是自治 Agent。
- 主链路：classify -> wiki -> retrieve -> context -> generate -> verify -> fallback。
- 三路检索：BM25 管精确，Vector 管语义，Graph 管关系。
- 动态路由：intent + query pattern，不只靠 LLM。
- Wiki：主题级知识层，做导航、扩展、沉淀、溯源。
- 缓存：可信语义缓存 = topic + similarity + version + permission + verified answer。
- 去重：SHA256 exact + embedding near-dedup。
- 幻觉控制：证据质量、锚定 Prompt、相似度校验、业务规则、拒答/重试/人工。
