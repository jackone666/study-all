### **1. Redis 有哪些数据结构？分别在什么场景下使用？**

| **数据类型** | **核心特点** | **典型应用场景** |
| --- | --- | --- |
| **String** | 二进制安全字符串，支持自增/自减 | 缓存用户信息（`set user:1 "{name:'张三'}"`）、计数器（`incr article:1001`阅读量）、分布式锁（`set lock:order NX PX 30000`） |
| **List** | 双向链表，支持两端操作 | 简单消息队列（`lpush queue:order 1001`+`brpop queue:order`）、最新消息排行（朋友圈动态，`lpush`+`lrange 0 9`） |
| **Hash** | 键值对集合（类似 HashMap） | 存储对象属性（`hmset user:1 name 张三 age 20`，可单独改`hset user:1 age 21`）、购物车（`hset cart:user1 sku1 2`） |
| **Set** | 无序唯一元素集合，支持交并差运算 | 去重（用户标签）、共同好友（`sinter user:1:follow user:2:follow`）、抽奖（`srandmember prize 1`） |
| **ZSet** | 带分数（score）的唯一元素集合，可排序 | 排行榜（`zadd rank:score 1000 user:1`+`zrevrange 0 9`）、延时任务（`zadd delay:order 1730000000 order:1001`+`zrangebyscore`） |
| **Bitmap** | 位运算（0/1），极致省内存 | 签到统计（`setbit sign:user1 20240501 1`）、在线状态（`setbit online:users 1001 1`） |
| **HyperLogLog** | 基数统计（UV），12KB 内存统计 2^64 数据 | 网站 UV 统计（`pfadd uv:20240501 user1 user2`+`pfcount uv:20240501`） |
| **Geo** | 地理位置存储与计算（经纬度） | 附近的人（`geoadd city:pos 116.4 39.9 beijing`+`georadius city:pos 116.4 39.9 10 km`） |
| **Stream** | 消息队列（支持持久化、消费者组、ACK） | 可靠消息传递（订单状态变更通知，`xadd stream:order * id 1001 status paid`+消费者组） |

### **2. Redis 为什么这么快？**
**金句**：“Redis 快的核心是‘内存操作+I/O多路复用+单线程无锁+高效数据结构’，把性能压榨到极致。”
**详细解答**：
1. **纯内存操作**：数据存储在内存中，读写速度比磁盘快 10 万倍以上（内存纳秒级，磁盘毫秒级）。
2. **I/O 多路复用**：单线程通过 epoll/kqueue 监听多个 socket 连接，高效处理并发请求（非阻塞 I/O）。
3. **单线程模型**：核心命令执行模块单线程，避免多线程上下文切换和锁竞争（6.0 后网络 I/O 多线程，命令执行仍单线程）。
4. **高效数据结构**：如 SDS（动态字符串，预分配空间）、跳表（ZSet 范围查询 O(logN)）、哈希表（O(1) 查找）。
5. **精简代码**：C 语言编写，核心逻辑简洁，无冗余操作。

### **3. Redis 是单线程的吗？6.0 之后呢？多线程到底解决了什么问题？**
**金句**：“Redis‘单线程’指核心命令执行单线程，6.0 后引入多线程 I/O 解决网络瓶颈，命令执行仍单线程保证无锁安全。”
**详细解答**：
**早期版本（<6.0）**：完全单线程，处理网络 I/O、命令执行、持久化等所有操作。
**6.0 之后**：**多线程 I/O（默认关闭）**，仅用于网络数据的读写（如 socket 接收/发送），**命令执行仍为单线程**（避免并发安全问题）。
**解决的问题**：网络 I/O 成为瓶颈（如高并发下大量连接握手/数据传输耗时），多线程并行处理网络读写，提升吞吐量（官方测试 QPS 提升 1 倍+）。
### **4. Redis 的持久化机制？RDB 和 AOF 分别是什么？优缺点？宕机丢失多少数据？**
**金句**：“RDB 是‘内存快照’（恢复快但丢数据），AOF 是‘操作日志’（数据安全但恢复慢），生产推荐混合持久化（RDB+AOF）。”
**详细解答**：

| **维度** | **RDB（快照持久化）** | **AOF（追加日志持久化）** |
| --- | --- | --- |
| **原理** | 定时 fork 子进程，生成二进制快照文件（dump.rdb） | 记录所有写命令到文本文件（appendonly.aof），重启时重放命令 |
| **优点** | 文件小（二进制压缩）、恢复快（直接加载内存） | 数据安全（可配置每秒同步）、支持实时记录 |
| **缺点** | 可能丢失最后一次快照后的数据（如 5 分钟快照丢 5 分钟数据）；fork 耗 CPU/内存（大内存时明显） | 文件大（文本格式）、恢复慢（重放所有命令） |
| **宕机丢失数据** | 最多丢失“快照间隔+写操作耗时”数据（如 save 60 10000，最多丢 60 秒数据） | 取决于同步策略：`always`（不丢）、`everysec`（最多丢 1 秒）、`no`（os决定什么时候缓存） |

### **5. AOF 重写的过程了解吗？重写时还能提供服务吗？BGREWRITE AOF 和 BGSAVE(rdb自动执行) 可以同时执行吗？**
**金句**：“AOF 重写是‘瘦身日志’，通过 COW 机制后台执行不阻塞服务，BGREWRITEAOF 和 BGSAVE 不能同时跑（争抢 fork 资源）。”
**详细解答**：
**重写过程**：
6. 主进程 fork 子进程，子进程读取当前内存数据，生成新的 AOF 文件（仅保留最终状态命令，如多次`set k v`合并为`set k v`）；
7. 主进程将重写期间的写命令缓存到“重写缓冲区”（aof_rewrite_buf_blocks）；
8. 子进程完成重写后，主进程将缓冲区命令追加到新 AOF 文件，替换旧文件。

**  是否提供服务**：**能**。重写期间主进程正常处理请求，写命令同时写入旧 AOF 和重写缓冲区。
**BGREWRITEAOF 和 BGSAVE 能否同时执行**：**不能**。两者都需 fork 子进程，Redis 会拒绝第二个请求（避免资源竞争）。
### **6. Redis 的内存淘汰策略有哪些？LRU 是精确的吗？Redis 怎么实现近似 LRU 的？**
**金句**：“内存淘汰是‘内存满时删 key’，8 种策略按需选；LRU 用‘采样近似’替代精确计算，平衡性能与效果。”
**详细解答**：
**8 种淘汰策略**：
**全量 key**：`allkeys-lru`（最近最少用，缓存首选）、`allkeys-lfu`（最不常用，热点数据）、`allkeys-random`（随机删）、`noeviction`（不删，报错）；
**过期 key**：`volatile-lru`（过期 key 中 LRU）、`volatile-lfu`（过期 key 中 LFU）、`volatile-random`（过期 key 随机删）、`volatile-ttl`（删 TTL 最短的）。
**LRU 是否精确**：**否**。精确 LRU 需维护所有 key 的访问时间链表，内存和时间开销大。
**近似 LRU 实现**：Redis 维护一个“候选池”（默认 5 个采样 key），淘汰时随机采样 N 个 key（可配置 maxmemory_samples，默认 5），选其中最久未访问的淘汰（采样越多越接近精确 LRU）。
### **7. Redis 过期策略 + 内存淘汰策略结合是怎么工作的？定期删除 + 惰性删除 + 定期删除的整个流程**
**金句**：“过期删除是‘定期扫+用时查’（控 CPU），内存淘汰是‘满了就删’（保服务），二者触发时机不同。”
**详细解答**：
**过期键删除策略（定期删除+惰性删除）**：
9. **定期删除**：Redis 每 100ms 随机抽取 20 个过期 key 删除，若过期比例 >25% 则重复（避免 CPU 占用过高）；
10. **惰性删除**：访问 key 时检查是否过期，过期则删除（如`get key`发现过期，立即删并返回空）。
- **内存淘汰策略**：当内存达到 maxmemory 时，触发淘汰（如`allkeys-lru`），删除“最该删”的 key（与过期无关，即使 key 未过期也可能被删）。
- **结合流程**：先通过定期+惰性删除清理过期 key，若内存仍满，则按淘汰策略删未过期 key。

### **8. Redis 的集群模式都有哪些？主从复制、哨兵、Cluster 分别解决什么问题？**
**金句**：“主从复制（数据备份+读写分离）、哨兵（自动故障转移）、Cluster（数据分片扩容），三者构成 Redis 高可用体系。”
**详细解答**：

| **集群模式** | **核心解决问题** |
| --- | --- |
| **主从复制** | 1. 数据备份（从库同步主库数据）；2. 读写分离（主库写、从库读，分担压力）；3. 故障恢复（从库升主）。 |
| **哨兵（Sentinel）** | 主从复制的高可用增强：自动监控主库状态，主库故障时通过 Raft 选举领头哨兵，执行故障转移（选从库升主）。 |
| **Cluster** | 数据分片扩容：将数据分散到多个节点（16384 个哈希槽），解决单机内存/性能瓶颈，支持横向扩展。 |

### **9. Redis Cluster 的槽位是怎么分配的？节点挂了槽位怎么迁移？客户端怎么定位到槽位？**
**金句**：“Cluster 用 16384 个哈希槽分片，节点挂了槽位迁移靠手动/自动，客户端通过 CRC16(key)%16384 定位槽位。”
**详细解答**：
**槽位分配**：预设 16384 个槽（0~16383），集群启动时手动/自动分配（如 3 节点，每节点约 5461 槽）。
**槽位迁移**：节点故障时，通过`redis-cli --cluster reshard`手动迁移槽位；或通过 Cluster 自动感知（从库升主后，接管原主库槽位）。
**客户端定位**：客户端缓存槽位-节点映射表，发送请求时先计算`CRC16(key) % 16384`得到槽位，再路由到对应节点；若槽位不属于该节点，返回`MOVED`重定向。
### **10. 缓存穿透、缓存击穿、缓存雪崩分别是什么？如何解决？**
**金句**：“穿透（查无数据）、击穿（热点过期）、雪崩（大面积过期）——核心区别是‘请求目标是否存在/是否集中’。” @
**详细解答**：

| **问题** | **定义** | **解决方案** |
| --- | --- | --- |
| **缓存穿透** | 查询**数据库不存在**的数据（如 ID=-1），缓存未命中，请求直达 DB。 | 1. 布隆过滤器（预存合法 key，过滤不存在请求）；2. 缓存空值（如`set key null ex 60`）。 |
| **缓存击穿** | **热点 key** 过期瞬间，大量并发请求穿透缓存直达 DB。 | 1. 互斥锁（`setnx lock:key 1 px 5000`，仅 1 个请求查 DB 重建缓存）；2. 逻辑过期（key 不物理过期，存`{value, expire}`，异步更新）。 |
| **缓存雪崩** | **大量 key 同时过期**，请求涌向 DB 导致崩溃。 | 1. 分散过期时间（过期时间+随机值，如 3600±300 秒）；2. 热点 key 永不过期（后台定时更新）；3. 高可用集群（Cluster/哨兵）。 |

### **11. Redis 如何实现分布式锁？Redisson 看门狗机制了解吗？红锁（RedLock）靠谱吗？**
**金句**：“单实例锁用 SET key random_value NX PX 实现，Redisson 看门狗自动续期防超时，红锁（多节点）提可靠但有争议。”
**详细解答**：
**基础实现**：`SET lock:order {random_value} NX PX 30000`（NX：不存在才设，PX：毫秒过期），释放时用 Lua 脚本判断 value 相等后删除（防误删）。
**Redisson 看门狗**：未指定过期时间时，启动后台线程每 10 秒（ttl/3）续期锁（重置为 30 秒），业务结束停止续期（解决“业务超时锁过期”问题）。
**红锁（RedLock）**：向 5 个独立 Redis 主节点发加锁请求，超半数（≥3）成功且总耗时 < ttl 则加锁成功。
**争议**：时钟漂移（节点时钟快导致锁提前过期）、性能开销（多节点通信）、实现复杂（工程中少用，推荐单实例+Redisson）。
### **12. Redis 事务了解吗？MULTI/EXEC/DISCARD，事务有原子性吗？WATCH 怎么实现乐观锁？**
**金句**：“Redis 事务是‘批量执行但不回滚’，原子性仅保证‘全执行或全不执行（语法错）’，WATCH 用 CAS 实现乐观锁。”
**详细解答**：
**命令**：`MULTI`（开启事务）、`EXEC`（执行事务队列）、`DISCARD`（取消事务）、`WATCH key`（监控 key，事务中若 key 被改则失败）。
**原子性**：**不支持回滚**。若命令语法错误（如`SE key v`），事务拒绝执行；若运行时错误（如对 String 执行`HSET`），仅该命令失败，其他继续执行（区别于 MySQL 事务）。
**WATCH 乐观锁**：事务执行前 WATCH 监控 key，若事务执行时 key 被其他客户端修改，则`EXEC`返回 nil（事务失败），需重试。
### **13. Redis 6.0 之后引入的多线程模型具体做了什么？读写分离？还是单线程写？**
**金句**：“6.0 多线程仅用于网络 I/O（读/写 socket），命令执行仍单线程（保无锁安全），并非读写分离。”
**详细解答**：
**多线程分工**：
**I/O 线程**（默认 4 个，可配置）：负责 socket 连接的**读取**（接收客户端命令）和**写入**（发送响应）；
**主线程**：负责**命令解析**、**执行**（单线程）、**结果返回**（将结果交给 I/O 线程发送）。
**核心目的**：解决网络 I/O 瓶颈（高并发下大量连接的数据传输耗时），命令执行仍单线程避免并发安全问题。
### **14. Redis 的内存用满了会怎么样？maxmemory-policy 配置过吗？**
**金句**：“内存满时按 maxmemory-policy 淘汰 key，默认 noeviction（报错），生产需配 allkeys-lru（缓存场景）。”
**详细解答**：
**现象**：若未配置淘汰策略（默认`noeviction`），写入新 key 时返回`OOM command not allowed`错误；若配置了淘汰策略，则按策略删除 key 后写入。
**maxmemory-policy 配置**：缓存场景推荐`allkeys-lru`（删最近最少用 key）；热点数据用`allkeys-lfu`；对数据一致性要求高的场景用`noeviction`（需人工介入）。
### **15. Redis 主从复制的原理？全量复制和增量复制的流程？复制积压缓冲区有什么用？**
**金句**：“主从复制用全量（RDB+buffer）初始化，增量（复制积压缓冲区）同步后续写操作，缓冲区存未同步命令防断连丢数据。”
**详细解答**：
**全量复制（首次同步/主库重启）**：
11. 从库发`PSYNC ? -1`请求同步；
12. 主库 fork 子进程生成 RDB，发送给从库，从库清空旧数据后加载 RDB；
13. 主库将复制缓冲区（repl_backlog_buffer）中 RDB 生成期间的写命令发送给从库，从库执行。

**增量复制（非首次同步）**：
14. 主库将写命令写入复制积压缓冲区（环形数组，默认 1MB）；
15. 从库发`PSYNC {runid} {offset}`请求同步，主库发送缓冲区中 offset 之后的命令。
- **复制积压缓冲区作用**：存储最近写命令，从库断连重连后，通过 offset 拉取缺失命令（避免全量复制）。

### **16. 哨兵（Sentinel）是如何实现高可用的原理？leader 选举、故障转移全流程？**
**金句**：“哨兵通过监控-通知-故障转移三步骤实现高可用，Raft 选举领头哨兵，故障转移选最优从库升主。”
**详细解答**：
**核心流程**：
16. **监控**：哨兵每 10 秒向主从节点发`INFO`，每 2 秒发`PUBLISH sentinel:hello`，每 1 秒发`PING`；
17. **主观下线（SDOWN）**：单个哨兵超过`down-after-milliseconds`（默认 30 秒）未收到节点响应，标记 SDOWN；
18. **客观下线（ODOWN）**：超半数哨兵认为主库 SDOWN，标记 ODOWN；
19. **领头选举**：哨兵通过 Raft 协议选举领头（先到先得，半数同意），避免脑裂；
20. **故障转移**：领头哨兵选最优从库（优先级高、复制进度近、运行 ID 小）升主，其他从库指向新主，通知客户端。

### **17. Redis 的 Key 是怎么寻址的？一致性哈希 vs 带虚拟节点的一致性哈希 vs Redis Cluster 的哈希槽**
**金句**：“一致性哈希（抗节点增减）、虚拟节点（均衡负载）、哈希槽（Redis Cluster 固定 16384 槽，分片更可控）。”
**详细解答**：

| **寻址方式** | **原理** | **优缺点** |
| --- | --- | --- |
| **一致性哈希** | key 和节点映射到 0~2^32-1 圆环，顺时针找最近节点。 | 优点：节点增减仅影响相邻节点数据；缺点：负载不均（热点节点压力大）。 |
| **带虚拟节点的一致性哈希** | 每个物理节点映射多个虚拟节点（如 200 个），均匀分布在圆环上。 | 优点：解决负载不均；缺点：实现复杂，虚拟节点数难配置。 |
| **Redis Cluster 哈希槽** | key 通过`CRC16(key) % 16384`计算槽位，节点负责部分槽（0~16383）。 | 优点：数据均匀分布（固定槽数）、平滑扩容（仅迁移槽位）；缺点：槽位固定（16384 个）。 |

### **18. Redis 的 Pipeline 有什么用？和事务、Lua 脚本的区别？**
**金句**：“Pipeline 减网络 RTT 提性能，事务保批量执行顺序，Lua 脚本保原子性+复杂逻辑，三者互补。”
**详细解答**：

| **维度** | **Pipeline** | **事务（MULTI/EXEC）** | **Lua 脚本** |
| --- | --- | --- | --- |
| **核心目标** | 减少网络往返（RTT），批量操作性能 | 保证命令批量执行（顺序性） | 原子性执行复杂逻辑（多条命令当一条） |
| **原子性** | 不保证（命令可能拆分执行） | 保证执行（不回滚） | 保证原子性（Redis 单线程执行脚本） |
| **错误处理** | 部分命令失败不影响其他命令 | 语法错则全拒，运行错仅单命令失败 | 脚本出错则全回滚（类似事务原子性） |
| **适用场景** | 批量写（如导入 10 万数据） | 简单批量执行（如转账扣减+增加） | 复杂逻辑（如“查+改”原子操作，防并发） |

### **19. Redis 的 AOF 文件太大怎么办？重写触发条件了解吗？**
**金句**：“AOF 重写‘瘦身日志’，触发条件是文件大小达上次重写后 N% 且最小 M MB（auto-aof-rewrite-percentage + min-size）。”
**详细解答**：
**问题**：AOF 文件过大导致恢复慢、占用磁盘。
**重写触发条件**（默认配置）：
`auto-aof-rewrite-percentage 100`：当前 AOF 文件大小是上次重写后大小的 2 倍（100% 增长）；
`auto-aof-rewrite-min-size 64mb`：当前 AOF 文件最小 64MB。
同时满足以上两个条件时，自动触发`BGREWRITEAOF`。
### **21. Redis 的慢查询怎么排查？slowlog 命令怎么用？**
**金句**：“慢查询用 slowlog 定位，配置 slowlog-log-slower-than（阈值微秒）+ slowlog-max-len（最大条数）。”
**详细解答**：
- **配置**：
`slowlog-log-slower-than 10000`：执行时间 >10ms（10000 微秒）的命令记录为慢查询（默认 10ms，设为 0 记录所有）；`slowlog-max-len 128`：慢查询日志最大条数（循环覆盖）。

**slowlog 命令**：
`slowlog get [n]`：获取 n 条慢查询（默认 10 条），包含命令、执行时间、客户端 IP；
`slowlog len`：查看慢查询日志长度；
`slowlog reset`：清空慢查询日志。
### **22. Redis 的内存碎片怎么产生的？怎么解决？**
**金句**：“内存碎片因频繁修改/过期删除导致，用 info memory 查碎片率，解决靠 jemalloc 调优+主动碎片整理。”
**详细解答**：
**产生原因**：
过期 key 删除后，内存未立即归还 OS（Redis 缓存分配器 jemalloc 复用碎片）。
**查看碎片率**：`info memory`中`mem_fragmentation_ratio = used_memory_rss / used_memory`（>1.5 表示碎片严重，<1 表示内存不足）。
**解决方法**：
21. 重启 Redis（彻底释放碎片，适合可接受短暂不可用场景）；
22. 主动碎片整理（`activedefrag yes`，自动整理碎片，需 Redis 4.0+）；
23. 优化数据（避免 Big Key，减少频繁修改）。

### **23. Redis 做异步队列都有哪几种方式？List、Stream、Pub/Sub、Zset 实现延迟队列，各自优劣？**
**金句**：“List 简单但不可靠，Stream 可靠但重，Pub/Sub 无持久化，Zset 延迟队列灵活，按需选。”
**详细解答**：

| **方式** | **实现** | **优点** | **缺点** | **适用场景** |
| --- | --- | --- | --- | --- |
| **List（LPUSH+BRPOP）** | 生产者 LPUSH 消息，消费者 BRPOP 阻塞取。 | 简单易用，轻量。 | 无 ACK，消息易丢；无消费者组。 | 简单临时队列（秒杀缓冲） |
| **Stream** | 生产者 XADD，消费者组 XREADGROUP+XACK。 | 可靠（持久化+ACK）、支持消费者组。 | 较重（需维护消费者组状态）。 | 可靠消息队列（订单通知） |
| **Pub/Sub** | 生产者 PUBLISH，消费者 SUBSCRIBE。 | 实时推送，多消费者广播。 | 无持久化（消费者离线丢消息）。 | 实时通知（聊天室） |
| **Zset 延迟队列** | 消息 score 设为到期时间戳，消费者 ZRANGEBYSCORE 取到期消息。 | 灵活支持延迟，可重试。 | 需手动维护消息状态（防重复消费）。 | 延时任务（订单超时取消） |

### **25. Redis 的 bigkey 怎么产生的？如何发现和解决？**
**金句**：“bigkey 是大 String（>10KB）、多元素 Hash/List（>5000 个），用 redis-cli --bigkeys 发现，拆分成小 key 解决。”
**详细解答**：
**产生原因**：
String：存储大文件（如图片）、未压缩的长文本；
Hash/List/Set/ZSet：元素过多（如 10 万字段的 Hash）、单个元素过大。
**发现方法**：
`redis-cli --bigkeys`：扫描所有 key，统计各类型最大 key；
`debug object key`：查看 key 内存占用（如`serializedlength`）。
**解决方法**：
拆分：如 10 万字段 Hash 拆为 10 个 Hash（每个 1 万字段）；
压缩：大 String 用 gzip 压缩后存储；
淘汰：非核心 bigkey 设过期时间，自动删除。
### **26. Redis 的 RDB 文件结构了解吗？是怎么压缩的？**
**金句**：“RDB 文件含头部（魔数+版本）、数据（键值对）、校验和，用 LZF 算法压缩（默认开启）。”
**详细解答**：
**文件结构**：
24. **头部**：5 字节魔数（`REDIS`）+ 4 字节版本号（如`0006`表示 RDB 6.0）；
25. **数据**：按类型存储键值对（String/List/Hash 等），含过期时间、数据类型、值；
26. **校验和**：8 字节 CRC64 校验和（验证文件完整性）。

**压缩**：RDB 生成时，对值部分用 LZF 算法压缩（默认开启`rdbcompression yes`），减少文件大小（压缩率约 30%-50%）。
### **27. Fork 操作对 Redis 性能的影响？子进程 copy-on-write 机制会卡主线程吗？**
**金句**：“Fork 是‘写时复制’，主线程不阻塞但耗内存，大内存实例 Fork 时卡顿（OS 拷贝页表）。”
**详细解答**：
- **Fork 原理**：主进程调用 fork() 创建子进程，子进程共享主进程内存页表（copy-on-write），仅当主进程修改数据时，才拷贝对应内存页。
- **性能影响**：
    - **时间**：Fork 耗时与内存大小正相关（如 10GB 内存实例 Fork 需 100ms+）；
    - **空间**：子进程共享内存页表，不立即占额外内存，但主进程修改数据时需拷贝页（可能瞬间内存翻倍）。
- **卡顿风险**：Fork 过程中主线程不阻塞，但大内存实例 Fork 时 OS 拷贝页表可能导致短暂卡顿（单线程模型下影响其他命令执行）。

### **30. 如果让你设计一个比 Redis 更快、更省内存的缓存系统，你会怎么设计？**
**金句**：“从内存管理（分层+压缩）、数据结构（定制高效编码）、分布式（一致性哈希+虚拟节点）、持久化（异步日志）四方面优化。”
**设计思路**：
27. **内存管理**：
分层存储：热数据放内存（DRAM），温数据放 PMEM（持久化内存），冷数据放 SSD（通过 mmap 映射）；
高效编码：如用 protobuf 替代 JSON 存对象，用 Roaring Bitmap 替代 HyperLogLog 做基数统计。
28. **数据结构**：定制轻量级结构（如简化的跳表、压缩哈希表），减少元数据开销（如 Redis 字典的 dictEntry 占 24 字节，可优化为 16 字节）。
29. **分布式**：
一致性哈希+虚拟节点（256 个虚拟节点/物理节点），均衡负载；
节点间异步复制（类似 Redis 主从，但用更高效的二进制协议）。
30. **持久化**：异步 AOF（批量攒批写入）+ RDB 快照（低峰期执行），用 LZ4 压缩日志。
31. **I/O 优化**：多线程网络 I/O（类似 Redis 6.0），但命令执行用协程（轻量级上下文切换）替代单线程，提升并发。
